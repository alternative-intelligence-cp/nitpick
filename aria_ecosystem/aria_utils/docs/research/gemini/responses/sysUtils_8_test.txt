Architectural Specification and Implementation Strategy for atest: The Hex-Stream Verification Framework
1. Introduction: The Crisis of Observability in Systems Verification
The validation of modern systems software has reached a critical inflection point where the legacy infrastructure of the Unix epoch no longer suffices for the demands of high-performance, concurrent, and safety-critical engineering. For over fifty years, the dominant paradigm for process interaction has been the three-stream Input/Output (I/O) model: Standard Input (stdin), Standard Output (stdout), and Standard Error (stderr). While elegant in its initial conception for teletype terminals, this model suffers from a fundamental architectural flaw known as "Semantic Conflation."
In a typical testing environment—whether it be pytest for Python, cargo test for Rust, or gtest for C++—the stdout stream is forced to serve multiple, often conflicting, masters. It acts as the User Interface (UI), rendering human-readable progress bars and "✓ Passed" ticks. Simultaneously, it serves as the data transport layer for machine-readable metrics (timing, coverage data). Frequently, it also becomes the dumping ground for application-level debug logs that developers lazily emit to the console. stderr, ostensibly reserved for errors, often captures thread sanitizer warnings, kernel signals, and expected failure outputs, further muddying the waters.
This conflation results in "Fragile Pipeline Syndrome." Continuous Integration (CI) systems must employ brittle regular expressions to scrape stdout for test results, attempting to distinguish between a test named "ErrorHandling" and an actual error message. When a developer inserts a print("DEBUG: reached here") statement, it often corrupts the expected output of a test, causing false negatives in the CI pipeline. Furthermore, the lack of a dedicated high-bandwidth telemetry channel means that comprehensive observability data—stack traces, heap profiles, and performance counters—must be written to sidecar files on the disk, introducing I/O latency and synchronization headaches.
The Aria programming language ecosystem proposes a radical restructuring of this paradigm through the Six-Stream I/O Topology. This report articulates the comprehensive design and implementation strategy for atest (the Aria Test Runner), a utility that serves as the proving ground for this new topology. By leveraging stddbg (FD 3) for telemetry, stddati (FD 4) for payload injection, and stddato (FD 5) for structured results, atest achieves a separation of concerns that is impossible in legacy systems. This document details the kernel-level mechanics, the type-system integration (specifically regarding Twisted Balanced Binary arithmetic), and the memory management strategies required to build atest as the cornerstone of the Aria toolchain.
2. Theoretical Framework: The Hex-Stream Topology
2.1 Deconstructing the Unix Legacy
To understand the necessity of the six-stream model implemented in atest, one must first analyze the limitations of the status quo. The POSIX standard defines three file descriptors (FDs) inherited by child processes:
* FD 0 (stdin): Read-only text input.
* FD 1 (stdout): Write-only output, typically line-buffered when connected to a TTY.
* FD 2 (stderr): Write-only error output, typically unbuffered.
This model assumes that "Output" is a monolithic concept. However, in the context of a test runner, "Output" comprises at least three distinct semantic categories:
1. Presentation Layer: The visual feedback for the human operator (e.g., green text, progress bars).
2. Data Layer: The verifiable outcome of the test (e.g., success/failure structs, code coverage vectors).
3. Observability Layer: The internal state of the system under test (e.g., variable values, allocation traces).
When these three layers compete for FD 1 and FD 2, distinct race conditions and data corruption events occur. For instance, if a multi-threaded test crashes while writing a large log entry to stderr, the crash report (Segmentation Fault) may be interleaved with the log data, rendering both indecipherable.
2.2 The Aria Topological Specification
Aria extends the process control block to support six standard streams. atest utilizes strictly defined semantics for each channel, ensuring that no data stream ever blocks or corrupts another.

	This topology fundamentally alters the architecture of the test runner.1 The atest process (the parent) spawns the test executable (the child) and establishes pipes for all six descriptors.
* The UI (FD 1) is essentially "quiet" during normal operation, emitting only the final status markers.
* The Result Channel (FD 5) carries a high-speed stream of TestResult structs. This allows the runner to aggregate statistics in real-time with zero string parsing overhead.
* The Telemetry Channel (FD 3) absorbs the high-volume debug logs. Even if this pipe fills up and blocks, it does not prevent the test result from being emitted on FD 5 (assuming non-blocking I/O or separate threads).
2.3 Kernel-Level Implementation Mechanics
Implementing this topology requires bypassing the standard library process spawning routines found in C (system, popen) or Python (subprocess). atest must interact directly with the OS kernel to ensure the inheritance of FDs 3, 4, and 5.
2.3.1 Linux Process Instantiation
On Linux 6.8 (the target kernel for AriaX), the implementation utilizes pipe2 and clone/execve.1


C




// Concept Code for Aria Spawn Strategy
int pipes; // 0:stdin, 1:stdout,... 5:stddato
for(int i=0; i<6; i++) pipe2(pipes[i], O_CLOEXEC); 

pid_t pid = fork();
if (pid == 0) {
   // Child Process
   // Remap pipe ends to fixed FDs 0-5
   dup2(pipes, 0); // stdin read
   dup2(pipes, 1); // stdout write
   dup2(pipes, 2); // stderr write
   dup2(pipes, 3); // stddbg write (Telemetry)
   dup2(pipes, 4); // stddati read (Payload In)
   dup2(pipes, 5); // stddato write (Payload Out)
   
   // Close original FDs to prevent leaks
   cleanup_pipes();
   
   // Exec the test binary
   execve(test_binary, args, env);
}

The critical innovation here is the explicit management of FD 3, 4, and 5. Standard shells (bash) often clobber these descriptors for internal use. The AriaX kernel patches specifically modify fs/file.c to reserve these descriptors, preventing the OS from assigning them to random file opens (like database connections), which would cause collisions.
2.3.2 Windows Compatibility Layer
On Windows, the concept of numbered file descriptors is an abstraction over HANDLE objects. The Win32 API CreateProcess accepts a STARTUPINFO structure which natively supports only hStdInput, hStdOutput, and hStdError. To support 6 streams, atest implements a Bootstrap Protocol.
1. The parent creates 6 Anonymous Pipes.
2. The handles are marked as inheritable (bInheritHandle = TRUE).
3. The parent serializes the integer values of the handles for streams 3, 4, and 5 into an environment variable block: __ARIA_FD_MAP=3:<handle_val>;4:<handle_val>;5:<handle_val>.
4. The Aria Runtime (aria_runtime.dll), upon initialization in the child process, parses this variable and reconstructs the io.stddbg, io.stddati, and io.stddato objects from these raw handles.
This cross-platform consistency ensures that atest behaves deterministically regardless of the underlying OS, a rarity in systems tooling.2
3. The atest Architectural Design
The atest utility is not a monolithic program but a distributed system of two components: the Runner (Supervisor) and the Harness (Agent).
3.1 Component 1: The Test Harness (std.test)
The harness is a static library (libaria_test.a) linked into every test executable. It acts as the "Agent" running inside the child process. It is responsible for:
* Test Registration: Using the Aria preprocessor to discover functions marked with @test during compilation.
* Isolation: Ensuring that a crash in one test case does not corrupt the runner.
* Reporting: Serializing test outcomes into the stddato binary format.
* TBB Integration: Handling exceptions related to Twisted Balanced Binary arithmetic (e.g., trit overflow).
The Harness is "headless"; it contains no logic for formatting text to the console. It simply emits data.
3.2 Component 2: The Test Runner (atest CLI)
The Runner is the "Supervisor." It is responsible for:
* Discovery: Scanning the filesystem using the globbing logic defined in AriaBuild to find test binaries.
* Orchestration: Spawning parallel test processes (Worker Pool).
* Aggregation: Reading stddato streams from multiple workers and aggregating the results (Total, Passed, Failed).
* Visualization: Rendering the TUI to stdout.
3.3 The stddato Binary Protocol
The communication between Harness and Runner over FD 5 (stddato) uses a strict binary protocol to ensure zero-overhead parsing. Text parsing is slow and error-prone; binary structs are instant.

3.4 The Event Loop (Reactor Pattern)
To manage multiple 6-stream processes simultaneously, atest employs a Reactor pattern using epoll (Linux) or IOCP (Windows).
* State 1 (Polling): The Runner calls epoll_wait().
* State 2 (Event Dispatch):
   * If data arrives on a FD 5 (stddato) pipe, the Runner reads the ATP packet. If it's a PASS packet, it increments the success counter and updates the progress bar on the Runner's stdout.
   * If data arrives on a FD 3 (stddbg) pipe, the Runner drains the buffer into a circular log buffer in memory. It does not print this to the console immediately.
* State 3 (Failure Handling): If a test fails, the Runner retrieves the buffered logs from the circular buffer (associated with that specific PID) and dumps them to the Runner's stdout, providing immediate context for the failure.
This architecture solves the "noisy neighbor" problem. In a parallel test run, the debug logs of passing tests are silently discarded (or archived), while only the logs of failing tests are presented to the user.
4. Integration with TBB and Aria Type System
Aria's type system features Twisted Balanced Binary (TBB) arithmetic, utilizing trit (base-3) and nit (base-9) types. These types introduce unique testing challenges that atest must address directly.
4.1 Testing Balanced Ternary Logic
Standard assertions (ASSERT_EQ) rely on boolean logic. However, trit types operate on three values: True (1), False (-1), and Unknown (0). Furthermore, TBB arithmetic uses a "Sticky Error" sentinel (e.g., -128 for tbb8) which propagates through calculations.
atest introduces specialized assertions in the std.test harness:
* ASSERT_TRIT_EQ(a, b): Verifies that two trits are identical.
* ASSERT_VALID(t): Verifies that a TBB value is not the ERR sentinel.
* ASSERT_STICKY(expr): Verifies that an expression correctly propagates the ERR sentinel.
Implementation Insight: The ASSERT macros use the Aria Preprocessor to inspect the types at compile time. If a TBB type is passed to a generic ASSERT_EQ, the preprocessor injects code to check for the ERR sentinel first.

5. Memory Model Integration: Wild vs. GC
Aria employs a hybrid memory model involving both a Garbage Collector (GC) and "Wild" (manual) allocation.1 Testing this requires distinct strategies.
5.1 The Arena Isolation Strategy
Traditional allocators (malloc/free) are global. In atest, the harness initializes a new Arena Allocator 1 for each test case.
* Setup: aria_pool_create() is called before the test function.
* Execution: All wild allocations inside the test are routed to this arena.
* Teardown: aria_pool_reset() is called after the test.
This strategy provides O(1) cleanup. Instead of freeing thousands of objects individually, the harness simply resets the arena pointer. This makes atest performance competitive with C++ unit tests, even when testing complex object graphs.
5.2 Leak Detection via Shadow Stacks
For wild pointers that escape the arena, atest leverages the Shadow Stack mechanism 1 utilized by the compiler for GC root tracking. The harness instruments aria_alloc and aria_free. If the counter of active wild allocations is non-zero at the end of a test scope, atest flags a Leak Error.
* Detection: wild_alloc_count > 0 at test exit.
* Reporting: The harness traverses the Shadow Stack to identify the instruction pointers of the allocation sites and writes the stack trace to stddbg.
* Protocol: The flags field in the stddato packet has a specific bit (0x02) for "Wild Leak," allowing the CI system to fail the build even if the test logic "passed."

6.1 The Runner's Main Loop
This code resides in src/tools/atest/main.cpp. It demonstrates the integration of the globbing system (borrowed from AriaBuild ) and the 6-stream spawning logic.


C++




#include "runtime/process.h"
#include "runtime/streams.h"
#include "tools/glob.h"

int main(int argc, char** argv) {
   // 1. Argument Parsing
   TestConfig config = parse_args(argc, argv);

   // 2. Discovery (Using AriaBuild Globbing)
   std::vector<std::string> test_files = glob::expand(config.pattern);
   
   // 3. Thread Pool Initialization
   ThreadPool pool(config.jobs);
   
   // 4. Execution Loop
   for (const auto& test_bin : test_files) {
       pool.submit([&] {
           // Aria Spawn with 6-Stream Mapping
           AriaProcessOptions opts;
           opts.flags = ARIA_PROCESS_DETACHED | ARIA_PROCESS_6_STREAM;
           
           // Spawn returns handles for all 6 streams
           AriaProcess* proc = aria_spawn(test_bin.c_str(), NULL, &opts);
           
           // Register FD 5 (stddato) and FD 3 (stddbg) with Epoll
           reactor.register(proc->stddato, HANDLE_RESULT);
           reactor.register(proc->stddbg, HANDLE_LOG);
           
           // Wait for completion
           int exit_code = aria_process_wait(proc);
           process_results(exit_code);
       });
   }
   
   return 0;
}

6.2 The Harness Entry Point
This code resides in stdlib/test/harness.cpp (part of libaria_test). It implements the logic to execute registered tests and emit binary packets.


C++




void run_tests() {
   // Walk the linker section for registered tests
   for (TestEntry* t = __start_test_registry; t < __stop_test_registry; ++t) {
       
       // 1. Initialize Arena for this test
       aria_pool_t* pool = aria_pool_create(4096); // 4KB block
       
       // 2. Emit START packet to stddato (FD 5)
       ResultPacket pkt = {.type = TEST_START };
       write(5, &pkt, sizeof(pkt));
       
       // 3. Run Test
       try {
           uint64_t start = aria_clock_ns();
           t->func(); // Execute the test function
           uint64_t duration = aria_clock_ns() - start;
           
           // Emit PASS packet
           pkt.type = TEST_PASS;
           pkt.duration = duration;
           write(5, &pkt, sizeof(pkt));
           
       } catch (AriaException& e) {
           // Emit FAIL packet
           pkt.type = TEST_FAIL;
           write(5, &pkt, sizeof(pkt));
           
           // Write detailed trace to stddbg (FD 3)
           dprintf(3, "Exception in %s: %s\n", t->name, e.what());
       }
       
       // 4. Reset Arena (Zero-cost cleanup)
       aria_pool_destroy(pool);
   }
}

7. Ecosystem Integration: atest, als, and acurl
The user's original request identified atest, als (List), and acurl (Transfer) as the "Hook" utilities for the Aria ecosystem. While atest is the focus of this implementation, it shares a "Common Core" with als and acurl that validates the 6-stream model across different domains.
7.1 Shared DNA: The Aria Utility Common Core
All three tools share a fundamental architectural dependency: Structured Separation of Data and UI.
* atest: Separates Test Results (Binary) from Progress Bars (Text).
* als: Separates File Metadata (Binary structs) from File Names (Text).
   * Implementation: als writes struct FileInfo { u64 size; u64 mtime; u8 type; } to stddato.
   * Benefit: A downstream tool like agrep can filter files based on binary size integers read from stddato without parsing ls -l text output, eliminating the "parsing ls is dangerous" trope common in Unix shell scripting.
* acurl: Separates Download Payload (Binary) from Progress/Speed UI (Text).
   * Implementation: acurl writes the raw downloaded bytes to stddato (FD 5) while writing the ASCII progress bar to stdout (FD 1).
   * Benefit: Users can pipe the download directly to tar (acurl... | tar -x) while still seeing the progress bar on their terminal. In bash, piping curl usually silences the progress bar because stdout is redirected. Aria's 6-stream model solves this by keeping UI on FD 1 and Data on FD 5.
7.2 The atest "Hook"
atest serves as the "Hook" because it demonstrates this capability most visibly to developers. The ability to run atest | acurl (streaming test results to a remote server) while seeing a clean UI locally proves the utility of the extra file descriptors immediately.
8. Continuous Integration (CI) and Future Proofing
8.1 CI Adapters
To support the requirement of "consuming stddato for statistics," atest includes a transpiler mode.


Bash




atest --adapter=junit > results.xml

In this mode, the Runner attaches to the stddato stream of the test harness but, instead of rendering a TUI, it converts the binary ATP packets into standard JUnit XML format on stdout. This allows Aria projects to integrate seamlessly with Jenkins, GitLab CI, and GitHub Actions without requiring those platforms to support the 6-stream model natively.
8.2 Distributed Testing
The binary nature of stddato makes atest trivially extensible to distributed testing. A "Satellite" runner can execute tests on a remote ARM64 machine, streaming the stddato packets over a TCP socket back to the "Master" runner. The Master runner aggregates these packets just as if they were local processes. The compactness of the binary protocol (28 bytes + name length per test) ensures minimal network overhead compared to streaming textual logs.
9. Conclusion
The implementation of atest described herein provides a definitive solution to the fragility of modern test automation. By rigorously applying the Aria Six-Stream Topology, atest achieves what legacy tools cannot: the complete decoupling of user experience from machine observability.
* Reliability: The CI system listens only to stddato (FD 5), receiving typed binary data that is immune to log pollution.
* Observability: The developer has access to a dedicated stddbg (FD 3) channel that can carry gigabytes of debug data without breaking the UI.
* Performance: The integration of the Arena allocator and TBB-aware assertions ensures that the testing harness is as performant as the systems code it validates.
This utility, alongside als and acurl, forms the triad of "Hook" applications that validate the Aria ecosystem's architectural thesis: that the 50-year-old Unix stream model is insufficient for modern computing, and that semantic stream separation is the path forward.

#pragma once
#include <cstdint>

namespace aria::test {

// Packet types for stddato stream
enum class PacketType : uint8_t {
   START = 0x01,
   PASS  = 0x02,
   FAIL  = 0x03,
   SKIP  = 0x04,
   LOG   = 0x05  // Structured log reference
};

// 32-byte Fixed Header (C++ code in appendix - using C++ syntax)
struct PacketHeader {
   uint32_t magic;       // 0x41524941 "ARIA"
   PacketType type;      // 1 byte
   uint8_t flags;        // 1 byte (TBB_ERR, MEM_LEAK)
   uint16_t reserved;    // Padding
   uint64_t duration_ns; // Timing
   uint64_t alloc_count; // Memory metrics
   uint32_t name_len;    // Length of following name string
   uint32_t payload_len; // Length of extra payload
};

} // namespace aria::test
