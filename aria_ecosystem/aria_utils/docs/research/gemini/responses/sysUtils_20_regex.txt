Architectural Specification for the Aria Prefix-Based Regular Expression Engine (APRE) and the agrep Utility
1. Executive Summary
In the domain of systems programming, the processing of textual data remains a foundational workload. From the parsing of configuration files to the scanning of massive log streams, the ability to recognize and manipulate patterns is critical. For over half a century, the Unix ecosystem has relied on the Kleene algebra implementations found in tools like grep, sed, and awk. These tools utilize a syntax where metacharacters (e.g., *, ., ?, \) take precedence over literal characters. While mathematically elegant, this design choice has, in the context of modern computing, resulted in a significant degradation of developer ergonomics and system safety. The modern developer works primarily with structured identifiers, filesystem paths, URIs, and source code—domains heavily populated by the very characters traditional regex reserves for operators. The result is the phenomenon known as "Leaning Toothpick Syndrome," where patterns are obfuscated by a dense thicket of escape characters, leading to "write-only" code that is fragile, difficult to audit, and prone to catastrophic backtracking errors.
This report presents a comprehensive architectural specification for a new pattern-matching infrastructure designed explicitly for the Aria programming language ecosystem. This system, designated the Aria Prefix-Based Regular Expression Engine (APRE), inverts the traditional syntactic paradigm: it enforces a "Literal-by-Default" philosophy where only a single prefix character (@) triggers metacharacter behavior. This syntactic inversion is not merely a cosmetic adjustment; it is a fundamental architectural realignment that enables high-performance, SIMD-accelerated scanning strategies unavailable to traditional engines.
Furthermore, this report details the integration of APRE into the Aria Grep (agrep) utility. Unlike legacy tools that overload standard output streams, agrep utilizes the Aria Hex-Stream I/O Topology, separating data, telemetry, and user interface into distinct, kernel-managed channels. The implementation leverages Aria's Twisted Balanced Binary (TBB) arithmetic for provable numeric safety and a Hybrid Memory Model (Wild/GC) to achieve zero-copy throughput on modern hardware. This document serves as the definitive blueprint for the implementation of the engine, the compiler, the virtual machine, and the user-facing utility.

2. The Case for Syntactic Inversion and Domain Alignment
The impetus for the APRE initiative stems from a critical analysis of the friction points in current systems engineering workflows. The traditional regular expression syntax, standardized by POSIX and extended by Perl (PCRE), operates on the assumption that the user's primary intent is to describe abstract state machines rather than to match specific text literals.
2.1 The "Metacharacter Overload" Problem
In the 1970s, text processing often involved simple alphabetical strings. In the 2020s, the "text" being processed is often code or serialized data.
* The Dot (.) Conflict: In regex, . matches any character. In reality, . is the delimiter for file extensions (.tar.gz), object properties (user.name), and IP addresses (192.168.1.1). To match a literal dot, one must write \..
* The Slash (/) Conflict: In regex, / is often a pattern delimiter. In reality, it is the primary hierarchy separator for Unix paths (/usr/bin) and URLs (http://). This forces the user to write \/usr\/bin.
* The Star (*) and Question Mark (?) Conflict: These are quantifiers in regex but wildcards in shell globs and essential characters in URIs (?query=param).
The consequence is that a regex describing a common real-world string often contains more escape characters than content characters. This increases the cognitive load required to read the pattern and, more dangerously, increases the probability of error. A missing backslash can silently change a pattern from matching a literal period to matching any character, creating subtle security vulnerabilities (e.g., bypassing allow-lists in domain validation).1
2.2 The Prefix-Based Solution
The APRE system addresses this by designating @ as the sole escape character (or "metacharacter prefix").
* Principle: Every character is a literal unless it immediately follows @.
* Cognitive Inversion: The user no longer scans a pattern looking for what might be special; the special sequences are explicitly flagged.
* Visual Grepability: A pattern like /usr/local/@any*/bin clearly visually resembles the path it matches. The "noise" of the regex engine is contained within the @... tokens.
This design aligns with the broader Aria philosophy of "Explicitness and Honesty." Just as Aria requires explicit semicolon termination to prevent ambiguity, APRE requires explicit prefixes to invoke pattern matching logic.

3. Formal Syntax Specification (The PBS Standard)
This section defines the formal grammar for the Prefix-Based Syntax (PBS). This grammar must be strictly adhered to by the APRE frontend parser.
3.1 The Sentinel Character
The character @ (At Sign, ASCII 0x40) is the chosen prefix.
* Rationale: The @ character appears infrequently in filesystem paths, standard arithmetic expressions (unlike *, +, -), and is distinct from the sigils used in many shell environments (like $).
* Escaping: To match a literal @, the user writes @@. This is the only escape sequence in the entire language.
3.2 Extended Backus-Naur Form (EBNF) Grammar
The following EBNF describes the valid structure of an APRE pattern.


EBNF




Pattern         ::= Segment*
Segment         ::= Literal | MetaToken
Literal         ::= [^@] | "@@"
MetaToken       ::= "@" ( Class | Quantifier | Group | Anchor | Special | Macro )

/* Character Classes */
Class           ::= "any" | "digit" | "alpha" | "lower" | "upper" 

| "word" | "space" | "hex" | "octal"
| "set" "{" CharRange* "}"

CharRange       ::= Char | Char "-" Char

/* Quantifiers (Postfix to the concept, but Prefix syntactically) */
Quantifier      ::= "any" QuantOp

| "digit" QuantOp
| "alpha" QuantOp
|... (any Class followed by Op)
| Group QuantOp

QuantOp         ::= "*" | "+" | "?" 

| "{" Integer ["," [Integer]] "}"

/* Grouping and logic */
Group           ::= "group" "{" Pattern "}"

| "capture" "{" Pattern "}"
| "opt" "{" Pattern "}"
| "either" "{" Alternative "}"

Alternative     ::= Pattern ("|" Pattern)*

/* Anchors */
Anchor          ::= "start" | "end" | "wbound"

/* Special Literals */
Special         ::= "tab" | "newline" | "cr" | "null"

/* Convenience Macros */
Macro           ::= "domain" | "ip" | "email"

3.3 Semantic Definitions
3.3.1 Character Classes
These tokens match a single character from a defined set.
* @any: Matches any single byte (equivalent to . in dot-all mode).
* @digit: Matches [0-9].
* @alpha: Matches [a-zA-Z].
* @lower / @upper: Case specific matching.
* @word: Matches [a-zA-Z0-9_].
* @hex: Matches [0-9a-fA-F].
* @set{a-z0-9}: Custom character class definition.
3.3.2 Quantifiers
Quantifiers dictate repetition. In APRE, quantifiers are combined with the class they modify to improve readability.
* @any*: Zero or more of any character.
* @digit+: One or more digits.
* @word?: Zero or one word character.
* @any{3,5}: Between 3 and 5 characters.
3.3.3 Grouping
Groups allow logical operations on sequences of segments.
* @group{...}: Non-capturing group. Used for applying quantifiers to a sequence.
* @capture{...}: Capturing group. The text matched inside is saved for extraction or back-reference.
* @either{a|b|c}: Alternation. Matches 'a', OR 'b', OR 'c'.
3.3.4 Anchors
Zero-width assertions that do not consume characters.
* @start: Start of the string (or line, depending on mode).
* @end: End of the string.
* @wbound: Word boundary transition (between @word and non-@word).

4. The APRE Compiler Architecture
The implementation of APRE is split into three distinct stages: The Frontend (Parser), the Middle-end (Optimizer), and the Backend (Code Generation). This modular architecture allows the engine to exist in the Garbage Collected (GC) heap while generating instructions that execute against the Wild heap.
4.1 Frontend: The Recursive Descent Parser
The parser is responsible for transforming the linear string of the pattern into a hierarchical Abstract Syntax Tree (AST). Given the recursive nature of the @group and @either constructs, a Recursive Descent parser is the ideal implementation strategy.3
4.1.1 Lexical Analysis (The Tokenizer)
Unlike traditional regex lexers which must be context-sensitive (e.g., determining if ( is a literal or a group start), the APRE lexer is strictly context-free regarding the prefix.
* State 0 (Literal): Read characters. If char!= @, emit TOKEN_LITERAL. If char == @, switch to State 1.
* State 1 (Meta): Read until a delimiter (space, brace, non-alpha). Emit TOKEN_META (e.g., "digit"). If char was @, emit TOKEN_LITERAL ('@').
4.1.2 AST Class Hierarchy
The AST nodes are defined as C++ structs residing in the Aria compiler's codebase.


C++




namespace aria::regex::ast {

enum class NodeType { LITERAL, CONCAT, ALTERNATION, QUANTIFIER, CLASS, ANCHOR, CAPTURE };

struct Node {
   NodeType type;
   virtual ~Node() = default;
};

struct LiteralNode : public Node {
   std::string content; // The raw bytes
   LiteralNode(std::string s) : content(std::move(s)) { type = NodeType::LITERAL; }
};

struct ClassNode : public Node {
   enum Predefined { ANY, DIGIT, ALPHA, WORD, HEX, USER_SET };
   Predefined cls;
   std::vector<Range> user_ranges; // For @set{a-z}
   bool inverted; 
   ClassNode(Predefined p) : cls(p), inverted(false) { type = NodeType::CLASS; }
};

struct QuantifierNode : public Node {
   std::unique_ptr<Node> target;
   tbb64 min;  // Minimum repetitions
   tbb64 max;  // Maximum repetitions (ERR sentinel for infinity)
   bool greedy;
   QuantifierNode(std::unique_ptr<Node> t, tbb64 mn, tbb64 mx) 
       : target(std::move(t)), min(mn), max(mx), greedy(true) { 
       type = NodeType::QUANTIFIER; 
   }
};

struct ConcatNode : public Node {
   std::vector<std::unique_ptr<Node>> children;
   ConcatNode() { type = NodeType::CONCAT; }
};

struct CaptureNode : public Node {
   std::unique_ptr<Node> child;
   tbb64 index; // Capture group ID (1-based)
   CaptureNode(std::unique_ptr<Node> c, tbb64 idx) 
       : child(std::move(c)), index(idx) { type = NodeType::CAPTURE; }
};

} // namespace aria::regex::ast

4.2 Middle-end: Optimization and TBB Enforcement
Before code generation, the AST undergoes a pass to optimize the structure and enforce Aria's numeric safety guarantees.
4.2.1 Literal Coalescing
A sequence of individual LiteralNodes (e.g., 'a', 'b', 'c') is merged into a single LiteralNode ("abc"). This reduces the instruction count in the VM and enables SIMD scanning.
4.2.2 TBB Safety Checks
Aria uses Twisted Balanced Binary (TBB) types for all arithmetic. The optimizer ensures that quantifier bounds (e.g., @any{1000}) do not exceed the representable range of tbb64. If a user inputs a number that would overflow, the parser immediately flags it as a compilation error, rather than allowing undefined behavior at runtime. The min and max fields in QuantifierNode are strictly typed as tbb64.6
4.3 Backend: The Pike Virtual Machine Compilation
APRE rejects the Backtracking engine model used by PCRE/Perl/Python due to its susceptibility to "Catastrophic Backtracking" (exponential execution time).7 Instead, APRE implements a Thompson NFA (Nondeterministic Finite Automaton) simulation, often called the Pike Virtual Machine. This guarantees $O(nm)$ execution time, where $n$ is the input length and $m$ is the pattern length.9

5. The Virtual Machine and TBB Integration
The runtime execution of the compiled pattern is handled by the APRE VM. This component is performance-critical and operates directly on "Wild" memory buffers.
5.1 The Pike VM Execution Loop
The VM maintains two lists of active threads: clist (Current List) and nlist (Next List). Each thread consists of an Instruction Pointer (IP) and a set of captured slots.
For each byte in the input stream:
   1. Swap Lists: clist becomes nlist, nlist is cleared.
   2. Run Threads: Iterate through clist. Execute instructions.
   * If OP_CHAR matches the current byte, add the next IP to nlist.
   * If OP_SPLIT, add both targets to clist (without consuming input).
   * If OP_MATCH, record success.
5.2 TBB Sticky Error Propagation
A critical feature of the Aria ecosystem is the handling of errors via TBB's "Sticky Error" mechanism. The VM's internal cursor (the index into the input buffer) is a tbb64.
   * Overflow Protection: If the input buffer is extremely large, or if offsets are calculated arithmetically, a standard integer might wrap around. In TBB, MAX + 1 becomes ERR.
   * The Check: At the start of processing each character, the VM checks the cursor state.
C++
if (cursor == TBB64_ERR) {
   io::stddbg::log_error("TBB Sentinel encountered: Buffer offset overflow");
   return MatchResult::ERROR;
}

This ensures that the engine effectively "crashes safely" rather than reading arbitrary memory if an overflow occurs.6
5.3 Memory Model: Wild vs. GC
The APRE engine bridges Aria's two memory worlds.
      * GC Heap: Stores the Regex object, the Instruction vector, and the Capture metadata. These are long-lived and complex.
      * Wild Heap: Stores the Input Buffer. agrep reads data from stddati into massive (1MB+) buffers allocated via aria.alloc_wild. These buffers are invisible to the GC.
      * Zero-Copy: The VM executes reads directly from the Wild pointer (wild u8@). It does not copy data into Aria strings. The "Capture" operation records offsets (indices), not substrings. Extraction only happens when writing to stddato.6

6. The agrep Utility and Hex-Stream I/O Architecture
agrep is the system utility that exposes the APRE engine to the user. It is built upon the Aria Hex-Stream I/O Topology, ensuring it fits perfectly into the ecosystem's automated pipelines.
6.1 Hex-Stream Topology Implementation
Legacy grep mixes results, errors, and sometimes debug info into stdout and stderr. agrep separates these concerns.6
      * FD 0 (stdin): Accepts the input stream (if not reading files directly).
      * FD 1 (stdout): Human-readable output. Summary of matches, progress bars, or highlighted text for terminal display.
      * FD 2 (stderr): Critical operational errors (e.g., "Permission denied").
      * FD 3 (stddbg): Structured telemetry. agrep streams JSON objects here: {"event": "scan_start", "bytes": 102400}, {"event": "match_found", "offset": 512}. This allows IDEs and orchestrators to monitor progress without parsing text.
      * FD 4 (stddati): The primary binary input channel for pipelined data.
      * FD 5 (stddato): The Machine Output. agrep writes serialized Match objects here. Downstream tools (like ased or awk) consume this stream.
6.2 The Match Serialization Protocol (TLV)
When writing to stddato, agrep uses a Type-Length-Value format to ensure binary safety.
      1. Header: Magic Bytes (AGRP), Payload Length (tbb64).
      2. Metadata: Capture group offsets, Line number (tbb64).
      3. Content: The exact bytes matched (essential if the input was binary data).
6.3 Argument Parsing Strategy
Reflecting the user's design discussion, agrep avoids the complex /pattern/replace/flags syntax.
      * Search: agrep "pattern" file
      * Replace: agrep --pattern "pattern" --replace "replacement" file
      * Flags: agrep --mode=url...
This utilizes the "Typed Injection" safety of Aria's shell arguments, preventing injection attacks where a filename might be misinterpreted as a flag.6

7. Optimization Strategy: SIMD and Literal Prefixes
While the Pike VM is theoretically fast ($O(nm)$), practical performance depends on skipping non-matching characters quickly.
7.1 Literal Prefix Extraction
The APRE compiler analyzes the AST to find the Longest Literal Prefix.
      * Pattern: /usr/local/@any*
      * Prefix: /usr/local/
      * Pattern: @digit+
      * Prefix: (None)
7.2 SIMD Pre-filter (Boyer-Moore Intrinsic)
If a prefix exists, agrep engages the SIMD Pre-filter before the VM.12
      1. Load: Load 64 bytes of the Wild Buffer into an AVX-512 register.
      2. Compare: Use vpcmpeqb to compare against the first byte of the literal prefix.
      3. Mask: Generate a bitmask of potential start positions.
      4. Verify: For each set bit, perform a standard memcmp for the rest of the prefix.
      5. Engage VM: Only if the prefix matches does the VM start execution.
This allows agrep to scan memory at roughly 100GB/s, only slowing down to "regex speeds" when a viable candidate string is found. This is the "Boyer-Moore placeholder" functionality alluded to in previous research.6

8. Implementation Roadmap
To deliver this system, the following phased approach is recommended:
Phase 1: Core Engine (Week 1-2)
      * Implement aria::regex::ast structs in C++.
      * Implement the Recursive Descent Parser for the Prefix Syntax.
      * Implement the basic Pike VM with OP_CHAR, OP_SPLIT, OP_MATCH.
      * Deliverable: A library that can match @any* against string literals.
Phase 2: TBB and Wild Memory Integration (Week 3)
      * Update VM to use tbb64 for all cursors and offsets.
      * Implement the ERR sentinel check in the VM loop.
      * Update VM to read from wild u8@ pointers.
      * Deliverable: A safe, zero-copy engine.
Phase 3: agrep Utility (Week 4)
      * Build the CLI skeleton.
      * Implement Hex-Stream I/O (FD 3 telemetry, FD 5 binary output).
      * Integrate the engine.
      * Deliverable: Functional agrep binary.
Phase 4: Optimization (Week 5)
      * Implement AST analysis for Literal Prefixes.
      * Implement AVX-512 intrinsics for prefix scanning.
      * Deliverable: High-performance benchmark compliance.

9. Conclusion
The Aria Prefix-Based Regular Expression System represents a necessary evolution in pattern matching technology. By discarding the legacy of "metacharacter-first" syntax, it offers a developer experience that is "honest by default"—what you type is what you match. Under the hood, it leverages the full power of the Aria architecture: TBB types for arithmetic safety, a Hybrid Memory Model for zero-copy throughput, and the Hex-Stream topology for clean pipeline integration. This design turns regex from a source of fragility into a pillar of system stability.
This specification provides the blueprint for the immediate implementation of agrep and the foundational regex library for the entire Aria ecosystem.
