Architectural Specification and Implementation of adf: A Native Hex-Stream Disk Usage Utility for the Aria Ecosystem
1. Executive Summary and Strategic Context
The administration of modern computing infrastructure sits at a precarious intersection between human legibility and machine parsability. For over four decades, the Unix philosophy—"Write programs that do one thing and do it well," and "Write programs to handle text streams, because that is a universal interface"—has guided the development of system utilities. However, as the complexity of automated orchestration systems has grown, the limitations of text as a universal interface have become acutely apparent. Tools like df (disk free) exemplify this crisis: they must simultaneously serve the human administrator, who requires "human-readable" formats (e.g., "52G"), and the monitoring script, which requires precise integers. The result is a fragile compromise where output formats are unstable, parsing heuristics are brittle, and critical diagnostic information is frequently lost in the noise of standard error streams.
This report presents a comprehensive design and implementation analysis of adf (Aria Disk Free), a reference utility for the Aria operating system ecosystem. adf is not merely a reimplementation of the classic df command; it is a foundational proof-of-concept for the Aria Hex-Stream Input/Output Model. By leveraging Aria's kernel-level extensions to the standard file descriptor table, adf strictly segregates its operational outputs into three orthogonal planes: a Presentation Plane (FD 1, stdout) for rich user interaction; a Data Plane (FD 5, stddato) for high-performance, typed binary serialization; and an Observability Plane (FD 3, stddbg) for structured diagnostic telemetry.
This document serves as the authoritative implementation guide for adf, detailing every aspect of its construction from the Foreign Function Interface (FFI) bindings for Linux kernel system calls to the Twisted Balanced Binary (TBB) arithmetic used for error-safe capacity calculations. It explores the theoretical underpinnings of the Six-Stream topology, the memory safety implications of Aria's "Wild" pointers during FFI traversal, and the design patterns required to build "Split-Brain" utilities that serve humans and machines with equal fidelity.
2. The Theoretical Crisis of Legacy System Utilities
To appreciate the architectural innovations of adf, one must first rigorously deconstruct the systemic failures of the legacy 3-stream model (stdin, stdout, stderr) in contemporary systems engineering.
2.1 The "Noisy Channel" Problem
The fundamental flaw in the traditional Unix pipeline is the overloading of stdout. A standard utility is expected to emit its primary functional output to File Descriptor 1. However, "functional output" is ambiguous. Does it refer to the data the program processed, or the visual representation of that data for the user? In the case of df, this conflation is total. The table printed to the console is both the UI and the API.
This conflation introduces the "Noisy Channel" problem. If a script is designed to parse disk usage via df -h | awk '{print $5}', it is relying on a specific visual layout. If the utility introduces a new column, changes alignment, or decides to print a warning message to stdout instead of stderr, the downstream parser breaks. This fragility necessitates the creation of complex, defensive regex-based parsers that consume significant CPU cycles merely to reconstruct structured data from unstructured text.
Furthermore, the lack of a dedicated debug channel forces developers to misuse stderr. In ideal Unix theory, stderr is for errors. In practice, it is a dumping ground for logs, progress bars, and debug traces. In modern CI/CD environments (like Jenkins or GitHub Actions), any output to stderr is often heuristically classified as a build failure. This forces tool authors to silence warnings or route diagnostics to stdout, further corrupting the data stream.1
2.2 The Fallacy of Text-Based Arithmetic
Consider the output "52G". To a human, this is intuitive. To a machine, it is an ambiguous string requiring normalization. Is "G" 10^9 bytes (Gigabytes) or 2^30 bytes (Gibibytes)? Does the utility round up or down? A monitoring agent triggering an alert at 90% usage must parse "52G", guess the base, convert it to an integer, and then perform the comparison. This process effectively serializes an integer into vague text, only for the next process to deserialize it back into an integer, wasting computational resources and introducing precision errors at every step.1
2.3 The Aria Six-Stream Resolution
The Aria ecosystem resolves these structural deficiencies by expanding the standard file descriptor table from three to six streams. This is not a user-space convention but a kernel-level contract enforced by the AriaX Linux distribution modifications.1

3. Kernel Interfaces and Low-Level bindings
The implementation of adf begins at the lowest layer of the software stack: the interface with the Linux kernel. Unlike higher-level scripting languages that provide abstract filesystem modules, Aria provides direct access to system calls via its Foreign Function Interface (FFI). This requires precise definitions of C data structures and rigorous management of memory boundaries.
3.1 The Discovery Interface: getmntent and /proc/mounts
To report disk usage, adf must first discover what filesystems are mounted. On Linux, the authoritative source of this information is the procfs pseudo-filesystem, specifically /proc/self/mountinfo or the legacy compatibility layer /proc/mounts (often symlinked to /etc/mtab).
The C standard library provides the getmntent family of functions to parse these files. However, getmntent returns a pointer to a static buffer, making it inherently thread-unsafe. While getmntent_r (the reentrant version) exists, adf is designed as a single-threaded utility for simplicity in this reference implementation.
The mntent structure, defined in <mntent.h>, contains pointers to null-terminated C strings.


C




struct mntent {
   char *mnt_fsname; /* name of mounted filesystem */
   char *mnt_dir;    /* filesystem path prefix */
   char *mnt_type;   /* mount type (see mntent.h) */
   char *mnt_opts;   /* mount options (see mntent.h) */
   int mnt_freq;     /* dump frequency in days */
   int mnt_passno;   /* pass number on parallel fsck */
};

In Aria, we must replicate this layout exactly to prevent memory corruption when the C function writes to the struct. We define this in the sys.fs module using the extern struct syntax. Crucially, we map C char* to Aria's wild byte*. The wild keyword is part of Aria's "Appendage Theory" memory model, signifying memory that is outside the control of the Garbage Collector (GC). This distinction is vital; trying to pass a GC-managed reference to a C function without pinning would be unsafe, but here we are receiving pointers from C, which must be treated as wild.1
3.2 The Capacity Interface: statvfs
Once a mount point is identified (e.g., /var), adf needs to query its statistics. The POSIX standard defines the statvfs system call for this purpose. It is preferred over the older, Linux-specific statfs because it provides a more portable struct definition, although field ordering can still vary by architecture.4
The layout of struct statvfs on Linux x86_64 is critical. Incorrectly mapping these fields would lead to reading garbage values—interpreting the block count as the fragment size, for example.


C




struct statvfs {
   unsigned long  f_bsize;    /* Filesystem block size */
   unsigned long  f_frsize;   /* Fragment size */
   fsblkcnt_t     f_blocks;   /* Size of fs in f_frsize units */
   fsblkcnt_t     f_bfree;    /* Number of free blocks */
   fsblkcnt_t     f_bavail;   /* Number of free blocks for unprivileged users */
   fsfilcnt_t     f_files;    /* Number of inodes */
   fsfilcnt_t     f_ffree;    /* Number of free inodes */
   fsfilcnt_t     f_favail;   /* Number of free inodes for unprivileged users */
   unsigned long  f_fsid;     /* Filesystem ID */
   unsigned long  f_flag;     /* Mount flags */
   unsigned long  f_namemax;  /* Maximum filename length */
};

In 64-bit Linux, unsigned long is 8 bytes (64 bits), and fsblkcnt_t / fsfilcnt_t are also 64 bits. Thus, the Aria definition can uniformly use uint64. Note the distinction between f_bsize (preferred I/O size) and f_frsize (fundamental block size). df calculations must use f_frsize * f_blocks to calculate total capacity. Using f_bsize is a common error that leads to incorrect reporting on filesystems where these values differ.4
3.3 Memory Safety Strategy: The Wild/GC Bridge
The interaction between getmntent and Aria's string system highlights a key architectural challenge. getmntent returns pointers to strings residing in a static buffer inside libc. If adf simply stored these wild byte* pointers, they would be overwritten by the next call to getmntent (as the iterator advances).
To solve this, adf must perform an explicit "promotion" of these strings. We use string.from_c_str(), a standard library function that performs a memcpy from the wild source to a new, immutable string object allocated on the Aria GC heap. This ensures that the MountPoint objects stored in our array have stable, independent lifecycles, decoupled from the low-level iteration state.1
4. TBB Arithmetic and Error Propagation
A unique feature of the Aria language is its use of Twisted Balanced Binary (TBB) arithmetic. Unlike standard two's complement integers which wrap around on overflow (often leading to silent bugs in disk usage calculations), TBB types utilize the minimum representable value as an explicit ERR sentinel (e.g., -128 for tbb8, huge negative for tbb64).
4.1 Sticky Error Propagation in adf
When adf calculates disk usage, it performs the operation: used = total - free.
In a standard language, if free were erroneously reported as larger than total (due to a filesystem corruption or race condition), used would underflow to a massive positive integer. adf would then report 18 Exabytes of used space, confusing the user.
With TBB types (tbb64), if free > total, the result is logically invalid but numerically handled. More importantly, if the statvfs call fails for a specific mount (e.g., an NFS stale handle), the wrapper can return ERR for the fields. Any subsequent calculation involving ERR (e.g., percentage = used / total) immediately results in ERR. This is "Sticky Error Propagation."
The adf UI layer detects this. When formatting the size for display, it checks if (value == ERR). If true, it prints a red "ERR" string instead of a number. This provides immediate visual feedback that a specific mount point is unhealthy, without crashing the entire utility or stopping the scan of other healthy drives.1
5. The Data Plane: stddato Binary Protocol
The defining feature of adf is its ability to pipe data to downstream tools. To do this efficiently, we eschew text parsing for a binary protocol.
5.1 Protocol Design Constraints
1. Compactness: The protocol should minimize bytes on the wire to allow high-frequency polling.
2. Zero-Copy Compatibility: The structure should align with memory pages to allow mechanisms like splice() to move data between pipes without user-space copying.
3. Versioning: The protocol must be versioned to allow future expansion (e.g., adding inode counts).
5.2 The V1 Wire Format
The stddato stream emits a sequence of variable-length packets. Each packet corresponds to one filesystem. We use a Type-Length-Value (TLV) inspired approach but optimized for fixed fields.1

5.3 Serialization Implementation
Aria's io.stddato stream is typed to accept wild buffers. To serialize a packet, adf could allocate a small buffer, pack the integers, and write. However, for maximum efficiency, we write fields sequentially. The io.stddato implementation is block-buffered (typically 64KB), so these small writes are coalesced in user space before a single write() syscall flushes them to the kernel pipe. This balances code simplicity with I/O throughput.1
6. The Observability Plane: stddbg Telemetry
While stdout shows the user what is happening, stddbg explains how it is happening. This is critical for debugging system state. For instance, if df hangs, it is often due to a blocked NFS mount. A user running df sees nothing until it times out. A user running adf can tail the stddbg stream (FD 3) and see exactly which mount point is currently being probed.
6.1 Structured Logging Strategy
adf emits logs in JSON format to stddbg. This allows log aggregators (like Splunk, Datadog, or a local jq pipeline) to index the events.
Example Telemetry Event:


JSON




{
 "ts": 1735001234567,
 "level": "DEBUG",
 "event": "mount_probe",
 "device": "/dev/sda1",
 "path": "/",
 "duration_ns": 4500
}

If a probe takes longer than a threshold (e.g., 100ms), adf emits a warning event. This provides immediate insight into "slow I/O" issues that are notoriously difficult to diagnose with legacy tools.1
6.2 Stream State Detection
A sophisticated feature of adf is its ability to detect if stddbg is connected to /dev/null. During startup, the runtime performs an fstat on FD 3. If the device ID corresponds to the null device, adf disables the generation of telemetry strings entirely. This "Observation Stability" optimization ensures that the overhead of formatting JSON strings is only incurred when someone is actually listening, saving CPU cycles in tight loops.1
7. User Interface Engineering: stdout
The user interface is the only part of adf that most humans will ever see. It must be concise, informative, and visually parsable.
7.1 ANSI Capability Detection
Before emitting color codes, adf checks if stdout is a TTY (Teletype). If the user is redirecting stdout to a file (adf > output.txt), color codes would corrupt the text. Aria's io.stdout.is_tty() method wraps the isatty() syscall to provide this boolean check. If false, adf falls back to a plain text mode similar to df -h.
7.2 The Visual Grammar
We employ a columnar layout with specific color coding rules to enhance cognition:
* Bar Chart: A visual progress bar [||||||....] provides an instant visual cue of capacity.
* Color Thresholds:
   * Green: < 75% usage.
   * Yellow: 75% - 90% usage.
   * Red: > 90% usage.
* Unit Normalization: We use IEC units (KiB, MiB, GiB) which are powers of 2 (1024), aligning with the kernel's binary block accounting. This avoids the "vendor gigabyte" (1000-based) confusion.6
8. Implementation: The adf Source Code
The following modules constitute the complete source code for adf. The project is structured into three files: sys/fs.aria (FFI), adf/core.aria (Logic), and adf/ui.aria (Presentation).
8.1 Module: sys.fs (Kernel Bindings)


Code snippet




// FILE: src/sys/fs.aria
// Low-level bindings to Linux filesystem APIs.
// Requires linking against libc.

module sys.fs;

use std.mem; // For wild pointers and unsafe blocks

// -----------------------------------------------------------------------------
// FFI Declarations
// -----------------------------------------------------------------------------

// Opaque handle for the FILE* stream returned by setmntent
extern struct:FILE = opaque;

// The mntent struct matches <mntent.h> layout.
// Strings are wild byte* because they point to static libc memory.
extern struct:mntent = {
   mnt_fsname: wild byte*, // Device/Server
   mnt_dir:    wild byte*, // Mount path
   mnt_type:   wild byte*, // FS Type (ext4, tmpfs, etc)
   mnt_opts:   wild byte*, // Options string
   mnt_freq:   int32,      // Dump freq
   mnt_passno: int32       // Fsck pass
};

// The statvfs struct matches <sys/statvfs.h> on x86_64.
// Using uint64 for all unsigned long fields to ensure 64-bit alignment.
extern struct:statvfs_t = {
   f_bsize:    uint64,     // Filesystem block size
   f_frsize:   uint64,     // Fragment size (Use this for calculations!)
   f_blocks:   uint64,     // Total blocks
   f_bfree:    uint64,     // Free blocks
   f_bavail:   uint64,     // Free blocks for unprivileged users
   f_files:    uint64,     // Total inodes
   f_ffree:    uint64,     // Free inodes
   f_favail:   uint64,     // Free inodes for unprivileged users
   f_fsid:     uint64,     // Filesystem ID
   f_flag:     uint64,     // Mount flags
   f_namemax:  uint64      // Max filename length
};

extern "C" {
   // Open the mount table. Usually "/proc/mounts" and "r".
   func:setmntent = wild FILE*(wild byte*:filename, wild byte*:type);
   
   // Iterate to next entry. Returns NULL on EOF.
   func:getmntent = wild mntent*(wild FILE*:stream);
   
   // Close the handle.
   func:endmntent = int32(wild FILE*:stream);
   
   // Query filesystem stats. Returns 0 on success, -1 on error.
   func:statvfs = int32(wild byte*:path, wild statvfs_t*:buf);
}

// -----------------------------------------------------------------------------
// High-Level Wrappers
// -----------------------------------------------------------------------------

// A safe, GC-managed representation of a mount point.
pub struct:MountPoint = {
   device: string,
   path:   string,
   fstype: string,
   options: string
};

// Usage statistics utilizing TBB types for safety.
pub struct:UsageStats = {
   total_bytes: tbb64,
   used_bytes:  tbb64,
   avail_bytes: tbb64,
   usage_pct:   flt32
};

// Generator that yields MountPoints
pub func:get_mounts = array<MountPoint>() {
   unsafe {
       // String literals are converted to C-strings
       wild byte*:proc_path = "/proc/mounts".to_c_str();
       wild byte*:mode = "r".to_c_str();
       
       wild FILE*:fp = setmntent(proc_path, mode);
       if (fp == null) {
           fail("Could not open /proc/mounts");
       }
       defer endmntent(fp); // RAII cleanup
       
       array<MountPoint>:list =;
       
       while (true) {
           wild mntent*:entry = getmntent(fp);
           if (entry == null) break;
           
           // Promotion: Copy wild C-strings to safe Aria Strings
           list.push(MountPoint {
               device: string.from_c_str(entry->mnt_fsname),
               path:   string.from_c_str(entry->mnt_dir),
               fstype: string.from_c_str(entry->mnt_type),
               options: string.from_c_str(entry->mnt_opts)
           });
       }
       
       pass(list);
   }
};

// Query statistics for a path. Returns TBB types.
pub func:get_stats = result<UsageStats>(string:path) {
   unsafe {
       stack statvfs_t:buf; // Allocate struct on stack for speed
       wild byte*:c_path = path.to_c_str();
       
       int32:res = statvfs(c_path, &buf);
       if (res!= 0) {
           return { err: "statvfs failed", val: null };
       }
       
       // Convert to TBB64. If values are impossibly large, TBB logic 
       // could be extended here to validate, but we trust the kernel mostly.
       tbb64:blk_size = buf.f_frsize; 
       tbb64:total = buf.f_blocks * blk_size;
       tbb64:free  = buf.f_bfree * blk_size;  // Total free
       tbb64:avail = buf.f_bavail * blk_size; // User free
       
       // Sticky Error logic: if total is 0 or ERR, this propagates.
       tbb64:used = total - free;
       
       // Calculate percentage for UI
       flt32:pct = 0.0;
       if (total > 0) {
           // TBB to Float conversion
           pct = (used.to_flt32() / total.to_flt32()) * 100.0;
       }
       
       pass(UsageStats {
           total_bytes: total,
           used_bytes: used,
           avail_bytes: avail,
           usage_pct: pct
       });
   }
};

8.2 Module: adf.core (Business Logic)


Code snippet




// FILE: src/adf/core.aria
module adf.core;

use sys.fs;
use std.io; // Access to stddato, stddbg

// Filesystems to ignore to reduce noise
const array<string>:PSEUDO_FS = [
   "sysfs", "proc", "devtmpfs", "devpts", "tmpfs", "cgroup", 
   "pstore", "bpf", "configfs", "debugfs", "tracefs", "autofs",
   "mqueue", "hugetlbfs", "fusectl", "rpc_pipefs", "overlay"
];

pub func:is_interesting = bool(sys.fs.MountPoint:mnt) {
   // 1. Filter by Type
   for (string:ignore in PSEUDO_FS) {
       if (mnt.fstype == ignore) return false;
   }
   
   // 2. Filter by Path (Snap packages clutter /snap)
   if (mnt.path.starts_with("/snap")) return false;
   if (mnt.path.starts_with("/var/lib/docker")) return false;
   
   return true;
};

// Serialize to stddato (FD 5) using TLV format
pub func:emit_data_packet = void(sys.fs.MountPoint:mnt, sys.fs.UsageStats:stats) {
   // Header: Magic (4) + PathLen (2)
   io.stddato.write_uint32(0xADDA7001); 
   io.stddato.write_uint16(mnt.path.length.to_uint16());
   
   // Body: Path String
   io.stddato.write_bytes(mnt.path.to_bytes());
   
   // Body: Metrics (Little Endian)
   // TBB types handled: if ERR, write MAX_UINT64 as sentinel
   if (stats.total_bytes == ERR) {
       io.stddato.write_uint64(0xFFFFFFFFFFFFFFFF);
       io.stddato.write_uint64(0);
       io.stddato.write_uint64(0);
   } else {
       io.stddato.write_uint64(stats.total_bytes.to_uint64());
       io.stddato.write_uint64(stats.used_bytes.to_uint64());
       io.stddato.write_uint64(stats.avail_bytes.to_uint64());
   }
};

// Telemetry to stddbg (FD 3)
pub func:log_telemetry = void(string:event, string:msg) {
   if (io.stddbg.is_null_device()) return; // Optimization
   
   int64:now = std.time.now_ns();
   // Use template literal for JSON construction
   string:json = `{ "ts": &{now}, "evt": "&{event}", "msg": "&{msg}" }\n`;
   io.stddbg.write(json);
};

## 10. Conclusion

The `adf` utility serves as a potent validation of the Aria ecosystem's core philosophy. By acknowledging that system tools serve two distinct masters—the human administrator and the automation script—and providing distinct architectural channels for each, Aria eliminates the compromises inherent in legacy Unix design.

Through the use of the Hex-Stream topology, `adf` achieves high-fidelity visualization without breaking script compatibility. Through the use of TBB arithmetic, it achieves arithmetic safety without performance penalties. And through the use of FFI with "Wild" pointers, it achieves systems-level performance with selective memory safety. As the first native utility of the AriaX distribution, `adf` sets the standard for the next generation of "Split-Brain" system tools.
