Architectural Specification and Implementation Strategy for ajq: The Aria JSON/Data Processor
1. Executive Introduction: The Post-Serialization Era of Systems Programming
The history of systems programming and the Unix philosophy has been dominated by a single, powerful abstraction: the text stream. The paradigm of "Everything is a File," established in the early 1970s, predicated that the universal interface between discrete processes should be unstructured sequences of bytes, typically interpreted as ASCII text.1 While this universality fueled the composability of the Unix shell—allowing tools like grep, sed, and awk to interoperate without shared schemas—it has fundamentally constrained the performance and safety of modern data-intensive pipelines. In the contemporary cloud-native landscape, where observability agents, high-frequency trading engines, and machine learning data loaders operate on gigabytes of structured telemetry per second, the text stream has become a liability.
This bottleneck is defined as the "Serialization Loop." In a traditional pipeline (e.g., curl | jq | grep), data undergoes a redundant cycle of transformation: internal binary memory representation → serialization to JSON text → transmission via pipe → parsing from JSON text → internal binary memory representation. This cycle consumes vast amounts of CPU instruction cache and memory bandwidth, generating heat and latency without adding computational value. Furthermore, the reliance on stdout (File Descriptor 1) for both data payload and user interaction creates a "noisy channel" where diagnostic messages or progress bars corrupt the data stream, leading to fragile "screen scraping" automation.1
The Aria programming language ecosystem introduces a structural remedy to these systemic inefficiencies: the Hex-Stream Input/Output Topology. By expanding the standard process interface from three streams (stdin, stdout, stderr) to six, specifically adding stddbg (FD 3), stddati (FD 4), and stddato (FD 5), Aria formally decouples the Control Plane (User Interface) from the Data Plane (Machine Interface) and the Observability Plane (Telemetry).1
This report presents a comprehensive architectural specification and implementation strategy for ajq (Aria JSON Query), the flagship data processing utility for the Aria ecosystem. Designed to supersede jq, ajq is not merely a filter for text; it is a JIT-compiled, zero-copy binary transducer. It operates directly on the stddati binary stream, leveraging Aria's Twisted Balanced Binary (TBB) arithmetic for sticky error propagation 1, Wild Memory arenas for allocation-free processing 1, and the Async Runtime for non-blocking I/O saturation.1 This document serves as the definitive blueprint for implementing ajq, detailing every subsystem from the kernel-level file descriptor management to the assembly generation of the JIT engine.
2. The Hex-Stream Topology and Data Plane Architecture
The viability of ajq as a high-performance successor to jq relies entirely on its integration with the underlying Hex-Stream model. Unlike legacy tools that must guess whether their input is data or commands, ajq operates within a strict topological contract enforced by the aria-shell and the modified Linux kernel hosting the Aria ecosystem.1
2.1 The Six-Stream Semantic Contract
The implementation of ajq strictly adheres to the semantic separation of concerns mandated by the Aria specifications.1 This separation allows ajq to process data at wire speed while simultaneously providing rich interactive feedback to the user and detailed telemetry to monitoring systems—capabilities that are mutually exclusive in the traditional 3-stream model.

	2.2 The Typed Pipeline Concept
The connection between acurl (Aria Curl) and ajq represents a Typed Pipeline.1 In a standard shell pipeline acurl... | ajq..., the shell creates a pipe. However, instead of connecting stdout to stdin, the aria-shell connects acurl's stddato (FD 5) to ajq's stddati (FD 4).
This architectural decision has profound implications for performance. acurl is aware that it is writing to a Data Plane. Instead of serializing the HTTP response into a generic JSON string, it promotes the data into a binary intermediate representation (IR), such as Universal Binary JSON (UBJSON) or Binary JData (BJData).1
ajq receives this binary stream. Because formats like UBJSON are length-prefixed and typed, ajq can perform skip-scanning. If the user query is .users, ajq reads the first array element and then uses the length of the array to calculate the byte offset of the end of the array, effectively seeking past gigabytes of irrelevant data without parsing a single byte of it. This operation is $O(1)$ relative to the skipped data size, whereas textual JSON parsing is $O(N)$ as every byte must be scanned for delimiters like { or }.
2.3 Integration with the Aria Async Runtime
The I/O model for stddati and stddato is built upon the Aria Async Runtime 1, which wraps the Linux io_uring interface. The ajq implementation does not use blocking read() or write() syscalls. Instead, it submits I/O requests to a submission queue (SQ) and yields execution to the scheduler.
The runtime manages the completion queue (CQ), waking the ajq processing task only when a buffer is filled. This allows ajq to achieve IO-CPU Overlap: while the CPU is executing the JIT-compiled filter logic on "Buffer A," the kernel (via DMA) is filling "Buffer B" from the network card or disk. This double-buffering strategy is essential to saturating modern NVMe storage and 100Gbps network links, aligning with the "High Performance" mandate of the Aria ecosystem.1
2.4 The Observability Channel (stddbg)
A critical requirement of the ajq specification is the use of stddbg for "Schema validation warnings, type coercion logs" [User Query]. In legacy tools, if a field is missing or has the wrong type, the tool must either abort (breaking the pipeline) or fail silently (hiding the bug). ajq uses stddbg (FD 3) to introduce a third option: Gradual Failure.
If ajq encounters a type mismatch (e.g., filtering .age > 21 on a string value), it emits a structured log event to stddbg.

3. The Aria Memory Model and Wild Pointers
To achieve the performance targets set by the "Zero serialization overhead" requirement, ajq cannot rely on the standard Garbage Collector (GC). GC allocations introduce non-deterministic pauses and memory overhead for object headers. ajq adopts the Hybrid Memory Model, utilizing "Wild" memory for the hot path of data processing.1
3.1 Arena Allocation Strategy
ajq implements a Region-Based Memory Management scheme, specifically using Arena Allocators (referenced in alloc_001_arena_allocator_patterns.txt 1). An Arena is a large, contiguous block of memory allocated directly from the OS (via mmap).
The ajq implementation maintains two primary arenas:
1. Ingress Arena: A rolling ring buffer where raw bytes from stddati are deposited.
2. Transducer Arena: A working memory area for constructing the output result.
When processing a "frame" (a single JSON object or record), ajq creates pointers that reference addresses within the Ingress Arena. It does not copy string data. A string in ajq's internal representation is simply a struct { length: uint64, data: wild byte* } where data points into the Ingress Arena.
Once the frame is processed and written to stddato, the arenas are reset, not freed. The allocation pointer is simply moved back to the start. This reduces the per-object allocation cost to a single pointer increment (approx. 1-2 CPU cycles), compared to the hundreds of cycles required for malloc/free.
3.2 Safety via the Borrow Checker
Using raw pointers ("wild" memory) usually invites buffer overflows and use-after-free errors. However, Aria's Borrow Checker 1 allows ajq to wrap these unsafe operations in safe abstractions.
The compiler enforces that no wild byte* referencing the Ingress Arena can outlive the current processing loop iteration. If the developer attempts to store a pointer to the input data in a global variable or pass it to a thread that outlives the arena reset, the Aria compiler will reject the code. This provides C++-level performance with Rust-style memory safety guarantees.
3.3 Zero-Copy Splicing
For the optimization of "pass-through" queries (e.g., ajq '.' or queries that select large sub-trees), ajq utilizes the Linux splice() and vmsplice() system calls. Because stddati and stddato are file descriptors, the kernel can move pages of memory from the read buffer to the write buffer without ever mapping them into ajq's userspace memory. This is the theoretical limit of I/O performance: the CPU effectively touches zero bytes of the payload, acting solely as a traffic controller for the DMA engine.
4. Twisted Balanced Binary (TBB) Arithmetic Integration
Aria's type system is unique in its support for Twisted Balanced Binary (TBB) types.1 These integer types (e.g., tbb64) utilize the entire bit range of a standard two's complement integer but reserve specific bit patterns as "sentinels" for error states (ERR).
4.1 Sticky Error Propagation
In data processing, handling missing or malformed data is the most common source of logic bugs. Standard integer arithmetic wraps on overflow or throws exceptions. TBB arithmetic is Sticky.
If a variable x holds the ERR sentinel value:
* x + 1 evaluates to ERR.
* x * 5 evaluates to ERR.
* x > 10 evaluates to ERR (or False with a side-effect flag, depending on the comparison mode).
ajq leverages this for robust query execution. If a user queries .users.age + 5, and one user object is missing the age field, the extraction logic returns ERR. The subsequent addition (+ 5) automatically propagates this ERR without requiring explicit if (x == null) checks in the JIT-compiled code.
4.2 TBB-Aware Serialization
When ajq writes the result to stddato, the serializer checks for TBB sentinels.
* Case 1: Valid Value. The integer is written as a standard binary integer (Network Byte Order).
* Case 2: ERR Sentinel. The serializer writes a specialized "Error Type" marker to the output stream.
This ensures that the error state is preserved across process boundaries. A downstream instance of aps reading this stream will see the value as "Error/Missing" rather than a confusing zero or negative number.

5.2 The Core Event Loop
The main function orchestrates the async event loop. It initializes the streams, compiles the filter, and enters the processing loop. Note the use of await for non-blocking I/O and unsafe blocks for memory manipulation.


Code snippet




async func:main = int8(string:args) {
   // 1. Argument Parsing & Stream Setup
   Config:config = parse_args(args);
   
   // Hex-Stream Initialization: Map explicit FDs
   // In Aria, global streams are singletons initialized by runtime
   BinaryStream:input = io.stddati;
   BinaryStream:output = io.stddato;
   TextStream:debug = io.stddbg;
   
   // 2. Filter Compilation (JIT)
   // Compile the query string into a native function pointer
   func<wild byte*, wild byte*>:filter_fn = compile_query(config.filter_expr);
   
   // 3. Buffer Allocation (Wild Memory Arena)
   // 64KB chunks for optimal L1/L2 cache usage
   const CHUNK_SIZE: int64 = 65536;
   wild byte*: buffer = mem.alloc_wild(CHUNK_SIZE);
   defer mem.free_wild(buffer); // Scope-based cleanup
   
   // 4. Processing Loop
   loop {
       // Async read from FD 4 (stddati)
       // Returns result<int64> containing bytes read or error
       var read_res: result<int64> = await input.read_async(buffer, CHUNK_SIZE);
       
       // Sticky error handling via TBB-like Result checking
       if (read_res.is_err()) {
           io.stderr.write("Fatal I/O Error: ");
           io.stderr.write(read_res.err.message);
           fail(1);
       }
       
       var bytes_read: int64 = read_res.val;
       if (bytes_read == 0) { break; } // EOF handling
       
       // 5. Execution (The Hot Path)
       unsafe {
           // Invoke JIT code. The function returns a pointer to the result
           // inside the output arena (managed internally by the JIT context)
           var result_ptr: wild byte* = filter_fn(buffer);
           var result_len: int64 = get_result_length(result_ptr);
           
           // Check for TBB ERR sentinel in the result pointer itself
           if (is_sentinel(result_ptr)) {
               debug.write("[ajq] Filter produced ERR result");
               // Write error marker to output stream
               write_error_marker(output);
           } else {
               // Write valid result to FD 5 (stddato)
               await output.write_async(result_ptr, result_len);
           }
       }
   }
   
   pass(0);
}

5.3 The JIT Compiler Implementation
The heart of ajq is the Just-In-Time compiler that converts the filter string (e.g., .age > 21) into x86-64 machine code. This eliminates the interpreter loop overhead found in jq. This section utilizes the Assembler class documented in aria_source_full.txt.1

5.3.2 The Code Generator (CodeGen)
The compile_query function iterates over the AST and emits machine code using the Assembler API.


Code snippet




func:compile_query = func<wild byte*, wild byte*>(string:query) {
   Assembler:asm = asm_new();
   QueryNode:ast = parse_query(query);
   
   // Standard Prologue: Setup stack frame
   // input pointer is in RDI (System V ABI)
   asm_prologue(asm); 
   
   while (ast!= null) {
       pick (ast.op) {
           (OpCode.LOAD_KEY) {
               // Emit code to find key in UBJSON object
               // 1. Check if RDI points to Object marker
               // 2. Loop through keys comparing hashes
               // 3. If found, move value pointer to RAX
               // 4. If not found, move TBB_ERR to RAX
               emit_key_lookup(asm, ast.arg);
           }
           (OpCode.COMPARE_GT) {
               // Emit comparison logic
               // 1. Compare RAX (value) with ast.val (immediate)
               // 2. Set Zero Flag / Sign Flag
               emit_compare_gt(asm, ast.val);
           }
           //... handle other ops
       }
       ast = ast.next;
   }
   
   // Epilogue: Restore stack and return
   asm_epilogue(asm);
   
   // Finalize: Mark page executable (RX)
   wild void@: code_ptr = asm_finalize(asm);
   
   // Cast to function type
   return (func<wild byte*, wild byte*>)code_ptr;
}

5.3.3 Optimizing Key Lookups
A naïve implementation of .key would perform string comparisons loop-by-loop. ajq optimizes this by Perfect Hashing. During the JIT phase, ajq computes the hash of the target key (e.g., "age"). The generated assembly computes the hash of keys in the input stream on the fly (using CRC32 or SIMD instructions) and compares integer hashes rather than strings. This reduces key lookup from $O(L)$ (length of string) to $O(1)$.
6. Binary Protocol Specification
ajq must support multiple input formats. The implementation mandates a format-agnostic internal representation based on Views.
6.1 Format Detection (Magic Bytes)
The stddati reader inspects the first 4 bytes of the stream:

In a chained pipeline acurl | ajq | aps:
   * Legacy: acurl downloads body -> Serializes JSON -> Pipe -> jq Parses JSON -> Process -> Serialize -> Pipe -> aps Parses.
   * Latency is additive. Serialization/Deserialization (SerDes) dominates.
   * Aria: acurl writes Binary -> Pipe -> ajq Scans Binary -> Process -> Writes Binary -> Pipe -> aps Reads Binary.
   * SerDes is eliminated. Latency is dominated by network I/O and memory bandwidth.
8. Integration within the Aria Ecosystem
8.1 Shell Integration (aria_shell)
The aria_shell plays a crucial role. When it detects an ajq command, it automatically handles stream mappings.
Command: ajq '.status'
   * Shell Action:
   * Is input coming from a pipe? Yes. -> Map pipe read-end to FD 4.
   * Is output going to a TTY? Yes. -> Map stddato (FD 5) to a temporary buffer.
   * Auto-Formatting: The shell detects that ajq finished. It reads the binary result from stddato, detects it's a TTY destination, and invokes a fast binary-to-text formatter to display the result on stdout (FD 1).

9. Conclusion
The specification for ajq represents a fundamental rethinking of the Unix filter. By discarding the legacy of text-based communication and embracing the Hex-Stream Topology, Twisted Balanced Binary types, and JIT compilation, ajq resolves the serialization bottleneck.
The implementation strategy detailed herein—focusing on zero-copy arenas, sticky error propagation via TBB sentinels, and async I/O saturation—provides a clear path to building a utility that is orders of magnitude faster and safer than existing solutions. As a test case for the Aria ecosystem, ajq demonstrates that the language's novel features (Wild Memory, TBB, 6-Stream I/O) are not merely academic curiosities but essential primitives for the next generation of high-performance systems infrastructure.
