Architectural Specification and Implementation Strategy for the Aria Configuration Language (ACL)
1. Introduction: The Strategic Imperative of a Bespoke Configuration Architecture
The modern software development landscape is currently fragmented by a dichotomy between rigorous systems programming languages and the fragile, loosely typed configuration formats that orchestrate them. In the specific context of the Aria programming language project, this friction is not merely an annoyance but a structural impediment to the language's core philosophy of deterministic control and safety. As delineated in the foundational project overview 1 and the AriaBuild specification 1, the reliance on legacy tooling—specifically GNU Make and its whitespace-sensitive syntax—constitutes a "significant friction point" that violates the Principle of Least Astonishment.
This report presents an exhaustive architectural specification for the Aria Configuration Language (ACL). It is not sufficient to merely define a file format; we must engineer a parsing infrastructure that is congruent with Aria's hybrid memory model 1, its Twisted Balanced Binary (TBB) arithmetic 1, and its "Configuration as Data" philosophy.1 The ACL is designed to be a strict superset of JSON, incorporating the ergonomic benefits of YAML (comments, unquoted keys) without importing its non-deterministic whitespace handling.
The necessity for this custom solution arises from the specific requirements of the Aria ecosystem. Standard JSON (RFC 8259) lacks comments, rendering it unsuitable for complex configuration files where documentation is essential.1 YAML and Python-based configuration (like SCons or Waf) rely on indentation, which introduces "invisible" syntax errors and creates friction when copying code between files with different whitespace settings.1 TOML, while deterministic, becomes unwieldy for deeply nested structures common in build dependency graphs.
Therefore, the ACL is architected to address these specific deficiencies through three primary design pillars:
1. Explicit Delimitation: Scope and structure are defined by visible characters ({, }, [, ]), making the parser robust against formatting variations.
2. Syntactic Homogeneity: The configuration syntax mirrors the Aria language's own object literal syntax, reducing cognitive load for developers.1
3. Deep System Integration: The parser is not a standalone library but an extension of the libaria_frontend, utilizing the same AriaLexer and DiagnosticEngine to provide compiler-grade error reporting and type safety.1
This document serves as the definitive implementation guide for the engineering team, detailing the theoretical underpinnings, formal grammar, compiler integration strategy, and robust error recovery mechanisms required to bring the ACL to production readiness.

2. Deconstructing the Legacy: Analysis of Existing Configuration Paradigms
To understand the architectural decisions behind ACL, we must first analyze the failures of existing formats in the context of systems programming. The research data suggests a pervasive dissatisfaction with "Configuration as Code" and whitespace-sensitive formats.1
2.1 The Failure of Significant Whitespace
The primary antagonist in the current build infrastructure is GNU Make. The AriaBuild research 1 explicitly calls out the "tab vs. space" distinction as a source of "invisible syntax errors." In a systems language environment, where correctness is paramount, invisible characters should never alter semantic meaning.
Python (and by extension, configuration DSLs based on it like Starlark) and YAML also rely on indentation to define scope. While visually clean, this introduces fragility:
* Refactoring Hazard: Moving a block of configuration from one level of nesting to another requires precise re-indentation of every line. Failure to do so results in logic errors that may not be caught until runtime.
* Copy-Paste Fragility: Pasting snippets from web resources or other files often corrupts indentation, leading to parse errors that are difficult to diagnose.
* Tooling Complexity: IDEs and editors must infer the indentation rules (2 spaces? 4 spaces? Tabs?), leading to inconsistent formatting across a team.
ACL rejects this model entirely. As specified in the AriaBuild design 1, ACL is strictly whitespace-insensitive. A file formatted on a single line is semantically identical to one spread across a thousand lines with arbitrary indentation.
2.2 The Limitations of Standard JSON
JSON is the lingua franca of data interchange, but it fails as a configuration language.
* Lack of Comments: Configuration files are code; they require documentation. JSON's ban on comments prevents developers from explaining why a specific flag is set or what a magic number represents.
* Strict Quoting: Requiring quotes around keys ("name": "value") adds significant syntactic noise, reducing readability.1
* Trailing Commas: JSON's disallowance of trailing commas makes generating configuration files programmatically (or editing lists manually) error-prone, as reordering lines can introduce syntax errors.
2.3 The "Configuration as Code" Trap
Tools like Gradle (Groovy/Kotlin), SCons (Python), or specialized DSLs often blur the line between configuration and scripting. While flexible, this leads to non-deterministic builds. A configuration script might check the system time, read random files, or fetch data from the network, causing the build graph to change unpredictably between runs.1
ACL adheres to the "Configuration as Data" philosophy. It is a declarative format. It supports variable substitution and simple logic, but it is not Turing-complete. This restriction ensures that:
* Introspection: External tools (IDEs, Language Servers) can statically analyze the configuration without executing a virtual machine.
* Determinism: The configuration evaluates to the same state every time, guaranteeing reproducible builds—a critical requirement for the Aria toolchain.

3. The Aria Configuration Language (ACL) Specification
The ACL is defined as a formal language with a rigorous grammar. It is designed to be parsed by a recursive descent parser with single-token lookahead (LL(1)), ensuring high performance and predictable error handling.
3.1 Lexical Structure and Tokenization
The ACL leverages the AriaLexer defined in src/frontend/lexer.h.1 This reuse strategy ensures that the configuration language evolves in lockstep with the main language and reduces the maintenance surface of the project.
3.1.1 Unicode and Encoding
The source input is assumed to be UTF-8. The lexer must handle multi-byte characters correctly, particularly in string literals. Aria's compiler infrastructure already includes robust UTF-8 handling for source code 1, which ACL inherits.
3.1.2 Whitespace Handling
The lexer treats the following characters as whitespace:
* Space (U+0020)
* Horizontal Tab (U+0009)
* Line Feed (U+000A)
* Carriage Return (U+000D)
* Form Feed (U+000C)
In the ACL mode, whitespace is strictly a token separator. It is consumed and discarded by the AriaLexer's skipWhitespace() method 1, ensuring that the parser never sees it. This fulfills the "no white space sensitivity" requirement.1
3.1.3 Comments
ACL supports two forms of comments, which are treated as whitespace (ignored by the parser but potentially preserved by a "doc-aware" mode):
1. Single-line: Starts with // and extends to the end of the line.
2. Multi-line: Delimited by /* and */. Unlike C, ACL supports nested block comments. This allows developers to comment out a block of code that already contains block comments—a frequent usability issue in C/C++.
The AriaLexer handles nesting by maintaining a depth counter when it encounters /* inside a comment block.1
3.1.4 Identifiers
Identifiers in ACL follow the Aria language specification.1 They are used for object keys and variable references.
* Regex: [a-zA-Z_][a-zA-Z0-9_]*
* Case Sensitivity: Identifiers are case-sensitive. debug and Debug are distinct keys.
* Unquoted Keys: A key in an object literal does not require quotes if it is a valid identifier. This reduces visual noise. { name: "Project" } is valid; { "name": "Project" } is also valid but less idiomatic.
3.2 Type System and Literals
ACL supports a richer set of data types than JSON, reflecting the underlying capabilities of the Aria runtime.1
3.2.1 Numeric Types and TBB Integration
Aria's numeric system is sophisticated, including standard integers, floats, and Twisted Balanced Binary (TBB) types. ACL must parse these correctly to support the "sticky error" propagation mechanism.1
* Integers:
   * Decimal: 42, -100
   * Hexadecimal: 0x1A, 0xFF (Critical for bitmasks and addresses)
   * Binary: 0b1010 (Useful for hardware configuration)
   * Octal: 0o755 (Useful for file permissions)
* Floats: Standard IEEE-754 formats. 3.14, 1.0e-5.
* TBB Types: The parser must be aware of TBB semantics. If a configuration field is typed as tbb8 (range -127 to +127), and the user provides -128 (the ERR sentinel) or 200 (overflow), the parser must handle this explicitly.
   * Sentinel Literal: We recommend adding an explicit ERR literal to the language to allow users to explicitly configure error states for testing.
   * Overflow Check: When parsing an integer for a specific config schema (e.g., max_retries: tbb8), the parser must validate the range. If 200 is parsed, it should not wrap around; it should trigger a compiler error or set the value to the ERR sentinel depending on the "Sticky Error" policy.1
3.2.2 String Literals and Interpolation
ACL supports two string types:
1. Standard Strings: Double-quoted "...". Support standard C-style escape sequences (\n, \t, \\, \", \uXXXX).
2. Raw/Template Strings: Backtick-delimited `...`. These support multi-line text and, crucially, Variable Interpolation.
Variable Interpolation Syntax:
The syntax &{VARIABLE_NAME} is reserved for interpolation.1 This syntax was chosen over the shell-style $VAR to avoid ambiguity and to align with Aria's string template syntax.
* Mechanism: When the lexer encounters &, it checks for {. If found, it enters a STATE_INTERPOLATION mode 1, parsing the content up to } as an identifier (or a simple expression path).
* Usage:
Code snippet
output_dir: "&{PROJECT_ROOT}/build"

3.2.3 Booleans and Null
   * true, false
   * null (representing the absence of a value, maps to std::monostate or nullptr in C++)
3.3 Formal Grammar (EBNF)
The grammar is defined below using Extended Backus-Naur Form.


EBNF




Configuration   ::= Object | Array ;

Object          ::= "{" [ MemberList ] "}" ;
MemberList      ::= Member { "," Member } [ "," ] ; (* Trailing comma support *)
Member          ::= Key ":" Value ;
Key             ::= Identifier | StringLiteral ;

Array           ::= "[" [ ValueList ] "]" ;
ValueList       ::= Value { "," Value } [ "," ] ;

Value           ::= Literal

| Object
| Array
| VariableRef ;

Literal         ::= StringLiteral

| IntegerLiteral
| FloatLiteral
| BoolLiteral
| NullLiteral
| TemplateString ;

VariableRef     ::= "&{" Identifier "}" ;  (* Standalone variable usage *)
TemplateString  ::= "`" { TemplateChar | VariableRef } "`" ;

This grammar is unambiguous and can be parsed by a simple top-down recursive parser. The reliance on structural delimiters ({, }, [, ]) ensures that whitespace is never needed to disambiguate constructs.

4. Parser Architecture and Compiler Integration
The implementation of the ACL parser is not a standalone utility. To ensure the "sane" and robust nature requested, it is integrated into the Aria compiler frontend (libaria_frontend).
4.1 Architectural Components
The configuration subsystem consists of four key C++ classes located in src/frontend/:
   1. AriaLexer (Existing): Handles tokenization. Reused without modification to ensure lexical consistency.1
   2. ConfigParser (New): Implements the recursive descent logic to build the configuration AST.
   3. ConfigAST (New): A lightweight, polymorphic node hierarchy representing the configuration data (ConfigObject, ConfigArray, ConfigValue).
   4. ConfigBinder (New): A template-based system to map ConfigAST nodes to C++ native structs (schema validation).
4.2 The ConfigAST Hierarchy
Unlike the main language AST, which must support complex control flow and expressions, the Configuration AST is purely data-centric.


C++




// src/frontend/ast/config_node.h

namespace aria {
namespace config {

enum class NodeType { OBJECT, ARRAY, STRING, INTEGER, FLOAT, BOOL, NULL_VAL };

struct SourceInfo {
   std::string filename;
   int line;
   int column;
};

class ConfigNode {
public:
   virtual ~ConfigNode() = default;
   NodeType type;
   SourceInfo loc; // Critical for error reporting
};

class ConfigObject : public ConfigNode {
public:
   std::vector<std::pair<std::string, std::shared_ptr<ConfigNode>>> members;
   // Fast lookup map, built on demand
   std::unordered_map<std::string, std::shared_ptr<ConfigNode>> lookup; 
};

class ConfigArray : public ConfigNode {
public:
   std::vector<std::shared_ptr<ConfigNode>> elements;
};

class ConfigInteger : public ConfigNode {
public:
   int64_t value;
   bool is_tbb_err; // Track if this is a TBB sentinel
};

//... Definitions for String, Float, Bool...

}
}

4.3 The ConfigParser Implementation
The parser implements the grammar defined in Section 3.3. It owns an instance of AriaLexer and consumes the token stream.
4.3.1 Error Recovery Strategy
A "sane" parser must provide helpful errors. If a syntax error occurs (e.g., a missing comma), the parser should not crash or stop immediately. Instead, it should employ Panic Mode Recovery 1:
   1. Report the error with precise location data.
   2. Skip tokens until a synchronization point is reached.
   * For objects, sync on } or ,.
   * For arrays, sync on ] or ,.
   * For the file, sync on EOF.
This allows the parser to report multiple errors in a single pass, drastically improving the developer experience.
4.3.2 Pseudo-Code: Parsing Objects


C++




std::shared_ptr<ConfigObject> ConfigParser::parseObject() {
   auto node = std::make_shared<ConfigObject>();
   expect(TOKEN_LBRACE); // Uses helper that throws/reports on mismatch

   while (!check(TOKEN_RBRACE) &&!isAtEnd()) {
       std::string key;
       
       // Handle Key: Identifier or String
       if (match(TOKEN_IDENTIFIER) |

| match(TOKEN_STRING_LITERAL)) {
           key = previous().value;
       } else {
           error("Expected property key (identifier or string)");
           synchronizeObject(); // Skip to next comma or brace
           continue;
       }

       consume(TOKEN_COLON, "Expected ':' after key");

       // Parse Value
       auto value = parseValue();
       node->members.push_back({key, value});

       // Handle Comma (Optional trailing, mandatory between items)
       if (!match(TOKEN_COMMA)) {
           if (!check(TOKEN_RBRACE)) {
               error("Expected ',' or '}' after property value");
           }
       }
   }

   consume(TOKEN_RBRACE, "Expected '}' to close object");
   return node;
}

4.4 The ConfigBinder: Schema Validation
Parsing yields a generic ConfigObject. However, the Aria compiler needs typed structs (e.g., ProjectConfig). We introduce a Data Binding layer to map the AST to C++ structs automatically.
This system uses C++ templates to "visit" the generic AST and populate the target struct, performing type checking along the way.


C++




// Example Usage
struct ProjectConfig {
   std::string name;
   int version;
   std::vector<std::string> authors;
};

// Binding definition
template<>
void bind(ConfigBinder& binder, ProjectConfig& cfg) {
   binder.bind("name", cfg.name);
   binder.bind("version", cfg.version);
   binder.bind("authors", cfg.authors);
}

The ConfigBinder handles:
      * Type Mismatch: If "version" is a string in the config but int in the struct, report "Type mismatch at line X: expected integer, got string".
      * Missing Keys: If a required field is missing, report an error.
      * TBB Ranges: If cfg.version was a tbb8, the binder checks if the parsed integer fits in the range.

5. Advanced Feature: TBB and Hex-Stream Integration
Aria's unique features—Twisted Balanced Binary arithmetic and the Hex-Stream I/O model—must be first-class citizens in the configuration format.
5.1 TBB Sticky Error Propagation in Configuration
TBB types use sentinels (e.g., -128 for tbb8) to represent errors.1 The configuration parser must respect this.
      * Scenario: A user configures a timeout: timeout_ms: 255 for a tbb8 field.
      * Standard Behavior: Wrap around to -1.
      * Aria Behavior: The ConfigBinder detects that 255 exceeds tbb8::max(). It has two valid behaviors configurable by the compiler:
      1. Strict Mode: Emit a compile error: "Value 255 exceeds range for tbb8".
      2. Sticky Mode: Set the value to ERR (-128). This effectively propagates the configuration error into the runtime, where it will trigger safety checks immediately upon use.1
5.2 Hex-Stream I/O Channels
Aria defines a 6-stream I/O model (stdin, stdout, stderr, stddbg, stddati, stddato).1 The configuration parser itself must adhere to this topology during execution.
      * Diagnostic Output: Parser errors and warnings must be emitted to stderr (FD 2).
      * Debug Telemetry: Internal parser states (e.g., "Parsing object at line 50") should be emitted to stddbg (FD 3). This allows developers to debug configuration loading without polluting the standard output.
      * Data Pipeline: If the parser is used as a standalone tool (e.g., aria-config), the parsed JSON/binary output should be sent to stddato (FD 5), keeping stdout (FD 1) clean for human-readable messages.

6. Implementation of the Variable Substitution Engine
The prompt highlights a discussion on "possible ideas," likely referring to the robust string interpolation required for build systems (e.g., specifying paths like src_dir: "&{ROOT}/src").
6.1 The Substitution Logic
Substitution happens during the Analysis Phase, after parsing but before Binding. The ConfigAST is traversed, and VariableRef nodes are resolved.
Resolution Hierarchy 1:
      1. Local Scope: The parser maintains a variables block.
Code snippet
variables: {
   ROOT: "/home/user/project",
   SRC: "&{ROOT}/src"
}

      2. Environment Scope: Identifiers prefixed with ENV. map to getenv().
         * &{ENV.HOME}
         * &{ENV.PATH}
         3. Global Scope: Compiler built-ins.
         * &{ARCH} (e.g., "x86_64")
         * &{OS} (e.g., "linux")
6.2 Cycle Detection
Recursive variable definitions (A: "&{B}", B: "&{A}") can cause infinite loops. The substitution engine must use a Detect-Cycle algorithm (likely Depth First Search with a "visiting" set) on the variable graph. If a cycle is detected, the parser must emit a detailed error trace: "Cyclic dependency detected: A -> B -> A".
6.3 Security: Typed Injection
Snippet 1 mentions "Typed Injection" to prevent attacks. In configuration, this means preventing an injected variable from altering the structure of the config.
         * Rule: Variable substitution acts on values, not structure.
         * If VAR contains }, "admin": true, {, interpolating it into a string "{ &{VAR} }" produces a string literal containing those characters, not a new JSON object. This is guaranteed because parsing happens before substitution. The parser sees a StringLiteral containing &{VAR}, creates a node, and the substitution engine merely changes the string content of that node. It does not re-invoke the parser.

7. Build System Integration (AriaBuild)
The primary consumer of ACL is AriaBuild.1 The specification for AriaBuild explicitly requests a whitespace-insensitive format to replace Makefiles.
7.1 The build.aria File Structure
Based on the ACL grammar, the standard build file structure is defined as:


Code snippet




// build.aria
{
   project: {
       name: "MyAriaProject",
       version: "0.1.0"
   },

   variables: {
       CC: "clang",
       CFLAGS: "-O3"
   },

   targets: [
       {
           name: "main",
           type: "binary",
           sources: ["src/**/*.aria"], // Globbing pattern
           depends_on: ["core_lib"]
       },
       {
           name: "core_lib",
           type: "library",
           sources: ["lib/core.aria"]
       }
   ]
}

7.2 Globbing Implementation
The configuration contains strings like "src/**/*.aria". The ACL parser treats these as simple strings. It is the responsibility of the AriaBuild logic (specifically the GlobEngine 1) to interpret these strings.
         * The parser passes the literal string src/**/*.aria to the Binder.
         * The Binder populates the std::vector<std::string> sources field.
         * AriaBuild iterates over this vector. If it detects * or ?, it invokes the std::filesystem::recursive_directory_iterator to find matching files.
         * Determinism: To ensure deterministic builds, the results of the glob expansion must be sorted alphabetically before being passed to the compiler.1

8. Development Roadmap and Testing Strategy
Phase 1: Core Parser Implementation (Week 1)
         * Goal: Implement ConfigParser, ConfigAST, and basic ConfigBinder.
         * Task 1: Create src/frontend/parser/config_parser.cpp utilizing AriaLexer.
         * Task 2: Implement the recursive descent methods (parseObject, parseArray, parseValue).
         * Task 3: Unit tests for basic parsing of JSON-like structures and Aria extensions (comments, unquoted keys).
Phase 2: Integration and Type Safety (Week 2)
         * Goal: Integrate TBB types and error reporting.
         * Task 1: Connect ConfigParser to DiagnosticEngine for pretty-printed errors (snippets, carets).
         * Task 2: Implement TBB range checking in ConfigBinder.
         * Task 3: Implement Hex/Binary literal parsing for integers.
Phase 3: Variable Substitution (Week 3)
         * Goal: Implement the string interpolation engine.
         * Task 1: Implement the variable resolution map and scope logic.
         * Task 2: Implement cycle detection for recursive variables.
         * Task 3: Verify "Typed Injection" security (fuzzing with malicious variable payloads).
Phase 4: AriaBuild Adoption (Week 4)
         * Goal: Replace the legacy build configuration parser in AriaBuild.
         * Task 1: Define the BuildConfig C++ structs.
         * Task 2: Update AriaBuild to use ConfigParser::parse("build.aria").
         * Task 3: Verify that globbing and dependency resolution work with the new format.
Testing Strategy
         * Unit Tests: Test individual parser methods with valid and invalid inputs.
         * Fuzz Testing: Feed the parser random byte streams to ensure it never crashes (segfaults), only reports errors. This is crucial for C++ implementations.
         * Golden File Tests: Parse a complex build.aria file and verify the resulting C++ struct matches the expected values exactly.

9. Conclusion
The Aria Configuration Language (ACL) as specified in this report fulfills every requirement of the original request and the associated research materials.
         * Sane & Custom: It rejects the idiosyncrasies of YAML/JSON in favor of a clean, C-style syntax.
         * Explicit Delimiting: It uses braces and brackets to define structure, ensuring robustness.
         * No Whitespace Sensitivity: It is purely token-based, eliminating "invisible" errors.
         * Parser Implementation: The report provides a detailed architectural blueprint for implementing the parser using Recursive Descent, integrated with the existing Aria compiler infrastructure.
By adopting ACL, the Aria ecosystem moves away from the fragility of legacy build tools and towards a unified, type-safe, and deterministic future, perfectly aligned with the language's guiding philosophy.

/**
* src/frontend/parser/config_parser.cpp
* 
* Implementation of the recursive descent parser for ACL.
* Leverages AriaLexer for tokenization.
*/

#include "config_parser.h"
#include "../lexer/tokens.h"
#include "diagnostics.h"

namespace aria {
namespace frontend {

// Parses a configuration file from source string
std::unique_ptr<ConfigNode> ConfigParser::parse(const std::string& source) {
   lexer = std::make_unique<AriaLexer>(source);
   advance(); // Prime the token stream
   
   if (match(TOKEN_LBRACE)) {
       return parseObject();
   } else if (match(TOKEN_LBRACKET)) {
       return parseArray();
   } else {
       // Use DiagnosticEngine for rich error reporting
       diags.report(currentToken.loc, DiagnosticLevel::Error, 
           "Configuration root must be an Object '{...}' or Array '[...]'");
       return nullptr;
   }
}

// Recursive function to parse values
std::unique_ptr<ConfigNode> ConfigParser::parseValue() {
   if (check(TOKEN_LBRACE)) {
       advance();
       return parseObject();
   }
   if (check(TOKEN_LBRACKET)) {
       advance();
       return parseArray();
   }
   if (check(TOKEN_STRING_LITERAL)) {
       auto node = std::make_unique<ConfigString>(currentToken.value);
       advance();
       return node;
   }
   if (check(TOKEN_INT_LITERAL)) {
       // TBB Logic: Check for sentinel values or overflow here
       int64_t val = std::stoll(currentToken.value);
       auto node = std::make_unique<ConfigInt>(val);
       advance();
       return node;
   }
   //... Handle Floats, Bools, Variable Subs...
   
   diags.report(currentToken.loc, DiagnosticLevel::Error, "Unexpected token in value position");
   return nullptr;
}
