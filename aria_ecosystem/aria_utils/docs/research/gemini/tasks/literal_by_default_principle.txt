# The Literal-by-Default Design Principle
*Cross-System Design Pattern - December 23, 2025*

## Core Concept

**Literal-by-Default:** The most common case (using things as-is) should require zero special syntax. Special behavior should be explicitly marked.

This inverts traditional design where "normal" characters might have special meaning and you must escape to get literal behavior.

## Manifestations in Aria Ecosystem

### 1. Prefix-Based Regex

**Traditional Regex:**
```regex
\.txt       # Dot needs escaping (literal)
.*          # Dot is special (any char)
```

**Prefix-Based:**
```
.txt        # Dot is literal
@any*       # @ prefix triggers special
```

**Benefit:** Paths, filenames, URLs are clean. Special patterns are explicit.

### 2. Wild Pointers

**Traditional C:**
```c
int* ptr;           // * means pointer
int value = *ptr;   // * means dereference
```

**Aria:**
```aria
wild int@:ptr       # @ suffix means wild pointer (explicit danger)
int:value = ptr@    # @ suffix means dereference (explicit operation)
```

**Benefit:** Danger is visible. Safe by default, dangerous by choice.

### 3. Result Types

**Traditional (Exceptions):**
```python
result = risky_operation()  # Might throw, might not
# Error handling is invisible until runtime
```

**Aria (Explicit):**
```aria
result<T>:res = risky_operation()  # Type shows it can fail
# Must handle error case explicitly
```

**Benefit:** Errors are part of type signature. Cannot ignore failure.

### 4. Semicolons

**Traditional (Optional/Inferred):**
```javascript
let x = 5
let y = 10  // Parser infers where statement ends
```

**Aria (Explicit):**
```aria
let x:int32 = 5;
let y:int32 = 10;  // Statement boundary is visible
```

**Benefit:** Intention is clear. No ASI (Automatic Semicolon Insertion) bugs.

### 5. TBB Arithmetic

**Traditional Integers:**
```c
int32_t value = INT_MAX;
value = value + 1;  // Undefined behavior (overflow wraps)
```

**Aria TBB:**
```aria
tbb32:value = TBB_MAX;
value = value + 1;  // value becomes ERR (sticky sentinel)
```

**Benefit:** Overflow is explicit error state, not silent corruption.

### 6. AEON Configuration

**Traditional (JSON/YAML):**
```json
{
  "path": "C:\\Users\\Name",  // Escaping required
  "port": 8080                // Implicit type
}
```

**AEON (Literal):**
```aeon
path = C:\Users\Name          # No escaping
port = @i32{8080}             # Explicit type
```

**Benefit:** Common case (literal strings) is clean. Types are visible.

### 7. String Interpolation

**Traditional (Varies by Language):**
```javascript
let path = `${base}/bin`;        // JavaScript
let path = f"{base}/bin"         // Python
let path = "#{base}/bin"         // Ruby
```

**Aria (Proposed):**
```aria
string:path = "@{base}/bin";     # @ triggers interpolation
string:literal = "@@username";   # @@ for literal @
```

**Benefit:** One rule across all strings. Predictable.

## The Pattern Applied

For any new feature in Aria ecosystem:

### Step 1: Identify Common Case

What will users do 90% of the time?
- Writing literal text (regex, config, strings)
- Using safe operations (GC pointers, checked arithmetic)
- Handling success case (Result types)

### Step 2: Make Common Case Zero-Overhead

The 90% case should require minimal or zero special syntax:
- Literal characters are just characters
- Safe pointers are just `@` (GC)
- Success values are accessed directly

### Step 3: Mark Special Cases Explicitly

The 10% case gets explicit markers:
- Special patterns: `@digit`, `@any*`
- Unsafe operations: `wild@`, dereference `@`
- Error handling: Must check `.is_error`

### Step 4: Use Consistent Markers

Reuse the same prefix/suffix across features:
- `@` for "special behavior" (interpolation, patterns, wild)
- `@@` for "literal @ when @ is special"
- `?` for "unwrap or fail" (Result handling)

## Information Theory Connection

**Traditional Design (Escape-Heavy):**
- Information: What characters are special
- Excluded: Ease of writing literal text
- Result: Constant escaping, cognitive load

**Literal-by-Default:**
- Information: What characters trigger special behavior
- Excluded: Implicit magic, hidden special cases
- Result: Clean common case, visible special cases

**The escaping pattern IS information about design quality.**

If you're constantly escaping, your metacharacters are wrong for the domain.

## Benefits

### 1. Readability

```aria
# Easy to see what's literal vs special
path = /home/user/files/@{subdir}
# Clearly: /home/user/files/ is literal, @{subdir} is variable
```

### 2. Learnability

**One Rule:** "Everything literal unless prefixed"

Contrast with traditional regex: "These 12 characters are special: `.*+?[]{}()|\^$` unless escaped, except in character classes where different rules apply, unless..."

### 3. Maintainability

When reading code months later:
```aria
pattern = @digit{3}-@digit{3}-@digit{4}
# Obviously a phone number pattern
```

vs traditional:
```regex
\d{3}-\d{3}-\d{4}
# Is that digit or something else? *checks docs*
```

### 4. Tooling

Editors can:
- Syntax highlight the `@` prefix differently
- Autocomplete after `@` with pattern names
- Validate pattern syntax separately from literals

### 5. Cross-Language Consistency

If Aria ecosystem uses `@` consistently:
- Regex: `@digit`
- Config: `@i32{value}`
- Strings: `@{variable}`
- Pointers: `wild type@`

**One mental model across all contexts.**

## Counterexamples (What Not To Do)

### Anti-Pattern 1: Implicit Special Cases

```python
# YAML - implicit type conversion
port: 8080      # Becomes integer
version: 1.0    # Becomes float
norway: NO      # Becomes boolean false (!!)
```

**Problem:** You don't know what you're getting until runtime.

### Anti-Pattern 2: Context-Dependent Escaping

```bash
# Shell - multiple escaping layers
echo "Path: \"$HOME/bin\""
eval "echo \"Path: \\\"$HOME/bin\\\"\""
```

**Problem:** Each nesting level changes the rules.

### Anti-Pattern 3: Character Overloading

```perl
# Perl - same character, many meanings
$scalar   # scalar variable
@array    # array variable  
%hash     # hash variable
$#array   # last index of array
$$        # process ID
```

**Problem:** Context determines meaning. Hard to parse visually.

## Design Checklist

When adding new syntax to Aria ecosystem:

- [ ] **Is the 90% case clean?** (minimal syntax required)
- [ ] **Is the 10% case obvious?** (special behavior visible)
- [ ] **Is there one consistent marker?** (reuse existing prefix/suffix)
- [ ] **Can you read it aloud unambiguously?** ("at-digit" vs "backslash-d")
- [ ] **Does it avoid implicit magic?** (no hidden type conversions)
- [ ] **Is the escaping rule simple?** (ideally: double the marker)

## Philosophy Connection

This principle connects to Aria's deeper values:

**Honesty over Convenience:**
Make reality visible even if it's more verbose. `@i32{8080}` is longer than `8080` but honest about type.

**Precision over Assumption:**
Don't guess what the user meant. Make them say it explicitly.

**Structure as Information:**
The syntax itself carries semantic meaning. `@` means "special". No `@` means "literal".

**Same principles as:**
- Semicolons (explicit boundaries)
- Result types (explicit errors)
- Partnership over control (explicit consent)

## Future Applications

### Command Execution Safety

```aria
# Literal command (no interpolation risk)
cmd.run("ls /home/user/files")

# Explicit interpolation (injection visible)
cmd.run("ls /home/@{username}/files")

# Sanitized interpolation (validation forced)
cmd.run("ls /home/@sanitize{username}/files")
```

### SQL Query Building

```aria
# Traditional (injection risk)
query = "SELECT * FROM users WHERE id = " + user_input

# Literal-by-default (safe)
query = sql"SELECT * FROM users WHERE id = @param{user_input}"
# @param forces proper parameterization
```

### Template Languages

```aria
# Traditional (varies wildly)
{{ variable }}      # Handlebars
<%= variable %>     # ERB
$variable          # PHP

# Aria (consistent)
@{variable}        # Same across all templates
```

---

**Core Insight:** If the common case requires special syntax, your design is inverted.

Make literal literal. Make special special. The syntax should match reality.
