# Prefix-Based Regex System
*Design Discussion - December 23, 2025*

## The Problem with Current Regex

Traditional regex has a fundamental design flaw: **metacharacters conflict with the domains they're most commonly used in**.

### Common Conflicts

**`.` (any character):**
- Metacharacter meaning: matches any single character
- Real-world usage: file extensions (`.txt`, `.yaml`, `.sh`)
- Result: Constant escaping `\.txt` or confusion

**`/` (delimiter in sed/perl):**
- Delimiter usage: separates pattern/replacement
- Real-world usage: path separator (`/usr/local/bin`)
- Result: Escape hell `\/usr\/local\/bin`

**`*` (zero or more):**
- Metacharacter: quantifier
- Real-world: glob patterns, wildcards
- Result: Context confusion

**`?` (zero or one):**
- Metacharacter: optional quantifier
- Real-world: URL query parameters (`page.php?id=123`)
- Result: Escaping in URLs

### Core Design Failure

Regex was designed abstractly for "text processing" without considering the actual domains:
- Filesystems (paths, extensions, special characters)
- URLs (schemes, queries, fragments)
- Source code (operators, punctuation)
- Common data formats (JSON, CSV, etc.)

**Result:** Users spend more time escaping than pattern matching.

## The Prefix-Based Solution

### Core Principle

**Invert the cognitive load:**
- Traditional: Everything might be special → escape the literals
- Prefix-based: Everything is literal → prefix the specials

### Single Escape Character

Choose ONE prefix character (e.g., `@`) that triggers special behavior.

**Everything else is literal by default.**

### Syntax Examples

```
# Match literal paths (no escaping needed):
/usr/local/bin/aria

# Match wildcards (explicit prefix):
/usr/@any*/bin/@any+

# Match file extensions:
config.yaml
test.txt

# Match version numbers:
aria-@digit.@digit.@digit.tar.gz

# Match emails:
@alpha+@@domain.@tld     # @@ = literal @
```

### Comparison with Traditional Regex

#### File Paths
```
# Traditional (escape hell):
\/usr\/local\/.*\/bin\/[a-zA-Z]+

# Prefix-based (clean):
/usr/local/@any*/bin/@alpha+
```

#### Version Numbers
```
# Traditional:
config-\d+\.\d+\.\d+\.tar\.gz

# Prefix-based:
config-@digit+.@digit+.@digit+.tar.gz
```

#### URLs
```
# Traditional:
https?:\/\/[^\/]+\/.*\?id=\d+

# Prefix-based:
http@opt{s}://@domain/@any*?id=@digit+
```

## Pattern Syntax

### Character Classes (Prefix-based)

```
@any        # any single character (was .)
@digit      # single digit [0-9]
@alpha      # single letter [a-zA-Z]
@lower      # lowercase letter
@upper      # uppercase letter
@word       # word character [a-zA-Z0-9_]
@space      # whitespace character
@hex        # hexadecimal digit
```

### Quantifiers (Postfix)

```
@any*       # zero or more
@any+       # one or more
@any?       # zero or one (optional)
@any{3}     # exactly 3
@any{2,5}   # between 2 and 5
@any{3,}    # 3 or more
```

### Groups and Alternatives

```
@group{pattern}     # grouping
@opt{pattern}       # optional group
@either{a|b|c}      # alternatives
@capture{pattern}   # capture group
```

### Anchors

```
@start      # start of line (was ^)
@end        # end of line (was $)
@wbound     # word boundary (was \b)
```

### Special Literals

```
@@          # literal @ character
@tab        # tab character
@newline    # newline
@cr         # carriage return
```

## Design Principles

### 1. Literal by Default
The most common case (matching literal text) should require zero special syntax.

### 2. Explicit Specials
When you need pattern matching, make it OBVIOUS with the prefix.

### 3. One Escape Rule
Only the prefix character needs escaping (by doubling: `@@`).

### 4. Readable Patterns
Patterns should be understandable without a reference manual:
- `@digit+` is clearly "one or more digits"
- `@any*` is clearly "zero or more any character"

### 5. Domain Awareness
Common domains (paths, URLs, emails) should be easy:
```
# Paths are clean:
/usr/local/@any*/bin

# URLs are clean:
https://@domain/@path

# Emails are clean:
@word+@@domain.@tld
```

## Implementation Strategy

### Parser Stages

1. **Literal Mode (default):**
   - Read characters literally
   - Only `@` triggers special handling

2. **Pattern Mode (after `@`):**
   - Parse pattern name (`digit`, `alpha`, `any`, etc.)
   - Parse optional quantifier
   - Return to literal mode

3. **Escape Mode (`@@`):**
   - Emit literal `@`
   - Return to literal mode

### Example Parse

Input: `/usr/@any*/bin`

```
Literal: /
Literal: u
Literal: s
Literal: r
Literal: /
Pattern: @any with quantifier *
Literal: /
Literal: b
Literal: i
Literal: n
```

## Argument Structure

### Separate Pattern and Replacement

Instead of:
```bash
sed 's/old/new/'
```

Use separate arguments:
```bash
agrep 'old' 'new'
# or
agrep --pattern 'old' --replace 'new'
```

**Rationale:** Arguments ARE delimiters. No need for internal delimiters that conflict with content.

### Benefits

1. No delimiter escaping
2. No ambiguity about what's pattern vs replacement
3. Clean, obvious syntax
4. Shell handles argument separation

## Use Cases

### File Operations

```bash
# Find all Python files
agrep --files '@any*.py'

# Replace in paths
agrep '/old/path' '/new/path' @any*.conf

# Version bumping
agrep 'version-@digit.@digit.@digit' 'version-2.0.0'
```

### Text Processing

```bash
# Email extraction
agrep '@word+@@domain.@tld'

# Phone numbers
agrep '@digit{3}-@digit{3}-@digit{4}'

# URLs
agrep 'https://@any+.@any+/@any*'
```

### Code Refactoring

```bash
# Function rename (no escaping parentheses!)
agrep 'oldFunction(@any*)' 'newFunction(@any*)'

# Import path updates
agrep 'from @word+ import' 'from newmodule import'
```

## Future Enhancements

### Context-Aware Modes

```bash
# Path mode: / and . are literal by default
agrep --mode=path '@any*/bin'

# URL mode: :, /, ? are literal by default  
agrep --mode=url 'http@opt{s}://@domain'

# Code mode: operators are literal by default
agrep --mode=code 'function(@any*)'
```

### Named Patterns

```bash
# Define reusable patterns
agrep --define email='@word+@@domain.@tld'
agrep --pattern '@{email}' --replace 'redacted'
```

### Backreferences

```bash
# Capture and reuse
agrep '@capture{@word+} = @capture{@digit+}' '@1 equals @2'
```

## Connection to agrep Implementation

**IMPORTANT:** This regex syntax design was developed independently but complements the agrep specification documented in `/home/randy/._____RANDY_____/REPOS/aria_utils/docs/research/gemini/responses/sysUtils_4_grep.txt`.

**Key Finding:** Gemini's agrep design focuses heavily on:
- Hex-Stream I/O topology (stddato, stddbg, etc.)
- Binary Match serialization protocols
- Memory safety (Wild pointers, TBB offsets)
- SIMD vectorization for performance

**However:** The actual pattern matching syntax is left unspecified. The agrep spec mentions "Boyer-Moore intrinsic" for literal string matching but provides no regex pattern language.

**Integration Point:** This prefix-based regex system should be integrated as agrep's pattern matching layer. The two designs are complementary:
- Gemini's work: Infrastructure and I/O architecture
- This design: Pattern matching syntax and semantics

When implementing agrep, use this prefix-based syntax for the `PATTERN` argument rather than traditional regex.

## Philosophy Connection

This design aligns with broader principles:

**Aria's semicolons:** Make intention explicit, don't rely on inference
**Information theory:** What you exclude (escaping) IS information about design quality
**Honesty as structure:** The syntax should reflect reality, not hide complexity

**Core insight:** If you're constantly escaping common characters, your metacharacter choice is wrong.

## Implementation Priority

1. **Core parser** (literal vs pattern mode)
2. **Basic patterns** (@any, @digit, @alpha, quantifiers)
3. **Separate arguments** (pattern and replacement)
4. **File operations** (search/replace in files)
5. **Advanced patterns** (groups, captures, anchors)
6. **Context modes** (path-aware, URL-aware, etc.)

## Open Questions

1. **Prefix character choice:** Is `@` the best? Alternatives: `%`, `$`, `~`
2. **Pattern names:** Keep verbose (`@digit`) or allow short (`@d`)?
3. **Compatibility mode:** Support traditional regex with flag?
4. **Performance:** Will prefix parsing be slower? (Probably negligible)

## Notes

- This came from frustration with `/pattern/replace/` requiring constant escaping
- Realized arguments can BE delimiters - no internal delimiters needed
- Extended to full regex redesign based on "literal by default" principle
- Same philosophy as Result types: make the common case easy, special case explicit
