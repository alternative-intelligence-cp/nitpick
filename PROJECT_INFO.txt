=== CORE PHILOSOPHY ===

1. NO DEFERRALS EVER
   - If feature is in aria_specs.txt, it MUST be implemented for v0.1.0
   - "Later" means "never" - creates exponentially harder work
   - Only exception: Features we haven't decided on (kernel integration)
   - Shortcuts waste more time than they save

2. IMPLEMENTATION > APPEARANCE
   - Working code > Looking like working code
   - Test must compile and run, not just exist
   - Specifications are plans, not accomplishments
   - Documentation describes reality, not aspirations

3. TEST EVERYTHING IMMEDIATELY
   - Feature implemented? Write test same day
   - Test must compile and execute successfully
   - If test fails, feature is not done
   - Never move to next feature until current one validated

4. HONEST DOCUMENTATION
   - Only claim features that have passing tests
   - "Implemented" = Has working test
   - "Specified" = In aria_specs.txt but not working
   - "Planned" = Idea not yet specified
   - Never use "coming soon" or "will be available"

5. NO TIMELINE PRESSURE
   - "When it's done" is the only deadline
   - Quality and completeness over arbitrary dates
   - Better to take extra time than release broken

=== ANTI-PATTERNS TO AVOID ===

❌ "Phase 7 tests pass, we're ready to release"
   → Tests only validated pipeline, not features
   
❌ "We have great specifications, language is complete"
   → Specifications ≠ Implementation
   
❌ "Let's defer the hard features for later"
   → Creates debt that compounds
   
❌ "Build tooling to show progress"
   → Core language must work before peripheral tools
   
❌ "Claim feature works, test it later"
   → Test immediately or fall back to training patterns
