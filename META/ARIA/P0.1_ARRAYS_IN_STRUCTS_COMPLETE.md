## Summary

Fixed long-standing TODO that prevented arrays from being used as struct fields. Arrays were being silently converted to pointers, causing them to disappear from struct type definitions.

## What Was Fixed

### Root Cause
- `resolveTypeNode()` line 6428 had: `return typeSystem->getPointerType(elementType);`
- Comment: `// TODO: Create proper ArrayType in TypeSystem`
- **All infrastructure existed** - just needed wiring!

### Implementation

**1. Type Resolution** (`src/frontend/sema/type_checker.cpp:6411-6447`)
- Replaced pointer hack with proper `ArrayType` creation
- Extract array size from literal expressions
- Handle both fixed-size (`T[N]`) and dynamic (`T[]`) arrays
- Validation: size must be non-negative integer literal

**2. Index Type Checking** (`src/frontend/sema/type_checker.cpp:5479-5483`)
- Added `ArrayType` recognition in `inferIndexExpr()`
- Returns element type for array indexing operations
- Enables: `c.watches[0].active = 1i32;`

**3. LLVM Type Mapping** (`src/backend/ir/codegen_expr.cpp:87-101`)
- Fixed arrays → `llvm::ArrayType::get(elementType, size)`
- Dynamic arrays → `llvm::PointerType`
- Proper memory layout in structs

## Known Limitations

### 1. Struct Literal Initialization (P0.7)
**Current**:
```aria
Container:c;  // ✅ Uninitialized declaration works
c.count = 3;  // ✅ Field-by-field init works
```

**Blocked**:
```aria
Container:c = {count: 3};  // ❌ "Object literal has 1 fields, requires 2"
// Missing array literal syntax for 'points' field
```

**Workaround**: Use uninitialized declaration, then assign fields

**Related**: P0.7 Complex Struct Initialization (40-60 hours)

### 2. Array Size Expressions
**Current**: Only integer literals supported
```aria
Point[3]:points;     // ✅ Works
Point[N]:points;     // ❌ Error - must be literal
Point[2+1]:points;   // ❌ Error - must be literal
```

**Future**: Const expressions support

### 3. Definite Assignment Analysis
Reading from uninitialized struct triggers error even if fields assigned:
```aria
Container:c;
c.count = 3;           // Assign field
if (c.count == 3) {}   // ❌ "Use of uninitialized variable 'c'"
```

**Reason**: Analysis doesn't track individual field initialization

**Workaround**: Only write to fields, don't read until properly initialized

## Design Notes

The fix followed Aria's design philosophy:
- **No special syntax**: Arrays use same `type:name` pattern
- **No context dependence**: Array fields work like any other field  
- **Structural correctness**: Replaced hack with proper type system support
- **Infrastructure reuse**: All necessary types existed, just needed wiring

Quote from user:
> "That's why a symbol means what it means. Not context dependent.
> `type:name = val;` doesn't matter if its a struct, an int, a float, a function. All the same."

This fix makes arrays in structs work the same way as primitive fields - no special cases, no surprises.
