Architectural Specification: Dependency Graph Engine and Scheduler for AriaMake
1. Executive Summary: The Imperative for a Deterministic Build Infrastructure
The construction of modern software systems is no longer a linear translation of source code to machine instructions; it is a complex orchestration of dependency resolution, parallel execution, and state management. For the Aria programming language, currently evolving through version v0.0.7, the build system—aria_make—serves as the critical nervous system connecting the developer's intent with the compiler's rigorous enforcement of safety and logic. This report articulates a comprehensive architectural specification for the DependencyGraph engine, the core subsystem responsible for modeling the build universe. The design mandates a shift from imperative, script-based build logic to a declarative, graph-theoretic model that guarantees determinism, maximizes concurrency via Kahn’s Algorithm, and ensures precise incremental compilation through Aria’s native Twisted Balanced Binary (TBB) arithmetic.
The contemporary landscape of systems programming has moved away from the loose, fragile scripting of early UNIX tools toward hermetic, reproducible build environments. Legacy tools like GNU Make, while foundational, suffer from "invisible" syntax errors due to significant whitespace and a reliance on shell-based execution that obscures the dependency graph.1 aria_make rejects this legacy in favor of a "Configuration as Data" philosophy. By ingesting the whitespace-insensitive Aria Build Configuration (ABC) format, the system constructs an in-memory Directed Acyclic Graph (DAG) where nodes represent compilation units and edges represent strict modification-order constraints. This model allows the build system to transcend the role of a simple task runner and become an authoritative source of truth for the project's structure, exporting precise compile commands for language servers and static analyzers.1
This specification addresses three critical engineering challenges. First, it defines the DependencyGraph class structure, utilizing modern C++ ownership semantics to manage the lifecycle of build nodes. Second, it details the implementation of Kahn’s Algorithm for topological sorting, transforming the static graph into a dynamic, parallel execution schedule that respects the unique "Blocking Paradox" of compiler invocation.1 Third, it establishes a robust mechanism for incremental build detection using tbb64 timestamps, eliminating the integer overflow risks associated with standard Unix time representations. Finally, it integrates a sophisticated cycle detection mechanism using Tri-Color Marking to provide actionable diagnostics for circular dependencies, a strict requirement of the Aria module system.1
2. Theoretical Framework: Graph-Theoretic Modeling of Compilation
To engineer a build system that is both correct and performant, one must first establish a rigorous mathematical model of the build process. The DependencyGraph is not merely a collection of files; it is a discrete mathematical structure $G = (V, E)$ that enforces the logical causality of software construction.
2.1 The Directed Acyclic Graph (DAG) as a Build Artifact
The build universe is modeled as a Directed Acyclic Graph. In this formalism, the vertices ($V$) represent the tangible artifacts of the build process, and the edges ($E$) represent the immutable constraints of causality.
* Vertices ($V$): The set $V$ encompasses all "Units of Work" or "Artifacts." In the context of the Aria compiler (ariac) and its ecosystem, these nodes map to distinct file types and logical entities:
   * Source Nodes: These are the leaves of the dependency tree (from the perspective of "requires") or the roots (from the perspective of data flow). They represent raw .aria source files discovered via the Globbing Subsystem.1
   * Intermediate Nodes: These represent the artifacts generated by the compiler's frontend and middle-end, such as LLVM Intermediate Representation (.ll) files or native object files (.o).
   * Target Nodes: These are the final outputs—executables, static libraries, or shared objects—resulting from the linkage of intermediate nodes.
* Edges ($E$): An edge $(u, v) \in E$ denotes a dependency relationship. The semantics of this directionality are critical. In this specification, we define a directed edge $u \to v$ to mean "Node $u$ depends on Node $v$." This implies that the action associated with $v$ must complete successfully before the action associated with $u$ can commence. This aligns with the "requires" relationship: app.exe $\to$ main.o $\to$ main.aria.
This model differs from the data-flow model where edges follow the transformation of data (main.aria $\to$ main.o). While the data flows forward, the dependency constraint points backward. Kahn's algorithm, used for scheduling, calculates "in-degree"—the number of dependencies that must be satisfied. Under the definition $u \to v$ ($u$ needs $v$), $u$ has an out-degree to $v$. However, for the purpose of the scheduler implementation, it is often computationally more efficient to store the reverse graph (edges pointing from dependency to dependent) to facilitate the "notification" step where a completed dependency unlocks its dependents. This report will explicitly distinguish between these logical and physical representations in the class design section.
2.2 The Necessity of Acyclicity
A fundamental invariant of the build graph is that it must be acyclic. A cycle $C = \{v_1, v_2,..., v_k\}$ such that $(v_1, v_2) \in E,..., (v_k, v_1) \in E$ represents a logical impossibility: $v_1$ cannot be built until $v_k$ is built, but $v_k$ requires $v_1$. In the context of Aria, such cycles most frequently arise from circular module imports—Module A uses Module B, while Module B uses Module A.1
The Aria compiler enforces a strict module hierarchy that forbids such cycles to ensure deterministic initialization orders. Therefore, the DependencyGraph engine serves as the first line of defense, validating the architectural integrity of the user's project. Unlike systems that might allow lazy evaluation to resolve certain cycles, aria_make must strictly reject them. This requirement necessitates the integration of a specific cycle detection algorithm, as Kahn's algorithm detects the presence of a cycle (by failing to process all nodes) but does not produce the diagnostic path required for the user to fix the error.1
2.3 Incrementalism and the Physics of Time
The performance of a build system is inversely proportional to the amount of redundant work it performs. A "clean build" is an infrequent event; the "incremental build" is the developer's inner loop. To support this, the graph model incorporates a temporal dimension.
Let $M(n)$ be the last modification timestamp of the artifact associated with node $n$. A node $u$ is considered "dirty" (requiring recompilation) if:
1. The artifact for $u$ does not exist on the filesystem.
2. There exists a dependency $v$ such that $M(v) > M(u)$.
This ostensibly simple logic is complicated by the realities of filesystem resolution (seconds vs. nanoseconds), clock skew between distributed build agents, and integer overflow in timestamp representation. Aria addresses these issues by adopting the tbb64 (Twisted Balanced Binary) type for all timestamp arithmetic. This ensures that time comparisons are mathematically robust and that error states (such as "file not found") are propagated safely through the "sticky error" mechanism inherent to TBB.1 The build system must rely on C++ system calls like stat or std::filesystem::last_write_time to bridge the gap between the host OS and Aria's runtime, as the current version v0.0.7 lacks native standard library support for these metadata operations.1
3. Architectural Design: The DependencyGraph Class
The DependencyGraph class acts as the central nervous system of the build tool. It is responsible for node lifecycle management, graph topology construction, state tracking, and the generation of execution schedules.
3.1 Memory Management and Ownership
In a complex build graph potentially containing thousands of nodes (e.g., a large standard library or application), pointer stability and memory safety are paramount. The architecture employs a "Container-Factory" pattern. The DependencyGraph instance exclusively owns all DependencyNode objects, typically storing them in a std::unordered_map keyed by the canonical file path. This ensures $O(1)$ lookup times and guarantees that pointers to nodes remain valid for the lifespan of the graph, preventing dangling pointer references during the asynchronous execution phase.1
3.1.1 The DependencyNode Structure
The DependencyNode is the atomic unit of the graph. It encapsulates the topology, build status, and incremental state of a single artifact. The design distinguishes between the immutable definition of the node and its mutable runtime state.

3.2 Thread-Safety and Concurrency Strategy
The build process has two distinct phases regarding concurrency: Construction and Execution.
1. Construction Phase: The graph is populated. This involves parsing configuration files (aria.toml) and source files (for use statements). This phase may be parallelized (e.g., parallel globbing), necessitating the graph_mutex_ to protect the node map during insertion.
2. Execution Phase: Once scheduling begins, the graph topology is immutable. However, the state of the nodes (in_degree, status) changes rapidly. By using atomic integers for in_degree, the system allows multiple worker threads to decrement dependency counts concurrently. The thread that observes the in_degree transition from 1 to 0 wins the race and claims the responsibility for enqueueing the newly ready node. This "lock-free" propagation logic is essential for scaling to high core counts.1
3.3 Implicit Dependency Discovery
AriaBuild constructs the graph from two sources. "Explicit" dependencies are declared in the aria.toml configuration (e.g., a test target depends on the main library). "Implicit" dependencies are discovered by parsing the source code. The DependencyGraph integrates with the parser to extract use statements. For example, if src/main.aria contains use std.io;, the system resolves std.io to its physical file path and calls graph.add_dependency("src/main.aria", "path/to/std/io.aria"). This requires the DependencyGraph to interface with the GlobEngine to resolve logical module paths to physical files.1
4. Topological Sorting: Implementation of Kahn’s Algorithm
To convert the static graph into a dynamic execution schedule, aria_make implements Kahn’s Algorithm. This algorithm is preferred over Depth-First Search (DFS) based topological sorts because it naturally produces a "frontier" of ready tasks suitable for consumption by a thread pool, rather than a strictly linearized list.
4.1 The Algorithm Mechanics
Kahn’s algorithm operates on the concept of "in-degree"—the number of incoming edges to a node. In our dependency model ($u \to v$ means $u$ requires $v$), the in-degree represents the number of unsatisfied dependencies.
Step 1: Initialization
The DependencyGraph iterates over all nodes to calculate their initial in-degree. In a valid DAG, there must be at least one node with an in-degree of 0. These are the "leaf" nodes (e.g., independent source files) that have no prerequisites.
Step 2: The Ready Queue
All nodes with in_degree == 0 are pushed into a ReadyQueue. This queue acts as the buffer for the thread pool.
Step 3: The Execution Loop (Parallel)
The scheduling logic is distributed across the worker threads.
1. A worker pops a node $u$ from the ReadyQueue.
2. The worker executes the build command for $u$ (e.g., invoking ariac).
3. Upon completion, the worker calls DependencyGraph::mark_complete(u).
4. The graph iterates over the dependents of $u$ (nodes $v$ that require $u$).
5. For each $v$, the graph atomically decrements v->in_degree.
6. If v->in_degree becomes 0, $v$ is added to the ReadyQueue.
4.2 The Blocking Paradox and Pool Sizing
An important insight from the research is the "Blocking Paradox".1 Although the build system logic is asynchronous, the actual compilation step involves spawning a child process (execvp) and waiting for it to terminate. This is a blocking operation. Consequently, a worker thread in the pool spends most of its time blocked on specific OS signals.
To maintain high CPU utilization, the thread pool size should inherently match or slightly exceed the number of logical cores (std::thread::hardware_concurrency()). If the pool is too small, the parallelism is artificially capped. If too large, context switching overhead degrades performance. AriaMake defaults to hardware_concurrency but allows user override via flags.
4.3 Handling Build Failures
If a node fails to build (non-zero exit code from ariac), the error must propagate. The DependencyGraph marks the node as FAILED. Crucially, it does not unlock its dependents. This naturally prunes the execution tree; any node requiring the failed artifact will never reach in_degree == 0 and will simply linger in the PENDING state. After the queue is empty, the system detects that the build was incomplete and reports the failure.1
5. Incremental Compilation and TBB Timestamping
Incremental compilation is the primary optimization feature of any build system. It avoids recompiling targets that are already up-to-date with respect to their inputs. aria_make implements a rigorous timestamp comparison logic rooted in Aria's unique type system.
5.1 The AriaTimestamp and TBB64
Standard Unix time (time_t) has historically suffered from ambiguity (32-bit overflow in 2038) and lack of error semantics (returning -1). Aria mitigates this by using tbb64 (Twisted Balanced Binary 64-bit integers) for all time representations.1
* Structure: A tbb64 is a signed 64-bit integer.
* Sentinel: The minimum representable value (0x8000...000) is reserved as the ERR sentinel.
* Sticky Propagation: In Aria's arithmetic, x + ERR = ERR. This property is invaluable for build logic. If a file does not exist, its timestamp is ERR. Any comparison involving ERR can be defined to trigger a rebuild (fail-safe).

	5.2 The StatCache Mechanism
During the build graph initialization, thousands of file paths are discovered via the globbing engine. Calling stat on these files individually can be expensive, especially on Windows filesystems or networked drives. aria_make implements a StatCache class that memoizes modification times.1
* Population: The GlobEngine often retrieves file metadata during directory traversal. This data is fed directly into the StatCache, avoiding redundant syscalls.
* Invalidation: When a target is rebuilt, its output file changes. The system must invalidate the cache entry for that file to ensure subsequent steps (or future builds in watch mode) see the correct new timestamp.
5.3 Implementation of is_dirty
The DependencyNode::is_dirty() method encapsulates the logic:
1. Retrieve $M(T)$ (Output Timestamp) from StatCache. If ERR, return true.
2. Iterate over all dependencies. Retrieve $M(S_i)$.
3. If $\max(M(S_i)) > M(T)$, return true.
4. Otherwise, return false.
This logic implements the check $\max(M(S_i)) > M(T)$ as specified in the architectural constraints.1
6. Cycle Detection: The Tri-Color Marking Algorithm
While Kahn’s Algorithm is excellent for scheduling, it has a significant deficiency: it detects cycles silently. If the algorithm terminates and the number of processed nodes is less than the total nodes, a cycle exists, but Kahn's logic does not isolate the nodes involved in the cycle. To provide developer-friendly error messages, aria_make implements a secondary traversal algorithm.
6.1 The Need for Diagnostics
A build error message stating "Circular dependency detected" is useless without the path. Developers need to see: Error: Cycle detected: A -> B -> C -> A. This requires reconstructing the path of the cycle.
6.2 Tri-Color DFS Implementation
We utilize the Tri-Color Marking algorithm (Depth-First Search) specifically on the set of nodes that remain unprocessed after Kahn's algorithm fails.
* States:
   * White: Unvisited.
   * Gray: Visiting (currently in the recursion stack).
   * Black: Visited (processed and safe).
Algorithm Steps:
1. The CycleDetector initializes all nodes remaining in the graph to White.
2. It begins a DFS from an arbitrary White node.
3. When visiting node $u$, mark it Gray. Push $u$ onto a path_stack.
4. Iterate over neighbors $v$ (dependencies).
   * If $v$ is Gray: A back-edge to the current recursion stack is found. Cycle Detected. The path_stack contains the exact cycle. Return the stack.
   * If $v$ is White: Recursively visit $v$.
   * If $v$ is Black: Skip (already verified acyclic subgraph).
5. If all neighbors are processed, mark $u$ Black and pop from path_stack.
This algorithm is $O(V+E)$ and is only run in the failure case, ensuring it incurs no overhead on successful builds.1
7. Integration with the Aria Ecosystem
The DependencyGraph relies on tight integration with other Aria components, specifically the standard library and the globbing subsystem.
7.1 Globbing and File Discovery
The inputs to the graph are generated by the GlobEngine.1
* Determinism: The GlobEngine enforces strict alphabetical sorting of discovered files. This is crucial for the DependencyGraph. If the graph construction order depended on the OS directory iteration order (which is non-deterministic), the topological sort might yield different valid schedules on different runs. While mathematically valid, this non-determinism can lead to reproducible build issues (e.g., varying link order). By consuming sorted input, the DependencyGraph guarantees a stable topology structure.1
7.2 The Six-Stream I/O Topology
Aria applications operate with a unique 6-stream I/O model (stdin, stdout, stderr, stddbg, stddati, stddato).1 The build system must respect this.
* Child Process output: When aria_make spawns ariac, it must correctly map these file descriptors.
* Log Separation: Compiler diagnostics (warnings/errors) go to stderr. However, structured build telemetry (e.g., timing data, memory usage) should be routed to stddbg (FD 3). The DependencyNode execution logic must set up the pipes to capture this structured data for build profiling, keeping the user's console (stdout) clean for progress bars and high-level status.
8. Conclusion
The architectural design of the DependencyGraph for aria_make represents a synthesis of rigorous graph theory and pragmatic systems engineering. By implementing Kahn’s Algorithm with atomic in-degree tracking, the system achieves maximal parallelism while maintaining thread safety without global locks. The integration of tbb64 timestamps ensures that incremental build logic is robust against the common pitfalls of integer overflow and sentinel ambiguity. Furthermore, the inclusion of Tri-Color Marking for cycle detection transforms the difficult problem of circular dependencies into actionable diagnostics. This design provides a scalable, deterministic foundation for the Aria build ecosystem, capable of handling the complexity of modern, large-scale software projects.

// TBB64 Timestamp wrapper
struct AriaTimestamp {
   int64_t val;
   static constexpr int64_t ERR = -9223372036854775807LL - 1; // INT64_MIN
   
   bool is_valid() const { return val!= ERR; }
   bool operator>(const AriaTimestamp& other) const {
       if (!is_valid() ||!other.is_valid()) return true; // Fail-safe dirty
       return val > other.val;
   }
};

enum class BuildStatus { PENDING, READY, BUILDING, COMPLETE, FAILED };

struct DependencyNode {
   std::string id;
   
   // Graph Edges
   std::vector<DependencyNode*> dependencies; // Outgoing (Requires)
   std::vector<DependencyNode*> dependents;   // Incoming (Required By)

   std::atomic<BuildStatus> status{BuildStatus::PENDING};

   std::string output_path;

   // Command to build this node
   std::string command;

};

9.2 Kahn's Algorithm Logic (Thread-Safe)


C++




// Called by worker thread when a task completes
std::vector<DependencyNode*> DependencyGraph::mark_complete(DependencyNode* node) {
   std::vector<DependencyNode*> ready_nodes;
   
   node->status.store(BuildStatus::COMPLETE);

   // Propagate completion to dependents
   for (DependencyNode* dependent : node->dependents) {
       // Atomic decrement. fetch_sub returns the PREVIOUS value.
       // We want to know if we just dropped it to 0.
       int prev_degree = dependent->in_degree.fetch_sub(1);
       
       // If previous was 1, it is now 0. Node is ready.
       if (prev_degree == 1) {
           dependent->status.store(BuildStatus::READY);
           ready_nodes.push_back(dependent);
       }
   }
   return ready_nodes;
}

9.3 Cycle Detection (Tri-Color DFS)


C++




bool CycleDetector::visit(DependencyNode* u, std::vector<DependencyNode*>& stack) {
   colors_[u] = Color::GRAY;
   stack.push_back(u);

   for (DependencyNode* v : u->dependencies) {
       if (colors_[v] == Color::GRAY) {
           return true; // Cycle found
       }
       if (colors_[v] == Color::WHITE) {
           if (visit(v, stack)) return true;
       }
   }

   stack.pop_back();
   colors_[u] = Color::BLACK;
   return false;
}
