1. Executive Summary
This report constitutes a rigorous, line-by-line safety audit of the Aria programming language compiler backend, specifically focusing on the Foreign Function Interface (FFI) and Application Binary Interface (ABI) boundary. The analysis was conducted under the "Life/Death" threat model mandated by the project architect, recognizing that the software generated by this compiler will control physical actuators in robotic assistants for vulnerable populations. In such a context, a single byte of memory corruption or a misinterpreted floating-point value is not merely a bug; it is a potential kinetic weapon.
The primary objective was to verify compliance with Safety Integrity Level 4 (SIL-4) standards, ensuring that the robotic control software generated by Aria operates with absolute predictability and stability when interacting with C-based system libraries and hardware drivers. The audit specifically targeted the translation logic within src/backend/ir/codegen_expr.cpp to determine if the "Explicitness, Predictability, and Stability" directives of the Aria specification are upheld at the binary interface level.
The analysis reveals three critical vulnerabilities and two major safety gaps that pose an immediate threat to physical safety if deployed. These are not theoretical concerns but deterministic failures rooted in the divergence between Aria's internal type representation and the System V AMD64 ABI.
1. Critical Alignment Violation (The LBIM int128 Trap): The implementation of large integers using the "Limb-Based Integral Model" (LBIM) successfully bypasses LLVM optimization bugs but introduces a fatal ABI alignment mismatch. Aria’s int128 structure aligns to 8 bytes, whereas the System V AMD64 ABI mandates 16-byte alignment for __int128. This guarantees General Protection Faults (#GP) when passing these values on the stack to GCC-compiled C functions utilizing SIMD instructions, leading to immediate process termination (watchdog trigger).
2. Type System Breach (The String ABI Mismatch): Aria maps its native str type to an LLVM opaque pointer (ptr). However, internal evidence confirms Aria strings are "fat pointers" (structs containing data pointer + length). When passing a str to a C function expecting a char*, Aria passes the address of the metadata struct rather than the character buffer. This results in the C runtime interpreting the pointer bytes as text data, leading to garbage output, information leakage, and potential heap corruption.
3. Floating Point Hazard (flt128 Compatibility): The compiler maps flt128 to LLVM’s fp128 (IEEE-754 binary128). On x86-64 Linux, the standard C long double is 80-bit extended precision (x86_fp80). This bit-level incompatibility causes catastrophic precision loss or the generation of Signaling NaNs (SNaN) if flt128 is passed to C functions expecting long double. In a physics-based AGI, this could result in force calculations being off by orders of magnitude.
The following sections detail the technical evidence, the mechanism of failure, and the necessary remediation for each audit item.

2. Threat Model & Audit Scope
2.1 Threat Landscape
In the context of a physics-based Artificial General Intelligence (AGI) controlling robotic actuators, the FFI boundary represents the "nervous system" connection between the high-level cognitive planner (Aria) and the low-level motor drivers (C/C++). The reliability of this interface is paramount. Any distortion in data transmission across this boundary results in a divergence between the AGI's internal model of the world and the physical reality of the robot's limbs.
The specific threat vectors analyzed in this report include:
* Stack Corruption via Struct Mismatch (ACE): If Aria passes a struct by value (consuming stack space) while the C callee expects a pointer (consuming a register), the stack frame will be corrupted. The Aria-generated code will overwrite the return address or frame pointer of the C function. Upon return, the instruction pointer (RIP) will load corrupted data, potentially leading to Arbitrary Code Execution (ACE) or a jump to a random location, causing unpredictable motor spasms.
* Force Limit Bypass via Type Confusion: If a force limit calculated as flt128 (IEEE quad) is passed to a C driver expecting long double (80-bit), the driver may interpret the exponent bits of the IEEE number as mantissa bits of the 80-bit number. This could result in a value magnitude interpretation error of factor $2^{1000}$, instantly commanding actuators to apply maximum physical force, bypassing safety limits.
* Watchdog Trigger via Alignment Fault: Misaligned stack arguments causing #GP faults in kernel-space drivers or real-time tasks will trigger immediate system halts. In a surgical or assistive context, a sudden limp-mode or freeze while supporting a patient's weight is a life-safety hazard.
2.2 Target Artifact Analysis
The audit strictly adhered to the provided source code artifacts, refusing to make assumptions about missing files. The primary focus was src/backend/ir/codegen_expr.cpp 1, which contains the definitive logic for mapping Aria AST types to LLVM IR types.
The analysis of codegen_expr.cpp confirms that it is the central "lowering" mechanism for the compiler. It includes:
* getLLVMType(Type* type): The recursive function converting high-level semantic types to LLVM types.
* getLLVMTypeFromString: A helper for primitive types and large integers.
* ExprCodegen class: The driver for expression generation.
The file explicitly includes llvm/IR/Function.h and llvm/IR/Intrinsics.h, confirming it has the capability to manipulate function attributes, though the specific logic for function declarations (signatures) is notably absent from the snippet. The analysis, therefore, focuses on the type compatibility of arguments generated during call expressions, which is sufficient to identify ABI violations.

3. Audit Item 1: Calling Convention Correctness
Requirement: The compiler must explicitly set the calling convention for all extern calls to match the platform ABI (e.g., CallingConv::C for standard functions, CallingConv::X86_StdCall for Windows API). The "Silent Defaults" prohibition in the user request mandates that reliance on LLVM's default behavior is unacceptable for SIL-4 software.
3.1 Implementation Analysis
The analysis of src/backend/ir/codegen_expr.cpp reveals a concerning absence of explicit calling convention management. While the file includes the necessary LLVM headers to manipulate calling conventions, a comprehensive search of the provided source text 1 confirms that setCallingConv is never invoked within the visible type mapping or expression generation logic.
In the LLVM infrastructure, if setCallingConv() is not explicitly called on a Function or CallInst, the compiler defaults to llvm::CallingConv::C (ccc). While ccc is the correct standard for the majority of Linux/Unix user-space functions (libc), assuming it universally is a violation of the stability requirement.
3.2 The Windows/RTOS Hazard
The reliance on the default C calling convention creates a latent hazard for cross-platform deployment or interaction with Real-Time Operating Systems (RTOS), which are common in robotic control stacks.
* Windows API Mismatch: Many Windows system libraries and hardware drivers utilize the stdcall convention. In stdcall, the callee is responsible for cleaning up the stack (popping arguments). In cdecl (the default), the caller cleans the stack.
   * Failure Mechanism: If Aria compiles a call to a stdcall driver function using cdecl semantics, the Aria code will adjust the stack pointer after the call returns. Since the driver function also adjusted the stack pointer before returning, the stack pointer will be double-adjusted. This misalignment will likely cause the program to crash upon the next ret instruction or access incorrect local variables.
* RTOS Hooks: High-performance robotic drivers often use fastcall or custom register-heavy conventions to minimize stack usage in interrupt contexts. A default ccc call will fail to pass arguments in the correct registers, passing garbage data to the driver.
3.3 Variadic Function Safety (isVarArg)
The audit searched for handling of variadic functions 1 and found no logic to detect or handle isVarArg flags in function types. This is critical for the System V AMD64 ABI used on Linux.
* The %al Register Requirement: The System V ABI mandates that for variadic calls (like printf), the caller must set the %al register (the lower 8 bits of %rax) to the number of vector registers used for floating-point arguments.
* LLVM Behavior: LLVM handles this automatically only if the function signature in the IR is correctly marked as variadic (...).
* The Hazard: If the Aria compiler generates a fixed-signature declaration for a variadic C function (e.g., declaring printf as taking exactly 2 arguments because the user only passed 2), LLVM will treat it as a standard function and will not set %al.
* Consequence: An optimized implementation of printf may use the value in %al to jump into a jump table for register saving. If %al contains garbage data (random bits from a previous operation), printf may jump to an invalid offset or attempt to save a nonsensical number of registers, leading to a segmentation fault or register corruption.
3.4 Verdict and Remediation
Status: ⚠️ Unclear / Hazard Detected
While the default behavior works for standard Linux C functions, it violates the "No silent defaults" and "Explicitness" rules of the project. The compiler blindly assumes the host platform's default C convention without providing a mechanism to override it.
Root Cause:
The ExprCodegen class lacks a mechanism to inspect the FuncDecl for ABI attributes (e.g., extern "stdcall").
Proposed Fix:
1. Modify AST: Update FuncDeclStmt to accept an optional ABI string.
2. Update Codegen: In codegen_stmt.cpp (implied existence), map these strings to llvm::CallingConv enums.
3. Enforce Validation: If an extern function is declared without an explicit ABI, the compiler should either require one or explicitly default to "C" with a warning, forcing the user to confirm the intent.

4. Audit Item 2: Struct Passing ABI & Layout
Requirement: Structs must be passed according to the System V AMD64 ABI classification rules. Layouts must match C compiler padding exactly to ensure data integrity across the FFI boundary.
4.1 Struct Layout & Natural Alignment
The source code in src/backend/ir/codegen_expr.cpp contains a robust and explicit implementation for struct layout generation.1


C++




// Set struct body with natural ABI padding (isPacked=false)
// This ensures platform-consistent alignment without manual padding calculation
llvmStruct->setBody(fieldTypes, /*isPacked=*/false);

Analysis:
The explicit use of isPacked=false is a crucial safety feature. It instructs LLVM to insert padding bytes between fields to satisfy the alignment requirements of the target architecture, mirroring the behavior of standard C compilers (GCC/Clang). This ensures that a struct defined in Aria as { u8, u64 } will occupy 16 bytes (1 byte data + 7 bytes padding + 8 bytes data) rather than 9 bytes. This alignment is vital for memcpy operations and direct pointer casting between Aria and C.
Furthermore, the code implements a specific fix (GEMINI BATCH 03 FIX: BUG-07-001) to handle recursive types. By creating an opaque struct first (StructType::create(context, structName)), the compiler correctly handles self-referential structures (like Linked Lists) without entering an infinite recursion loop during type resolution.
Verdict: ✅ Compliant. The struct layout generation logic is safe and standard-compliant.
4.2 The "Fix256" Physics Hazard
The audit identified a specific custom type, fix256, defined for deterministic physics simulations.1


C++




if (typeName == "fix256") {
   //... [2 x i64] integer part + [2 x i64] fractional part
   llvm::Type* limbArray = llvm::ArrayType::get(llvm::Type::getInt64Ty(context), 4);
   fix256Struct = llvm::StructType::create(context, {limbArray}, "struct.fix256");
}

ABI Analysis:
* Size: 4 elements * 64 bits = 256 bits (32 bytes).
* System V ABI Rule: The System V ABI dictates that aggregates (structs) larger than 16 bytes (two eightbytes) are classified as Class MEMORY.2 They must be passed on the stack, not in registers.
* LLVM Lowering: When a function accepts an aggregate type by value that is classified as MEMORY, LLVM's backend lowers this to a pointer to a copy (an invisible reference) or performs a memcpy to the stack slots depending on the specific optimization level and target attributes (byval).
The Threat Scenario:
Consider a C physics engine function defined as:
void update_physics(const fix256_t* state); (Pass by pointer, common for large structs)
If the Aria developer writes:
extern func update_physics(state: fix256); (Pass by value, implied by lack of *)
Aria generates IR: call void @update_physics(%struct.fix256 %val).
Because fix256 is 32 bytes, LLVM (following SysV ABI) will likely place a pointer to a copy of the struct in RDI (as if it were a hidden argument) or push the entire 32 bytes onto the stack. However, the C function expects RDI to contain the address of the user's struct.
If LLVM pushes 32 bytes to the stack, but C expects a pointer in RDI, RDI will effectively contain garbage. The C function will dereference this garbage pointer, leading to an immediate segmentation fault or, worse, a read from a random memory location that happens to be valid.
Verdict: ⚠️ User-Dependent / Hazard.
The compiler's generation is technically correct for the types provided (it lowers a struct to a struct). However, the burden of ABI matching is placed entirely on the user. Given the "Life/Death" scenario, relying on the user to manually add * (e.g., fix256*) for large structs is a risk.
Remediation:
Implement a safety warning in the semantic analysis phase. If an extern function takes a struct by value that is larger than the platform's register-passing threshold (16 bytes on x86-64), emit a warning: "Performance/Safety Warning: Struct 'fix256' is 32 bytes. Verify if the C function expects a pointer or a value. Passing large structs by value is uncommon in C APIs."

5. Audit Item 3: The str (String) Fat Pointer Mismatch
Requirement: String types passed to C functions must be compatible with C-style strings (char* null-terminated) or the C function must explicitely accept the Aria string layout.
5.1 The Evidence
The audit uncovered a definitive mismatch in how strings are defined versus how they are passed.
1. Aria String Layout: In codegenLiteral 1, string literals are explicitly created as global constants of type struct.AriaString. The code constructs a struct with two fields:
C++
std::vector<llvm::Type*> fields = {
   llvm::PointerType::get(llvm::Type::getInt8Ty(context), 0), // data pointer (char*)
   llvm::Type::getInt64Ty(context)                            // length (i64)
};

This confirms str is a Fat Pointer (Struct of { ptr, i64 }).
2. Type Mapping: In getLLVMType 1, the mapping for the str type is:
C++
if (type_name == "str"...) {
   return llvm::PointerType::get(context, 0);
}

This maps the str type to a single LLVM opaque pointer (ptr).
5.2 The Crash Mechanism
The mismatch arises because the str type in the type system maps to a pointer to the metadata struct, not a pointer to the character data.
Consider this call:
extern func puts(s: str);
puts("Active");
   1. Aria creates a global AriaString struct instance in static memory: { data_ptr -> "Active\0", length -> 6 }.
   2. The str variable in Aria holds the address of this struct instance.
   3. When calling puts, Aria passes the address of the struct.
   4. The C function puts(const char* s) receives this address in register RDI.
   5. puts expects RDI to point to the character 'A'.
   6. Instead, RDI points to the AriaString struct.
   7. The first 8 bytes of the struct are the data_ptr.
   8. puts reads these 8 bytes (which form a memory address) and tries to print them as characters.
Implication:
   * On Little Endian systems (x86-64), the bytes of the pointer will be printed as garbage characters.
   * If the address bytes happen to contain a 0x00 (very likely for heap/stack addresses), printing stops, resulting in garbled, truncated output.
   * If the address does not contain a null byte, puts continues reading into the length field and beyond until it hits a null byte or unmapped memory, causing a Segfault.
Safety Impact:
This is a catastrophic ABI mismatch. In a robotic system, if strings are used for command serialization (e.g., sending "MOVE_X" to a controller), the controller will receive the binary bytes of a memory address instead of the command. The robot will fail to execute the command or execute a random command corresponding to the address bytes.
Verdict: ❌ Critical Bug Found (Line 114 in codegen_expr.cpp).
Root Cause:
The compiler treats str as a pointer to the AriaString struct but allows it to be passed directly to C functions that expect char*.
Proposed Fix:
The compiler must strictly distinguish between str (Aria String) and cstr (C String).
   1. Code Change: In ExprCodegen::getLLVMType, do not treat str merely as a pointer. It requires special handling at call sites.
   2. Automatic Decay: In visitCallExpr (implied), if the target function is extern "C" and an argument is of type str, automatically emit a GEP (GetElementPtr) instruction to load the data pointer from the struct and pass that to the C function.
   3. Alternative: Introduce a dedicated cstr type for FFI and forbid passing str to extern functions, forcing the user to write s.data.

6. Audit Item 4: Platform-Specific Quirks (The Alignment Trap)
Requirement: Types must adhere to platform alignment requirements to prevent hardware exceptions. On x86-64 System V ABI, __int128 must be 16-byte aligned.
6.1 The LBIM Implementation
Aria implements int128 to bypass LLVM bugs using the Limb-Based Integral Model (LBIM).1


C++




if (typeName == "int128" |

| typeName == "uint128") {
   //...
   llvm::Type* limbArray = llvm::ArrayType::get(llvm::Type::getInt64Ty(context), 2);
   int128Struct = llvm::StructType::create(context, {limbArray}, "struct.int128");
}

This creates an LLVM type equivalent to { [2 x i64] }.
6.2 Alignment Analysis
LLVM calculates the alignment of a struct based on the maximum alignment of its members.
   * i64 has an alignment of 8 bytes on x86-64.
   * Therefore, struct.int128 has a natural alignment of 8 bytes.
The ABI Conflict:
The System V AMD64 ABI specification explicitly states:
"The __int128 type is stored in memory with 16-byte alignment."
If Aria passes an int128 to a C function expecting __int128, and that value is passed on the stack (e.g., because registers are exhausted, or it is passed by value as a struct member):
   1. LLVM (Aria side) aligns the stack slot to 8 bytes (natural alignment of the LBIM struct).
   2. GCC (C side) compiles the function assuming 16-byte alignment for __int128.
   3. GCC emits movdqa (Move Aligned Double Quadword) to load the value from the stack into an XMM register for processing.
   4. movdqa throws a General Protection Fault (#GP) if the memory address is not 16-byte aligned.
Failure Scenario:
A robot controller function calculate_torque(__int128 force_vector) is called. Aria calculates the force and passes it. The call happens to place the argument at address 0x...08 (8-byte aligned but not 16-byte aligned). The CPU executes movdqa in the C library. The CPU detects the misalignment and triggers a hardware exception. The operating system catches the exception and terminates the process (SIGSEGV or SIGBUS). The robot's control loop dies instantly.
Safety Impact:
This is a "Life/Death" failure mode. A watchdog timer might eventually catch the freeze, but the immediate loss of control authority is unacceptable.
Verdict: ❌ Critical Bug Found (Line 132 in codegen_expr.cpp).
Proposed Fix:
Explicitly force the alignment of the LBIM structs to 16 bytes (for int128) and higher for larger types if SIMD instructions are used.
In getLLVMTypeFromString:


C++




// Proposed Fix
if (typeName == "int128" |

| typeName == "uint128") {
   //... creation logic...
   // Note: LLVM's StructType::create doesn't take alignment directly.
   // The alignment must be enforced at the AllocaInst level (variable declaration)
   // and the Load/Store level.
   // However, we can trick LLVM by adding a dummy private field or using alignas
   // semantics if mapped to a higher level construct.
   // BEST FIX: Map int128 to llvm::Type::getInt128Ty(context) for the ABI boundary,
   // and only use LBIM for internal arithmetic. Conversions required at FFI boundary.
}

Given the strict requirement to use LBIM to avoid optimizer bugs, the compiler must generate code that manually aligns these structs on the stack to 16 bytes when passing them to extern functions.

7. Audit Item 5: Floating Point Logic (flt128 vs long double)
Requirement: Match C long double semantics on the target platform to ensure force calculations are interpreted correctly.
7.1 Implementation


C++




if (typeName == "flt128") return llvm::Type::getFP128Ty(context);

This maps flt128 to LLVM's fp128 type, which is strictly defined as IEEE-754 binary128 (128-bit precision).
7.2 Platform Quirk (x86-64)
On x86-64 Linux (System V ABI):
   * long double in C is 80-bit Extended Precision (x87 format, stored as 128 bits/16 bytes for alignment).
   * __float128 in C (GCC extension) is IEEE-754 binary128.
The Hazard:
The term flt128 suggests "128-bit float." A developer might assume this corresponds to the standard C long double type, which is the widest standard float available.
   1. Aria passes a 128-bit IEEE float (1 sign, 15 exp, 112 mantissa).
   2. C function reads 80-bit Extended float (1 sign, 15 exp, 1 int, 63 fraction).
   3. Result: The bit representations are completely different. The exponent bias is different ($16383$ for 128-bit vs $16383$ for 80-bit, but the fields are in different locations). The 80-bit format has an explicit integer bit that IEEE 128 lacks.
Example Calculation:
A force value of 1.0 in IEEE 128 has a hex representation (approx) of 3FFF 0000....
If interpreted as x87 80-bit:
   * The exponent might align, but the explicit integer bit (required by x87) would be drawn from the high bits of the IEEE mantissa.
   * If that bit is 0, the number is a generic Unnormal or Denormal.
   * The value could be interpreted as a Signaling NaN or a value close to zero, or a massive number depending on specific bit patterns.
Verdict: ❌ Dangerous Mismatch (Line 183 in codegen_expr.cpp).
Remediation:
   1. Rename Type: Change flt128 to bin128 or ieee128 to explicitly signal it is NOT long double.
   2. Add f80 Type: Introduce an explicit f80 type that maps to llvm::Type::getX86_FP80Ty(context) for interoperability with legacy C math libraries.
   3. Documentation: Explicitly warn in the documentation that flt128 requires __float128 on the C side, not long double.

9. Conclusion
The Aria compiler backend, in its current state as analyzed in src/backend/ir/codegen_expr.cpp, contains latently fatal flaws in its FFI implementation. While the internal type system (LBIM, fix256) is robustly designed to avoid LLVM optimization bugs, the interface with the external world is porous and misaligned.
Deploying this code to a robotic actuator system interacting with C drivers (e.g., via CAN bus or EtherCAT libraries utilizing int128 or long double) carries a near certainty of runtime failure. The str passing bug alone guarantees corrupt data exchange, and the int128 alignment issue is a ticking time bomb for SIMD-optimized drivers.
