Architectural Specification: Borrow Checker Fixpoint Iteration and Loop Analysis for the Aria Compiler
1. Executive Summary and Architectural Context
The evolution of the Aria programming language has reached a decisive phase where the theoretical elegance of its design must be reconciled with the rigorous demands of static analysis. While the backend infrastructure, exemplified by the ExprCodegen class in src/backend/ir/codegen_expr.cpp, demonstrates a mature capability to lower complex types like Twisted Balanced Binary (TBB) integers and extended-precision floating-point values to LLVM IR, the semantic analysis layer faces a critical vacuum.1 Specifically, the Borrow Checker—the subsystem responsible for enforcing memory safety without garbage collection—remains unimplemented, representing a "CRITICAL Priority" gap with "zero existing research" in the current codebase.1
This report establishes a comprehensive architectural specification for the Borrow Checker, focusing specifically on the non-trivial problem of Loop Analysis via Fixpoint Iteration. In linear control flow, lifetime validation is a straightforward application of the "Appendage Theory" inequality ($D(Host) \le D(Appendage)$).1 However, loops introduce cyclic dependencies in the Control Flow Graph (CFG) that render single-pass analysis insufficient. To prove safety in the presence of loops—where a reference created in iteration $N$ might be invalidated in iteration $N+1$—the compiler must employ iterative dataflow analysis to compute the set of active loans until a stable fixed point is reached.
The proposed architecture integrates modern static analysis techniques, drawing heavily from the "Polonius" model developed for the Rust compiler, which shifts the analytical domain from lexical lifetimes to "origins" (sets of loans).2 This shift is essential for Aria to support its unique "Hybrid Memory Model," which seamlessly interweaves Stack, GC, and "Wild" (manual) memory allocation strategies. Furthermore, the analysis framework incorporates "Allocation Site Abstraction" to track the state of "Wild" pointers across loop back-edges, enabling the detection of complex Use-After-Free (UAF) vulnerabilities and memory leaks that simpler lexical scopes would miss.4
By defining the lattice structures, transfer functions, and widening operators required for this analysis, this document provides the blueprint for transforming Aria from a structurally sound prototype into a memory-safe systems language capable of self-hosting.
2. Theoretical Foundations of the Aria Memory Model
To construct a valid static analysis algorithm, one must first rigorously define the operational semantics of the memory model it is intended to safeguard. Aria's memory model is distinct in its tripartite division of storage strategies, each governed by a specific set of safety invariants that the Borrow Checker must enforce.
2.1 The Tripartite Memory Segmentation
The Aria compiler backend categorizes every variable into one of three storage classes, defined in the AllocStrategy enum found within the compiler's symbol table architecture.1 The Borrow Checker does not treat memory as a monolith; rather, it applies a strategy pattern to validation based on the region.
2.1.1 The Stack (Lexical Scope)
Stack memory in Aria is governed by the rigid Last-In, First-Out (LIFO) semantics of function activation records. Variables allocated here correspond to alloca instructions in the LLVM backend.1
* Invariant: Strict Lexical Containment. A reference to a stack variable is valid if and only if the reference's lifetime is a strict subset of the variable's scope.
* Analysis Role: The Borrow Checker tracks the "Scope Depth" ($D$) of every stack variable. Loop analysis allows the checker to verify that a reference taken inside a loop does not escape to an outer scope unless the referent itself lives in that outer scope.
2.1.2 The GC Heap (Managed & Moveable)
The default allocation strategy for high-level objects (obj, string, dyn) is the Garbage Collected heap. Aria utilizes a high-performance, moving collector to eliminate fragmentation.1
* The Moving Problem: Because the GC may relocate objects during compaction, raw pointers to GC memory are inherently unsafe.
* The Pinning Solution: The # operator (Pin) temporarily "pins" a GC object in memory, returning a stable "Wild" pointer. The Borrow Checker must enforce the Pinning Contract: the object cannot be unpinned or moved as long as any Wild pointer derived from it is live.1 This requires flow-sensitive analysis to track the liveness of the derived pointer across control flow boundaries.
2.1.3 The Wild Heap (Manual & Fixed)
The wild keyword introduces unmanaged memory, allocated via aria.alloc (similar to malloc) and freed via aria.free. This region offers C-like performance and interoperability but lacks runtime safety nets.
* Invariant: Resource Hygiene. Every allocation must be paired with exactly one deallocation on every execution path.
* Analysis Role: The Borrow Checker acts as a static leak detector and UAF preventer. It must prove that a wild pointer is never accessed after it has been passed to free. This is the most complex aspect of loop analysis, as the state of a pointer (Allocated vs. Freed) can change across iterations.
2.2 Formalizing Appendage Theory
"Appendage Theory" is the central theoretical contribution of Aria's safety model, designed to abstract the complex relationships of ownership and borrowing into a topological inequality based on scope depth.1
Let $\mathbb{S}$ be the set of all lexical scopes in a program. We define a depth function $\Delta: \mathbb{S} \to \mathbb{N}$ such that the global scope has depth 0, and any nested scope has depth $\Delta(parent) + 1$. For any variable $v$, let $\Delta(v)$ be the depth of the scope in which $v$ is declared.
The theory defines two roles:
1. Host ($H$): The owner of the data (e.g., a stack variable or a pinned GC object).
2. Appendage ($A$): A dependent reference to the Host.
The Appendage Invariant states:




$$\Delta(H) \le \Delta(A)$$
This inequality ensures that the Host strictly encloses the Appendage in the scope hierarchy. In linear code, this is trivial to verify. However, loops break this simplicity. A variable declared outside a loop ($\Delta_{outer}$) might be assigned a reference created inside the loop ($\Delta_{inner}$). If that reference points to a loop-local variable, the assignment violates the invariant because $\Delta_{local} > \Delta_{outer}$. The Borrow Checker must detect these "Scope Widening" attempts.
2.3 The Necessity of Flow-Sensitive Analysis
A purely scope-based (flow-insensitive) check is insufficient for Aria because it cannot reason about the state of a variable at a specific program point. Consider a wild pointer p. Its scope might be the entire function, but its validity ends the moment aria.free(p) is called.
More critically, in the context of loops, flow sensitivity is required to detect Loop-Carried Dependencies.6 A reference created in iteration $i$ might be stored in a variable declared outside the loop. In iteration $i+1$, that variable still holds a reference to the data from iteration $i$. If iteration $i+1$ attempts to mutate the underlying data, a data race (or aliasing violation) occurs. The Borrow Checker must track these dependencies across the back-edge of the loop to enforce the "Mutable XOR Shared" aliasing rule.8
3. Dataflow Analysis Framework and Lattice Design
To solve the problem of loop safety, Aria employs Dataflow Analysis, a technique that propagates facts about the program state through the edges of the Control Flow Graph (CFG) until the system stabilizes at a fixed point.10 The mathematical structure underpinning this analysis is the Lattice.
3.1 The Borrowing Lattice Construction
We define the domain of our analysis as a Lattice $\mathcal{L}$. The elements of this lattice represent the set of Active Loans at any given point in the program.12
Let $\mathbb{O}$ be the set of all "Origins" in the function. Adopting the Polonius model, an Origin is a set of loans that are valid for a particular region of code.2 A loan $l \in \mathbb{L}$ is a tuple $(h, k, s)$, where:
* $h$: The Host variable being borrowed.
* $k$: The kind of borrow (Immutable $ or Mutable $mut).
* $s$: The statement (AST node) where the borrow was introduced.
The lattice element at program point $P$ is a subset $S_P \subseteq \mathbb{L}$.
The partial order $\sqsubseteq$ is defined by set inclusion ($\subseteq$).
* Bottom ($\bot$): The empty set $\emptyset$ (no active loans).
* Top ($\top$): The set of all possible loans (conservative over-approximation).
Since we are performing a May Analysis (a loan may be active), the Join Operator ($\sqcup$) is Set Union ($\cup$).14 At a control flow merge point (e.g., the header of a loop where the entry edge meets the back-edge), a loan is considered active if it is active on any incoming path.


$$In(Block) = \bigsqcup_{P \in \text{preds}(Block)} Out(P) = \bigcup_{P \in \text{preds}(Block)} Out(P)$$
This formulation is critical. If a loop conditionally creates a borrow, the union ensures that the analyzer assumes the borrow exists in subsequent code, forcing a conservative safety check.
3.2 The Wild Memory State Lattice
While the Borrowing Lattice tracks references, the Wild Heap requires a separate state machine to track resource lifecycles. This uses a different lattice structure tailored for Must Analysis (Must be valid) and May Analysis (May be freed).
We define the state of a wild variable as:
* UNINIT: Uninitialized.
* ALLOC: Allocated (Valid).
* FREED: Deallocated (Invalid).
* MOVED: Ownership transferred.
* ERROR: Double-free or use-after-free detected.
The Join Operator for Wild States ($\sqcup_W$) handles the merging of conflicting states 16:
* $\text{ALLOC} \sqcup_W \text{FREED} = \text{MAY_FREED}$ (Requires runtime check or conservative error).
* $\text{FREED} \sqcup_W \text{FREED} = \text{FREED}$.
* Any $\sqcup_W \text{ERROR} = \text{ERROR}$.
This state tracking enables the compiler to detect "Conditional Leaks" inside loops. If a variable is allocated in the loop body but not freed on all paths before the back-edge, the state at the loop header will transition to a potential leak state in the second iteration of the fixpoint algorithm.
3.3 The Transfer Functions
The transfer function $f_S(In)$ computes the set of active loans after executing statement $S$. It is defined in terms of Gen (generate) and Kill sets.10


$$Out(S) = f_S(In(S)) = (In(S) \setminus Kill(S)) \cup Gen(S)$$
3.3.1 Gen Set (Loan Creation)
A statement $S$ generates a loan if it contains a borrow expression ($x or #x).
* Example: let r = $x;
* $Gen(S) = \{ (x, \text{Immutable}, S) \}$
3.3.2 Kill Set (Loan Termination)
A loan is killed (removed from the set) when:
1. Scope Exit: The reference variable holding the loan goes out of scope.19
2. Overwrite: The reference variable is reassigned. If r = $y, the previous loan in r is killed.
3. End of Lifetime: For temporary borrows (e.g., foo($x)), the loan is killed immediately after the call.
3.4 Allocation Site Abstraction
Static analysis cannot track an infinite number of heap allocations. For loops that allocate memory (e.g., while (...) { wild p = aria.alloc(...) }), the compiler uses Allocation Site Abstraction.4 Each aria.alloc call in the source code is treated as a single abstract memory location.
If a loop allocates memory and assigns it to a variable p without freeing the previous value, the analysis detects that the abstract location Site_Alloc_Line_X is being overwritten while still live. This triggers a "Resource Leak" error, forcing the developer to insert a free or use a container that manages ownership.
4. The Loop Analysis Algorithm: Fixpoint Iteration
The presence of loops necessitates an iterative approach. A single pass cannot determine the safety of the loop body because the state at the beginning of the loop depends on the state at the end of the previous iteration (the back-edge).
4.1 The Round-Robin Iterative Algorithm
Aria employs a Round-Robin Iterative Solver to compute the Least Fixed Point (LFP) of the dataflow equations.10


Algorithm Step
	Description
	1. Initialization
	Initialize $In(B) = \bot$ (empty) and $Out(B) = \bot$ for all blocks $B$ in the loop. Push the Loop Header block to the Worklist.
	2. Selection
	While Worklist is not empty, pop a block $B$.
	3. Merge (Join)
	Compute $In(B) = \bigcup_{P \in \text{preds}(B)} Out(P)$. This merges state from the entry edge and the loop back-edge.
	4. Transfer
	Compute $Out(B) = f_B(In(B))$ by applying the transfer functions of all statements in $B$.
	5. Convergence Check
	Compare $Out(B)_{new}$ with $Out(B)_{old}$. If the state has changed (i.e., the set of active loans has grown), add all successor blocks of $B$ to the Worklist.
	6. Termination
	Repeat until the Worklist is empty. The lattice has finite height (finite number of possible loans), so termination is guaranteed.20
	4.2 Handling Loop-Carried Dependencies
A Loop-Carried Dependency occurs when a resource borrowed in iteration $i$ is accessed in iteration $i+1$. The fixpoint algorithm naturally handles this:
Scenario:


Code snippet




wild int8@:prev = null;
while (...) {
   if (prev!= null) {
       $prev = 5; // Use borrow from iteration i-1
   }
   wild int8@:curr = aria.alloc(10);
   prev = curr; // Carry dependency to iteration i
}

1. Iteration 1: prev is null. The use $prev is skipped. At the end, prev points to Alloc_Site_A.
2. Back-Edge: The state { prev -> Alloc_Site_A } flows back to the loop header.
3. Iteration 2: The merge at the header updates the input state. Now prev is Alloc_Site_A. The usage $prev is analyzed. The checker verifies if Alloc_Site_A is valid.
4. Leak Detection: If curr overwrites prev without freeing, the analysis sees prev holding a live resource (Alloc_Site_A) being overwritten. It flags a leak.
4.3 Widening Operators for Convergence Speed
While the algorithm is guaranteed to terminate, deep lattices can lead to slow compilation times (many iterations). To mitigate this, Aria employs a Widening Operator ($\nabla$).22
If the set of active loans continues to grow after a fixed threshold of iterations (e.g., $k=3$), the Widening Operator jumps to a conservative upper bound.
* Standard Join: $S_{new} = S_{old} \cup S_{delta}$
* Widening: $S_{new} = S_{old} \nabla S_{delta} = \top_{\text{Loop}}$
This effectively says, "If the loop keeps generating new, distinct borrows that I can't resolve, assume everything reachable in the loop is borrowed." This may cause false positive errors (rejecting valid code), but it ensures the compiler remains responsive.
5. Polonius Integration: From Lifetimes to Origins
The Aria Borrow Checker draws significant inspiration from the Polonius project in Rust.2 Traditional borrow checking relies on "Lifetimes" defined as continuous regions of the CFG. This model often fails for "Conditional Returns" or "Loop-Carried Borrows" where a reference might be valid only on some paths.
5.1 The Origin Model
Aria adopts the Polonius concept of Origins. Instead of tracking the lifetime of a reference r, we track its origin—the set of loans that r might be referring to.


$$Origin(r) = \{ L_1, L_2,..., L_n \}$$
In a loop, a reference variable might point to a loan created in the current iteration ($L_{curr}$) or a loan from a previous iteration ($L_{prev}$). The Origin set is the union of these possibilities.
5.2 Problem Case #3: Conditional Returns
A classic difficulty in borrow checking is returning a reference from a loop only on specific conditions.2


Code snippet




func find_elem(list: List, target: int) -> int$ {
   loop list.iter() {
       if $.value == target {
           return $$.value; // Return borrow
       }
       // Continue iteration
   }
   return NIL;
}

A standard lifetime checker might extend the borrow of list to the end of the function because of the return, incorrectly locking list for the entire loop duration. The Origin-based model sees that the loan returned in the if block flows into the return value, while the loan in the else path dies at the end of the iteration. This granularity allows Aria to accept code that strictly lexical checkers would reject.
5.3 Subset Relationships
The analysis tracks subset relationships between origins.24 If r1 = r2, then $Origin(r2) \subseteq Origin(r1)$. In a loop, this creates a graph of constraints. The fixpoint iteration propagates these subset constraints until the graph stabilizes. Safety is proved if, for every active loan $L$ in $Origin(r)$, the actions performed on $r$ (read/write) are compatible with the permissions of $L$.

   std::string host_name;

struct BorrowState {
   // The set of active loans in the current flow
   std::unordered_set<LoanID> active_loans;

   bool merge(const BorrowState& other); // Implements the Join Operator

   std::vector<BorrowState> scope_stack;
   BorrowState current_state;

6.2 Implementing checkWhileStmt
The checkWhileStmt method is the driver for the fixpoint iteration.1


C++




void BorrowChecker::checkWhileStmt(WhileStmt* stmt) {
   // 1. Snapshot Entry State
   BorrowState entry_state = ctx.current_state;
   BorrowState loop_head_state = entry_state;
   
   bool changed = true;
   int iterations = 0;
   
   // 2. Fixpoint Loop
   while (changed) {
       changed = false;
       iterations++;
       
       // Start analysis of body with the current assumption of head state
       ctx.current_state = loop_head_state;
       
       // Analyze Condition
       checkExpression(stmt->condition.get());
       
       // Analyze Body
       ctx.enterScope();
       if (stmt->body) checkStatement(stmt->body.get());
       
       // Capture Back-Edge State (state at end of body)
       BorrowState back_edge_state = ctx.current_state;
       ctx.exitScope();
       
       // 3. Merge Back-Edge into Header (The Join Operation)
       // If back_edge_state adds new loans/states to loop_head_state, we must iterate again.
       if (loop_head_state.merge(back_edge_state)) {
           changed = true;
       }
       
       // Widening safeguard
       if (iterations > WIDENING_THRESHOLD) {
           loop_head_state.widen(); // Promote to Top/Unknown to force termination
           changed = false; // Stop iterating
       }
   }
   
   // 4. Final State
   // The state after the loop is the Fixed Point Header State
   // (We assume the loop might not execute, or might execute N times)
   ctx.current_state = loop_head_state;
}

6.3 State Merging Logic
The merge function implements the core lattice logic:
* Active Loans: Union. $S_{out}.loans = S_{in1}.loans \cup S_{in2}.loans$.
* Wild States: Conservative merge.
   * ALLOC + UNINIT $\to$ ALLOC (Maybe allocated).
   * ALLOC + FREED $\to$ ERROR (Ambiguous ownership—unsafe).
   * FREED + FREED $\to$ FREED.
This logic ensures that if a variable is freed inside a loop but potentially used in the next iteration (where it enters as FREED), the merge will flag the ALLOC (entry) vs FREED (back-edge) conflict if usage occurs.
7. Aria's Specific Constraints and Optimizations
7.1 Integration with TBB Sticky Errors
Aria's Twisted Balanced Binary (TBB) types introduce a unique "Sticky Error" semantic.1 If an operation results in the ERR sentinel (e.g., -128 for tbb8), that error propagates.
The Borrow Checker must respect this. If a loop index variable is a tbb type, the loop condition while (i!= ERR) serves as a validity check. The dataflow analysis should recognize that inside the loop block, i is guaranteed not to be ERR. This "Range Refinement" prevents false positives where the checker might assume i could be ERR and thus invalid for array indexing.
7.2 Hex-Stream I/O Topology
AriaX Linux introduces a 6-stream I/O model (stdin, stdout, stderr, stddbg, stddati, stddato).1 These streams are represented by file descriptors (FDs). The Borrow Checker must track these FDs as "Wild" resources.
* Constraint: The standard streams (FD 0-5) are "Static Wild" resources. They are implicitly allocated at startup and must not be closed by user code (unless explicitly intending to daemonize or redirect).
* Analysis: The LifetimeContext should initialize with these FDs in a GLOBAL_ALLOC state. A close(0) call should trigger a warning, while close(fd) on a user-opened file is mandatory.
7.3 Bootstrapping and Self-Hosting
The implementation of this Borrow Checker is the final hurdle for Aria's self-hosting capability. Currently, the compiler relies on C++ logic.1 By implementing the fixpoint analysis in C++ now, we enable the compiler to eventually compile a version of itself written in Aria. The Borrow Checker will verify the Aria-based compiler's own memory management, closing the loop.
8. Conclusion
The absence of the Borrow Checker in the current Aria codebase is a fundamental safety deficit. Without it, the language's "Wild" and "Pinning" features are unchecked, inviting memory corruption. The architectural specification provided here—centered on Lattice-Based Fixpoint Iteration—solves the critical problem of validating lifetimes across loop boundaries.
By adopting the Polonius-style "Origin" model and integrating it with "Allocation Site Abstraction" for wild memory, Aria can achieve a level of safety comparable to Rust, while retaining the flexibility of its hybrid memory model. The C++ implementation of the BorrowCheckVisitor and LifetimeContext described herein is the immediate next step for the engineering team, enabling the v0.1.0 release to meet its "Non-Negotiable" safety guarantees.
