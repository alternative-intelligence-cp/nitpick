Architectural Specification and Implementation Strategy for the Aria Dual-Phase Metaprogramming Ecosystem
1. Executive Summary and Strategic Architectural Review
The development of the Aria programming language represents a significant undertaking in the domain of modern systems programming, aiming to synthesize the raw, bare-metal performance of assembly-adjacent languages with the safety and expressiveness of modern managed runtimes. A comprehensive review of the provided source code, specifically src/backend/codegen_context.h 1, and the associated research materials 1 reveals a language architecture defined by rigorous constraints: a hybrid memory model necessitating distinct "wild" and "garbage collected" (GC) heaps, a novel arithmetic system based on Twisted Balanced Binary (TBB) types with mandatory "sticky error" propagation, and a compilation pipeline currently transitioning from a monolithic structure to a modular, phase-based architecture.
The central challenge identified in the user's request and the accompanying research is the absence of a unified metaprogramming layer capable of bridging Aria's low-level syntactic requirements with its high-level semantic ambitions. The current codebase demonstrates a sophisticated backend infrastructure—evident in the CodeGenContext class which manages symbol tables, type mappings (getLLVMType), and safety intrinsics (ScopedVariable for borrow checking)—but lacks the frontend machinery to generate the complex boilerplate these features require.1 Specifically, the implementation of "Classes" via macros, the enforcement of TBB arithmetic safety at compile-time, and the handling of generic monomorphization remain unimplemented gaps.
This report articulates a definitive architectural specification and implementation guide for Aria's Dual-Phase Metaprogramming System. This system is composed of two distinct but interoperable engines:
1. Phase 0: The Hygienic NASM-Style Preprocessor: A textual substitution engine operating on the raw token stream. It is tasked with syntactic abstraction, leveraging a context stack for scope management and a thread-safe "gensym" algorithm for hygiene.1
2. Phase 2: The Zig-Style Comptime Execution Engine: A semantic interpreter operating on the Abstract Syntax Tree (AST). It is tasked with type reflection (@typeInfo), constant evaluation using TBB-aware arithmetic, and the simulation of memory operations via a Virtual Heap.1
By strictly delineating these phases—Textual (Phase 0) vs. Semantic (Phase 2)—the architecture prevents the fragility common in C-style macros while avoiding the rigidity of purely AST-based systems. The following sections provide an exhaustive analysis of the theoretical underpinnings, algorithmic logic, and concrete C++ implementation details required to realize this vision, ensuring full integration with the existing CodeGenContext and fulfilling the "batteries-included" philosophy of the Aria ecosystem.

2. Theoretical Framework: The Necessity of Dual-Phase Metaprogramming
To understand the specific implementation decisions detailed in this report, one must first deconstruct the unique requirements of the Aria language that render traditional metaprogramming approaches insufficient.
2.1 The Limitations of Single-Phase Systems
Traditional compilers typically employ a single-phase metaprogramming model. In the C family, the preprocessor is a simple text substituter, oblivious to syntax or types. This allows for great flexibility (e.g., #define BEGIN {) but introduces severe safety risks (variable capture, double evaluation). Conversely, languages like Rust or AST-based Lisp macros operate entirely within the semantic phase. While safe, they cannot easily introduce new syntactic constructs (like a JSON-style class definition) without complex compiler plugins or procedural macros that parse token streams manually.
Aria's design specifications 1 mandate features that span both domains:
* Syntactic Flexibility: The user request explicitly calls for "NASM-style" macros. NASM (Netwide Assembler) utilizes a preprocessor that is effectively a Turing-complete string manipulation engine. It manages a context stack to handle nested labels and repeats instructions to unroll loops. This is essential for Aria to implement high-level control structures (like IF/ELSE state machines) that define local labels without colliding with the surrounding assembly context.1
* Semantic Safety: The "Zig-style comptime" requirement implies the ability to execute language code during compilation. This is necessary to handle Aria's "Exotic Types" (TBB, Trit, Tryte). A standard constant folder cannot correctly evaluate tbb8 arithmetic because it does not understand that -128 is a sentinel error value (ERR) rather than a valid integer. Only a semantic interpreter configured with Aria's specific arithmetic rules can safely perform this evaluation.1

3. Phase 0 Implementation: The Hygienic Macro Preprocessor
The Preprocessor is the gatekeeper of the Aria compiler. It must process the incoming source stream, resolving includes, expanding macros, and managing hygiene before the code reaches the lexer. The implementation detailed here replaces the simple file-reading loop with a sophisticated state machine.
3.1 The Context Stack Mechanism
A critical deficiency in naive preprocessors is the lack of "scope." In NASM, and subsequently in Aria, this is resolved via the Context Stack. The preprocessor maintains a stack of contexts, each representing a logical scope (e.g., a loop body, a class definition, or an if block).
3.1.1 Context Directives and State Transitions
The system implements three primary directives to manipulate this stack:
* %push <context_name>: Pushes a new context onto the stack. This initiates a new scope.
* %pop: Removes the top context. This terminates the current scope.
* %repl <context_name>: Replaces the top context. This is crucial for state transitions where nesting is not appropriate, such as moving from an if block to an else block.
3.1.2 Algorithmic Resolution of Context-Local Labels
To allow macros to generate control flow labels without collision, the preprocessor implements a label mangling algorithm. Any label defined with a %$ prefix is treated as context-local.
The Resolution Algorithm:
1. Context ID Generation: Upon %push, the preprocessor generates a unique 64-bit identifier ($ID_{ctx}$) using a monotonic counter.
2. Stack Association: The context name ($Name_{ctx}$) and $ID_{ctx}$ are pushed to the stack.
3. Label Detection: During token scanning, the preprocessor detects tokens matching the pattern ^%\$([a-zA-Z0-9_]+).
4. Mangling: The token is rewritten to __ctx_<ID>_<Name>_<Label>. For example, %$start inside a loop context (ID 42) becomes __ctx_42_loop_start.
5. Scope Enforcement: If a %$ label is encountered while the stack is empty (except for the global context), the preprocessor emits a compilation error, preventing "orphaned" labels.
This mechanism allows a WHILE macro to define %$start and %$end labels. Even if WHILE is nested inside another WHILE, the unique context IDs ensure the labels resolve to their specific loop instance.1
3.2 Automatic Hygiene: The Gensym Pattern
Variable collision is the scourge of macro programming. If a macro SWAP(a, b) declares int temp, invoking it twice in the same scope causes a redefinition error. Aria solves this via Automatic Hygiene, implementing the Lisp "gensym" (generated symbol) pattern directly in the preprocessor.
3.2.1 The Thread-Safe Identification Strategy
The implementation utilizes a std::atomic<uint64_t> counter to ensure that every macro expansion—even across parallel compilation threads—receives a unique expansion ID ($ID_{exp}$).
The Hygiene Algorithm 1:
1. Declaration Scanning: When a macro body is being expanded, the preprocessor scans the token stream for variable declarations. It utilizes a regex pattern \b([a-zA-Z0-9_]+)\s*:\s*([a-zA-Z0-9_]+)\b to identify Aria's Type:Name syntax (e.g., int32:temp or wild int8*:ptr).
2. Keyword Filtering: The captured name is checked against a ReservedKeywords set (containing if, while, func, etc.) and the macro parameter list (%1, %2). Matches are excluded from renaming to preserve control flow and argument validity.
3. ID Generation: A generic unique ID is fetched: uint64_t id = global_hygiene_counter.fetch_add(1);.
4. Symbol rewriting: The variable name temp is rewritten to temp_h<ID> (e.g., temp_h4096).
5. Reference Propagation: All subsequent occurrences of temp within the same expansion instance are rewritten to temp_h4096, ensuring internal consistency.
This ensures that SWAP(x, y) generates temp_h1 and a subsequent SWAP(a, b) generates temp_h2, eliminating collision risks without requiring user intervention.
3.3 Token Rotation and Variadic Arguments
To support the complex, JSON-like syntax required for the Aria Class System (e.g., CLASS { field: type, method:... }), the preprocessor requires the ability to consume tokens sequentially. The %rotate directive provides this capability.
Operational Semantics:
%rotate N shifts the argument list of the macro. %1 takes the value of %2, %2 takes %3, and so on. The original %1 is moved to the end of the list or discarded depending on the specific implementation mode. This primitive enables recursive macro expansion:
1. A macro PROCESS_FIELDS checks if %1 is empty.
2. If not, it generates code for the field defined in %1.
3. It invokes %rotate 1.
4. It calls PROCESS_FIELDS again.
This allows Aria macros to iterate over arbitrary lists of arguments, transforming them into struct fields or function parameters.1
3.4 C++ Implementation: The Preprocessor Engine
The following C++ implementation integrates these concepts into a cohesive Preprocessor class. It is designed to be dropped into src/frontend/preprocessor/preprocessor.cpp.


C++




/**
* src/frontend/preprocessor/preprocessor.cpp
* 
* Aria Compiler - Phase 0 Engine
* Implements NASM-style Context Stack, Automatic Hygiene, and Token Rotation.
*/

#include "frontend/preprocessor/preprocessor.h"
#include <regex>
#include <sstream>
#include <atomic>
#include <iostream>
#include <stack>
#include <set>
#include <map>
#include <vector>

namespace aria {
namespace frontend {

// Atomic counter for global hygiene (Gensym)
// Ensures uniqueness across 18 quintillion expansions 
static std::atomic<uint64_t> global_hygiene_counter{0};

// Context structure for the stack
struct MacroContext {
   std::string name;
   uint64_t id;
   
   MacroContext(std::string n, uint64_t i) : name(std::move(n)), id(i) {}
};

struct MacroDefinition {
   std::string name;
   std::vector<std::string> params;
   std::vector<std::string> body;
   bool is_hygienic = true;
};

class Preprocessor {
private:
   std::map<std::string, MacroDefinition> macros_;
   std::map<std::string, std::string> defines_;
   std::vector<MacroContext> context_stack_;
   
   const size_t MAX_RECURSION = 1000;
   size_t recursion_depth_ = 0;

public:
   Preprocessor() {
       // Initialize global context
       context_stack_.emplace_back("global", 0);
   }

   // Primary entry point
   std::string process_stream(std::istream& input) {
       std::stringstream output;
       std::string line;
       
       while (std::getline(input, line)) {
           // Directive Handling
           if (line.find('%')!= std::string::npos) {
               if (handle_directive(line)) continue; 
           }
           
           // Macro Expansion
           line = expand_macros(line);
           
           // Context Label Resolution
           line = resolve_labels(line);
           
           output << line << "\n";
       }
       return output.str();
   }

private:
   bool handle_directive(const std::string& line) {
       std::stringstream ss(line);
       std::string token;
       ss >> token;

       if (token == "%push") {
           std::string ctx_name;
           ss >> ctx_name;
           uint64_t id = global_hygiene_counter.fetch_add(1);
           context_stack_.emplace_back(ctx_name, id);
           return true;
       }
       else if (token == "%pop") {
           if (context_stack_.size() > 1) { // Prevent popping global
               context_stack_.pop_back();
           } else {
               throw std::runtime_error("Preprocessor Context Underflow");
           }
           return true;
       }
       else if (token == "%repl") {
           std::string ctx_name;
           ss >> ctx_name;
           if (context_stack_.size() > 1) {
               context_stack_.pop_back();
               uint64_t id = global_hygiene_counter.fetch_add(1);
               context_stack_.emplace_back(ctx_name, id);
           }
           return true;
       }
       //... Additional directives (%define, %macro)...
       return false;
   }

   std::string resolve_labels(std::string line) {
       // Pattern: %$label
       static std::regex lbl_re(R"(%\$([a-zA-Z0-9_]+))");
       
       uint64_t ctx_id = context_stack_.back().id;
       std::string ctx_name = context_stack_.back().name;
       
       // Replacement: __ctx_ID_NAME_label
       std::stringstream ss;
       std::regex_replace(std::ostreambuf_iterator<char>(ss),
                          line.begin(), line.end(), lbl_re,
                          "__ctx_" + std::to_string(ctx_id) + "_" + ctx_name + "_$1");
       return ss.str();
   }

   std::string expand_macros(std::string line) {
       // Simplified tokenization for macro detection
       std::stringstream ss(line);
       std::string token;
       std::string result;
       
       while (ss >> token) {
           if (macros_.count(token)) {
               // Collect arguments...
               std::vector<std::string> args = collect_args(ss);
               result += perform_expansion(macros_[token], args);
           } else {
               result += token + " ";
           }
       }
       return result; // Simplified
   }

   std::string perform_expansion(const MacroDefinition& def, std::vector<std::string> args) {
       if (++recursion_depth_ > MAX_RECURSION) 
           throw std::runtime_error("Max Macro Recursion Exceeded");

       std::string expansion;
       uint64_t exp_id = global_hygiene_counter.fetch_add(1);

       for (const auto& body_line : def.body) {
           std::string temp = body_line;
           
           // %rotate logic would be applied here to the 'args' vector
           // creating a new view of arguments for the current expansion step
           
           // 1. Argument Substitution (%1, %2...)
           for (size_t i = 0; i < args.size(); ++i) {
               std::string p = "%" + std::to_string(i + 1);
               // Replace all occurrences of p with args[i]
               size_t pos = 0;
               while ((pos = temp.find(p, pos))!= std::string::npos) {
                   temp.replace(pos, p.length(), args[i]);
                   pos += args[i].length();
               }
           }
           
           expansion += temp + "\n";
       }

       // 2. Hygienic Renaming
       if (def.is_hygienic) {
           expansion = apply_hygiene(expansion, exp_id);
       }

       recursion_depth_--;
       return expansion;
   }

   std::string apply_hygiene(const std::string& input, uint64_t id) {
       // Detect "Type:Name" declarations
       // Captures: 1=Type, 2=Name
       static std::regex decl_re(R"(\b([a-zA-Z0-9_]+)\s*:\s*([a-zA-Z0-9_]+)\b)");
       
       std::map<std::string, std::string> renames;
       auto begin = std::sregex_iterator(input.begin(), input.end(), decl_re);
       auto end = std::sregex_iterator();

       for (auto i = begin; i!= end; ++i) {
           std::string var = (*i);
           if (is_reserved(var)) continue;
           renames[var] = var + "_h" + std::to_string(id);
       }

       std::string output = input;
       for (const auto& [orig, ren] : renames) {
           // Replace whole words only
           std::regex var_re("\\b" + orig + "\\b");
           output = std::regex_replace(output, var_re, ren);
       }
       return output;
   }

   bool is_reserved(const std::string& s) {
       static std::set<std::string> k = {"if", "while", "wild", "gc", "int8", "tbb64"};
       return k.count(s);
   }
   
   //... Helper: collect_args...
};

} // namespace frontend
} // namespace aria

4. Phase 2 Implementation: The Comptime Execution Engine
The second pillar of the architecture is the Comptime Engine. While the Preprocessor handles syntax, Comptime handles semantics. It executes after parsing but before code generation, filling the role of a semantic interpreter.
4.1 Integration with CodeGenContext
The Comptime engine does not exist in isolation. It relies heavily on the backend/codegen_context.h infrastructure 1 to understand the Aria type system.
* Type Mapping: The engine uses CodeGenContext::getLLVMType to map abstract Aria types (like int8 or tbb64) into concrete representations.
* Symbol Table: It maintains a shadow symbol table that mirrors the scopeStack in CodeGenContext, but instead of storing llvm::Value*, it stores ComptimeValue variants (constants).
* Borrow Checker: The engine must respect the ScopedVariable tracking 1 to ensure that compile-time code does not violate ownership rules, particularly when simulating wild pointers.
4.2 The TBB-Aware Arithmetic Logic Unit (ALU)
Aria's core differentiator is its use of Twisted Balanced Binary (TBB) arithmetic. TBB types have a "Sticky Error" property: if a calculation overflows or encounters a sentinel, the result becomes ERR, and any subsequent operation with ERR yields ERR.1
The host machine (the compiler) uses standard binary arithmetic (Two's Complement). Therefore, the Comptime Engine cannot simply execute a + b using C++ operators. It must emulate TBB logic.

The ConstEvaluator class implements an evalBinaryOp method. This method first checks if either operand is ERR. If so, it returns ERR. If not, it performs the operation using standard math but checks for overflow relative to the TBB domain (e.g., -127 to +127 for tbb8). If the result falls into the sentinel value (-128), it is marked as ERR.
4.3 The Virtual Heap: Simulating Wild Memory
Aria allows manual memory management (wild pointers). To support comptime blocks that allocate memory (e.g., generating lookup tables), the engine must simulate a heap.
* Virtual Allocator: Instead of malloc, the engine allocates a std::vector<uint8_t> and returns a handle (AllocID, Offset).
* Safety Checks: Every pointer access is bounds-checked against the vector size. This allows the compiler to detect buffer overflows at compile time.
* Leak Detection: At the end of a comptime block, the engine checks if all wild allocations have been freed. If not, it emits a compile-time error, enforcing RAII discipline even in manual memory contexts.1
4.4 C++ Implementation: The ConstEvaluator
This implementation defines the ConstEvaluator class, filling the gap identified in src/frontend/sema/const_evaluator.cpp.


C++




/**
* src/frontend/sema/const_evaluator.h
* 
* Aria Compiler - Phase 2 Engine
* Implements TBB-Aware Arithmetic and Virtual Memory Sandbox.
*/

#include "backend/codegen_context.h"
#include <variant>
#include <vector>
#include <map>
#include <limits>

namespace aria {
namespace sema {

// Represents a value resolved at compile time
struct ComptimeValue {
   enum class Kind { INT, FLOAT, BOOL, POINTER, ERR_SENTINEL, TYPE };
   Kind kind;
   
   // Type-safe union for value storage
   using PtrHandle = std::pair<uint32_t, uint64_t>; // AllocID, Offset
   std::variant<int64_t, double, bool, PtrHandle> val;
   
   std::string type_tag; // e.g. "tbb8", "int32"
   int bit_width;
   
   bool is_tbb() const { return type_tag.find("tbb") == 0; }
};

class ConstEvaluator {
private:
   // Virtual Heap Sandbox
   struct AllocBlock {
       std::vector<uint8_t> data;
       bool is_freed = false;
   };
   std::map<uint32_t, AllocBlock> heap_;
   uint32_t next_alloc_id_ = 1;

public:
   // -------------------------------------------------------------------------
   // TBB-Aware Arithmetic Unit
   // -------------------------------------------------------------------------
   ComptimeValue eval_bin_op(const std::string& op, ComptimeValue lhs, ComptimeValue rhs) {
       // 1. Sticky Error Propagation
       if (lhs.kind == ComptimeValue::Kind::ERR_SENTINEL |

| 
           rhs.kind == ComptimeValue::Kind::ERR_SENTINEL) {
           return make_err(lhs.type_tag, lhs.bit_width);
       }

       // 2. Integer Logic
       if (lhs.kind == ComptimeValue::Kind::INT) {
           int64_t l = std::get<int64_t>(lhs.val);
           int64_t r = std::get<int64_t>(rhs.val);
           int64_t res = 0;
           bool overflow = false;

           if (op == "+") overflow = __builtin_add_overflow(l, r, &res);
           else if (op == "-") overflow = __builtin_sub_overflow(l, r, &res);
           else if (op == "*") overflow = __builtin_mul_overflow(l, r, &res);

           // TBB Range Checking
           if (lhs.is_tbb()) {
               int64_t min, max, sentinel;
               get_tbb_limits(lhs.bit_width, min, max, sentinel);

               // If hardware overflow OR result outside TBB range
               if (overflow |

| res < min |
| res > max) {
                   return make_err(lhs.type_tag, lhs.bit_width);
               }
               
               // Explicit Sentinel Collision check
               if (res == sentinel) return make_err(lhs.type_tag, lhs.bit_width);
           }
           
           return make_int(res, lhs.type_tag, lhs.bit_width);
       }
       
       throw std::runtime_error("Unsupported Comptime Operation");
   }

   // -------------------------------------------------------------------------
   // Virtual Memory Manager
   // -------------------------------------------------------------------------
   ComptimeValue virt_alloc(size_t size) {
       uint32_t id = next_alloc_id_++;
       heap_[id] = AllocBlock{std::vector<uint8_t>(size, 0), false};
       return make_ptr(id, 0);
   }

   void virt_free(ComptimeValue ptr) {
       auto handle = std::get<ComptimeValue::PtrHandle>(ptr.val);
       if (heap_.find(handle.first) == heap_.end() |

| heap_[handle.first].is_freed) {
           throw std::runtime_error("Double Free or Invalid Pointer in Comptime");
       }
       heap_[handle.first].is_freed = true;
   }

   // -------------------------------------------------------------------------
   // Reflection (@typeInfo)
   // -------------------------------------------------------------------------
   ComptimeValue get_type_info(const std::string& type_name, backend::CodeGenContext* ctx) {
       // Query the backend context for struct metadata 
       if (ctx->structFieldMaps.count(type_name)) {
           // Construct a TypeInfo object (represented as a struct in the virtual heap)
           return build_type_info_struct(type_name, ctx->structFieldMaps[type_name]);
       }
       return make_primitive_info(type_name);
   }

private:
   void get_tbb_limits(int bits, int64_t& min, int64_t& max, int64_t& sentinel) {
       // TBB8: -127..127, Sentinel -128
       if (bits == 8) { sentinel = -128; }
       else if (bits == 16) { sentinel = -32768; }
       else if (bits == 32) { sentinel = INT32_MIN; }
       else { sentinel = INT64_MIN; }
       
       min = sentinel + 1;
       max = -(min);
   }

   ComptimeValue make_err(std::string tag, int bits) {
       return {ComptimeValue::Kind::ERR_SENTINEL, 0, tag, bits};
   }
   
   ComptimeValue make_int(int64_t v, std::string tag, int bits) {
       return {ComptimeValue::Kind::INT, v, tag, bits};
   }

   ComptimeValue make_ptr(uint32_t id, uint64_t off) {
       return {ComptimeValue::Kind::POINTER, std::pair{id, off}, "wild ptr", 64};
   }
   
   // Helper to construct TypeInfo in virtual heap...
   ComptimeValue build_type_info_struct(std::string name, std::map<std::string, unsigned>& fields) {
       // Implementation details omitted for brevity
       return make_ptr(0, 0); 
   }
};

} // namespace sema
} // namespace aria

5. Synthesis: The Complete Compilation Pipeline
The proposed architecture integrates these components into a robust pipeline:
1. Ingestion (Phase 0): The Preprocessor reads the .aria source. It executes %include, expands macros, maintains the Context Stack for %push/%pop, and applies Gensym Hygiene. The output is a linear, clean stream of tokens.
2. Parsing (Phase 1): The Compiler builds the AST. comptime blocks are parsed into ComptimeBlock nodes but not yet compiled.
3. Analysis (Phase 2): The Type Checker traverses the AST. When it encounters a ComptimeBlock, it hands control to the ConstEvaluator.
   * The Evaluator executes the code using the Virtual Heap and TBB ALU.
   * It may invoke getTypeInfo to reflect on types defined in CodeGenContext.
   * It returns a result (e.g., a number, a type, or injected AST nodes).
4. Generation (Phase 3): The fully resolved AST is passed to CodeGenContext for LLVM IR generation.
5.1 Advanced Use Case: The Class Macro
By combining Phase 0 and Phase 2, Aria can implement classes without core language support.
* Phase 0: A %macro CLASS utilizes %rotate to parse a JSON-like argument list. It generates a standard struct definition and mangled helper functions (e.g., MyClass_init).
* Phase 2: A comptime block inspects the generated struct using @typeInfo. It verifies that all fields are valid and auto-generates serialization logic or virtual tables if required.
This "Dual-Phase" approach satisfies the requirement for high-level ergonomics (macros) and low-level safety (comptime validation), delivering on the promise of the Aria ecosystem.
