Architectural Specification for the Implementation of TBB Sticky Error Lowering Strategies in the Aria Compiler Backend
1. Introduction: The Imperative of Safety in Integer Arithmetic
The design and implementation of modern systems programming languages are increasingly defined by their approach to safety. Where legacy languages like C and C++ prioritized raw performance and hardware mapping—often at the cost of undefined behavior and security vulnerabilities—the Aria programming language posits a new paradigm: "Safe by Semantics." At the heart of this paradigm lies the Twisted Balanced Binary (TBB) type system, a revolutionary approach to integer arithmetic that eliminates the historical asymmetries of two's complement representation and introduces a robust, mathematically rigorous error propagation model.
This report serves as the definitive architectural specification for the TBB Sticky Error Lowering Strategy. It addresses a critical deficiency in the current Aria compiler version v0.0.7: the absence of automatic translation (lowering) from high-level TBB arithmetic expressions to safety-enforced LLVM Intermediate Representation (IR). Currently, the compiler's CodeGenContext correctly tracks TBB types via metadata but fails to utilize this information during instruction emission, defaulting to standard, unsafe integer wrapping.1 This behavior violates the core safety guarantees of the language, where an operation like tbb8: 127 + 1 should yield a sticky ERR sentinel rather than wrapping to -128.
The scope of this document encompasses the complete design of the TBBLowerer backend subsystem. It details the theoretical underpinnings of TBB arithmetic, the specific LLVM IR sequences required for validation and overflow detection, and the complex Control Flow Graph (CFG) modifications necessary to implement "sticky" short-circuiting. By enforcing these checks at the compiler level, we move the burden of safety from the programmer's cognitive load to the compiler's mechanical rigor, ensuring that every arithmetic operation is mathematically sound or explicitly erroneous.
2. Theoretical Framework: The TBB Type System
To architect a correct lowering strategy, one must first rigorously define the mathematical domain of TBB types. The "Twisted" nature of these types refers to their deviation from standard hardware integers to achieve symmetry and safety.
2.1 The Asymmetry of Two's Complement
Standard signed integers in virtually all modern CPU architectures use two's complement representation. For a bit-width $N$, the representable range is $[-2^{N-1}, 2^{N-1}-1]$.
* 8-bit (int8): Range $[-128, +127]$.
* 16-bit (int16): Range $[-32768, +32767]$.
This representation is fundamentally asymmetric: there is one more negative number than positive number. The value $-2^{N-1}$ (e.g., -128) has no positive counterpart. This creates a "singular point" in arithmetic logic:
1. Negation Failure: The operation -(-128) cannot be represented. In C/C++, this is often Undefined Behavior (UB).
2. Absolute Value Failure: abs(-128) returns -128 (identity) or traps, as +128 is out of range.
3. Division Anomaly: INT_MIN / -1 attempts to produce INT_MAX + 1, causing a hardware trap (SIGFPE) on x86 architectures.1
2.2 The TBB Symmetric Domain
Aria's TBB types resolve this by artificially restricting the domain to be perfectly symmetric. A TBB type of width $W$ is defined over the set of integers:




$$\mathbb{Z}_{TBB} = \{ z \in \mathbb{Z} \mid -(2^{W-1} - 1) \le z \le (2^{W-1} - 1) \}$$
This definition explicitly excludes the minimum representable value of the underlying storage container.
* tbb8: Range $[-127, +127]$. The value -128 (binary 10000000) is excluded from the number line.
* tbb16: Range $[-32767, +32767]$. The value -32768 is excluded.
By enforcing this symmetry, Aria guarantees that the negation operator is closed over the set $\mathbb{Z}_{TBB}$. For all valid $x$, $-x$ is also valid. This eliminates the entire class of bugs related to INT_MIN handling.
2.3 The Error Sentinel (ERR)
The excluded bit pattern is not wasted; it is repurposed as a dedicated Error Sentinel, denoted as ERR. This value is topologically distinct from the number line. It represents "invalid data," "overflow," or "computation failure."
Sentinel Value Definitions 1:
* tbb8: ERR = -128 (0x80)
* tbb16: ERR = -32768 (0x8000)
* tbb32: ERR = -2,147,483,648 (0x80000000)
* tbb64: ERR = INT64_MIN (0x8000000000000000)
2.4 Sticky Error Semantics
The "Sticky Error" principle dictates that the ERR value acts as an absorbing element in the algebra of TBB types. Once a computation enters the ERR state—whether via input propagation, overflow, or explicit assignment—it cannot leave that state through standard arithmetic.
For any binary operator $\odot$ (add, sub, mul, div):




$$x \odot y = \begin{cases} \text{ERR} & \text{if } x = \text{ERR} \lor y = \text{ERR} \\ \text{ERR} & \text{if } \text{calc}(x, y) \notin \mathbb{Z}_{TBB} \\ \text{calc}(x, y) & \text{otherwise} \end{cases}$$
The lowering strategy is essentially the mechanical translation of this piecewise function into executable machine code (LLVM IR). The "Sticky" property ensures that errors propagate downstream, allowing a developer to check for validity at the end of a computation chain rather than after every step, significantly improving code readability without sacrificing safety.2
3. Backend Architecture: The TBBLowerer Class
The implementation of TBB support requires a dedicated component within the Aria compiler's backend. This component, the TBBLowerer, acts as a bridge between the semantic richness of the frontend and the low-level instructions of the LLVM IR Builder.
3.1 Context Analysis: The CodeGenContext
The CodeGenContext class 1 is the central repository of state during compilation. It manages the symbol table, scope stack, and most critically, the exprTypeMap.
* exprTypeMap: A std::map<Value*, std::string> that associates every LLVM Value* with its high-level Aria type name (e.g., "tbb8"). This map is the "Source of Truth" for the TBBLowerer. Since LLVM IR is type-erased regarding signedness (an i8 is just 8 bits), the TBBLowerer relies entirely on this map to decide when to inject safety checks.
* Current Deficiency: The current visit(BinaryOp) implementation likely checks this map but fails to diverge into a safe code generation path, simply calling builder->CreateAdd. The remediation involves intercepting this call.
3.2 Class Design and interface
The TBBLowerer is designed as a stateless helper class (or lightweight stateful wrapper around the Builder) that encapsulates the logic for generating safe arithmetic.


C++




/**
* src/backend/ir/tbb_lowerer.h
* 
* Encapsulates the logic for lowering TBB arithmetic operations into 
* safe, checked LLVM IR sequences.
*/
class TBBLowerer {
private:
   llvm::LLVMContext& context;
   llvm::IRBuilder<>& builder;
   llvm::Module* module;
   
   // Internal helper to get the ERR sentinel for a given width
   llvm::Value* getSentinel(llvm::Type* type);

public:
   TBBLowerer(llvm::LLVMContext& ctx, llvm::IRBuilder<>& bld, llvm::Module* mod)
       : context(ctx), builder(bld), module(mod) {}

   // Predicate to check if a type requires TBB lowering
   static bool isTBBType(const std::string& typeName);

   // Primary Lowering Methods
   llvm::Value* createAdd(llvm::Value* lhs, llvm::Value* rhs);
   llvm::Value* createSub(llvm::Value* lhs, llvm::Value* rhs);
   llvm::Value* createMul(llvm::Value* lhs, llvm::Value* rhs);
   llvm::Value* createDiv(llvm::Value* lhs, llvm::Value* rhs); // Includes Div-by-Zero check
   
   // Unary Lowering
   llvm::Value* createNeg(llvm::Value* val);
   llvm::Value* createAbs(llvm::Value* val);
};

3.3 Integration Point: ExprCodegen
The TBBLowerer is instantiated inside the ExprCodegen::visit(BinaryExpr*) method.


C++




// Pseudocode for Integration
llvm::Value* ExprCodegen::visit(BinaryExpr* expr) {
   llvm::Value* L = expr->left->accept(this);
   llvm::Value* R = expr->right->accept(this);
   
   std::string typeL = context.exprTypeMap[L];
   
   if (TBBLowerer::isTBBType(typeL)) {
       TBBLowerer tbb(context.llvmContext, *context.builder, context.module.get());
       switch (expr->op) {
           case TOKEN_PLUS: return tbb.createAdd(L, R);
           case TOKEN_MINUS: return tbb.createSub(L, R);
           //...
       }
   }
   // Fallback to standard unsafe math
   return context.builder->CreateAdd(L, R);
}

4. Lowering Logic and Sentinel Injection
This section details the instruction-by-instruction generation logic. The prompt explicitly requires the specification for injecting icmp eq %val, -128 checks.
4.1 Input Validation (The Pre-Check)
Before any arithmetic occurs, the operands must be validated. If an operand is ERR, the operation is conceptually invalid.
* Mechanism: Generate icmp eq instructions against the sentinel.
* Logic:
Code snippet
%sentinel = -128
%l_is_err = icmp eq i8 %lhs, %sentinel
%r_is_err = icmp eq i8 %rhs, %sentinel
%inputs_bad = or i1 %l_is_err, %r_is_err

This boolean %inputs_bad flag will later control the result selection or control flow.
4.2 Arithmetic with Overflow Detection
Standard LLVM add instructions wrap on overflow. To implement TBB safety, we must detect when the mathematical result exceeds the $N$-bit storage. We utilize LLVM Intrinsics for this purpose.4
   * Intrinsic: @llvm.sadd.with.overflow.i8 (for tbb8).
   * Return Type: A struct { i8, i1 } where the first element is the wrapped result and the second is the overflow bit.
   * Usage:
Code snippet
%res_struct = call {i8, i1} @llvm.sadd.with.overflow.i8(%lhs, %rhs)
%raw_val = extractvalue {i8, i1} %res_struct, 0
%overflow_bit = extractvalue {i8, i1} %res_struct, 1

4.3 The Sentinel Collision Problem
Detecting signed overflow is necessary but not sufficient. We must also detect Sentinel Collision.
      * Scenario: Consider tbb8. Range is $[-127, +127]$. Sentinel is -128.
      * Case 1 (Overflow): 127 + 1. Mathematical result 128. Representation in two's complement is -128. LLVM sets overflow_bit = true. We catch this.
      * Case 2 (Underflow): -127 - 1. Mathematical result -128. Representation is -128. LLVM does not set overflow_bit because -128 is a valid int8.
      * Crucial Insight: In int8, -128 is valid. In tbb8, -128 is ERR. Therefore, the intrinsic thinks the operation succeeded, but semantically, it failed (result is sentinel).
      * Solution: We must explicitly check if the result equals the sentinel, regardless of the overflow bit.
Code snippet
%is_sentinel_pattern = icmp eq i8 %raw_val, -128

4.4 Aggregating the Failure State
The final "is error" condition is a logical OR of all failure modes:
         1. Input was ERR.
         2. Hardware/Intrinsic Overflow occurred.
         3. Result collided with Sentinel bit pattern.


Code snippet




%fail_1 = or i1 %inputs_bad, %overflow_bit
%fail_final = or i1 %fail_1, %is_sentinel_pattern

4.5 Result Selection (Dataflow Approach)
In a basic implementation without short-circuiting (see Section 5), the result is selected using the select instruction (conditional move).


Code snippet




%final_result = select i1 %fail_final, i8 -128, i8 %raw_val

This ensures that if any error condition is met, the result is forced to ERR.
5. Control Flow Graph (CFG) Modifications for Short-Circuiting
The prompt specifically requests "CFG modifications required to implement 'sticky' error propagation where any ERR input short-circuits the operation."
While select instructions are efficient for pipelining, they still perform the arithmetic operation (the intrinsic call). "Short-circuiting" implies skipping the operation entirely if the inputs are invalid. This is strictly required when the right-hand operand has side effects (e.g., a + function_call()) or when the operation is expensive (e.g., division).
5.1 The BasicBlock Split Strategy
To implement short-circuiting, we must modify the CFG by splitting the current basic block into three:
         1. Entry Block: Checks inputs. Branches to Error or Compute.
         2. Compute Block: Performs the math. Branches to Exit.
         3. Exit Block: Merges the results using a $\phi$ (Phi) node.
5.2 Detailed CFG Construction Algorithm
The TBBLowerer will manipulate the IRBuilder to inject these blocks dynamically.
Step 1: Input Validation in Current Block


Code snippet




Label: entry
 %lhs_err = icmp eq i8 %a, -128
 br i1 %lhs_err, label %merge_block, label %check_rhs_block

Note: Strictly speaking, if we want to short-circuit the evaluation of RHS, that logic belongs in the AST traversal (skipping rhs->accept()). If we assume operands are already evaluated (loaded into registers), we short-circuit the arithmetic instruction.
Step 2: The Compute Block (Safe Zone)
If inputs are valid, we enter the compute block.


Code snippet




Label: compute_block
 %res_struct = call @llvm.sadd...
 %val = extractvalue...
 %ovf = extractvalue...
 %collision = icmp eq %val, -128
 %bad_math = or %ovf, %collision
 %safe_val = select %bad_math, -128, %val
 br label %merge_block

Step 3: The Merge Block (Phi Node)
The merge block reconciles the two paths: one where we skipped math (input was ERR), and one where we did math (and potentially found a new error).


Code snippet




Label: merge_block
 %final_res = phi i8 [ -128, %entry ], [ %safe_val, %compute_block ]

5.3 Short-Circuiting Operand Evaluation
A deeper level of short-circuiting involves the CodeGenVisitor. If the expression is A + B and A evaluates to ERR, we should not even execute the code to compute B. This mirrors the behavior of && and || in C++.
CFG Design for Full Short-Circuiting:
         1. Eval A Block: Compute A. Check if ERR.
         * If ERR: Jump to Merge Block (push ERR to Phi).
         * If Valid: Jump to Eval B Block.
         2. Eval B Block: Compute B. Check if ERR.
         * If ERR: Jump to Merge Block (push ERR to Phi).
         * If Valid: Jump to Arithmetic Block.
         3. Arithmetic Block: Perform A + B. Check Overflow/Collision.
         * Jump to Merge Block (push Result/ERR).
         4. Merge Block: Phi node with 3 incoming edges.
This approach is computationally optimal for TBB chains but significantly increases code size and branch prediction pressure. For standard arithmetic (a + b where a, b are locals), the select approach (Section 4) is preferred. For complex expressions or function calls, this CFG splitting is mandatory.
6. Implementation Detail: The C++ TBBLowerer Class
The following C++ implementation integrates the theoretical logic into a concrete LLVM IR generator. It assumes the select strategy for instruction-level operations but can be adapted for branching.
6.1 TBBLowerer Implementation (src/backend/ir/tbb_lowerer.cpp)


C++




#include "backend/codegen_context.h"
#include "llvm/IR/Intrinsics.h"

using namespace llvm;

namespace aria::backend {

// Helper to get the sentinel constant (e.g., -128 for i8)
Value* TBBLowerer::getSentinel(Type* type) {
   unsigned width = type->getIntegerBitWidth();
   // Signed minimum value: 100...0
   return ConstantInt::get(context, APInt::getSignedMinValue(width));
}

Value* TBBLowerer::createAdd(Value* lhs, Value* rhs) {
   Type* type = lhs->getType();
   Value* sentinel = getSentinel(type);

   // 1. Inject Input Checks (Sticky Logic)
   // Generates: icmp eq %lhs, -128
   Value* lhsIsErr = builder.CreateICmpEQ(lhs, sentinel, "lhs_is_err");
   Value* rhsIsErr = builder.CreateICmpEQ(rhs, sentinel, "rhs_is_err");
   Value* inputErr = builder.CreateOr(lhsIsErr, rhsIsErr, "input_err");

   // 2. Arithmetic with Overflow Intrinsic
   // We use the intrinsic to catch standard overflow (e.g., 127 + 1)
   Function* intrinsic = Intrinsic::getDeclaration(module, 
                                                   Intrinsic::sadd_with_overflow, 
                                                   {type});
   Value* resStruct = builder.CreateCall(intrinsic, {lhs, rhs}, "sadd");
   Value* rawRes = builder.CreateExtractValue(resStruct, 0, "raw_res");
   Value* ovfBit = builder.CreateExtractValue(resStruct, 1, "ovf_bit");

   // 3. Sentinel Collision Check
   // Catch cases like (-127 - 1) which produces -128 (valid in i8, ERR in tbb8)
   // Even though sadd_with_overflow might say "no overflow", the result is ERR.
   Value* resIsSentinel = builder.CreateICmpEQ(rawRes, sentinel, "res_collision");

   // 4. Combine Errors
   Value* mathErr = builder.CreateOr(ovfBit, resIsSentinel, "math_err");
   Value* totalErr = builder.CreateOr(inputErr, mathErr, "sticky_err");

   // 5. Select Final Result
   // If any error occurred, return Sentinel. Otherwise raw result.
   return builder.CreateSelect(totalErr, sentinel, rawRes, "tbb_res");
}

// Subtraction is nearly identical, using ssub_with_overflow
Value* TBBLowerer::createSub(Value* lhs, Value* rhs) {
   //... (similar setup)...
   Function* intrinsic = Intrinsic::getDeclaration(module, 
                                                   Intrinsic::ssub_with_overflow, 
                                                   {type});
   //... (rest is identical logic)...
}

// Division requires Branching for safety (No Trap)
Value* TBBLowerer::createDiv(Value* lhs, Value* rhs) {
   Type* type = lhs->getType();
   Value* sentinel = getSentinel(type);
   Value* zero = ConstantInt::get(type, 0);
   Value* minusOne = ConstantInt::get(type, -1);

   // 1. Input Checks
   Value* lhsIsErr = builder.CreateICmpEQ(lhs, sentinel);
   Value* rhsIsErr = builder.CreateICmpEQ(rhs, sentinel);
   Value* inputErr = builder.CreateOr(lhsIsErr, rhsIsErr);

   // 2. Div-by-Zero Check
   Value* isZero = builder.CreateICmpEQ(rhs, zero);

   // 3. Overflow Check (MIN / -1 traps on x86)
   Value* lhsIsMin = builder.CreateICmpEQ(lhs, sentinel); // Sentinel is MIN
   Value* rhsIsMinusOne = builder.CreateICmpEQ(rhs, minusOne);
   Value* isOverflow = builder.CreateAnd(lhsIsMin, rhsIsMinusOne);

   // 4. Safe Execution Logic (Prevent Trap)
   // If unsafe, we divide by 1 (dummy op) and return ERR later
   Value* isUnsafe = builder.CreateOr(isZero, isOverflow);
   
   // Select safe RHS: if unsafe, use 1, else use rhs
   Value* safeRhs = builder.CreateSelect(isUnsafe, ConstantInt::get(type, 1), rhs);
   
   // Perform Division
   Value* rawRes = builder.CreateSDiv(lhs, safeRhs);

   // 5. Final Selection
   Value* logicErr = builder.CreateOr(inputErr, isUnsafe);
   
   // Note: Division result cannot collide with Sentinel unless it was the Overflow case
   // which we already caught in step 3. 
   return builder.CreateSelect(logicErr, sentinel, rawRes, "tbb_div");
}

} // namespace aria::backend

7. Optimization and Edge Case Verification
7.1 Minimizing Overhead via LLVM Passes
The generated code contains many icmp checks. While correct, this is verbose.
         * Redundancy: If we compute a + b + c, we check a and b. The result t1 is checked against sentinel. Then t1 is checked again as input to the second add.
         * Optimizer: Standard LLVM passes (InstCombine, SimplifyCFG) will aggressively optimize this. Since t1 comes from a select that outputs sentinel on error, LLVM's range analysis (ValueTracking) can often deduce that t1 is safe or known-sentinel, folding subsequent checks.
         * Custom TBB Pass: We recommend a custom LLVM pass that recognizes the TBB intrinsic patterns and performs "Global Sticky Optimization," hoisting error checks out of loops.1
7.2 Verification Scenarios
To validate the design, we trace execution of edge cases:
         * Case A: 127 + 1 (tbb8)
         * Intrinsic sadd returns -128, overflow=1.
         * mathErr becomes true (via overflow).
         * Result: select returns -128 (ERR). Correct.
         * Case B: -127 - 1 (tbb8)
         * Intrinsic ssub returns -128, overflow=0 (valid int8).
         * resIsSentinel checks -128 == -128. Returns true.
         * mathErr becomes true (via collision).
         * Result: select returns -128 (ERR). Correct.
         * Case C: ERR + 5
         * lhsIsErr is true.
         * inputErr is true.
         * Result: select returns -128 (ERR). Correct.
8. Conclusion
The implementation of the TBBLowerer class represents a fundamental shift in the Aria compiler's backend, moving from naive instruction emission to a safety-enforcing code generation pipeline. By systematically injecting sentinel validation, utilizing intrinsics for overflow detection, and employing conditional selection for sticky error propagation, the compiler guarantees that the runtime behavior matches the strict TBB specification.
This architecture satisfies the critical requirements:
         1. Sentinel Injection: Explicit icmp eq val, -128 logic is central to input and result validation.
         2. Sticky Propagation: Implemented via logical OR chains and select instructions, ensuring errors are irreversible.
         3. CFG Modification: While the default path uses branchless selection for performance, the design supports block splitting for true short-circuiting of complex operands.
The deployment of this logic will immediately resolve the current a + b wrapping bugs, elevating Aria to a truly memory-safe and arithmetic-safe language suitable for high-reliability systems programming.
