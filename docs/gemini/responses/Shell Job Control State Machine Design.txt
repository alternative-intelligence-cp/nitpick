Architectural Specification: AriaSH Job Manager State Machine and Hex-Stream Controller Integration
1. Executive Summary and Theoretical Foundation
The architecture of the command-line shell has remained largely static since the standardization of POSIX job control in the 1980s. While robust for the teletype-era systems for which they were designed, traditional shells (Bash, Zsh, Tcsh) suffer from fundamental architectural deficiencies when placed in the context of modern, high-concurrency, distributed systems. The Aria Shell (AriaSH) represents a ground-up redesign of the shell paradigm, specifically engineered to support the Aria programming language's unique features—Twisted Balanced Binary (TBB) arithmetic, "Wild" memory models, and a typed, six-stream I/O topology.
This specification details the design of the AriaSH Job Manager, the kernel-integrated subsystem responsible for the lifecycle management of processes. It departs from the synchronous fork-exec-wait loop of legacy shells, adopting instead a fully asynchronous, event-driven architecture powered by modern OS primitives such as Linux pidfd and Windows Job Objects. Central to this design is the integration of the Hex-Stream Controller, a threaded subsystem that manages six parallel channels of I/O (stdin, stdout, stderr, stddbg, stddati, stddato) to resolve the "Noisy Channel" problem and prevent pipe deadlocks in complex orchestration pipelines.
The analysis presented herein defines the rigorous state machine governing process states (Foreground, Background, Stopped), the hybrid signal handling logic necessitated by the shell's "Raw Mode" input architecture, and the precise kernel-level mechanisms required to implement this system on both POSIX and Windows NT kernels.
1.1 The Legacy Job Control Paradigm and Its Failures
To appreciate the architectural necessity of the AriaSH Job Manager, one must first deconstruct the limitations of the existing POSIX job control model. The traditional Unix shell manages jobs—collections of processes sharing a Process Group ID (PGID)—through a synchronous mechanism deeply tied to the TTY line discipline.
In a standard shell:
1. Foregrounding: The shell calls tcsetpgrp() to grant a specific PGID exclusive write access to the terminal.1
2. Blocking: The shell calls waitpid(), effectively sleeping until the child process changes state.
3. Signal Interception: Signals like SIGINT (Ctrl+C) and SIGTSTP (Ctrl+Z) are interpreted by the kernel's TTY driver and delivered directly to the foreground process group. The shell typically does not see these signals unless it is monitoring for them specifically.3
This model presents three critical failure modes in modern computing:
* PID Recycling Race Conditions: Standard waitpid operations rely on numeric PIDs. In containerized environments with high process churn, PIDs wrap around quickly. There exists a race window where a child process exits, its PID is reaped, and the kernel reassigns that PID to a new, unrelated process before the shell handles the exit signal. This can lead to the shell sending signals to the wrong process.5
* The Noisy Channel: Standard stdout is critically overloaded. It carries user prompts, progress bars, and binary data simultaneously. This prevents the reliable composition of tools; a single debug message printed to stdout corrupts a binary pipeline.7
* Synchronous Blocking: The blocking nature of waitpid is incompatible with the high-throughput, async-first nature of the Aria runtime. A shell that sleeps cannot service other I/O events, specifically the telemetry and data streams required by the Hex-Stream topology.7
1.2 The AriaSH Architectural Shift
AriaSH addresses these issues by inverting the control model. It moves the shell from a synchronous supervisor to an asynchronous orchestrator.
* Handle-Based Identity: Processes are tracked not by PID, but by stable file descriptors (pidfd on Linux) or Kernel Handles (Windows). These handles guarantee identity stability even if the underlying OS identifier is recycled.7
* Hex-Stream Topology: The shell manages six distinct streams per process, separating control messages (text) from data payloads (binary) and telemetry (structured logs). This allows a background compiler to emit detailed JSON logs to stddbg without corrupting the object code flowing through stddato.7
* Raw Mode Signal Mediation: The shell forces the terminal into Raw Mode, disabling the kernel's automatic signal generation (ISIG). The shell intercepts every keystroke, including control chords, and generates signals programmatically. This gives the Job Manager absolute authority over signal delivery, preventing the ambiguity of TTY-driven signaling.7
2. The Hex-Stream Controller Architecture
The Hex-Stream Controller (StreamController) is the data plane of the Job Manager. While the Job Manager handles state transitions, the StreamController handles the flow of bytes. Its primary design goal is to prevent "Pipe Deadlock"—a scenario where a child process blocks because it has filled its output buffer, while the parent process is blocked waiting for the child to exit, resulting in a system hang.

2.2 The Threaded Draining Model (The Active Pump)
To manage these streams without blocking the main event loop, the StreamController employs a Threaded Draining Model.
* Worker Allocation: For every spawned process, the controller initializes a pool of std::jthread workers. Typically, this involves four threads per process, dedicated to reading stdout, stderr, stddbg, and stddato.7
* Continuous Pumping: These threads perform blocking read() calls on the process's output pipes. As soon as data appears, it is moved into a User-Space Ring Buffer.
* Deadlock Prevention: The kernel pipe buffer on Linux is typically 64KB. If a process writes 65KB and nobody reads it, the process sleeps. By having dedicated threads that do nothing but read, the StreamController ensures the kernel buffer is always drained. This decouples the child process's execution speed from the shell's processing speed.7
2.3 Zero-Copy Optimization (Linux)
For the binary streams (stddati and stddato), the StreamController implements an optimization path utilizing the Linux splice() system call.
* Mechanism: When the shell detects a pipe connecting Process A's stddato to Process B's stddati, it instructs the kernel to move pages directly between the pipe buffers without copying data into the shell's user-space memory.
* Efficiency: This reduces context switches and memory bandwidth usage, aligning with Aria's performance goals for "wild" memory operations.7
2.4 Stream Buffering Strategies
The StreamController applies different buffering logic based on the Job State:
* Foreground Jobs: stdout and stderr bypass the ring buffer and are written directly to the TTY to minimize latency. stddbg is buffered to allow for post-execution analysis.
* Background Jobs: All output streams are drained into ring buffers. If a background job writes to stdout, the data is captured but not displayed immediately to avoid corrupting the user's prompt. The shell may trigger a notification ("Job 1 has output pending").7
3. The Job Manager State Machine
The Job Manager maintains a rigorous Finite State Machine (FSM) for every job under its control. This FSM is the authoritative source of truth for process status, dictating how the shell interacts with the TTY and the OS kernel.
3.1 The Job Control Block (JCB)
Each job is represented internally by a JobControlBlock structure. This structure aggregates the process handles, PGID, state metadata, and the associated StreamController instance.


C++




struct JobControlBlock {
   uint32_t job_id;                // Shell-assigned ID (e.g., %1)
   pid_t pgid;                     // Process Group ID
   std::vector<ProcessHandle> processes; // PIDs or Handles in the pipeline
   
   // State Tracking
   std::atomic<JobState> state;    // FG, BG, STOPPED, TERMINATED
   std::atomic<bool> notified;     // Notification flag for status changes
   
   // Resources
   std::unique_ptr<StreamController> streams; // Hex-Stream manager
   termios saved_tmodes;           // Terminal modes specific to this job
   
   // Logic
   int exit_code;                  // Aggregated exit code (Result<T>)
};

3.2 State Definitions
3.2.1 FOREGROUND (State: FG)
The FOREGROUND state implies exclusive ownership of the terminal's input capabilities.
* Terminal Ownership: The Job's PGID is the designated foreground group of the controlling terminal. This is established via tcsetpgrp(tty_fd, job_pgid) on POSIX systems.1
* Input Access: The job has unrestricted access to stdin. Reads do not trigger SIGTTIN.
* Signal Disposition: The shell actively propagates signals generated by the user (Ctrl+C, Ctrl+Z) to this job.
* Hex-Stream Behavior: stdout and stderr are wired to the physical terminal. stddbg continues to be buffered by the StreamController.
3.2.2 BACKGROUND (State: BG)
The BACKGROUND state implies asynchronous execution. The job runs concurrently with the shell's main loop.
* Terminal Ownership: The shell (or another job) owns the terminal.
* Input Access: Reads from stdin are prohibited. If a background process attempts to read, the kernel sends SIGTTIN, suspending the process.
* Output Behavior: Writes to stdout and stderr are captured by the StreamController's active pumps into ring buffers. This prevents the "Noisy Channel" effect where background output garbles the foreground prompt.7
* Execution: The process is scheduled by the OS and running.
3.2.3 STOPPED (State: STP)
The STOPPED state represents a suspended execution context.
* Trigger: Transition occurs via SIGTSTP (user suspend), SIGSTOP (kernel enforcement), or SIGTTIN/SIGTTOU (background I/O violation).
* Stream Behavior: The StreamController threads attempting to drain the pipes will eventually block on read() as the suspended process stops producing data. This is a natural, resource-efficient pause; explicit thread suspension is not required.7
* Resumption: Transition out of this state requires a SIGCONT signal.
3.2.4 TERMINATED (State: TERM)
The job has ceased execution.
* Cleanup: The Job Manager must reap the exit status (preventing zombies), close all Hex-Stream pipes, and release the pidfd or Windows Handles.
* Error Propagation: The exit code is analyzed. If it corresponds to a TBB ERR sentinel (e.g., -128), the shell flags a "Sticky Error".7

	3.4 Raw Mode and the Input State Machine
AriaSH operates in Raw Mode, meaning the TTY's ISIG flag is disabled. The kernel does not automatically send SIGINT or SIGTSTP. The shell's Input Engine acts as the intermediary.
* Input State Machine: The input engine cycles through Idle, Chord Analysis, and Submission states.
* Implication: When a user types Ctrl+Z, it is not a signal; it is a byte sequence (0x1A). The Input Engine parses this, determines it is a "Suspend" command, and then instructs the Job Manager to execute the transition logic (sending the actual signal). This allows the shell to implement complex logic (like double-tap Ctrl+C) that standard line disciplines cannot support.7
4. Signal Handling and Orchestration Logic
The handling of signals in AriaSH is hybrid: it combines the precision of modern file-descriptor-based monitoring with the necessity of legacy signal propagation for child processes.
4.1 SIGINT (Interrupt / Ctrl+C)
* Standard Behavior: Terminate the foreground process.
* AriaSH Logic:
   1. Input: User presses Ctrl+C. Input Engine captures byte 0x03.
   2. Check Context: The Job Manager checks if there is an active FOREGROUND job.
   3. Action (Job Exists): The shell calls kill(-pgid, SIGINT). The negative PGID ensures the signal is delivered to the entire pipeline (e.g., grep | sort), not just the leader.
   4. Action (No Job): If the shell is idle, the Input Engine cancels the current input buffer and renders a new prompt (aria> ), mimicking the behavior of zsh or fish.10
   5. Shell Immunity: The shell process itself installs a SIG_IGN or a specialized handler for SIGINT to ensure it never terminates itself upon receiving the signal meant for a child.
4.2 SIGTSTP (Suspend / Ctrl+Z)
* Standard Behavior: Suspend the foreground process and return control to the shell.
* AriaSH Logic:
   1. Input: User presses Ctrl+Z. Input Engine captures 0x1A.
   2. Action: The shell calls kill(-pgid, SIGTSTP).
   3. Confirmation: The shell must confirm the stop. It uses waitpid(pid, &status, WUNTRACED) or polls the pidfd. When the kernel reports WIFSTOPPED(status), the transition is committed.11
   4. TTY Restoration: The Job Manager saves the child's terminal modes (tcgetattr) and restores the shell's own modes (tcsetattr), then calls tcsetpgrp(shell_pgid) to reclaim the foreground.
   5. Notification: The shell prints + Stopped <command>.
4.3 SIGCHLD and Asynchronous Reaping
Legacy shells rely on a signal handler for SIGCHLD to detect when children die. This is prone to race conditions (signals can be coalesced). AriaSH abandons this for a file-descriptor-based approach.
* The pidfd Strategy (Linux):
   * Upon spawning, the shell obtains a pidfd for the child.7
   * This pidfd is registered with the shell's main epoll instance with the EPOLLIN event flag.
   * When the child terminates, the pidfd becomes readable. The epoll_wait loop wakes up, and the shell processes the specific job associated with that FD. This provides precise, O(1) complexity notification without signal handler overhead.5
* State Changes (Stop/Continue): While pidfd handles termination perfectly, detecting state changes (Stopped/Continued) sometimes still requires SIGCHLD. AriaSH uses a "Self-Pipe" or signalfd integration where the SIGCHLD handler writes a byte to a specialized fd monitored by epoll, triggering a non-blocking waitid scan to update status flags without blocking the main thread.14
4.4 SIGTTIN and SIGTTOU
These signals enforce the TTY access rules.
* Shell Immunity: The shell itself runs in the background while a job is foregrounded. If the shell attempts to write to the TTY (e.g., a debug log) during this time, it would receive SIGTTOU and suspend itself, causing a deadlock. Therefore, AriaSH must mask these signals (SIG_IGN) in its own process context.15
* Handling: When a background job triggers these, it stops. The Job Manager detects the state change (via SIGCHLD/waitid) and updates the job state to STOPPED.
5. OS-Specific Implementation Strategies
The Job Manager essentially acts as a hardware abstraction layer over the OS's process primitives. The implementation diverges significantly between Linux and Windows.
5.1 Linux Implementation (The pidfd/epoll Model)
The Linux backend leverages features available in kernel 5.3+.
1. Process Creation:
   * The shell creates the Hex-Stream pipes using pipe2(fds, O_CLOEXEC).
   * It calls clone3() with the CLONE_PIDFD flag. This atomic syscall returns the child PID and the pidfd simultaneously, eliminating the race condition where a PID is recycled before the parent can open a handle to it.5
   * Inside the child: setpgid(0, 0) establishes the new process group. dup2 maps the pipes to FDs 0-5.
2. Terminal Handoff:
   * The parent calls setpgid(child_pid, child_pid) to ensure the group exists (redundancy against race conditions).
   * The parent calls tcsetpgrp(tty_fd, child_pid) to hand over the TTY.
3. Event Loop Integration:
   * The pidfd is added to the epoll set. The StreamController adds the read-ends of the pipes to epoll as well.
   * The main loop handles data events (draining pipes) and lifecycle events (process exit) in a unified manner.
5.2 Windows Implementation (Job Objects & IOCP)
Windows lacks fork, exec, and process groups. AriaSH emulates these features using the Windows NT Object Manager.
1. Hex-Stream Inheritance:
   * Standard CreateProcess allows inheriting handles, but it's "all or nothing" by default.
   * AriaSH uses InitializeProcThreadAttributeList with PROC_THREAD_ATTRIBUTE_HANDLE_LIST. This allows the shell to explicitly whitelist exactly the 6 handles (Hex-Stream) to be inherited, securing the child sandbox.7
   * Bootstrap Protocol: Since Windows handles are not fixed integers (like 0, 1, 2), AriaSH sets an environment variable __ARIA_FD_MAP=3:<HandleVal>;4:<HandleVal>.... The Aria Runtime (aria_rt.dll) inside the child process parses this on startup and wraps these handles into the io module's file objects.7
2. Job Objects as Process Groups:
   * The shell creates a Job Object for every logical job.
   * AssignProcessToJobObject binds the child process to the Job.
   * This allows the shell to terminate the entire pipeline (TerminateJobObject) or suspend it, mimicking Unix signal behavior.
3. Asynchronous Waiting:
   * The Job Object is associated with the shell's I/O Completion Port (IOCP).
   * When a process in the job exits, the kernel queues a JOB_OBJECT_MSG_EXIT_PROCESS packet to the IOCP. The shell's event loop pops this packet to handle job termination.7
6. Integration with Aria Language Semantics
The Job Manager is not just a system utility; it is a runtime component of the Aria language.
6.1 Exit Codes and TBB Safety
Aria uses Twisted Balanced Binary (TBB) integers which include an explicit error sentinel (ERR or -128 for tbb8).
* Sticky Error Integration: When a process exits, the Job Manager retrieves the status code. If the exit code maps to the TBB ERR value, the shell does not simply treat it as "Command Failed" (like Bash's $?!= 0). Instead, it raises a Sticky Error Violation. This halts the entire orchestration script immediately, preventing the error from propagating silently through subsequent logic.7
6.2 Typed Injection and the "&" Operator
In traditional shells, the & operator is syntax for backgrounding. In AriaSH, backgrounding is a property of the spawn command, while & is repurposed for Typed Injection in variable substitution.
* Concept: To prevent command injection (rm -rf / inside a filename variable), AriaSH uses &{VAR}. The Job Manager parses this and passes the variable content as a single argument to the exec syscall, ignoring any spaces or metacharacters contained within. This guarantees that spawn("ls", ["&{file}"]) is safe regardless of the content of file.7
7. Performance and Security Analysis
7.1 Throughput Analysis
The Threaded Draining Model significantly outperforms standard shell redirection in high-throughput scenarios.
* Metric: By using splice() on the Data Plane (stddati/stddato), AriaSH achieves near-native memory bandwidth for IPC, as data never crosses the user-kernel boundary.
* Latency: The use of epoll and pidfd eliminates the polling overhead of waitpid in tight loops, allowing the shell to manage thousands of concurrent background jobs with negligible CPU impact.
7.2 Security Considerations
* Handle Leaks: The Windows implementation's use of PROC_THREAD_ATTRIBUTE_HANDLE_LIST prevents the accidental leakage of sensitive handles (e.g., open sockets, file locks) from the shell to the child, a common vulnerability in CreateProcess implementations.7
* Signal Spoofing: By operating in Raw Mode and handling signals programmatically, AriaSH prevents "Signal Spoofing" attacks where malicious output sequences could trick a TTY into generating signals for the shell.
8. Conclusion
The AriaSH Job Manager represents a radical departure from forty years of shell design tradition. By rejecting the synchronous waitpid model in favor of pidfd and epoll, and by replacing the "Noisy Channel" of stdout with a typed Hex-Stream topology, it provides a robust foundation for modern systems programming. The state machine defined herein ensures that foreground and background jobs coexist without resource contention, while the deep integration with Aria's TBB types and memory model ensures that this performance does not come at the cost of safety. This architecture successfully bridges the gap between the ergonomic needs of an interactive shell and the rigorous requirements of a system automation runtime.
