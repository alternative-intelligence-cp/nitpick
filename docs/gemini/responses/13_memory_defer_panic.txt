SIL-4 Safety Audit Report: Aria Compiler Backend, Type System Integrity, and Runtime Resilience
1. Executive Summary and Compliance Determination
This document presents a comprehensive, SIL-4 (Safety Integrity Level 4) safety audit of the Aria programming language compiler backend and associated runtime architecture. The audit was conducted under the strictest possible parameters, treating the target system—a physics-based Artificial General Intelligence (AGI) designed for "Teacher" roles with vulnerable populations—as a life-critical application. The primary objective was to verify the integrity of memory safety mechanisms, specifically defer statement execution, panic unwinding, and resource cleanup guarantees, ensuring that no software failure could result in uncontrolled actuator movement or physical harm to a human subject.
The audit methodology relied on a forensic examination of the provided source code artifacts, specifically the Intermediate Representation (IR) generation logic found in src/backend/ir/codegen_expr.cpp, the semantic analysis definitions in src/frontend/sema/, and the runtime library implementations in src/runtime/. The analysis was guided by the "Fail Hard" philosophy mandated by the project requirements: assumptions were prohibited, and all unknowns were treated as potential catastrophic failure modes.
1.1 Critical Non-Compliance Finding: Missing Artifacts
The audit identified a critical gap in the provided verification materials. The specific target files requested for audit—src/backend/ir/codegen_stmt.cpp (responsible for defer statement generation) and src/backend/ir/codegen.cpp (responsible for the main compilation loop and potentially panic unwinding)—were listed in the repository directory structure but were absent from the source content provided for inspection.
Without src/backend/ir/codegen_stmt.cpp, it is technically impossible to verify the implementation of the LIFO (Last-In, First-Out) execution order for defer blocks, nor can we confirm that defer statements are deterministically injected into all exit paths (returns, breaks, and panics). Consequently, Compliance Item 1 (Defer statement code generation) and Compliance Item 5 (Defer with early returns) cannot be verified as safe. Under SIL-4 protocols, unverifiable code is functionally equivalent to unsafe code.
1.2 Audit Verdict
Current Status: ❌ NON-COMPLIANT / UNVERIFIABLE
The system cannot be certified for deployment in robotic assistants interacting with children in its current state. While the backend demonstrates high-quality defensive engineering in its type system and arithmetic generation, the absence of proof regarding the defer cleanup mechanism constitutes an unacceptable risk. If the cleanup logic is missing or flawed, a runtime error could leave a robotic limb efficiently powered and moving (due to the robust arithmetic) but effectively lobotomized (due to the control loop crashing without disengaging motors).
1.3 High-Assurance Findings (Positive Safety Indicators)
Despite the missing files, the deep analysis of src/backend/ir/codegen_expr.cpp and the runtime environment revealed several "Best in Class" safety features that suggest a strong architectural foundation:
* Limb-Based Integral Model (LBIM): The compiler implements a custom, rigorous handling of large integers (int128 through int4096) that purposefully bypasses LLVM's standard optimization passes to avoid known backend bugs. This indicates a "paranoia-first" approach suitable for SIL-4.1
* Deterministic Physics (fix256): The implementation of a Q128.128 fixed-point type (fix256) as a first-class citizen guarantees bit-identical physics calculations across different hardware architectures, preventing "butterfly effect" drifts in motion planning.1
* Twisted Balanced Binary (TBB) Safety: The backend generates branchless, constant-time logic for TBB arithmetic that propagates "sticky" error sentinels. This prevents timing side-channels and ensures that arithmetic errors (like overflow) are captured deterministically.1
* Recursion Depth Guarding: The expression generator includes explicit stack depth checks (MAX_EXPR_DEPTH) to prevent stack overflow Denial of Service (DoS) attacks or accidents during the compilation of deeply nested expressions.1
The following sections detail the rigorous analysis of these components, the inferred risks regarding the missing defer logic, and the mandatory remediation steps required to achieve certification.

2. Threat Model: The "Teacher" System and Actuator Control
To understand the severity of the findings, one must contextualize the software within the physical reality of the "Teacher" system. This is not a desktop application; it is the central nervous system of a machine capable of exerting force on fragile human bodies.
2.1 The Kinetic Consequence of Software Failure
In a standard software environment, a "memory corruption" or "type error" results in a segmentation fault or a crash report. In the context of a robotic assistant for neurodivergent children or hospital patients, these abstract software concepts translate into kinetic energy.
* Scenario A: The Frozen Actuator (Defer Failure)
   * Trigger: The AGI attempts to parse a student's vocal response. The audio buffer is malformed, triggering a runtime panic (index out of bounds) in the listen() function.
   * Mechanism: The listen() function was called inside a scope where the robot's arm was explicitly enabled to gesture. A defer motor_disable() statement was registered to ensure the arm relaxes after the gesture.
   * Failure: Due to a bug in codegen_stmt.cpp (currently unverified), the panic unwinding mechanism bypasses the defer stack. The program terminates the control loop, but the process does not exit immediately, or the OS driver retains the last command.
   * Outcome: The motor remains locked at the last commanded torque. If the child bumps into the arm, it does not yield (impedance control is dead). The rigid limb becomes a battering ram.
* Scenario B: The Force Limit Bypass (Type Error)
   * Trigger: The physics engine calculates the required force to lift a toy. The result is a precise fix256 value.
   * Mechanism: Due to a compiler bug in type widening (specifically checked in Audit Item 4), the 256-bit fixed-point value is implicitly cast to a 32-bit integer, truncating the fractional part or overflowing the sign bit.
   * Failure: The value 0.0001 (hold gently) is interpreted as 1000 (crush) due to misalignment of the decimal point during the cast.
   * Outcome: The robotic hand closes with maximum hydraulic pressure, crushing the object or the child's hand.
* Scenario C: The Drifting Teacher (Determinism Failure)
   * Trigger: The system is trained on an x86 server farm but deployed on an ARM-based robot.
   * Mechanism: Standard floating-point math (double) has minute differences in rounding behaviors between architectures.
   * Failure: Over thousands of simulation steps, these micro-drifts accumulate. The AGI believes the student is 1.5 meters away (safe zone), but physical reality has drifted such that the student is 0.5 meters away (contact zone).
   * Outcome: The robot executes a "pat on the head" gesture that connects with the student's face.
2.2 SIL-4 Compliance Matrix
Safety Integrity Level 4 demands that the probability of dangerous failure per hour of operation be less than $10^{-9}$ (continuous) or $10^{-5}$ to $10^{-4}$ (on demand). To achieve this, the software architecture must demonstrate:
1. Completeness: All cleanup code must execute.
2. Correctness: Arithmetic must not produce silent errors.
3. Containment: Failures must be trapped before reaching the actuators.
This audit evaluates the Aria compiler against these rigid constraints.

3. Deep Analysis of Type System Integrity and Code Generation
While the statement generation logic is missing, the expression generation logic (src/backend/ir/codegen_expr.cpp) is available and contains critical information regarding how data is represented and manipulated. This file is the "ground truth" for the Type System Breach threat.
3.1 The "Fail Hard" Policy on Unknown Types
Audit Item: Verify EVERY type mapping. Fail hard on unknowns. No silent defaults.
Status: ✅ COMPLIANT
Analysis:
Legacy compilers often default undefined integer types to i32 (32-bit integer) to allow compilation to proceed. In a safety-critical context, this is unacceptable. A review of ExprCodegen::getLLVMType 1 reveals a strict adherence to the "Fail Hard" requirement.
The code explicitly implements safeguards against "Type System Breaches" (referenced as ARIA-026).


C++




// ARIA-026: SAFETY FIX - Fail hard on non-primitive types instead of defaulting to i32
// Gemini Safety Audit Fix #1: Type System Breach
if (!type->isPrimitive()) {
   if (type->isPointer()) {
       return llvm::PointerType::get(context, 0);
   }
   throw std::runtime_error("ICE: Code generation encountered unmapped non-primitive type kind: " + 
                           std::to_string(static_cast<int>(type->getKind())) + 
                           ". This is a compiler bug - all types must be explicitly mapped.");
}

Furthermore, within the primitive type mapping, there is a second catch-all clause:


C++




// ARIA-026: SAFETY FIX - No silent i32 default for unknown types
throw std::runtime_error("ICE: Code generation encountered unknown primitive type: '" + type_name + 
                       "'. This is a compiler bug - all primitive types must be explicitly mapped.");

Implication:
This confirms that the compiler will panic (crash itself) rather than generate incorrect machine code if it encounters a corrupted or unknown type definition. This is the desired behavior for a safety-critical toolchain. It eliminates the risk of a 16-byte struct being silently truncated to 4 bytes, which prevents stack corruption that could lead to arbitrary code execution or variable overwrites.
3.2 Limb-Based Integral Model (LBIM) for Large Integers
Audit Item: Numerical Stability and Precision.
Status: ✅ COMPLIANT (Architecturally)
Analysis:
The system uses int128, int256, up to int4096 for high-precision calculations. Relying on LLVM's native support for arbitrary-width integers (e.g., i4096) is risky because rare optimization passes in LLVM (like IPSCCP or ConstraintElimination) have historically contained bugs when handling non-standard bit widths.
The Aria backend implements a custom Limb-Based Integral Model (LBIM).1


C++




// ARIA-024: Large integers use Limb-Based Integral Model (LBIM)
// Stored as structs of i64 limbs to bypass LLVM IPSCCP/ConstraintElimination bugs
if (typeName == "int128" |

| typeName == "uint128") {
   llvm::StructType* int128Struct = llvm::StructType::getTypeByName(context, "struct.int128");
   if (!int128Struct) {
       llvm::Type* limbArray = llvm::ArrayType::get(llvm::Type::getInt64Ty(context), 2);
       int128Struct = llvm::StructType::create(context, {limbArray}, "struct.int128");
   }
   return int128Struct;
}

This pattern repeats for all large integer types up to int4096 (64 limbs). By treating these numbers as arrays of i64 (standard 64-bit integers) and presumably using runtime library calls for arithmetic (aria_lbim_add, etc., inferred from function names in the source), the compiler bypasses the opaque optimization logic of LLVM.
Safety Insight:
This shifts the burden of correctness from the LLVM optimizer (which is vast and opaque) to the Aria Runtime Library (which is smaller and auditable). While the runtime library source for LBIM arithmetic was not provided, the decision to use LBIM is a strong positive signal for safety. It ensures that a compiler optimization update will not silently break the cryptography or physics calculations.
3.3 Deterministic Fixed-Point Physics (fix256)
Audit Item: Simulation-Reality Parity.
Status: ✅ COMPLIANT
Analysis:
The file codegen_expr.cpp reveals a dedicated type fix256 designed specifically for deterministic physics.


C++




// ARIA-025: fix256 deterministic fixed-point (Q128.128 for physics simulations)
if (typeName == "fix256") {
   // fix256 uses 4-limb structure: [2 x i64] integer part + [2 x i64] fractional part
   llvm::Type* limbArray = llvm::ArrayType::get(llvm::Type::getInt64Ty(context), 4);
   fix256Struct = llvm::StructType::create(context, {limbArray}, "struct.fix256");
   return fix256Struct;
}

Implication:
Floating-point arithmetic (IEEE 754) is notoriously non-deterministic across different CPU architectures and compiler optimization levels. A simulation run on a developer's x86 workstation might yield different trajectories than the same code running on an ARM-based robot controller.
By implementing fix256 (Q128.128 fixed-point) as a struct of integers, the Aria compiler forces the math to be bit-exact everywhere. This guarantees that if the "Teacher" AGI simulates a movement as safe, the physical execution will match that simulation exactly, removing "drift" as a threat vector.

4. The Twisted Balanced Binary (TBB) Safety Protocol
One of the most unique and critical safety features identified in the codegen_expr.cpp file is the implementation of Twisted Balanced Binary (TBB) arithmetic. This system provides a sophisticated mechanism for error handling that avoids the pitfalls of exceptions (non-deterministic timing) and flags (easy to ignore).
4.1 "Sticky Error" Propagation
Mechanism:
TBB types (tbb8, tbb16, tbb32, tbb64) are symmetric signed integers where the most negative value (e.g., -128 for int8) is reserved as a specialized ERR sentinel.
The compiler generates code that checks inputs for this sentinel. If any operand is ERR, or if an operation overflows, the result becomes ERR.
Source Evidence (generateTBBBinaryOp) 1:


C++




// Step 1: Input validation - check if either operand is ERR
llvm::Value* lhsIsErr = builder.CreateICmpEQ(left, sentinel, "lhs_is_err");
llvm::Value* rhsIsErr = builder.CreateICmpEQ(right, sentinel, "rhs_is_err");
llvm::Value* inputInvalid = builder.CreateOr(lhsIsErr, rhsIsErr, "input_invalid");

//... (Operation logic using intrinsics)...

// Step 5: Branchless select - return sentinel if any failure, else return mathResult
return builder.CreateSelect(failFinal, sentinel, mathResult, "tbb_result");

4.2 Branchless Logic and Timing Determinism
Audit Finding:
The implementation uses builder.CreateSelect (mapping to the cmov or csel assembly instructions) rather than conditional branches (br).
Safety Insight:
This is critical for a real-time control loop. Branching (if/else) introduces variable execution time depending on whether the branch is taken or not (due to pipeline flushes and branch prediction failures).
* With Branches: A division-by-zero error might take less time to execute than a successful division, causing the control loop to finish early and potentially destabilize the jitter.
* With Branchless Select: The error path takes the exact same number of CPU cycles as the success path. The control loop timing remains rigid and predictable regardless of data validity.
4.3 The Vulnerability: Unchecked Propagation
Status: ❌ POTENTIAL BUG / RISK
Risk Analysis:
While the arithmetic logic correctly propagates the error sentinel, there is no evidence in ExprCodegen that the system enforces a check of this sentinel before the value is used in a critical context.
If a tbb64 value resulting in ERR (the bit pattern for INT64_MIN) is passed to an external C function (e.g., a hardware driver set_motor_voltage(int64_t v)), the C function will likely interpret INT64_MIN as "Maximum Negative Voltage" rather than "Error".
Threat Scenario:
1. tbb64 speed = calculate_speed(); // Returns ERR due to divide-by-zero.
2. hal_set_motor(speed); // Compiler effectively casts TBB64_ERR to int64_t.
3. Robot interprets 0x8000... as max reverse speed.
4. Robot snaps backwards instantly.
Required Remediation:
The compiler must implicitly inject a check whenever a TBB type is cast to a non-TBB type or passed to an extern function. This check logic is not visible in ExprCodegen. It is possible it resides in ExprCodegen::codegenCall (which calls codegenExpressionNode), but without the full implementation of call handling or cast handling, this remains a severe open risk.

5. Audit of Defer and Resource Cleanup (The Missing Link)
This section addresses the primary focus of the user's request. Due to the absence of codegen_stmt.cpp, we must perform a gap analysis based on the available interfaces.
5.1 Audit Item 1: Defer Statement Code Generation
Requirement: Verify defer blocks registered correctly; check LIFO order; confirm execution on all exit paths.
Status: ❌ UNVERIFIABLE (Critical Failure)
Architectural Inference:
The ExprCodegen class references stmt_codegen.


C++




void ExprCodegen::setStmtCodegen(StmtCodegen* stmt_gen) {
   stmt_codegen = stmt_gen;
}

This confirms that StmtCodegen exists and likely manages the scope stack. However, without the code, we cannot confirm:
1. Memory Allocation: Does defer allocate closure context on the heap? If so, what happens if that allocation fails? A safety-critical system cannot rely on malloc for control flow integrity. It should ideally use stack allocation (alloca) for defer contexts.
2. Panic Hooks: Does the panic handler walk the stack and invoke these defers? codegen_expr.cpp uses C++ exceptions (std::runtime_error) for compiler errors, but the generated code (LLVM IR) needs its own unwinding mechanism (e.g., landingpad instructions in LLVM). There is no evidence of landingpad generation in the expression code, which is expected, but its absence in the provided materials leaves the panic safety unproven.
5.2 Audit Item 2: Defer Block Scope Management
Requirement: Verify defers scoped to function/block; check nested defers.
Status: ⚠️ WARNING: Stack Allocation Risks
Evidence:
In ExprCodegen::codegenTemplateLiteral 1, we see direct use of alloca:


C++




if (totalSegments <= MAX_STACK_SEGMENTS) {
   stringsArray = builder.CreateAlloca(arrayType, nullptr, "template_strings");
}

Risk:
The compiler uses alloca for temporary storage. If defer also uses alloca to store captured variables for the cleanup block, and if a defer is placed inside a loop (e.g., while(true) { defer... }), the stack frame will grow indefinitely until a stack overflow occurs.
* SIL-4 Requirement: The compiler must either forbid defer inside loops or implement llvm.stacksave / llvm.stackrestore at the loop body boundaries. We cannot verify this protection exists.
5.3 Audit Item 4: Resource Cleanup Guarantees
Requirement: Verify file handles/sockets closed; memory freed.
Status: ❌ Bug Found (Implementation Detail in String Interning)
Analysis:
The file codegen_expr.cpp implements string interning to save memory (ARIA-026).


C++




// String literal - create global AriaString struct
//...
llvm::GlobalVariable* str_gv = new llvm::GlobalVariable(...);
string_pool_[str] = str_gv; // String interning

The Flaw:
While this optimization is standard for static compilation, the "Teacher" system implies a long-running, potentially adaptive AI process. If this compiler is part of a JIT (Just-In-Time) compilation loop running on the robot to generate new lesson plans, the string_pool_ represents a permanent memory leak. llvm::GlobalVariable objects are generally not reclaimed until the entire LLVM module is destroyed.
* Threat: Over weeks of operation, the string pool consumes all available RAM. The OOM killer terminates the control process mid-operation.
* Fix: The JIT compiler context must implement a garbage collection strategy for interned strings or scope their lifetime to the lesson plan module.

6. Runtime Environment and Memory Safety
The provided snippets included headers for the runtime environment (src/runtime/), allowing us to audit the "Wild" and "GC" memory models referenced in the user's specs.
6.1 WildX Executable Memory (W^X Security)
File: include/runtime/allocators.h 1
Analysis:
The runtime defines a WildXGuard for JIT compilation.


C




typedef enum { 
   WILDX_STATE_WRITABLE = 1, 
   WILDX_STATE_EXECUTABLE = 2 
} WildXState;

This enforces the W^X (Write XOR Execute) security policy. Memory is either writable (for generating code) or executable (for running it), never both simultaneously.
* Safety Verdict: ✅ COMPLIANT. This prevents JIT-spraying attacks where an attacker (or a corrupted neural net output) writes malicious bytecode and executes it. This is crucial for a system that "learns" and modifies its behavior.
6.2 The "Nursery" Allocator
File: src/runtime/gc/allocator.cpp 1
Analysis:
The GC uses a "Nursery" with a bump-pointer allocator.


C++




void* new_bump_ptr = (char*)alloc_ptr + total_size;
if (new_bump_ptr <= end_addr) {... }

Safety Verdict: ✅ COMPLIANT.
Bump allocation is extremely fast (pointer addition) and deterministic compared to complex free-list allocators like malloc. Using mmap for the nursery ensures memory is acquired in large, contiguous pages. This reduces heap fragmentation, which is a major cause of non-deterministic failure in long-running robotic systems.

7. Audit of Panic Unwinding (Item 3)
Status: ❌ NON-COMPLIANT
Requirement: Verify panic triggers defer execution before stack unwind.
Evidence:
The codegen_expr.cpp file does not generate invoke instructions (LLVM's exception-handling call instruction) but rather standard call instructions.


C++




// Example from generateTBBBinaryOp
llvm::Value* resStruct = builder.CreateCall(saddFunc, {left, right}, "sadd_ovf");

Implication:
If the called function panics, the standard call instruction assumes the program terminates or the panic is handled internally. Without invoke and associated landingpad blocks, the LLVM IR does not support C++-style stack unwinding for these operations.
* Hypothesis: Aria uses a "panic = abort" model or a setjmp/longjmp style mechanism, or the TBB "sticky error" model is intended to replace exceptions entirely.
* The Danger: If Aria relies on TBB errors instead of unwinding, then defer blocks will NOT run automatically when a math error occurs. The error simply propagates as a value.
* Conclusion: This confirms the suspicion raised in Section 4.3. If defer relies on unwinding, and TBB relies on value-propagation, there is a mismatch. A TBB error will not trigger a defer cleanup unless explicitly checked.

8. Recommendations for Certification
To achieve SIL-4 certification and ensure the safety of the "Teacher" system, the following actions are mandatory:
8.1 Immediate Remediation (Blockers)
1. Provide Source Code: The file src/backend/ir/codegen_stmt.cpp must be audited to verify the defer implementation.
2. Implement TBB Gating: A compiler pass must be added to enforce that no TBB type can be passed to an extern function or cast to a raw integer without an intervening is_err() check. This prevents the "Reverse Velocity" bug.
3. Fix String Interning: Modify ExprCodegen to use a Module-scoped string pool rather than a global map, or ensure the JIT context destroys the compiler instance frequently to reclaim memory.
8.2 Architectural Hardening
1. Loop-Safe Stack Allocation: Verify that codegen_stmt.cpp uses llvm.stacksave and llvm.stackrestore around loop bodies if defer uses alloca.
2. Runtime Integrity Checks: The generated code should include a preamble that verifies the integrity of the Nursery heap and the WildX state before executing any motion control block.
8.3 Testing
1. Fuzzing: Fuzz the compiler with deeply nested defer statements inside try/catch and loop blocks to prove LIFO execution order.
2. Fault Injection: Inject TBB_ERR sentinels into the inputs of every actuator control function and verify that the robot performs a safe stop rather than chaotic movement.

9. Conclusion
The Aria compiler backend exhibits sophisticated safety engineering, particularly in its "Fail Hard" type system, Deterministic Fixed-Point (fix256) arithmetic, and Limb-Based Integral Model (LBIM). These features are well-aligned with the requirements of a high-assurance physics engine.
However, the safety verification is incomplete due to missing source code for the statement generator. Furthermore, the analysis of the expression generator reveals a critical hazard: the reliance on "sticky error" propagation (TBB) without visible enforcement of error checking at the hardware boundary. This creates a plausible path where a software math error translates into a maximum-force actuator command.
Final Verdict: The system is NOT YET SAFE for deployment with vulnerable populations. The "TBB Sentinel" vulnerability must be patched, and the defer mechanism must be audited, before the "Teacher" system can be trusted with a child's safety.
