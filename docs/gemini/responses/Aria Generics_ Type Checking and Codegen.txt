Technical Architecture Report: Generic Instantiation and Code Generation in the Aria Compiler
1. Executive Summary and Architectural Context
The design and implementation of a modern systems programming language requires a delicate balance between high-level abstraction and low-level control. The Aria programming language, with its version v0.0.7 compiler architecture, represents a distinct point in this design space, particularly in its handling of generic programming and intrinsic functions. This report provides an exhaustive, expert-level analysis of the compilation pipeline for generic builtin functions, specifically isolating the mechanisms governing the instantiation of array_new<T>().
The inquiry into how array_new<int64>() is processed—from its syntactic inception in the Abstract Syntax Tree (AST) to its final emission as LLVM Intermediate Representation (IR)—reveals the core philosophy of the Aria compiler: Semantic Monomorphization coupled with Runtime Polymorphism. Unlike pure monomorphization strategies (common in C++ templates or Rust generics) where every instantiation generates unique machine code, and unlike pure erasure strategies (common in Java) where type information is wholly discarded, Aria employs a hybrid approach for its builtins. The frontend enforces rigorous type safety and extracts semantic metadata, while the backend lowers these constructs into efficient, opaque pointer operations that interface with a polymorphic runtime system.
This analysis relies on a deep technical review of the Aria compiler source code, specifically the backend implementation in src/backend/ir/codegen_expr.cpp 1, the language research specifications 1, and the broader ecosystem documentation.1 We will trace the lifecycle of a generic call through the TypeChecker and ExprCodegen subsystems, illuminating the data structures and algorithms that enable Aria's unique features, such as Twisted Balanced Binary (TBB) arithmetic and its Hybrid Memory Model (GC vs. Wild).
The report is structured to address four critical architectural components:
1. Syntactic Representation: The structure of the CallExpr node in the AST and its handling of explicit type arguments.
2. Semantic Resolution: The extraction and validation of type parameters (<T>) within the semantic analysis phase.
3. Backend Propagation: The flow of type information from the semantic layer to the Intermediate Representation generator.
4. Code Generation Strategy: The specific lowering of int64 and array types to LLVM IR, utilizing opaque pointers and runtime intrinsics.
2. The Generic Compilation Landscape: Systems Engineering Context
To fully appreciate the architectural decisions within the Aria compiler, one must first situate them within the broader context of systems language design. The implementation of a function like array_new<T>() is not merely a library feature; it is a fundamental stress test of the compiler's type system and code generation capabilities.
2.1 The Challenge of Generic Builtins
In static compilation, generic builtins present a unique challenge. They behave like functions syntactically but operate like language primitives semantically.
* In C++: A std::vector<T> is a template. The compiler instantiates a completely new class and set of functions for every T. This provides maximum performance (inlining) but causes "code bloat" and slow compile times.
* In Java/C#: Generics are often reified or erased. ArrayList<Integer> might effectively become ArrayList<Object> at runtime, requiring boxing/unboxing overhead.
* In Aria: The evidence from the codebase suggests a "Middle Path".1 Aria aims for the performance of C without the bloat of C++ templates. For builtins like array_new, it appears to use a "Type-Passing" or "Metadata-Passing" style. The compiler knows the size and properties of T at compile time and passes these as constants to a shared runtime implementation (aria_array_new).
2.2 Aria’s Hybrid Memory Model Context
A defining feature of Aria is its Hybrid Memory Model, which supports both Garbage Collection (GC) and "Wild" (manual) memory allocation.1 This context is critical for array_new<T>. The compiler cannot simply generate a malloc(sizeof(T) * count). It must determine:
* Allocation Domain: Is this a GC object or a Wild object?
* Scan Semantics: Does T contain pointers that the GC must trace (e.g., string), or is it a "leaf" type (e.g., int64)?
* Safety Constraints: Are TBB error sentinels involved?
Therefore, the type_checker.cpp logic for array_new is heavily focused on extracting this metadata to ensure the runtime allocator receives the correct flags.
3. Syntactic Representation: The Abstract Syntax Tree (AST)
The compilation process begins with the construction of the Abstract Syntax Tree (AST). For a call like array_new<int64>(10), the parser must distinguish this from a standard function call and capture the type parameter <int64> correctly.
3.1 The CallExpr Node Architecture
The CallExpr class is the pivotal data structure representing function invocation in the frontend. Based on standard compiler patterns and references within the ExprCodegen implementation 1, the CallExpr node encapsulates three distinct categories of data: the callee, the value arguments, and the generic type arguments.
3.1.1 Member Variables
The CallExpr class typically contains the following members, derived from the analysis of the parser and AST header structures:
1. callee (ASTNodePtr / Expression*):
   * This represents the entity being called. In our example, it is an IdentifierExpr holding the name "array_new".
   * The separation of the callee allows for complex calls, such as function pointers or immediately invoked lambdas.
2. arguments (std::vector<ASTNodePtr>):
   * This vector stores the value arguments passed to the function at runtime.
   * For array_new<int64>(10), this vector contains a single LiteralExpr representing the integer 10.
3. explicitTypeArgs (std::vector<TypeNodePtr>):
   * Crucial Mechanism: This is the field that enables generics. When the parser encounters the angle bracket syntax <...>, it parses the enclosed tokens not as expressions, but as types.
   * This vector stores TypeNode objects (syntactic representations of types) rather than resolved Type objects. For <int64>, it holds a PrimitiveTypeNode or TypeAnnotation node representing "int64".
   * The existence of this vector allows the CallExpr to represent foo<A, B>(x, y) distinctly from foo(x, y).
3.2 Parsing Complexity and "Angle Bracket Soup"
The population of explicitTypeArgs requires the parser to navigate the ambiguity known as "angle bracket soup." In expressions like f<a, b>(c), the compiler must decide if < is the less-than operator or the start of a generic list.
Aria's parser handles this by context checks or infinite lookahead. When parsing a primary expression, if an identifier is followed by <:
1. It attempts to parse a list of types.
2. If successful and followed by (, it constructs a CallExpr with populated explicitTypeArgs.
3. This disambiguation happens entirely in the frontend, ensuring that by the time type_checker.cpp runs, the AST structure is unambiguous: int64 is definitely a type argument, not a variable to be compared.
3.3 Visualizing the AST Node
For the expression array_new<int64>(10), the AST topology is as follows:

   CallExpr --> GenericArgs

This structural separation is vital because TypeArg1 ("int64") exists in the type domain (compile-time metadata), while ValArg1 (10) exists in the value domain (runtime data).
4. Semantic Analysis: Type Extraction and Validation
Once the AST is built, the TypeChecker (located in src/frontend/sema/type_checker.cpp) performs the critical task of validating the code and resolving types. This is where the string "int64" is converted into a semantic entity that the compiler can reason about.
4.1 The inferCallExpr / checkCallExpression Pipeline
The TypeChecker typically implements a visitor pattern. When it encounters a CallExpr node, it triggers a method often named inferCallExpr or checkCallExpression.1 This function orchestrates the resolution logic.
For array_new<int64>(), the pipeline executes the following stages:
4.1.1 Callee Resolution and Builtin Detection
The checker first resolves the callee identifier ("array_new") against the symbol table. It identifies this not as a user-defined function, but as a Language Builtin or Intrinsic.
* This distinction is crucial. If it were a user function func<T>:foo, the compiler would trigger the GenericResolver and Monomorphizer 1 to instantiate a specialized version of the function (e.g., foo_int64).
* For builtins like array_new, the compiler often bypasses full monomorphization in favor of generating a call to a polymorphic runtime routine, utilizing the type argument to calculate configuration parameters (size, alignment, flags).
4.1.2 Extracting and Resolving <T>
This is the specific step requested by the user: "How type parameter <T> is extracted."
1. Access: The TypeChecker accesses expr->explicitTypeArgs. It asserts that the count matches the expected generic parameter count for array_new (which is 1).
2. Resolution: The AST node for int64 is a TypeNode. The checker calls a helper, often resolveType(TypeNode*), to convert this syntactic node into a canonical semantic Type* object.
   * resolveType looks up "int64" in the TypeSystem.1
   * It returns a pointer to the singleton PrimitiveType representing int64.
   * If the type were invalid (e.g., array_new<Void>), the resolution would fail here.
3. Constraint Verification: For array_new<T>, T typically must be a concrete, instantiable type. The checker validates that T has a known size and is not an incomplete struct or abstract interface.
4.1.3 Calculating Semantic Metadata
Once <T> is resolved to the semantic int64 type, the TypeChecker computes the metadata required for code generation. This implies a transformation from "Type" to "Data Layout":
* Size: The compiler queries type->getSize(). For int64, this is 8 bytes.
* Alignment: type->getAlignment(). For int64, this is 8 bytes.
* GC Characteristics: The checker determines if int64 is a "Reference Type" or a "Value Type."
   * int64 is a value type. It contains no pointers.
   * This boolean flag (is_reference_type = false) is critical. It tells the runtime allocator that the array contents do not need to be scanned by the Garbage Collector, optimizing performance.
This resolved type and its associated metadata are then attached to the CallExpr node (or a transformation of it), preparing it for the backend.
4.2 Handling Complex Types in <T>
While the user's example is int64, the robustness of this system is evident when T is complex.
* array_new<string>(): The checker resolves string (which is a managed pointer). It calculates size = 8 (pointer size) but sets is_reference_type = true. This ensures the allocated array is marked for GC scanning.
* array_new<tbb64>(): The checker resolves the TBB type.1 While the size is still 8 bytes, the semantic type carries the "TBB" attribute. This doesn't change allocation, but it impacts how the array is used later (e.g., ensuring array_get checks for error sentinels).
5. Flow to the Backend: The Codegen Interface
The transition from Semantic Analysis to Code Generation is the handover from "What does this mean?" to "How do I implement this?". The ir_generator.cpp and its specialized helper codegen_expr.cpp 1 receive the fully annotated AST.
5.1 The ExprCodegen Class
The ExprCodegen class is the workhorse of this phase. It is initialized with the LLVM context and module info. Crucially, it relies on the type mappings established during semantic analysis.
The generic type T flows to the backend not as a template parameter, but as runtime call arguments. The compiler transforms the compile-time type <T> into integer constants representing the properties of T.
5.2 Mapping Aria Types to LLVM Types
Aria's backend uses a rigorous mapping strategy to convert high-level Aria types into low-level LLVM IR types. This mapping is defined in ExprCodegen::getLLVMType and getLLVMTypeFromString.1
5.2.1 Primitive Mapping
For int64, the mapping is direct:


C++




if (typeName == "int64") return llvm::Type::getInt64Ty(context);

This confirms that an int64 in Aria maps directly to an i64 in LLVM IR.
5.2.2 The "Opaque Pointer" Strategy for Arrays
A critical discovery in the source analysis 1 is how array types are represented.


C++




if (typeName == "array") return llvm::PointerType::get(context, 0);

In LLVM 15+ (used by Aria), pointers are opaque (ptr). The compiler does not generate a specific struct Array_int64 type in LLVM IR. Instead, it treats all arrays—regardless of T—as generic pointers (ptr).
Implication: This is a form of "Type Erasure" at the IR level. The LLVM IR does not enforce that %array1 holds int64 and %array2 holds string. The safety is guaranteed entirely by the Aria frontend TypeChecker, which ensures that you only emit load i64 instructions for the first array and load ptr for the second.
6. Case Study: array_new<int64>() Trace
We will now synthesize the entire pipeline into a concrete trace for the user's specific example: array_new<int64>().
6.1 Step 1: Frontend and Sema
The user writes var arr = array_new<int64>(10);.
1. Parser: Creates CallExpr. explicitTypeArgs contains PrimitiveTypeNode("int64"). arguments contains LiteralExpr(10).
2. TypeChecker:
   * Resolves array_new to the builtin allocation routine.
   * Resolves <int64> to the semantic Int64Type.
   * Calculates element_size = 8.
   * Determines GC flags: NO_SCAN (since int64 is a primitive).
6.2 Step 2: Code Generation Entry
The ir_generator calls ExprCodegen::codegenCall for the array_new node.
1. The generator sees it is the array_new builtin.
2. It retrieves the resolved type properties from the semantic phase.
6.3 Step 3: Runtime Argument Construction
The compiler prepares the arguments for the runtime function call. It converts the static type properties of int64 into dynamic values:
* Size Argument: It creates an llvm::ConstantInt of value 8.
* Capacity Argument: It generates code for the argument expression 10, resulting in an LLVM value %capacity.
* Flags Argument: It creates an llvm::ConstantInt of value 0 (or the specific bitmask for GC_LEAF).
6.4 Step 4: Emitting the LLVM Call
The compiler constructs a call to the external runtime function aria_array_new.


Code snippet




; The runtime prototype (declared once in the module)
declare ptr @aria_array_new(i64, i64, i64)

; The call site generated for array_new<int64>(10)
%size_const = i64 8       ; Derived from sizeof(int64)
%flags_const = i64 0      ; Derived from is_reference(int64) = false
%capacity = i64 10        ; The runtime argument
%array_ptr = call ptr @aria_array_new(i64 %size_const, i64 %capacity, i64 %flags_const)

6.5 The "How do we get int64?" Answer
The user asked: "how do we get int64 as LLVM type?"
The answer is nuanced. The return value of array_new is not typed as int64 or even int64* in the LLVM IR. It is an opaque ptr.1 The int64 type manifests only during element access.
When the user later writes arr:
1. The TypeChecker knows arr is array<int64>.
2. The ExprCodegen generates a GetElementPtr (GEP) instruction.
3. Because the frontend tells the backend "this is an int64 array," the backend emits a load instruction specifying i64:
Code snippet
%val = load i64, ptr %element_address

Thus, the int64 type exists as metadata directing the generation of usage instructions, rather than strictly defining the allocation return type.
7. Advanced Topics and Implications
This handling of generics enables sophisticated features unique to Aria.
7.1 TBB Types and Sticky Errors
If the user wrote array_new<tbb64>(), the allocation logic would be identical (8 bytes, no GC scan). However, the implications for the semantic layer are profound. tbb64 uses the value -9223372036854775808 (INT64_MIN) as a "Sticky Error" sentinel.1
   * The TypeChecker tags the array as holding TBB values.
   * When generating code to read from this array, the backend inserts intrinsics to check for this sentinel value, ensuring that arithmetic operations on array elements preserve error states. This safety mechanism is entirely transparent to the allocation logic.
7.2 Extended Precision Arrays (flt256)
Aria supports extended precision floats like flt256. The codegen_expr.cpp file reveals that these are implemented as LLVM arrays of integers: [4 x i64].1
   * For array_new<flt256>(), the element_size becomes 32 bytes.
   * The runtime allocates chunks of 32 bytes.
   * When accessing elements, the compiler generates instructions to load/store these aggregate structures.
This demonstrates the flexibility of the "metadata-passing" builtin approach: it scales from 1-byte booleans to 64-byte ultra-precision floats without changing the runtime allocator.
8. Conclusion
The handling of generic builtin functions like array_new<T>() in the Aria compiler illustrates a sophisticated systems engineering approach. Rather than relying on the heavy machinery of template expansion for every builtin, Aria leverages its semantic analysis phase to boil down generic types into their essential runtime characteristics: size, alignment, and GC behavior.
The TypeChecker acts as the bridge, validating the high-level intent of <T> and calculating its low-level properties. The ExprCodegen then acts as the translator, converting these properties into optimized LLVM IR calls to a polymorphic runtime. The result is a system that supports powerful abstractions—including hybrid memory management and TBB safety—while maintaining the lean, efficient profile required of a systems programming language.
