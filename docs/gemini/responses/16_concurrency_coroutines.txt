1. Executive Summary
This report constitutes a comprehensive Safety Integrity Level 4 (SIL-4) audit of the concurrency and coroutine subsystems within the Aria programming language. The audit was commissioned to verify the stability, security, and predictability of the language's async/await primitives, specifically in the context of controlling physical actuators in robotic assistants designed for vulnerable populations, including neurodivergent children and patients in extended hospital care.1 The nature of the target application—a "Teacher" system capable of physical interaction—mandates a zero-tolerance policy for logic errors, memory corruption, or non-deterministic behavior. In this domain, a software failure does not result in a mere crash; it results in kinetic trauma.
The audit has identified FIVE (5) CRITICAL VULNERABILITIES that fundamentally compromise the safety of the system. These vulnerabilities are not theoretical edge cases; they are deterministic defects in the compiler's code generation logic and runtime integration that guarantee system instability. The findings include vectors for silent memory corruption via Garbage Collection (GC) blindness, deterministic deadlocks in actuator control loops, and widespread resource leakage during error conditions.
Immediate Action Required: The current implementation of the async runtime and backend code generation is unsafe for deployment. The robotic control systems must NOT be operated using code compiled with the current version of Aria. The risk of "uncontrolled motor movement" and "force limit bypass" described in the threat model is actively realized by the current codebase.1

2. System Architecture and Threat Landscape
To understand the severity of the findings, it is necessary to contextualize the Aria async architecture within the constraints of a SIL-4 robotic control system. The source code provided 1 reveals that Aria utilizes a compiler-driven transformation for asynchronous code, leveraging LLVM's coroutine intrinsics to implement a state-machine-based concurrency model.
2.1 The Coroutine Lowering Model
Aria employs LLVM's switched-resume lowering model for coroutines. In this model, a function marked as async in the Aria source is not compiled into a single contiguous block of machine code. Instead, the StmtCodegen::codegenFuncDecl function 1 transforms the routine into three distinct logical parts:
1. The Ramp Function: This code executes when the coroutine is first called. It is responsible for allocating the "Coroutine Frame"—a block of memory used to store the function's state (instruction pointer, local variables, and spilled registers) across suspension points.
2. The Resume Function: This contains the body of the user's code, transformed into a state machine. It includes a jump table that allows execution to resume at the correct point after an await.
3. The Destroy Function: This handles the deallocation of the frame and the destruction of any local variables that were in scope when the coroutine completed or was cancelled.
This transformation is invisible to the Aria developer but is the single most critical component for memory safety. If the Ramp function allocates memory incorrectly, or if the Resume function mismanages pointers, the safety guarantees of the high-level language are nullified.
2.2 The Threat Model: Robotic Actuation
In a physics-based AGI system, the control loop typically operates at a high frequency (100Hz to 1kHz). A typical control cycle might look like this:


Code snippet




async func:control_loop = void() {
   while (true) {
       // 1. Read sensors (Suspendable I/O)
       SensorData:data = await sensors.read_all();
       
       // 2. Compute safety limits
       if (data.force > MAX_FORCE) { panic("Force limit exceeded"); }
       
       // 3. Actuate
       await motors.apply_torque(calculate_pid(data));
   }
}

The audit focuses on the "Suspension Points" (the await keywords). In a coroutine model, await involves saving the current state to the heap and yielding the CPU.
* The Threat of Corruption: If the state saved to the heap is corrupted or lost during the await, the variable data might contain garbage when the function resumes. A force reading of 0.5N could become NaN or 2,000,000N, causing the safety check to fail or the PID controller to output a maximum torque command.
* The Threat of Liveness: If the await mechanism fails to wake the control_loop after sensors.read_all() completes, the loop stops. However, in many robotic architectures, the motor drivers continue executing the last valid command until a watchdog timer fires. If the watchdog timeout is 100ms, a robot arm moving at 2m/s can travel 20cm—enough to cause severe injury to a child—before the system halts.

3. Audit Item 1: Coroutine State Capture & Memory Allocation
Compliance Requirement: "State capture MUST be complete. Verify all live variables saved before suspension. Confirm no stack corruption from state save/restore."
Specific Threat: The interaction between the coroutine's memory allocation strategy and the language's Garbage Collector (GC). If the GC is unaware of the coroutine frame, it will erroneously reclaim objects referenced only by that frame.
3.1 Analysis of Frame Allocation Logic
We examined the src/backend/ir/codegen_stmt.cpp file 1 to verify how the compiler generates the setup code for async functions. Specifically, we analyzed the codegenFuncDecl method, which handles the lowering of function declarations.
The code generation logic for async functions (Phase 4.6 in the source comments) performs the following steps to initialize a coroutine:
1. Generates the llvm.coro.id intrinsic to identify the function as a coroutine.
2. Calculates the required frame size using llvm.coro.size.
3. Allocates memory for the frame.
The critical defect lies in how this memory is allocated. The source code explicitly generates a call to the system's raw malloc function:


C++




// src/backend/ir/codegen_stmt.cpp 

// 3. Allocate coroutine frame
llvm::Function* malloc_func = module->getFunction("malloc");
if (!malloc_func) {
   llvm::FunctionType* malloc_type = llvm::FunctionType::get(
       llvm::PointerType::get(llvm::Type::getInt8Ty(context), 0),
       {llvm::Type::getInt64Ty(context)},
       false
   );
   malloc_func = llvm::Function::Create(
       malloc_type,
       llvm::Function::ExternalLinkage,
       "malloc",
       module.get()
   );
}
llvm::Value* coro_mem = builder.CreateCall(malloc_func, {coro_size}, "coro.alloc");

3.2 Vulnerability: GC Blindness (Invisible Roots)
This implementation constitutes a critical safety violation. Aria is a garbage-collected language.1 The correctness of the GC relies on "Reachability Analysis"—the ability to traverse all live objects starting from a set of "roots" (global variables and the active stack).
When a coroutine is running, its local variables are on the CPU stack, which the GC scans. However, when a coroutine suspends:
1. State Spilling: LLVM spills live variables (e.g., RobotArm* arm) from the CPU stack into the Coroutine Frame.
2. Heap Storage: This frame is stored in the memory block returned by malloc.
3. GC Cycle: If a GC cycle triggers while the coroutine is suspended, the collector scans the known roots. Because the coroutine frame was allocated via malloc and not the GC's allocator, and because there is no mechanism shown to register this raw pointer as a GC root, the Coroutine Frame is invisible to the GC.
4. Erroneous Collection: The GC sees no references to the RobotArm object (since the only reference is hidden inside the opaque malloc'd frame). It concludes the object is dead and reclaims its memory.
5. Use-After-Free: When the coroutine resumes, it loads the arm pointer from the frame. This pointer now points to unallocated memory (or memory reallocated to a different object). The next call to arm->stop() will corrupt the heap or crash the system.
3.3 Theoretical Implications
This failure mode is particularly insidious because it is non-deterministic. It depends on the timing of the GC cycles relative to the coroutine suspension. In a test environment with low memory pressure, the GC might not run, and the code will appear correct. in a "teacher" scenario with heavy asset loading (images, speech synthesis), memory pressure increases, triggering collections during suspensions, leading to sudden, irreproducible crashes or behavior drifts in the robot.
3.4 Remediation Requirement
Status: ❌ Bug found.
The compiler must be modified to use the runtime's GC-aware allocator instead of malloc. The include/runtime/async/gc_integration.h file 1 defines __aria_coro_alloc_gc, which is designed for this exact purpose. It likely registers the new frame as a "root" or allocates it within a manageable heap region.
Proposed Fix:
In src/backend/ir/codegen_stmt.cpp, replace the malloc generation with a call to the runtime intrinsic:


C++




// Corrected logic for codegen_stmt.cpp

// Get the GC-aware allocator
llvm::Function* gc_alloc_func = getOrDeclareGCAlloc(); // Assuming this helper exists or define __aria_coro_alloc_gc

// Call the allocator with the size and a Task ID (for tracking)
// Signature: i8* __aria_coro_alloc_gc(i64 size, i64 task_id)
llvm::Value* task_id = llvm::ConstantInt::get(builder.getInt64Ty(), 0); // Placeholder or actual ID
llvm::Value* coro_mem = builder.CreateCall(gc_alloc_func, {coro_size, task_id}, "coro.alloc");

4. Audit Item 2: Await Expression Safety
Compliance Requirement: "Verify awaited value lifetime extends across suspension. Confirm no dangling pointers in resumed context."
Specific Threat: The "Frozen Robot" scenario. The await mechanism is the bridge between the asynchronous operation (the Promise/Future) and the execution flow. If this bridge is broken, the robot enters a state of indefinite suspension.
4.1 Analysis of Await Lowering
We analyzed the src/backend/ir/codegen_expr.cpp file 1, specifically the codegenExpression function handling the ASTNode::NodeType::AWAIT case. This section is responsible for generating the LLVM IR that suspends the current coroutine and schedules it for resumption when the awaited task completes.
The implemented logic is as follows:


C++




// src/backend/ir/codegen_expr.cpp 

case ASTNode::NodeType::AWAIT: {
   // 1. Evaluate the operand (the Future/Task)
   llvm::Value* future_value = codegenExpression(awaitExpr->operand.get());

   // 2. Resume the child (if it's a handle)
   if (future_value->getType()->isPointerTy()) {
       builder.CreateCall(coroResumeFunc, {future_value});
   }

   // 3. Save state and suspend the CURRENT coroutine
   llvm::Value* save_token = builder.CreateCall(coroSaveFunc, {current_handle}, "await.save");
   llvm::Value* suspend_result = builder.CreateCall(coroSuspendFunc, {save_token, is_final}, "await.suspend");

   // 4. Switch on suspend result (Resume, Destroy, Suspend)
   //...
}

4.2 Vulnerability: The "Fire-and-Forget" Deadlock
The code exhibits a fundamental misunderstanding of asynchronous control flow. It performs two disconnected actions:
1. It resumes the child coroutine (coroResumeFunc).
2. It suspends the parent coroutine (coroSuspendFunc).
The Missing Link: There is no mechanism connecting the completion of the child to the resumption of the parent.
* In a correct implementation (e.g., C++20 co_await or Rust poll), the parent registers a "continuation" (a callback or a handle to itself) with the child before suspending.
* When the child finishes, it looks at this registered continuation and calls resume() on the parent.
In the current Aria implementation, the parent calls resume on the child and then immediately goes to sleep. The child runs until it hits its own suspension point (e.g., waiting for hardware IO). When the child suspends, control returns... nowhere. The parent is already suspended. The executor has no signal that the parent is waiting on this specific child.
Result: The parent coroutine is suspended indefinitely. It is never placed back into the readyQueue because the child doesn't know it needs to wake the parent. The control loop freezes.
4.3 Theoretical Implications: Deterministic Deadlock
This is not a race condition; it is a logic error that guarantees deadlock for any await operation that doesn't complete synchronously.
* Synchronous Completion: If the child completes immediately (before the parent suspends), the coroResumeFunc call might return "done", but the parent still proceeds to call coroSuspendFunc unconditionally (the logic doesn't check coro.done). This leads to the parent suspending after the child has already finished, meaning the wakeup signal (if one existed) would have been missed (The "Lost Wakeup Problem").
* Asynchronous Completion: The parent suspends. The child completes later. The child destroys itself. The parent remains suspended forever.
4.4 Remediation Requirement
Status: ❌ Bug found.
The await codegen must be rewritten to implement a handshake between the parent and the child (or the Future object).
Proposed Fix:
The runtime must provide a registration intrinsic, likely aria_future_set_continuation or similar.


C++




// Corrected logic for codegen_expr.cpp

// 1. Get the current coroutine's handle
llvm::Value* parent_handle = current_handle; 

// 2. Register the parent as the continuation of the future
// Signature: void aria_register_continuation(void* future, void* parent_handle)
llvm::Function* reg_func = module->getFunction("aria_register_continuation");
builder.CreateCall(reg_func, {future_value, parent_handle});

// 3. NOW suspend. The runtime is responsible for re-scheduling the parent
// when the future completes.
//... existing suspend logic...

5. Audit Item 3: Suspension Point Atomicity & Type Safety
Compliance Requirement: "Verify no partial operations visible at suspension boundary. Confirm invariants maintained. Type errors = force limit bypass."
Specific Threat: Type Confusion. If the compiler provides a pointer to a value instead of the value itself, mathematical operations will operate on the memory address. In safety-critical code, this typically results in "Max Force" violations.
5.1 Analysis of Resume Logic
We continued the analysis of src/backend/ir/codegen_expr.cpp 1 to see what happens after the coroutine resumes from the await. This is the point where the result of the asynchronous operation should be made available to the caller.


C++




// src/backend/ir/codegen_expr.cpp 

// Step 7: Generate resume block - continue execution here
builder.SetInsertPoint(resume_block);

// For now, return the future value itself
// TODO: Extract actual result from Future<T> when trait system is ready
return future_value; 

5.2 Vulnerability: Type Confusion / Pointer Leak
The compiler explicitly returns future_value as the result of the await expression.
* future_value is the pointer to the coroutine handle or the Future object (an i8* or void*).
* The expression expects a value of type T (e.g., int32 or fix256).
The Failure Scenario:
Consider the following Aria code:


Code snippet




int32:current_force = await sensor.read_force(); // Returns 5
int32:safe_limit = 10;
if (current_force < safe_limit) {... }

In the compiled LLVM IR:
1. future_value is a heap address, e.g., 0x00007F80A0B0.
2. The compiler returns 0x00007F80A0B0 as the result of the await.
3. The compiler casts this pointer to int32 to match the variable current_force.
4. current_force becomes 2,139,136,176 (the decimal representation of the address).
5. The check 2,139,136,176 < 10 evaluates to FALSE.
6. The safety check fails erroneously, OR (worse), if the logic was inverted (force + adjustment), the system calculates a target force of 2 billion Newtons.
5.3 Theoretical Implications: Force Limit Bypass
This is a classic "Type Confusion" vulnerability. In a managed language, the type system promises that an int32 holds an integer value. By returning a raw pointer masked as an integer, the compiler breaks the fundamental contract of the language. This bypasses all application-level logic designed to constrain forces, velocities, or temperatures, as the "measured" values are effectively random large numbers (heap addresses).
5.4 Remediation Requirement
Status: ❌ Bug found.
The code generation must include an "Unwrap" phase after resumption. It must call a runtime function to extract the actual payload from the Future object.
Proposed Fix:


C++




// Corrected logic for codegen_expr.cpp

// At Step 7 (Resume Block):
// 1. Call runtime to get pointer to the result data inside the future
// Signature: void* aria_future_get_result_ptr(void* future)
llvm::Function* get_val_func = module->getFunction("aria_future_get_result_ptr");
llvm::Value* data_ptr_void = builder.CreateCall(get_val_func, {future_value});

// 2. Cast the void* to the expected result type pointer (e.g., int32*)
llvm::Type* resultType = getLLVMType(awaitExpr->type); 
llvm::Value* data_ptr = builder.CreatePointerCast(data_ptr_void, llvm::PointerType::get(resultType, 0));

// 3. Load the actual value
llvm::Value* result = builder.CreateLoad(resultType, data_ptr, "await.result");

return result;

6. Audit Item 4: Coroutine Frame Allocation and Deallocation
Compliance Requirement: "All coroutine frames MUST be freed (no leaks). Confirm no memory leaks from abandoned coroutines."
Specific Threat: Memory Exhaustion (DoS). In a robotic system, memory is a finite resource. If tasks that are cancelled or encounter errors do not free their stack frames, the heap will eventually fill up. When malloc fails, the system crashes or enters an unpredictable state.
6.1 Analysis of Task Lifecycle
We examined include/runtime/async/executor.h 1 to verify the Task class destructor, which is responsible for cleaning up task resources when a Task object is destroyed (e.g., when a request is cancelled).


C++




// include/runtime/async/executor.h 

class Task {
   //...
   ~Task() {
       // Cleanup result storage if allocated
       if (resultStorage) {
           // TODO: Proper cleanup based on result type
       }
       // NOTE: No code here destroys the coroutine handle
   }
};

We also cross-referenced this with src/runtime/async/work_stealing.cpp 1 to see how tasks are normally cleaned up. The normal cleanup happens inside the worker loop only if the coroutine reports __aria_coro_done.
6.2 Vulnerability: Zombie Frames on Cancellation
The coroutine frame (allocated via malloc in BUG-01) is owned by the coro_handle. The generated LLVM code includes a call to coro.free inside the coroutine's "cleanup" block. This block is only entered if the coroutine runs to completion (natural return).
However, if a Task is cancelled externally (e.g., the user cancels a lesson plan, or the system initiates an emergency stop), the Task object is destroyed by the executor. The ~Task() destructor runs, but it does not call coro_destroy.
Result: The Task wrapper is freed, but the potentially large (kilobytes or megabytes) stack frame allocated for the coroutine remains on the heap forever. This is a classic memory leak. In a long-running "Teacher" system that spawns thousands of short-lived tasks (e.g., interpreting distinct sentences), this will rapidly deplete available RAM.
6.3 Theoretical Implications: Reliability Drift
Memory leaks introduce a time-dependent failure mode. A system that passes all unit tests and short-duration integration tests will inevitably fail after $N$ hours of operation. In a hospital setting, this "drift" toward failure is unacceptable. A robot might function perfectly during the morning rounds but fail critically in the evening due to heap fragmentation or exhaustion caused by leaked frames.
6.4 Remediation Requirement
Status: ❌ Bug found.
The Task destructor must enforce ownership of the coroutine handle and ensure it is destroyed if the task is being torn down before completion.
Proposed Fix:


C++




// Corrected logic for include/runtime/async/executor.h

~Task() {
   // Check if the handle is valid
   if (handle) {
       // Check if it's already done to avoid double-free (if the runtime doesn't track this)
       // Or simply call destroy, which should be safe if implemented correctly via intrinsics
       aria_coro_destroy(handle); 
   }
   //... existing cleanup...
}

7. Audit Item 5: Exception/Panic Interaction with Suspension
Compliance Requirement: "Verify panic during coroutine triggers cleanup. Check defer blocks run even if suspended."
Specific Threat: Resource Deadlock. If a coroutine acquires a hardware lock (e.g., mutex.lock()) and then crashes due to a panic, the lock must be released. If the panic mechanism bypasses the defer block responsible for unlocking, the hardware remains locked forever.
7.1 Analysis of Control Flow Generation
We examined src/backend/ir/codegen_stmt.cpp 1 to understand how function calls and statements are generated. Specifically, we looked for the use of LLVM's invoke instruction versus the standard call instruction.
Findings:
The code generator uses builder.CreateCall for function invocations.


C++




// src/backend/ir/codegen_expr.cpp
llvm::Value* result = builder.CreateCall(runtimeFunc, {left, right},...);

7.2 Vulnerability: The "Unwind Gap"
In LLVM IR, the call instruction assumes the called function returns normally. If the called function throws an exception (which is how panic or fail is typically implemented in runtimes to unwind the stack), and the current function does not use an invoke instruction with a landingpad, the behavior is undefined or results in immediate program termination (std::terminate).
The Implication for defer:
Aria implements defer blocks by generating code at the end of the function (in codegenReturn).


C++




void StmtCodegen::codegenReturn(ReturnStmt* stmt) {
   executeFunctionDefers(); // Generates cleanup code here
   //...
   builder.CreateRet(ret_value);
}

This logic only runs on the normal return path. It does not hook into the exception unwinding mechanism.
Scenario:
1. Coroutine A locks MotorMutex.
2. defer { MotorMutex.unlock(); } is registered.
3. Coroutine A calls sensor.read().
4. sensor.read() panics (e.g., hardware fault).
5. The stack unwinds. Because codegenReturn is skipped, the defer block is never executed.
6. MotorMutex remains locked.
7. The panic is caught by the top-level supervisor, which tries to reset the robot.
8. The reset procedure tries to lock MotorMutex and deadlocks.
7.3 Theoretical Implications: State Corruption
This vulnerability violates the fundamental "Resource Acquisition Is Initialization" (RAII) guarantee that defer is supposed to provide. In a safety-critical system, error paths are as important as happy paths. A system that cannot safely recover from a partial failure is not SIL-4 compliant.
7.4 Remediation Requirement
Status: ❌ Bug found.
The compiler must wrap all calls that might panic (which is essentially all calls in a dynamic system) in an invoke instruction if there are active defer statements in the scope.
Proposed Fix:
The StmtCodegen class needs to manage a stack of "cleanup blocks". When generating a call:
1. Check if there are active defers.
2. If yes, create a BasicBlock for the exception handler (LandingPad).
3. Use builder.CreateInvoke(func, normal_block, exception_block, args).
4. In the exception_block, generate the defer execution code, then resume the unwinding.

8. Conclusion and Recommendations
The audit of the Aria v0.0.7 compiler and runtime has revealed a system that is structurally unsound for safety-critical applications. While the language design documents 1 articulate a strong vision for safety (TBB types, immutability), the backend implementation 1 fails to uphold these guarantees in the asynchronous domain.
The five identified bugs form a "Swiss Cheese" failure model:
1. BUG-01 blinds the memory manager to the robot's state.
2. BUG-02 ensures that complex control flows will freeze.
3. BUG-03 allows type system violations that bypass safety limits.
4. BUG-04 ensures the system will eventually crash from memory exhaustion.
5. BUG-05 ensures that recovery from errors leads to deadlock.
Recommendation:
1. Halt Deployment: Do not deploy this version to any hardware capable of physical motion.
2. Remediation Phase: Implement the five proposed fixes.
3. Re-Audit: A second audit is required after remediation to verify the fixes and ensure no regression.
