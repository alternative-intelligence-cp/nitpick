Architectural Specification for the Aria GlobEngine Foreign Function Interface (FFI) Bridge
1. Executive Summary and Systems Architecture Context
The architectural maturity of the Aria programming language ecosystem relies heavily on the seamless integration between its high-level runtime environment, capable of garbage collection and structured concurrency, and the low-level systems capabilities provided by its host implementation language, C++17. A critical component within this ecosystem is the GlobEngine, a specialized subsystem responsible for recursive directory traversal, pattern matching, and file discovery.1 While the C++ implementation of the GlobEngine (located in aria_utils/src/aglob/) has been verified for correctness and functionality, a significant architectural chasm exists: the std.fs module within the Aria standard library lacks the mechanism to invoke these capabilities directly.1
This report details the comprehensive design and specification for the GlobEngine Foreign Function Interface (FFI) Bridge. This bridge serves as a translation layer, or "FFI Firewall," enabling the Aria runtime to safely consume the functionality of the C++ host without violating memory safety guarantees or ABI (Application Binary Interface) constraints.1
The design specifically addresses the complex challenges of cross-language interoperability: marshaling dynamic data structures like variable-length string arrays, mapping C++ exceptions to FFI-safe error codes, and establishing a strict memory ownership protocol to prevent leaks in Aria’s hybrid memory model—a model that mixes Garbage Collection (GC) with "Wild" manual allocation.1 The interface is strictly defined using extern "C" linkage to ensure compatibility with the Aria compiler’s symbol resolution logic and the system linker.1
Implementation of this specification resolves the "bootstrapping paradox" currently affecting the ecosystem, where the build system (aria_make) depends on file system capabilities that the language's standard library has not yet exposed, creating a circular dependency that hinders self-hosted compilation.1 Furthermore, this architecture establishes a precedent for future native extensions, defining the canonical pattern for integrating high-performance C++ subsystems into the Aria runtime.
2. Theoretical Framework of the FFI Bridge
The fundamental engineering challenge in bridging C++ and Aria lies in their divergent runtime models. C++ utilizes Resource Acquisition Is Initialization (RAII) for deterministic resource management, exceptions for error propagation, and the Standard Template Library (STL) for collections. Aria, conversely, employs a hybrid model involving result<T> types for error handling, a distinction between managed (gc) and unmanaged (wild) memory, and a specific structure layout for arrays and strings.1 The bridge must mediate these differences without introducing overhead that would negate the performance benefits of the underlying C++ engine.
2.1 The "FFI Firewall" Architectural Pattern
To mediate the disparate runtime models, the glob_ffi.cpp implementation adopts the "FFI Firewall" architectural pattern.1 This pattern dictates that no C++ specific features—exceptions, classes, references, or STL containers—may leak across the API boundary. The bridge acts as a stateless service layer that functions as follows:
1. Interception: The layer intercepts calls from the Aria runtime, which are made using standard C calling conventions.
2. Unmarshaling: Primitive C-types are converted into C++ objects (e.g., converting char* to std::filesystem::path or std::string). This step often involves data copying to satisfy the differing memory layout requirements of C-strings versus C++ strings.
3. Execution: The core GlobEngine logic is invoked. This logic operates entirely within the C++ domain, utilizing std::regex and std::filesystem.
4. Marshaling: The results are converted back into C-compatible structures (char**, flat structs).
5. Exception Containment: A global try-catch block wraps the execution logic. All C++ exceptions are caught and translated into integer error codes defined by the GlobError enumeration.1
This design ensures that the Aria runtime remains insulated from the complexities of the host filesystem implementation, receiving only deterministic, flat data structures that it can safely consume.1 If a C++ exception were to propagate uncaught across the FFI boundary, it would attempt to unwind the stack through stack frames generated by the Aria runtime or the C linker, leading to undefined behavior and likely process termination.1
2.2 Memory Ownership and Lifecycle Management Protocol
A defining characteristic of this FFI design is the "Caller Owns Memory" protocol. Because the GlobEngine allocates memory for the result set (the list of discovered file paths) on the heap (using malloc or new via strdup), the Aria runtime must accept responsibility for freeing this memory.1
Aria's Garbage Collector (GC) cannot automatically traverse or free memory allocated by the C++ system allocator unless it is explicitly registered as a root or allocated via aria_alloc. Since the GlobEngine is a host-side utility potentially used by tools other than the Aria runtime (e.g., the compiler driver ariac), it minimizes dependencies by avoiding aria_alloc. Instead, it uses standard system allocation, requiring the Aria side to utilize a dedicated deallocation function (aria_glob_free) within a defer block to ensure resource cleanup.1 This creates a clear contract: the bridge allocates "wild" memory, and the Aria runtime must manually release it, typically by transferring the data into managed structures and then invoking the cleanup routine.
2.3 Segment-Based Anchoring and Performance Implications
The GlobEngine employs segment-based anchoring, a technique where the pattern is analyzed to find the longest literal prefix before the first wildcard.1 For example, in the pattern src/core/**/*.aria, the anchor is src/core. The engine only initiates directory traversal from this anchor, drastically reducing the number of stat and readdir calls compared to scanning from the root.
This architectural detail impacts the FFI design. The interface must accept a base_dir parameter separate from the pattern parameter. While the engine could infer the base directory, explicit separation allows the Aria runtime to contextualize the search (e.g., relative to a project root or a specific module path) without string manipulation overhead on the C++ side. The FFI must validate that this base_dir exists before instantiating the engine to provide fast-fail feedback.1
3. Data Structure Specification
The FFI interface relies on two primary data structures to facilitate bidirectional communication: AriaGlobOptions for input configuration and AriaGlobResult for output data. These structs are designed with standard C data layout rules (POD - Plain Old Data) to ensure binary compatibility across compilers (GCC, Clang, MSVC) and platforms (Linux, Windows, macOS).1
3.1 The Configuration Structure: AriaGlobOptions
The C++ GlobEngine utilizes a GlobOptions struct with boolean fields to control behavior (e.g., case sensitivity, recursive depth).1 However, the size of a bool is implementation-defined in C++ (usually 1 byte, but not guaranteed) and is not a standard type in C89. To ensure robust interoperability and avoid structure padding issues, the FFI version of this struct, AriaGlobOptions, utilizes explicit integers (uint8_t and uint32_t) to represent boolean states and numeric limits.1
Structure Definition (C Header Context)


C




/**
* Configuration options for the GlobEngine FFI.
* Maps 1:1 with the C++ GlobOptions struct but uses strict C types
* to ensure ABI compatibility across language boundaries.
*/
typedef struct AriaGlobOptions {
   uint8_t case_sensitive;         // 1 = True, 0 = False
   uint8_t follow_symlinks;        // 1 = True, 0 = False
   uint8_t files_only;             // 1 = True, 0 = False
   uint8_t directories_only;       // 1 = True, 0 = False
   uint8_t include_hidden;         // 1 = True, 0 = False
   uint8_t skip_permission_errors; // 1 = True, 0 = False
   uint32_t max_depth;             // Maximum recursion depth (default 64)
} AriaGlobOptions;

Field Semantics and Mapping Strategy
* case_sensitive: Controls whether "File.txt" matches "file.txt". The C++ GlobEngine defaults this to true on Linux and false on Windows/macOS.1 The FFI layer implementation must replicate this logic: if the user passes a null pointer for options, the engine uses these platform-specific defaults. If an AriaGlobOptions struct is provided, the bridge explicitly casts the uint8_t (0 or 1) to a C++ boolean.
* follow_symlinks: Critical for security and infinite loop prevention. The GlobEngine includes graph traversal logic to detect symlink cycles.1 Enabling this flag allows the iterator to follow links, but the engine maintains a set of visited inodes to prevent recursion loops. The FFI exposes this safety toggle directly.
* max_depth: This field prevents stack overflow errors on deeply nested directory trees. The C++ engine defaults this to 64.1 The FFI struct exposes this as a uint32_t, allowing Aria users to override it for specific deep-search queries or restrict it for performance-critical scans.
* include_hidden: On Linux/Unix, files starting with . are treated as hidden. On Windows, file attributes are checked. The FFI exposes this control to allow build systems to ignore .git or .vscode directories efficiently.1
3.2 The Result Structure: AriaGlobResult
The result of a glob operation is a variable-length list of strings (paths). In C++, this is represented as std::vector<std::filesystem::path>.1 The FFI translation layer must flatten this into a C-compatible array of strings to be consumable by Aria's raw pointer handling capabilities.
Structure Definition (C Header Context)


C




/**
* Container for the results of a glob match operation.
* Memory for 'paths' and the strings it points to is allocated
* by the FFI layer on the system heap and must be freed via aria_glob_free().
*/
typedef struct AriaGlobResult {
   char** paths;        // Dynamic array of null-terminated C strings
   size_t count;        // Number of paths in the array
   int error_code;      // 0 = OK, >0 = GlobError code mapping
} AriaGlobResult;

Memory Layout Analysis
* char** paths: This member is a pointer to a block of memory containing pointers to char. Each char* within that block points to a separate, disjoint memory block containing a null-terminated string.
   * Allocation Strategy: The C++ bridge calculates the total number of matches, allocates the master array (sizeof(char*) * count), and then iterates through the std::vector, allocating new memory for each string using strdup (or malloc + memcpy).1
   * Rationale: This "array of pointers" approach (argv style) is the standard C method for representing string lists. While a single contiguous buffer (arena) containing all strings packed back-to-back would be more cache-efficient, it increases the complexity of iteration on the Aria side (calculating offsets). The array-of-pointers style allows Aria to iterate using simple array indexing on the char** pointer.
* count: A size_t representing the number of valid pointers in paths. This maps to uint64 in Aria (assuming a 64-bit architecture).1
* error_code: This integer corresponds directly to the GlobError enum defined in the C++ engine.
   * 0: OK
   * 1: INVALID_BASE_DIR
   * 2: PATTERN_SYNTAX_ERROR
   * 3: ACCESS_DENIED
   * 4: FILESYSTEM_ERROR
   * 99: UNKNOWN_ERROR.1

	4. API Function Specification
The FFI Bridge exposes three primary functions and one utility function. All functions utilize extern "C" linkage to prevent C++ name mangling. Name mangling encodes function parameters into the symbol name (e.g., _Z15aria_glob_matchPKcS0_), which makes dynamic linking from Aria difficult as it requires predicting the compiler-specific mangling scheme. extern "C" forces the linker to use the bare function name (aria_glob_match), simplifying symbol resolution.1
4.1 The Core Match Function: aria_glob_match
This is the primary entry point for simple glob queries. It instantiates the engine with default options, suitable for the majority of use cases.
Signature


C




extern "C" AriaGlobResult aria_glob_match(const char* base_dir, const char* pattern);

Implementation Logic
1. Input Validation: The function immediately validates that base_dir and pattern are not NULL. If either is null, it returns an AriaGlobResult initialized with error_code = UNKNOWN_ERROR (or a specific invalid argument code) and paths = NULL.1 This prevents segmentation faults in the C++ layer.
2. String Conversion: The const char* inputs are implicitly converted to std::filesystem::path objects. This conversion handles character encoding (assuming UTF-8) required by the GlobEngine.1
3. Engine Instantiation: A temporary GlobEngine instance is created on the stack. This ensures thread safety, as no global state is shared between calls.
4. Execution & Exception Handling: The match method is invoked within a try-catch block.
   * std::filesystem::filesystem_error: Caught and translated to GlobError::FILESYSTEM_ERROR (Code 4).
   * std::regex_error: Caught and translated to GlobError::PATTERN_SYNTAX_ERROR (Code 2).1
   * std::exception: Catch-all handler translating to GlobError::UNKNOWN_ERROR (Code 99).
5. Marshaling: Upon successful execution, the resulting std::vector is converted to char**.
   * The implementation allocates the pointer array: char** ptrs = (char**)malloc(vec.size() * sizeof(char*));.
   * It loops through the vector, allocating strings: ptrs[i] = strdup(vec[i].string().c_str());.
   * Failure Recovery: If malloc fails at any point (returning NULL), the implementation must trigger a cleanup routine that iterates backwards from the current index, freeing all successfully allocated strings, freeing the main array, and returning an out-of-memory error code.
4.2 The Configurable Match Function: aria_glob_match_with_options
This variant allows the Aria runtime to pass specific flags (e.g., to enable case insensitivity on Linux or restrict recursion depth).
Signature


C




extern "C" AriaGlobResult aria_glob_match_with_options(
   const char* base_dir, 
   const char* pattern, 
   AriaGlobOptions* options
);

Implementation Logic
1. Option Mapping: The critical step differentiates this function from the basic match. It maps the AriaGlobOptions C struct to the C++ GlobOptions struct.
   * cpp_opts.case_sensitive = (options->case_sensitive!= 0);
   * cpp_opts.max_depth = options->max_depth;
   * This manual mapping ensures that the internal representation of the C++ class remains decoupled from the rigid C struct layout, allowing the C++ class to evolve (e.g., adding private members) without breaking the ABI.1
2. Execution: The process proceeds identically to aria_glob_match, but the GlobEngine is initialized with the explicitly constructed cpp_opts object.
4.3 The Combined Match Function: aria_glob_match_all
The GlobEngine supports matching against multiple patterns simultaneously (logical OR semantics). This is useful for build systems (e.g., "include src/*.cpp AND include/*.h").
Signature


C




extern "C" AriaGlobResult aria_glob_match_all(
   const char* base_dir, 
   const char** patterns, 
   size_t pattern_count, 
   AriaGlobOptions* options
);

Implementation Logic
1. Vector Conversion: The function iterates over the patterns C-array, converting each const char* into a std::string and pushing it into a std::vector<std::string>.
2. Invocation: It calls the match_all method of the GlobEngine.1
3. Result Aggregation: The engine returns a unified, sorted, and deduplicated list of paths, which is then marshaled using the standard logic described in section 4.1.
4.4 The Memory Management Function: aria_glob_free
Because the result structure contains pointers to heap-allocated memory that the C++ runtime does not track (once returned), explicit cleanup is mandatory to avoid memory leaks.
Signature


C




extern "C" void aria_glob_free(AriaGlobResult* result);

Implementation Logic
1. Safety Check: If result is NULL or if result->paths is NULL, the function returns immediately (no-op).1 This allows safe calling on empty or failed results.
2. Deep Free: The function iterates from 0 to result->count - 1, calling free(result->paths[i]) for each string.
3. Container Free: It calls free(result->paths) to release the array of pointers itself.
4. Struct Cleanup:
   * result->paths = NULL;
   * result->count = 0;
   * This defensive programming step prevents double-free errors if aria_glob_free is accidentally called twice on the same struct instance (though the caller should avoid this).
5. Integration with the Aria Runtime
The bridge design requires corresponding declarations within the Aria language to enable consumption. This section defines how the C-side specifications map to Aria definitions, establishing the syntax developers will use to interact with the system.
5.1 FFI Declaration in Aria (std.fs.glob)
To use the FFI functions, the Aria standard library must declare them in an extern block. Aria’s type system supports void* and int32 types that map directly to the C ABI.1 Note the use of wild pointers, which signifies unmanaged memory that the GC will not trace.


Code snippet




// std/fs/glob.aria

// Define the C-compatible struct layout for options
// Order must match the C struct exactly to ensure alignment.
extern struct:GlobOptions = {
   uint8:case_sensitive,
   uint8:follow_symlinks,
   uint8:files_only,
   uint8:directories_only,
   uint8:include_hidden,
   uint8:skip_permission_errors,
   uint32:max_depth
};

// Define the Result struct layout
extern struct:GlobResult = {
   wild char**:paths, // Pointer to array of char* (wild pointers)
   uint64:count,      // size_t maps to uint64 on 64-bit systems
   int32:error_code
};

// External function declarations
extern "C" {
   func:aria_glob_match = GlobResult(wild char*:base, wild char*:pattern);
   func:aria_glob_match_with_options = GlobResult(wild char*:base, wild char*:pattern, wild GlobOptions*:opts);
   func:aria_glob_free = void(wild GlobResult*:res);
}

5.2 The Safe Wrapper API Implementation
Directly calling the extern functions is unsafe due to manual memory management requirements and raw pointer manipulation. The standard library must expose a high-level wrapper that handles the conversion between Aria's managed types and the raw FFI types. This wrapper serves as the "safe" interface for end-users.


Code snippet




// High-level safe wrapper in std.fs
pub func:glob = result<array<string>>(string:pattern, string:base_dir = ".") {
   // 1. Convert Aria strings to C-strings
   // Aria strings are not null-terminated by default; conversion allocates wild memory.
   wild char*:c_pattern = pattern.to_c_str();
   defer aria.free(c_pattern); 
   
   wild char*:c_base = base_dir.to_c_str();
   defer aria.free(c_base);

   // 2. Call FFI
   // The result 'res' contains wild pointers managed by the C host.
   GlobResult:res = aria_glob_match(c_base, c_pattern);
   
   // 3. Ensure cleanup of FFI result
   // The defer ensures aria_glob_free is called even if we return early with error.
   defer aria_glob_free(&res);

   // 4. Check for errors
   if (res.error_code!= 0) {
       // Map integer error code to a descriptive Aria result error
       return result_err("Glob failed with error code: " + res.error_code);
   }

   // 5. Convert results to safe Aria array
   // Allocate a managed array with sufficient capacity
   array<string>:matches = array_new<string>(res.count);
   
   // Iterate using raw pointer arithmetic (allowed in unsafe blocks or via helper)
   // We copy data from wild memory to managed memory here.
   for (i in 0.. res.count) {
       // Extract C-string at index i
       wild char*:path_str = res.paths[i];
       
       // Convert to managed Aria string and push to array
       // string.from_c_str performs a copy, so the source can be freed safely.
       matches.push(string.from_c_str(path_str)); 
   }

   pass(matches);
}

This wrapper demonstrates the complete lifecycle: allocation of temporary C-strings, invocation of the FFI, handling of the result, converting data back to managed types, and ensuring the FFI memory is freed via aria_glob_free regardless of success or failure.1
6. Implementation Strategies and Edge Cases
6.1 Deterministic Sorting for Reproducible Builds
The implementation of the C++ match function must guarantee deterministic output. The GlobEngine specification requires a "Canonical Sort Phase".1 The underlying std::filesystem::recursive_directory_iterator does not guarantee order; the traversal order depends on the operating system's inode table layout or hash table states. This non-determinism is unacceptable for build systems (AriaBuild), where the link order of object files must be identical across builds to ensure binary reproducibility.
Strategy: Before the std::vector is marshaled into the char** array, the C++ implementation must explicitly sort the paths.
* Mechanism: Use std::sort with a custom comparator that performs byte-by-byte lexicographical comparison.
* Implication: This ensures that src/a.txt always appears before src/b.txt, regardless of whether the build runs on Linux (ext4, case-sensitive) or macOS (APFS, case-insensitive).1
6.2 Path Normalization and Cross-Platform Consistency
Paths returned by the OS may contain backslashes on Windows (\) and forward slashes on Unix (/). To simplify the Aria runtime's handling of these strings, the FFI bridge acts as a normalization layer.
Specification: The GlobEngine is configured to output POSIX-style paths (forward slashes) even on Windows. The FFI layer ensures that the char* strings returned in AriaGlobResult conform to this standard.1 This allows Aria code to split paths by / reliably without platform-specific logic in every script.
6.3 Handling "Zero Matches" vs. "Error"
A common ambiguity in globbing implementations is the distinction between finding no files and encountering an error. The FFI bridge clarifies this through explicit return values.
Design Decision:
* Empty Set: If the pattern is valid but matches nothing (e.g., src/*.xyz where no .xyz files exist), error_code must be 0 (OK), and count must be 0. The paths pointer may be NULL or a valid pointer to an empty array. The aria_glob_free function handles NULL gracefully, ensuring this case is safe.1
* Error Condition: If the base directory does not exist, this is treated as an error (INVALID_BASE_DIR), not zero matches. This distinction allows the developer to catch configuration errors (wrong path) versus logic outcomes (empty set).
7. Performance and Safety Analysis
7.1 Overhead Considerations
The FFI design incurs overhead due to the double-copying of strings, which is a trade-off made for safety and isolation.
1. Copy 1: C++ Internal -> std::string / fs::path -> strdup (FFI Heap).
2. Copy 2: FFI Heap -> Aria string (Managed GC Heap).
While a zero-copy approach is theoretically possible (by having the FFI write directly into Aria-allocated buffers), this would tightly couple the C++ engine to Aria's internal memory layout and Garbage Collector. This violation of the isolation principle would make the engine fragile to runtime updates. Given that directory traversal involves costly I/O operations (stat/readdir), the relative CPU cost of string copying is negligible for typical build system workloads.1
7.2 Safety Guarantees
* Null Safety: All pointers passed into the FFI are checked for NULL. This defensive coding prevents the host application from crashing due to script errors.
* Exception Safety: The try-catch block in glob_ffi.cpp guarantees that the C stack is never unwound by a C++ exception. Unwinding across language boundaries is undefined behavior in the Itanium C++ ABI and MSVC, often leading to immediate aborts.1
* Type Safety: The use of explicit struct layouts prevents misalignment between the Aria definition and the C compilation unit. The synchronization of uint8_t in C and uint8 in Aria ensures that bit-widths match perfectly.
8. Conclusion
The FFI interface designed herein provides a robust, safe, and platform-agnostic bridge between the Aria runtime and the high-performance C++ GlobEngine. By strictly adhering to C ABI standards, employing the "FFI Firewall" pattern to contain exceptions, and enforcing a clear "Caller Owns Memory" protocol, this specification ensures that Aria's standard library can gain powerful filesystem capabilities without compromising stability. The decoupling of memory management via aria_glob_free allows the Aria runtime to integrate these external resources seamlessly into its own lifetime management constructs (defer), paving the way for the completion of the std.fs module and the broader Aria ecosystem. This bridge not only solves the immediate bootstrapping issue for aria_make but establishes a resilient pattern for all future native integrations.
