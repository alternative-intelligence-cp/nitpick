ARCHITECTURAL SPECIFICATION AND IMPLEMENTATION STRATEGY FOR THE ARIA DEBUGGER SUBSYSTEM
1. Executive Summary and Strategic Context
The maturation of the Aria programming language ecosystem, as it transitions from a theoretical specification to a production-grade compiler and runtime environment, necessitates a parallel evolution in its developer tooling. While the Aria compiler (ariac) currently provides robust mechanisms for code generation via LLVM 20.1.2, including complex semantic analysis for "borrow checking" and "Twisted Balanced Binary" (TBB) arithmetic, the absence of a dedicated debugging infrastructure represents a critical gap in the developer experience.1 The current reliance on generic tools like GDB or basic logging restricts the developer's ability to inspect Aria's unique runtime characteristics—specifically the hybrid memory model (distinguishing between wild, gc, and wildx regions), the semantic nuances of TBB error propagation, and the non-blocking state machines of the asynchronous runtime.1
This report articulates a comprehensive architectural blueprint for aria-debug, a monolithic, self-contained debugging utility designed to provide deep introspection into Aria programs. Unlike generic C/C++ debuggers which view memory as untyped byte arrays, aria-debug is architected as a semantic lens that understands Aria's invariants. It leverages the robust backend of LLDB (via the C++ SB API) while presenting a dual-interface frontend: a standard Debug Adapter Protocol (DAP) interface for IDE integration (VS Code, Neovim) and a bespoke, high-fidelity Web Interface for visualizing complex memory topologies and async dependency graphs.1
The design philosophy detailed herein prioritizes "Batteries-Included Introspection." This means the debugger is not merely a passive observer of register states but an active interpreter of the runtime. It decodes 64-bit object headers to visualize GC generations, validates TBB error sentinels to detect "sticky" arithmetic errors before they propagate, and reconstructs logical call stacks from heap-allocated coroutine frames.1 By adhering to the constraints of C++17 and zero external dependencies, this specification ensures the debugger remains a portable, single-binary distribution capable of running on Linux, macOS, and Windows, fully aligning with the cross-platform nature of the LLVM infrastructure already integrated into the project.1

2. Architectural Design and System Topology
The architecture of the Aria Debugger is predicated on a layered abstraction model that decouples the user interface from the low-level mechanics of process control. This separation is vital for supporting multiple concurrent clients—such as a VS Code extension for line-by-line stepping and a browser-based memory profiler for heap visualization—without duplicating the core debugging logic.
2.1 The Monolithic Server Architecture
The system operates as a single executable process, aria-debug, which acts as a bridge between the external world (editors, browsers) and the target inferior process. The decision to implement this as a monolithic server rather than a library or a set of disparate tools is driven by the need for coherent state management. When a user pauses execution in the editor, the Web UI must immediately reflect this state change; keeping both interfaces driven by a shared event loop ensures atomic synchronization of the debugging session.
The internal topology is stratified into four distinct layers:
1. The Transport Layer: This layer manages physical communication channels. It simultaneously listens on stdin/stdout for standard DAP messages (used by IDEs) and on a TCP port for WebSocket connections (used by the Web UI). An authentication gate ensures that only authorized clients can attach to the session, mitigating security risks in shared development environments.
2. The Protocol Layer: This layer is responsible for marshaling and unmarshaling data. It implements the Microsoft Debug Adapter Protocol (DAP) as the canonical language for control flow (breakpoints, stepping, stack traces). Crucially, it extends DAP with Aria-specific custom requests (e.g., ariaMemoryMap, ariaTraceDefer) to expose features not covered by the standard protocol specification.
3. The Semantics Engine: This is the "brain" of the debugger. It contains the logic that differentiates aria-debug from a raw LLDB session. It includes the TBBInspector for decoding semantic integers, the MemoryVisualizer for classifying heap regions, and the AsyncFlowTracker for reconstructing coroutine chains.
4. The Backend Abstraction: This layer wraps the liblldb C++ API (SBTarget, SBProcess, SBThread). It translates high-level semantic requests into low-level ptrace or Mach-port operations, handling the platform-specific idiosyncrasies of Linux, macOS, and Windows.
2.2 Integration with LLVM and LLDB
The choice to build upon LLDB (Low Level Debugger) is strategic. Since the Aria compiler emits LLVM IR 1, the resulting binaries are compatible with standard DWARF debugging information. LLDB provides the most mature C++ API for inspecting this information. By linking against liblldb (which is already part of the project's LLVM 20.1.2 dependency), aria-debug inherits:
* JIT Support: LLDB’s Just-In-Time compilation capabilities allow the debugger to compile and execute Aria expressions typed into the console at runtime. This is essential for evaluating complex expressions in the context of the current stack frame.
* Platform Support: LLDB handles the grim details of unwinding stacks on x86_64, ARM64, and Windows ABI, allowing the Aria team to focus on language semantics rather than register spilling conventions.
* Symbolication: The SBModule and SBSymbolContext classes provide efficient access to the DWARF debug info, mapping instruction pointers to source file locations without requiring a custom DWARF parser implementation.2
2.3 The Event Loop Concurrency Model
A critical architectural decision is the threading model for the debugger process. Debugging is inherently asynchronous; the target process runs, stops, emits signals, and exits unpredictably. A naive single-threaded approach would cause the UI to freeze while waiting for the target.
The aria-debug architecture employs a dual-thread model managed by a central EventLoop:
1. The Command Thread (Main): This thread owns the Transport Layer. It performs blocking reads on stdin (for DAP) and manages the epoll / kqueue loop for WebSocket connections. It processes incoming requests (e.g., "set breakpoint") and dispatches them to the debugger engine.
2. The Event Thread: This thread calls SBListener::WaitForEvent in a loop. It is dedicated to receiving asynchronous notifications from the LLDB backend (e.g., "process stopped at breakpoint", "process exited", "stdout output"). When an event occurs, this thread serializes a DAP Event (e.g., stopped, output) and pushes it to the Transport Layer for broadcast to all connected clients.
This split ensures that the debugger remains responsive to user input even when the target process is running or hung. Synchronization is managed via thread-safe queues for outgoing messages and a std::recursive_mutex protecting the SBDebugger instance, ensuring that the Command Thread does not attempt to inspect memory while the Event Thread is handling a resume operation.1

3. Protocol Design and Transport Specifications
The robustness of the debugger depends on a rigorously defined communication protocol. We adopt the Debug Adapter Protocol (DAP) as the foundation but extend it significantly to support Aria's novel features.
3.1 The Debug Adapter Protocol (DAP)
DAP is a JSON-RPC based protocol. Messages are framed using HTTP-like headers, specifically Content-Length.
Standard Request Flow:
1. Client: Sends initialize.
2. Server: Responds with capabilities (supports configurationDone, supportsSetVariable, etc.).
3. Client: Sends launch (with program path) or attach (with PID).
4. Server: Spawns the process via SBTarget::Launch.
5. Client: Sends setBreakpoints.
6. Client: Sends configurationDone.
7. Server: Resumes execution via SBProcess::Continue.
Custom Aria Extensions:
Standard DAP lacks verbs for memory visualization or type-specific introspection. We define the following custom requests which are transmitted via the standard customRequest mechanism in DAP clients:
* ariaMemoryMap Request:
   * Arguments: None.
   * Response: A JSON object containing a list of MemoryRegion structures. Each structure defines start_addr, end_addr, type (GC_NURSERY, GC_OLD, WILD, WILDX), and fragmentation_score.
   * Use Case: Populating the Web UI heatmap.
* ariaTraceDefer Request:
   * Arguments: threadId (int).
   * Response: A JSON array of DeferBlock structures, each containing file, line, and creation_timestamp.
   * Use Case: Visualizing the stack of deferred cleanup actions, which often differ from the call stack.
* ariaValidateScope Request:
   * Arguments: pointer_address (hex string).
   * Response: JSON object { "valid": boolean, "scope_id": int, "alloc_id": int, "error": string }.
   * Use Case: On-demand validation of "Fat Pointers" to debug Use-After-Free errors without crashing the process.
3.2 The WebSocket Transport for Web UI
To support the Web UI, the debugger implements a lightweight WebSocket server. This runs alongside the stdio transport.
Handshake and Authentication:
Upon startup, aria-debug generates a cryptographically secure random token (e.g., 32 bytes hex encoded). This token is printed to stdout for the user to copy.
The Web UI must include this token in the initial HTTP Upgrade request:
GET /debug?token=<TOKEN> HTTP/1.1
Upgrade: websocket
If the token matches, the connection is upgraded. If not, the socket is closed immediately. This prevents malicious scripts on websites from connecting to a developer's local debugger and inspecting memory.
Protocol Tunneling:
Once connected, the WebSocket tunnel carries standard DAP JSON messages. This simplifies the frontend implementation, as the JavaScript client can use the same logic as a VS Code extension to parse stack traces and variable lists.

4. Deep Dive: Memory Model Introspection
One of the most complex aspects of debugging Aria is visualizing its Hybrid Memory Model. Aria programs partition memory into four distinct classes: Stack, GC Heap, Wild Heap, and WildX.1 A generic debugger sees these all as identical virtual memory pages. aria-debug must apply semantic classification.
4.1 Visualizing the Hybrid Heap
The MemoryVisualizer component is responsible for generating the "God's Eye View" of the process memory.
Classification Algorithm:
1. OS Query: The debugger queries the operating system for the memory map. On Linux, it reads /proc/<pid>/maps. On macOS, it calls mach_vm_region. On Windows, it uses VirtualQueryEx. This yields a list of ranges with permissions (rwx).
2. WildX Detection: Any range with r-x or rwx permissions that is not backed by the main executable file or shared libraries is classified as WILDX. These are JIT-compiled regions.1
3. GC Heap Detection: The Aria runtime exports global symbols __aria_gc_heap_start and __aria_gc_heap_end. The debugger reads these symbol values to identify the contiguous range reserved for the Garbage Collector.
4. Nursery vs Old Gen: Within the GC range, the runtime exports __aria_gc_nursery_limit. Addresses below this limit are GC_NURSERY (green); above are GC_OLD_GEN (blue).
5. Wild Allocations: Ranges that are rw- (anonymous mappings) but fall outside the GC heap and Stack are classified as WILD (grey). These are large buffers allocated via aria_alloc (mimalloc wrapper).
4.2 Object Header Decoding
Inside the GC Heap, memory is not an amorphous blob; it is structured as objects prefixed with 64-bit headers. The debugger allows users to inspect these headers to diagnose corruption or pinning issues.
The Object Header (ObjHeader) Layout:
According to the specification 1, the header is a packed 64-bit integer containing:
* Bits 0-7: Flags (mark_bit, pinned_bit, forwarded_bit, is_nursery).
* Bits 8-31: size_class (24 bits) - Index into the size class table.
* Bits 32-63: type_id (32 bits) - Runtime type identifier.
Synthetic Provider Implementation:
The debugger registers a synthetic child provider for any pointer type aliased to gc_ptr<T>. When the user expands a variable x of type gc MyStruct* in the IDE:
1. The debugger reads the address addr = x.
2. It reads 8 bytes from addr - 8 (the header).
3. It performs bitwise masking to extract the fields.
4. It presents synthetic children [header]:
   * pinned: true (Visualized as a lock icon in Web UI).
   * generation: nursery.
   * size_class: 32 bytes.
   * type_id: 0x4F2A (Debuggers can map this back to type names using DWARF lookup).
4.3 Fat Pointers and Safety Mode
In debug builds (ARIA_DEBUG), the compiler promotes raw pointers to "Fat Pointers" to track provenance and bounds.1 The debugger must understand this struct layout to display pointers correctly.
Struct Layout:


C++




struct aria_fat_pointer {
   void* ptr;      // 0x00: The actual pointer
   void* base;     // 0x08: The base of the allocation
   uint64_t size;  // 0x10: The size of the allocation
   uint64_t id;    // 0x18: The unique allocation ID
};

Validation Logic:
When the user requests to view a Fat Pointer, the debugger performs a consistency check:
1. Bounds Check: Is base <= ptr < base + size? If not, display (OUT OF BOUNDS) warning.
2. Liveness Check: The debugger queries the runtime's "Active Allocation Table" (a hash map in the debug runtime) using id. If the ID is not found, or marked as freed, display (USE AFTER FREE) warning. This transforms cryptic segfaults into immediate, actionable error messages.

5. TBB Type Inspection and Error Propagation
Twisted Balanced Binary (TBB) arithmetic is a cornerstone of Aria's design, providing symmetric ranges and a dedicated ERR sentinel. However, to a standard debugger, tbb8 is just a signed int8. The Aria Debugger solves this impedance mismatch through custom data formatters.
5.1 The Mathematics of TBB Visualization
Standard integer types use two's complement. int8 ranges from -128 to +127.
Aria's tbb8 uses the range -127 to +127. The value -128 (0x80) is reserved as the ERR sentinel.
Formatter Logic:
The debugger implements a TBBTypeSummaryProvider C++ class. This class is registered to handle any type matching the regex ^tbb(8|16|32|64)$.
* Step 1: Sentinel Calculation:
The sentinel is always the minimum representable value for the underlying signed integer.

$$S_{tbbN} = -2^{(N-1)}$$
   * tbb8: -128
   * tbb16: -32768
   * tbb32: -2,147,483,648
   * tbb64: -9,223,372,036,854,775,808
   * Step 2: Value Classification:
The formatter reads the raw scalar value V.
      * Case A (Sentinel): If V == S, the formatter returns the string literal "ERR". This is crucial; seeing "ERR" is infinitely more informative than seeing "-128".
      * Case B (Overflow): While the storage allows standard two's complement values, the valid semantic range is symmetric: $[-(2^{N-1}-1), +(2^{N-1}-1)]$. If V falls outside this range (but is not S), it represents a transient overflow state that hasn't yet collapsed to ERR. The debugger displays "{V} (OVERFLOW)".
      * Case C (Normal): Otherwise, display "{V}".
5.2 Sticky Error Visualization
The "Sticky Error" property means that ERR + x = ERR. In a long chain of calculations, it can be hard to find where the error originated.
Web UI Feature: The Error Trace:
In the Web UI, the debugger tracks the history of TBB variables (if "Time Travel" debugging is enabled via rr integration, or simply by step-logging). If a variable transitions from a valid value to ERR, the timeline highlights that exact step in red. Even without time travel, the UI visualizes TBBs as gauges.
      * Gauge Visualization: A linear bar centered at 0.
      * Green Zone: The valid symmetric range.
      * Red Marker: The ERR sentinel.
      * Needle: The current value.
This allows developers to instantly spot values that are dangerously close to the overflow boundary.

6. Asynchronous Control Flow Debugging
Debugging async/await code is notoriously difficult because the "physical" call stack (what the CPU executes) often just shows the Event Loop and a generic poll function, having lost the context of the "logical" caller.
6.1 Coroutine State Machine Reconstruction
Aria compiles async functions into state machines. The local variables are spilled into a heap-allocated CoroutineFrame.
Structure of CoroutineFrame:


C++




struct CoroutineFrame {
   void* resume_fn;
   void* destroy_fn;
   uint32_t state_index; // Current await point
   void* promise;        // Reference to the Future<T>
   //... captured variables...
};

The Logical Stack Algorithm:
When the debugger stops inside an async function:
         1. Identify Frame: It detects that the current function is a coroutine ramp/resume function (via DWARF attributes or symbol naming conventions like _Aria_Async_...).
         2. Locate Frame Pointer: It reads the this pointer (or first argument) which points to the CoroutineFrame.
         3. Inspect Promise: It reads the promise field. In Aria's runtime, the Promise object contains a list of awaiters—other coroutines that are waiting for this one to complete.
         4. Traverse Upward: The debugger recursively walks this chain of awaiters. For each awaiter, it looks up the source location corresponding to its current state_index.
         5. Synthesize Stack: It constructs a "Virtual Stack Trace" composed of these awaiters and presents it to the UI. This allows the developer to see main -> await fetch -> await connect, even though main is not physically on the stack.
6.2 Visualizing pick and fall
The pick statement with explicit fall() is another unique control flow construct. The compiler lowers this to a series of Basic Blocks.
Debugging fall():
When stepping through a pick statement, a fall(label) command acts like a goto. Standard debuggers handle this fine, but the Aria Debugger adds semantic value.
         * Transition Prediction: In the CLI/Web UI, when the cursor is on a fall() statement, the debugger calculates the target label and displays a "phantom arrow" pointing to where execution will jump next.
         * Cycle Detection: If a pick statement contains a fall cycle (Case A falls to Case B which falls to Case A), the debugger's static analysis module (integrated from the compiler) flags this as a potential infinite loop in the UI.

7. Web Interface Architecture
The Web Interface is designed to be a lightweight, single-page application (SPA) served directly from the aria-debug binary.
7.1 Zero-Dependency Embedded Server
To satisfy the "No External Dependencies" constraint, we cannot link against heavy frameworks like Node.js or Python. The server is implemented in C++ using Berkeley Sockets.
Architecture:
         1. Asset Storage: The HTML, CSS, and JS files for the UI are compressed and embedded into the C++ source code as static byte arrays (xxd -i) during the build process.
         2. HTTP Handling: The server listens on a loopback port. Upon receiving a GET / request, it serves the embedded index.html.
         3. WebSocket Upgrade: Upon receiving Connection: Upgrade, it performs the SHA-1 handshake and switches the socket to binary mode for DAP traffic.
7.2 Frontend Technology Stack
         * Framework: Vanilla ES6 JavaScript (No React/Vue/Angular). This ensures the debugger remains functional for decades without suffering from JS ecosystem churn.
         * Rendering:
         * Memory Map: HTML5 <canvas> API. This allows rendering millions of heap blocks (pixels) efficiently (60fps), which DOM elements cannot handle.
         * Text/Code: standard DOM nodes with Monaco Editor (loaded from CDN or embedded) for source viewing if local file access is permitted.
         * State Management: A simple Store pattern (Observer) handling JSON-RPC events from the WebSocket.

8. Implementation Roadmap
The implementation is structured into four distinct phases, prioritizing the CLI and fundamental LLVM integration before moving to advanced visual features.
Phase 1: The Foundation (Weeks 1-4)
Goal: A working CLI debugger that can launch, break, and step.
         * Task 1.1: Integrate llvm::DIBuilder into IRGenerator.1 Ensure line tables and subprograms are emitted correctly.
         * Task 1.2: Implement the DAPServer skeleton handling initialize and launch.
         * Task 1.3: Link liblldb and implement the SBTarget creation logic.
         * Task 1.4: Verify setBreakpoints using source-to-line mapping.
Phase 2: Runtime Awareness (Weeks 5-8)
Goal: Correct display of Aria types and basic memory inspection.
         * Task 2.1: Implement TBBTypeSummaryProvider (C++) to handle ERR sentinels.
         * Task 2.2: Update CodeGenContext to emit DW_TAG_typedef for TBB types so LLDB recognizes them.
         * Task 2.3: Implement GCPointerSyntheticProvider to decode object headers.
         * Task 2.4: Implement Result<T> pretty-printing (Ok(v) vs Error(c)).
Phase 3: The Web Experience (Weeks 9-12)
Goal: Interactive memory visualization.
         * Task 3.1: Implement the embedded C++ WebSocket server.
         * Task 3.2: Build the HTML5/Canvas frontend for the Memory Heatmap.
         * Task 3.3: Implement the backend MemoryVisualizer to scan /proc/self/maps.
         * Task 3.4: Create the ariaMemoryMap custom DAP request.
Phase 4: Async & Safety (Weeks 13-16)
Goal: Deep concurrency debugging and safety validation.
         * Task 4.1: Implement AsyncDebugger to walk coroutine promise chains.
         * Task 4.2: Integrate FatPointer validation logic (checking alloc_id).
         * Task 4.3: Implement trace-defer to visualize the cleanup stack.

9. Conclusion
The architectural specification provided herein offers a robust, "batteries-included" path to delivering world-class debugging tooling for the Aria language. By leveraging the industrial strength of LLVM/LLDB for the heavy lifting of process control and symbolication, the Aria team can focus its resources on the high-value, language-specific domains: TBB semantics, Hybrid Memory visualization, and Asynchronous flow reconstruction.
The resulting tool, aria-debug, will not only satisfy the immediate needs of systems programmers but will serve as a differentiator for the language itself. The ability to visualize memory fragmentation in real-time or instantly identify the source of a "Sticky Error" provides a developer experience superior to that of established languages like C++ or Go. This deep integration of tooling and runtime is the hallmark of a modern, mature systems language ecosystem.

B.1 LLVM Debug Info Setup (Compiler Side)
This C++ snippet demonstrates how the compiler backend initializes the DIBuilder to emit DWARF metadata compatible with the debugger.


C++




// src/backend/ir/ir_generator.cpp
void IRGenerator::initDebugInfo(std::string filename, std::string directory) {
   if (!debug_enabled) return;
   
   // Initialize the DWARF Builder attached to the current module
   diBuilder = std::make_unique<llvm::DIBuilder>(*module);
   
   // Create the Compile Unit (Root Scope)
   diCU = diBuilder->createCompileUnit(
       llvm::dwarf::DW_LANG_C, // Placeholder; eventually register DW_LANG_Aria
       diBuilder->createFile(filename, directory),
       "Aria Compiler v0.0.7", // Producer String
       false,                  // isOptimized
       "",                     // Flags
       0                       // Runtime Version
   );
   
   // Register TBB Typedefs so the Debugger can recognize semantics
   // We map 'tbb8' to an 8-bit signed integer, but wrapped in a typedef.
   auto int8Ty = diBuilder->createBasicType("int8", 8, llvm::dwarf::DW_ATE_signed);
   diBuilder->createTypedef(int8Ty, "tbb8", diFile, 0, diCU);
   
   // This allows the debugger's data formatter to match on "tbb8" string.
}

B.2 TBB Data Formatter (Debugger Side)
This implementation shows how the debugger intercepts variable display to render TBB semantics using the LLDB C++ API.


C++




// src/tools/debugger/aria_formatters.cpp
bool TBBTypeSummaryProvider::GetSummary(lldb::SBValue valobj, lldb::SBStream& stream, 
                                     const lldb::SBTypeSummaryOptions& options) {
   if (!valobj.IsValid()) return false;

   // 1. Extract Bit Width from Type Name (e.g., "tbb8", "tbb32")
   std::string typeName = valobj.GetTypeName();
   int bitWidth = 0;
   if (typeName == "tbb8") bitWidth = 8;
   else if (typeName == "tbb16") bitWidth = 16;
   else if (typeName == "tbb32") bitWidth = 32;
   else if (typeName == "tbb64") bitWidth = 64;
   
   if (bitWidth == 0) return false; // Not a TBB type

   // 2. Calculate Sentinel and Range
   // Sentinel is -2^(N-1) (e.g., -128 for 8-bit)
   int64_t rawValue = valobj.GetValueAsSigned();
   int64_t sentinel = -(1LL << (bitWidth - 1));
   int64_t maxVal = (1LL << (bitWidth - 1)) - 1;

   // 3. Format based on Semantic Rules
   if (rawValue == sentinel) {
       stream.Printf("ERR"); 
   } else {
       // Check symmetric range bounds for overflow detection
       if (rawValue < -maxVal |

| rawValue > maxVal) {
            stream.Printf("%lld (OVERFLOW)", rawValue);
       } else {
            stream.Printf("%lld", rawValue);
       }
   }
   return true;
}
