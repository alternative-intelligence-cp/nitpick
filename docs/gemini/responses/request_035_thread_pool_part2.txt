Architectural Specification: High-Throughput Thread Pool Integration for Aria Language Server
1. Executive Summary and Strategic Context
The Aria Language Server (aria-ls) currently operates as a synchronous, single-threaded application. While this architecture—referencing the current implementation in src/tools/lsp/server.cpp—has successfully verified the core correctness of the Language Server Protocol (LSP) handling, including handshake lifecycles and basic document synchronization, it faces an insurmountable scalability barrier. The current design processes requests serially on the main thread. This means that parsing operations, semantic analysis, and diagnostic generation—tasks with algorithmic complexities ranging from $O(N)$ to $O(N^2)$ relative to document size—block the I/O loop.
For a systems programming language like Aria, which aims to support high-performance applications including kernels and game engines, the developer experience (DX) must match the performance of the runtime. A "stop-the-world" pause during a compilation phase results in an unresponsive editor: typing lags, cancellation requests are ignored, and the "spiral of death" occurs where the server processes obsolete document states while the user continues to type.
This research report specifies a comprehensive architectural transition to a Thread Pool Model for Phase 7.3.6. This transition shifts aria-ls from a synchronous Reactor pattern to an asynchronous Proactor-like architecture (specifically the Half-Sync/Half-Async pattern). By decoupling the Input/Output (I/O) multiplexing from the computational workload, the server guarantees responsiveness to critical protocol notifications (like $/cancelRequest and textDocument/didChange) regardless of the depth of the analysis queue.
The design adheres strictly to the C++17 standard, leveraging std::thread, std::condition_variable, and std::shared_mutex to ensure portability and compliance with the existing codebase constraints.1 It integrates with the established VirtualFileSystem (VFS) and maintains strict ordering semantics required by the LSP specification 2, particularly for document mutation events.

2. Architectural Analysis and Pattern Selection
2.1 The Limits of the Current Single-Threaded Model
The current implementation residing in src/tools/lsp/server.cpp utilizes a polling loop or blocking read on stdin.1 When a message arrives, the server deserializes it using nlohmann::json and immediately dispatches it to a handler function (e.g., handle_did_change).
This model is insufficient for the following reasons:
1. I/O Starvation: While the parser is traversing a 10,000-line Aria source file to generate diagnostics, the operating system's pipe buffer for stdin fills up. If the user issues a $/cancelRequest (because they moved the cursor), the server cannot read this request until after the expensive operation completes, rendering the cancellation logic useless.3
2. Latency Spikes: Operations like textDocument/completion (IntelliSense) require sub-50ms latency to feel "instant." If a background indexing task is running on the only available thread, the user sees "Loading..." indefinitely.
3. Resource Underutilization: Modern developer workstations possess high core counts (8-32 cores). A single-threaded compiler leaves 90%+ of the CPU idle while the user waits for feedback.
2.2 Alternative Concurrency Models
Before selecting the Thread Pool pattern, we evaluate alternatives against the C++17 and Aria constraints:
* Thread-Per-Request: Spawning a std::thread for every incoming JSON-RPC request.
   * Analysis: This creates unbounded resource consumption. If a client sends 100 requests in a burst (e.g., rapid hovering or typing), 100 threads are spawned. The overhead of context switching and stack allocation (typically 1MB-8MB per thread) becomes prohibitive.4 It also creates complex synchronization issues for didChange ordering.
* std::async (Task-Based Parallelism):
   * Analysis: While std::async is simpler, the C++17 standard does not guarantee thread reuse (launch policy implementations vary). More critically, std::future destructors can block the main thread, defeating the purpose of async I/O. std::async also lacks a mechanism for request prioritization or cancellation, which are strictly required by LSP.6
* Coroutines (C++20):
   * Analysis: While efficient, this violates the strict C++17 constraint imposed by the project requirements.
2.3 Selected Pattern: Half-Sync/Half-Async (Leader/Followers)
The optimal architecture for AriaLS is a fixed-size Worker Thread Pool fed by a thread-safe Priority Queue, managed by a dedicated Main Thread I/O Pump.
2.3.1 Main Thread Responsibilities (The Sync Layer)
The Main Thread acts as the high-availability "control plane." Its duties are restricted to operations with $O(1)$ or $O(L)$ complexity (where $L$ is message length):
1. Transport I/O: Reading raw bytes from stdin, verifying Content-Length headers, and extracting JSON payloads.
2. Unmarshalling: Parsing JSON text into objects.
3. Routing: Classifying messages as Requests vs. Notifications and determining dispatch priority.
4. Queueing: Encapsulating the work into Task objects and pushing them to the WorkQueue.
5. Cancellation Management: Intercepting $/cancelRequest and flagging the corresponding task's cancellation token immediately.7
6. Response Marshalling: Serializing JSON responses (generated by workers) to stdout in a thread-safe manner.
2.3.2 Worker Pool Responsibilities (The Async Layer)
The Worker Pool constitutes the "data plane."
1. Compilation: Instantiating the Lexer, Parser, and DiagnosticEngine.
2. Analysis: Traversing the AST to answer hover, definition, or references queries.
3. VFS Access: Acquiring Read locks (std::shared_lock) or Write locks (std::unique_lock) on the global VirtualFileSystem.
4. Fault Tolerance: Acting as a firewall for compiler crashes to prevent server termination.

3. C++ Implementation Details (Phase 7.3.6)
To meet the requirements, we must construct a robust ThreadPool class using C++17 primitives. The implementation must handle task prioritization, cancellation, and graceful shutdown.
3.1 The Task Structure
A simple std::function<void()> is insufficient because we need metadata for prioritization and cancellation.8 We introduce a Task struct.


C++




// include/tools/lsp/task.h
#include <functional>
#include <memory>
#include <atomic>
#include <nlohmann/json.hpp>

namespace aria::lsp {

enum class TaskPriority {
   CRITICAL = 0, // didChange, didOpen (State mutation)
   HIGH = 1,     // completion, signatureHelp (Latency sensitive)
   NORMAL = 2,   // hover, definition
   LOW = 3       // diagnostics, indexing
};

// Lightweight cancellation token shared between Main Thread and Worker
class CancellationToken {
public:
   CancellationToken() : cancelled_(std::make_shared<std::atomic<bool>>(false)) {}
   
   void cancel() { cancelled_->store(true, std::memory_order_relaxed); }
   bool is_cancelled() const { return cancelled_->load(std::memory_order_relaxed); }

private:
   std::shared_ptr<std::atomic<bool>> cancelled_;
};

struct Task {
   std::string method;
   nlohmann::json id; // Null if notification
   TaskPriority priority;
   CancellationToken token;
   std::function<void()> action;
   uint64_t sequence_id; // To maintain FIFO within priority levels

   // Priority queue operator: Highest priority (lowest enum value) first
   bool operator<(const Task& other) const {
       if (priority!= other.priority) {
           return priority > other.priority; // Lower value = higher priority
       }
       return sequence_id > other.sequence_id; // Lower ID = earlier submission
   }
};

}

3.2 Thread-Safe Work Queue
We utilize std::priority_queue protected by std::mutex and signaled via std::condition_variable.8 The queue must handle the "Shutdown" signal gracefully.


C++




// include/tools/lsp/work_queue.h
#include <queue>
#include <mutex>
#include <condition_variable>
#include <optional>
#include "task.h"

namespace aria::lsp {

class WorkQueue {
public:
   void push(Task task) {
       {
           std::lock_guard<std::mutex> lock(mutex_);
           queue_.push(std::move(task));
       }
       cv_.notify_one();
   }

   std::optional<Task> pop() {
       std::unique_lock<std::mutex> lock(mutex_);
       // Wait until queue has items OR shutdown is requested
       cv_.wait(lock, [this] { return!queue_.empty() |

| shutdown_; });

       if (shutdown_ && queue_.empty()) {
           return std::nullopt; // Signal worker to exit
       }

       Task task = std::move(queue_.top());
       queue_.pop();
       return task;
   }

   void shutdown() {
       {
           std::lock_guard<std::mutex> lock(mutex_);
           shutdown_ = true;
       }
       cv_.notify_all(); // Wake up all workers
   }

private:
   std::priority_queue<Task> queue_;
   std::mutex mutex_;
   std::condition_variable cv_;
   bool shutdown_ = false;
};

}

3.3 The Thread Pool Implementation
The pool manages the lifecycle of the worker threads. It initializes them on construction and joins them on destruction.


C++




// include/tools/lsp/thread_pool.h
#include <vector>
#include <thread>
#include "work_queue.h"

namespace aria::lsp {

class ThreadPool {
public:
   explicit ThreadPool(size_t num_threads = std::thread::hardware_concurrency()) {
       // Fallback if hardware_concurrency returns 0
       size_t count = num_threads > 0? num_threads : 2;
       
       for (size_t i = 0; i < count; ++i) {
           workers_.emplace_back([this] { worker_loop(); });
       }
   }

   ~ThreadPool() {
       queue_.shutdown();
       for (auto& worker : workers_) {
           if (worker.joinable()) worker.join();
       }
   }

   void submit(Task task) {
       queue_.push(std::move(task));
   }

private:
   void worker_loop() {
       while (true) {
           auto task_opt = queue_.pop();
           if (!task_opt) return; // Shutdown

           Task& task = *task_opt;

           // PRE-EXECUTION CANCELLATION CHECK
           if (task.token.is_cancelled()) {
               // Task was cancelled while in queue. 
               // Skip execution. Main thread has likely already cleaned up.
               continue;
           }

           try {
               task.action();
           } catch (const std::exception& e) {
               // "Firewall" - prevent worker death
               std::cerr << "Worker Error: " << e.what() << std::endl;
               // In production, send window/showMessage to client
           } catch (...) {
               std::cerr << "Unknown Worker Error" << std::endl;
           }
       }
   }

   std::vector<std::thread> workers_;
   WorkQueue queue_;
};

}

3.4 Result Passing
Instead of std::future, which implies blocking the main thread to get() the result, we use a callback mechanism. The Task's action lambda will capture the Server instance and call Server::send_response. This method must be thread-safe (mutex-protected writing to stdout).

	4.2 The didChange Ordering Problem
The LSP specification guarantees that the client sends didChange notifications in the correct order.2 However, a thread pool can violate this order if two didChange events are processed by two different workers simultaneously. If Version 2 finishes before Version 1, the VFS will be left in an inconsistent state (Version 1).
Ordering Strategy: The Mutation Barrier
1. Critical Priority: didChange tasks are assigned CRITICAL priority.
2. Sequence IDs: The WorkQueue enforces FIFO for same-priority tasks via sequence_id.
3. Exclusive Locking: didChange tasks acquire a std::unique_lock (Write Lock) on the VFS.
   * Since std::shared_mutex allows only one writer (and no readers) at a time, the OS scheduler effectively serializes these tasks. Even if two threads pick up two consecutive didChange tasks, one will block on the mutex until the other finishes.
   * Note: This relies on the OS mutex implementation being fair or the tasks execution time being significantly larger than scheduling noise. For strict correctness in high-load scenarios, a "strand" (a serialized queue per file URI) is theoretically superior, but given the "Full Sync" model of Aria (sending full text on change), simply ensuring the latest version wins is often sufficient. Strict Barrier: The worker pool logic can enforce that if a CRITICAL task is running, no other tasks for that Document URI run.
4.3 VFS Synchronization
The VirtualFileSystem in include/tools/lsp/vfs.h uses std::shared_mutex.
* Writers (didOpen, didChange): Use std::unique_lock. This blocks all readers.
* Readers (hover, definition): Use std::shared_lock. Multiple readers can access the VFS simultaneously.
The "Snapshot" Optimization:
Holding a VFS Read Lock for the entire duration of a parse (which might take 200ms) is dangerous because it starves Writers (didChange). If a user types, the didChange will block waiting for the hover to release the lock, freezing the editor.
Solution:
1. Worker acquires Read Lock.
2. Worker copies the source code string from VFS to a local std::string.
3. Worker releases Read Lock immediately.
4. Worker parses the local copy.
This reduces lock contention from $O(ParseTime)$ to $O(CopyTime)$, effectively decoupling readers from writers.
4.4 Handling $/cancelRequest
The cancellation workflow:
1. Client sends $/cancelRequest with id: 123.
2. Main Thread receives it. It looks up id: 123 in a std::map<json, std::shared_ptr<CancellationToken>>.
3. Main Thread calls token->cancel().
4. Worker Thread (processing id: 123):
   * Checks token->is_cancelled() inside the loop (e.g., in the Parser).
   * If true, throws OperationCancelledException.
5. ThreadPool catches exception.
6. Server sends JSON-RPC Error -32800 (RequestCancelled).3

5. Error Handling and Resilience
5.1 Worker Exception Firewall
The worker_loop must contain a global try-catch block.1 If the Aria compiler (Lexer/Parser) throws an unhandled exception (e.g., std::bad_alloc or a logic error on malformed code), it must not terminate the process.
* Action: Catch std::exception. Log the error to stderr. Send an LSP InternalError (-32603) to the client for that specific request ID. The worker thread then returns to the pool to service new requests.
5.2 Thread Pool Shutdown
On exit notification:
1. Main thread calls thread_pool.shutdown().
2. WorkQueue sets shutdown_ = true and calls cv_.notify_all().
3. Workers wake up, see shutdown_ is true and queue is empty, and exit their loops.
4. Main thread calls worker.join() for all threads.
This ensures no "detached" threads are left accessing freed memory during application tear-down.
5.3 Deadlock Prevention
Deadlocks occur when locks are acquired in inconsistent orders.
   * Locks in System: VFS_Mutex (RW), Stdout_Mutex (Exclusive), Queue_Mutex (Exclusive), TokenMap_Mutex (Exclusive).
   * Hierarchy Rule:
   1. Main Thread: Stdout_Mutex (write) -> Queue_Mutex (push) -> TokenMap_Mutex (access). These are usually independent operations.
   2. Worker: VFS_Mutex (read/write) -> Stdout_Mutex (send response).
   * Constraint: Never hold VFS_Mutex while waiting on Queue_Mutex or TokenMap_Mutex.
   * Constraint: Never hold Stdout_Mutex while performing compilation.

6. Testing Strategy
6.1 Unit Testing the Thread Pool
Using tests/test_thread_pool.cpp:
   * Concurrency: Queue 1000 tasks that increment an std::atomic<int>. Verify final count is 1000.
   * Ordering: Queue 5 CRITICAL tasks and 5 LOW tasks. Ensure via timestamps that CRITICAL tasks started execution first.
   * Cancellation: Submit a task that sleeps for 100ms. Cancel it immediately. Assert that the task returns early or sets a "Cancelled" flag.
6.2 Race Condition Detection
Compile with ThreadSanitizer (TSan) (-fsanitize=thread). Run the full test suite. TSan will detect:
   * Accessing VFS without locks.
   * Writing to stdout from multiple threads without synchronization.
   * Data races on the request_tokens_ map.
6.3 Performance Benchmarking
Measure the "Time to Cancel":
   1. Send textDocument/didChange (large file).
   2. Send textDocument/hover.
   3. Send $/cancelRequest (for the hover).
   4. Assert that the cancellation response is received < 10ms. (In single-threaded mode, this would take > 500ms).

7.2 Modifications to dispatch_message
The dispatch_message method in src/tools/lsp/server.cpp acts as the router.


C++




void Server::dispatch_message(const JsonRpcMessage& msg) {
   if (msg.type == MessageType::REQUEST) {
       //... (Logic to create CancellationToken)...
       Task task;
       task.method = msg.method;
       task.id = msg.id;
       task.priority = get_priority(msg.method);
       task.action = [this, msg, token](){ 
           // Worker Logic
           this->handle_request_async(msg, token); 
       };
       thread_pool_.submit(task);
   }
   //...
}

7.3 Integration with Existing Code
   * Transport Layer: Ensure Transport::write uses a std::mutex to protect std::cout.
   * VFS: No changes needed to vfs.h (already has shared_mutex), but Server.cpp must be updated to use std::shared_lock explicitly when calling vfs_.get_content().
   * DiagnosticEngine: Needs to be instantiated per-task (on the stack) or use thread_local storage to avoid cross-thread pollution of error lists.
8. Conclusion
Implementing the Thread Pool architecture described above transforms aria-ls into a production-grade language server. By adhering to C++17 standards and the LSP specification's concurrency requirements, this design ensures that the Aria developer experience remains fluid and responsive, even as the complexity of the compiler analysis grows. The segregation of I/O and computation, combined with the "Snapshot" pattern and atomic cancellation, provides a robust foundation for future features.
Implementation Time: ~45 minutes for core ThreadPool, Task, and WorkQueue classes plus Server.cpp refactoring.
