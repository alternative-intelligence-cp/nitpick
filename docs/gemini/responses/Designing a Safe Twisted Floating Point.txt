1.1 The Failure of IEEE 754 in High-Dimensional Toroidal Physics
Current industry standards for floating-point arithmetic, specifically IEEE 754 (binary64), are fundamentally unsuited for the Nikola architecture due to inherent topological asymmetries and "silent" failure modes.1
1.1.1 The Asymmetry of Signed Zero
In IEEE 754, the number line contains a topological tear at the origin, represented by $+0.0$ and $-0.0$. While mathematically equivalent ($x + 0 = x$), computationally they behave differently in critical contexts. For instance, $\frac{1}{+0.0} = +\infty$ while $\frac{1}{-0.0} = -\infty$. In a 9-dimensional wave processor, determining the "zero crossing" of an interference pattern is the mechanism for retrieving memory.3 If the phase cancellation results in $-0.0$ instead of $+0.0$ due to a minor rounding artifact, the resulting vector could flip sign, projecting the memory retrieval into the wrong octant of the 9D torus. This non-determinism creates a "butterfly effect" where micro-level numerical noise amplifies into macro-level cognitive dissociation.
1.1.2 The Ambiguity of NaN
The IEEE standard defines a complex family of NaN (Not a Number) values, distinguishing between "Quiet" and "Signaling" NaNs. In practice, most systems treat NaNs as quiet, allowing them to propagate silently through calculations.4 For the Nikola AGI, this is catastrophic. If a tensor operation in the physics engine produces a NaN (e.g., $\sqrt{-1}$ during a complex phase rotation), and that NaN is effectively "hidden" inside a massive weight matrix, the AGI might continue operating with a corrupted worldview for millions of cycles before the error manifests—likely as incoherent behavior towards the child. There is no mechanism to trace the error back to its source, violating the requirement for "100% safe" operation.
1.2 The TBB Solution: Twisted Balanced Binary Integration
The Aria language's existing "Twisted Balanced Binary" (TBB) paradigm offers a robust solution for integer arithmetic by strictly enforcing domain symmetry and reserving the most negative value as a sticky error sentinel.3 This ensures that $127 + 1$ results in ERR, not a wrapped negative number, and that any operation involving ERR results in ERR.
This report specifies the design and implementation of tfp64 (Twisted Floating Point, 64-bit), a novel numerical primitive that extends TBB semantics to the real number line. By constructing a floating-point number from two TBB integers (one for the exponent, one for the mantissa), tfp64 eliminates the signed zero problem, unifies all error states into a single verifiable sentinel, and guarantees bit-exact determinism across all platforms. This fulfills the user's requirement for a "special float using tbb for exponent and mantissa" designed specifically for the rigorous tolerances of the AGI's toroidal memory.

2. Theoretical Architecture of tfp64
To satisfy the feature freeze at v0.1.0, tfp64 must be implemented as a primitive type that fits within the existing Aria type system 3 while providing the mathematical rigor required by the physics engine.
2.1 The "Twisted" Topological Definition
Standard floating-point numbers utilize a scientific notation form:




$$V = (-1)^S \times 1.M \times 2^{E - \text{bias}}$$


This reliance on a sign bit $S$ and an exponent bias creates the asymmetries discussed above. tfp64 rejects this structure in favor of a "Twisted" tuple of TBB integers:




$$V_{\text{tfp}} = \text{Mantissa}_{\text{tbb}} \times 2^{\text{Exponent}_{\text{tbb}}}$$
Here, the sign is not a separate bit; it is intrinsic to the Two's Complement representation of the Mantissa.
2.1.1 Component Allocation
The 64-bit word is partitioned to maximize precision for the 9-dimensional interference patterns while maintaining a dynamic range sufficient for high-energy physics simulations.

	Semantics

	Total Width: 64 bits.
2.1.2 Domain Symmetry and Normalization
Because both components are TBB types, they possess symmetric valid ranges relative to zero, excluding the sentinel.
* Mantissa ($M$): Range $[- (2^{47}-1), + (2^{47}-1)]$. The sentinel is $-2^{47}$.
* Exponent ($E$): Range $[- (2^{15}-1), + (2^{15}-1)]$. The sentinel is $-2^{15}$.
This implies:
1. Unique Zero: If $M = 0$, the value is $0$. We canonically enforce $E=0$ whenever $M=0$. This eliminates $-0.0$. There is only one zero in tfp64: 0x0000...0000.
2. Symmetric Negation: For any valid $V$, $-V$ is strictly representable. In IEEE 754, negating the smallest normal number can sometimes trigger edge cases. In tfp64, negation is a simple 2's complement of the mantissa $M$. Since $M$ is symmetric (TBB), this never overflows.
3. No Subnormals: tfp64 does not support denormalized numbers. If a result is too small to be represented with the minimum exponent and a normalized mantissa, it is flushed to Zero (or ERR, depending on strictness configuration). This eliminates the performance cliff associated with denormals.4
2.2 The Unified Sticky Error State
The "Life or Death" safety requirement implies that the system must never operate on corrupted data. tfp64 defines a single, absorbing ERR state.
Definition: A tfp64 value is ERR if:




$$(M == \text{TBB48\_ERR}) \lor (E == \text{TBB16\_ERR})$$
However, to simplify comparison logic in the runtime, we define a Canonical Error Pattern. Any operation that results in an invalid state (Overflow, Underflow, Division by Zero, Domain Error) must return this exact bit pattern.
Canonical Sentinel:
* Exponent: 0x8000 (TBB16_ERR representation)
* Mantissa: 0x800000000000 (TBB48_ERR representation)
* Hex Representation: 0x8000800000000000
This ensures that the "Infected Hypercube" theory 3 holds: a single ERR value in a matrix multiplication will propagate to the output, alerting the self-improvement loop that the memory inference failed, rather than producing a hallucination.
2.3 Precision Analysis for 9D Torus Physics
The Nikola torus requires high precision to resolve interference patterns.
* IEEE 754 Double: 53 bits of significand.
* tfp64: 48 bits of significand (47 magnitude + 1 sign).
While tfp64 has 5 fewer bits of precision than a standard double, this trade-off is mathematically justified by the gain in stability. In high-dimensional vector spaces, the accumulation of rounding errors from non-associative floating-point math often exceeds the theoretical precision of the last 5 bits. By enforcing strict, deterministic software rounding in tfp64 (which we will implement in the runtime), we ensure that the simulation trajectories are reproducible, which is more valuable for a stable AGI than raw theoretical precision that is swamped by noise.

3.3 Literal Conversion Logic (The "Parser Twist")
The most critical safety step in the frontend is converting the user's string input (e.g., "0.000000001tf") into the tfp64 binary format. This must happen at compile-time. We cannot use atof or strtod because they do not respect TBB semantics.
We implement a state machine parser that computes the TBB Mantissa ($M$) and Exponent ($E$) directly.
3.3.1 The Conversion Algorithm
Given a string input $S$:
1. Parse Integers: Extract the integer part ($I$) and fractional part ($F$).
2. Determine Sign: Check for leading -.
3. Compute Magnitude: Combine $I$ and $F$ into a high-precision integer representation.
   * Find the position of the most significant non-zero digit.
   * Shift the value such that it fits into the 48-bit signed range $[-2^{47}+1, 2^{47}-1]$.
   * This is the Normalization Step. The goal is to maximize the magnitude of $M$ to preserve precision.
4. Compute Exponent:
   * Start with the base-10 exponent from the literal (e.g., e-9).
   * Adjust for the binary shift performed during normalization.
   * Convert the base-10 scale factor to a base-2 exponent ($E$).
5. Validation:
   * If $E$ exceeds TBB16_MAX (32767), return TFP64_ERR (Overflow).
   * If $E$ is less than TBB16_MIN (-32767), return TFP64_ERR (Underflow).
   * This strict validation ensures that constants defined in the code are mathematically valid before the program ever runs.

5.2 Packing and Normalization Logic
The "Twist" in tfp64 is the normalization. Unlike IEEE which assumes a leading 1, we explicitly shift the mantissa to maximize its magnitude within the 48-bit window. This ensures that the interference patterns in the torus are calculated with maximum available precision at any scale.


C++




// src/runtime/tfp/tfp64.cpp

#include "tfp64.h"

// Unpack opaque u64 into components
static UnpackedTfp unpack(tfp64_t val) {
   if (val == TFP_ERR) return {0, 0, true};

   // Extract Exponent (Top 16 bits)
   int16_t e = (int16_t)(val >> 48);

   // Extract Mantissa (Bottom 48 bits)
   // Sign-extend 48-bit to 64-bit
   int64_t m = (int64_t)(val << 16) >> 16;

   // Check TBB sentinels within components
   if (m == TBB48_ERR |

| e == TBB16_ERR) {
       return {0, 0, true};
   }

   return {m, e, false};
}

// Normalize and Pack
// This is the critical safety valve. It ensures uniqueness of representation.
static tfp64_t pack(int64_t m, int16_t e) {
   // 1. Check Error Sentinels
   if (m == TBB48_ERR |

| e == TBB16_ERR) return TFP_ERR;

   // 2. Canonical Zero
   if (m == 0) return TFP_ZERO;

   // 3. Normalization Loop
   // Shift left until the magnitude is large enough to use the full 48-bit precision.
   // We want the MSB (bit 46, since 47 is sign) to be significant.
   // Range target: |m| >= 2^46
   
   // Bounds check to prevent infinite loops or undefined shifts
   if (m > TBB48_MAX |

| m < TBB48_MIN) return TFP_ERR;

   const int64_t TARGET_THRESHOLD = 1LL << 46;

   // Shift Left (Increase Precision)
   while ((m > -TARGET_THRESHOLD && m < TARGET_THRESHOLD) && m!= 0) {
       // Check for exponent underflow
       if (e <= TBB16_MIN) {
           // Cannot decrease exponent further.
           // Flush to zero or return sub-optimal precision?
           // For safety, we flush to zero if it becomes too small.
           return TFP_ZERO;
       }
       m <<= 1;
       e--;
   }

   // Shift Right (Handle Overflow from Math Ops)
   while (m > TBB48_MAX |

| m < TBB48_MIN) {
       // Check for exponent overflow
       if (e >= TBB16_MAX) return TFP_ERR;
       
       // Sticky Bit Logic for Rounding
       // If we shift out a '1', we track it to round correctly (nearest even).
       bool sticky = (m & 1);
       m >>= 1;
       e++;
       if (sticky && (m & 1)) m++; // Simple round-up logic
   }

   // 4. Final Pack
   uint64_t m_bits = (uint64_t)m & 0x0000FFFFFFFFFFFFULL;
   uint64_t e_bits = (uint64_t)e << 48;
   return e_bits | m_bits;
}

5.3 Arithmetic Operations (The Safety Core)
The arithmetic operations must implement "Sticky Error Propagation." If any input is ERR or if any overflow occurs, the result is ERR.
5.3.1 Addition (tfp64_add)
Addition involves aligning exponents. The number with the smaller exponent is right-shifted. This loses precision, which is physically correct (adding a quark's mass to a galaxy's mass doesn't change the galaxy's mass perceptibly).


C++




extern "C" tfp64_t aria_tfp64_add(tfp64_t a, tfp64_t b) {
   UnpackedTfp ua = unpack(a);
   UnpackedTfp ub = unpack(b);

   // Sticky Error Propagation
   if (ua.is_err |

| ub.is_err) return TFP_ERR;

   // Optimization: Identity
   if (ua.m == 0) return b;
   if (ub.m == 0) return a;

   // Align Exponents
   // Ensure ua has the larger exponent
   if (ua.e < ub.e) {
       UnpackedTfp temp = ua; ua = ub; ub = temp;
   }

   int32_t diff = ua.e - ub.e;

   // If diff is too large, ub is negligible
   if (diff > 48) return a;

   // Shift ub.m right by diff
   // Implementation Detail: We assume 2's complement arithmetic shift
   int64_t mb_shifted = ub.m >> diff;

   // Perform Addition
   // We need a wider type to catch overflow before it wraps
   // __int128_t is available on GCC/Clang 64-bit targets.
   __int128_t sum = (__int128_t)ua.m + (__int128_t)mb_shifted;

   // The pack() function handles normalization and overflow checking
   // But we need to fit sum back into int64 first.
   
   // If sum overflows 64-bit int, we must preshift
   if (sum > INT64_MAX |

| sum < INT64_MIN) {
       sum >>= 1;
       ua.e++;
   }

   return pack((int64_t)sum, ua.e);
}

5.3.2 Multiplication (tfp64_mul)
Multiplication is the most dangerous operation for the 9D torus because exponents add up ($E_{new} = E_A + E_B$), leading to rapid overflow toward Infinity/Error. tfp64 clamps this strictly.


C++




extern "C" tfp64_t aria_tfp64_mul(tfp64_t a, tfp64_t b) {
   UnpackedTfp ua = unpack(a);
   UnpackedTfp ub = unpack(b);

   if (ua.is_err |

| ub.is_err) return TFP_ERR;
   if (ua.m == 0 |

| ub.m == 0) return TFP_ZERO;

   // Multiply Mantissas
   // 48-bit * 48-bit = 96-bit result. Fits in __int128_t.
   __int128_t prod = (__int128_t)ua.m * (__int128_t)ub.m;

   // Add Exponents
   int32_t new_e = (int32_t)ua.e + (int32_t)ub.e;

   // The product is now essentially "denormalized" relative to the new exponent 
   // because we multiplied two normalized numbers.
   // e.g. (M1 * 2^E1) * (M2 * 2^E2) = (M1*M2) * 2^(E1+E2)
   // Since M1, M2 are ~2^46, M1*M2 is ~2^92.
   // We need to shift it back down to fit in 48 bits (~2^46).
   // Shift right by 46.
   
   prod >>= 46;
   new_e += 46; // Compensate for the conceptual binary point?
   // Actually, simply passing the raw product to pack() might fail 
   // if pack() expects int64. We must shift manually.
   
   // Safety check for exponent overflow before packing
   if (new_e > TBB16_MAX |

| new_e < TBB16_MIN) return TFP_ERR;

   return pack((int64_t)prod, (int16_t)new_e);
}

5.3.3 Division (tfp64_div)
Division by zero is a classic source of AGI instability. In IEEE, $1/0 = \infty$. In tfp64, $1/0 = \text{ERR}$. This halts the branch of calculation immediately.


C++




extern "C" tfp64_t aria_tfp64_div(tfp64_t a, tfp64_t b) {
   UnpackedTfp ua = unpack(a);
   UnpackedTfp ub = unpack(b);

   if (ua.is_err |

| ub.is_err) return TFP_ERR;
   
   // Division by Zero Safety
   if (ub.m == 0) return TFP_ERR; 
   if (ua.m == 0) return TFP_ZERO;

   // Subtract Exponents
   int32_t new_e = (int32_t)ua.e - (int32_t)ub.e;

   // Mantissa Division
   // To preserve precision, we shift the dividend left (scale up) before dividing.
   // Scale by 2^48 to ensure the result has integer precision.
   __int128_t dividend = (__int128_t)ua.m << 48;
   __int128_t quot = dividend / ub.m;

   // Adjust exponent for the shift
   new_e -= 48;

   return pack((int64_t)quot, (int16_t)new_e);
}

6. Advanced Math Integration: The Wave Processor
The 9-dimensional torus relies on transcendental functions to simulate wave interference. We must provide sin, cos, and sqrt implementations that respect the tfp64 error semantics.
6.1 Taylor Series Expansion for TFP
Since we cannot use the hardware fsin instructions (they work on IEEE floats), we implement Taylor series approximations in software. While slower, this guarantees that sin(x) on an x86 machine produces the exact same bit pattern as sin(x) on an ARM machine—critical for the AGI's distributed model coherence.
Example: Square Root (tfp64_sqrt)
$\sqrt{x}$ is undefined for $x < 0$ in the real domain. IEEE returns NaN. tfp64 returns ERR.


C++




extern "C" tfp64_t aria_tfp64_sqrt(tfp64_t a) {
   UnpackedTfp ua = unpack(a);
   if (ua.is_err) return TFP_ERR;
   if (ua.m < 0) return TFP_ERR; // Negative input -> ERR
   if (ua.m == 0) return TFP_ZERO;

   // Sqrt(M * 2^E) = Sqrt(M) * 2^(E/2)
   // If E is odd, we adjust: Sqrt(M * 2 * 2^(E-1)) = Sqrt(2M) * 2^((E-1)/2)
   
   int64_t m_prime = ua.m;
   int16_t e_prime = ua.e;

   if (e_prime % 2!= 0) {
       m_prime <<= 1; // Multiply mantissa by 2
       e_prime--;     // Make exponent even
   }

   // Integer Square Root of Mantissa (using integer algorithm)
   // We scale m_prime up to preserve precision
   // Sqrt(m << 48) approx Sqrt(m) * 2^24
   __int128_t m_scaled = (__int128_t)m_prime << 48;
   int64_t m_sqrt = isqrt(m_scaled); // Implementation of integer sqrt required

   int16_t e_res = e_prime / 2 - 24; // Adjust for the scale factor

   return pack(m_sqrt, e_res);
}
