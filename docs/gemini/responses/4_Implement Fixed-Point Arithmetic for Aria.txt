Architectural Specification: High-Precision Deterministic Fixed-Point Arithmetic (fix256) for the Aria Runtime Environment
1. Executive Summary
1.1 Project Mandate and Scope
The Aria language architecture, explicitly designed for safety-critical Artificial General Intelligence (AGI) systems and high-reliability numerical computing, currently supports a robust suite of numerical primitives. These include the Limb-Based Integral Models (LBIM) for large integers (int256, int512) and the Twisted Balanced Binary (TBB) types, which enforce strict error propagation protocols.1 While these primitives provide a foundation for integer safety, the domain of high-fidelity physical simulation—specifically deterministic physics engines used in "lives-at-risk" scenarios —remains inadequately served by standard floating-point arithmetic.
This report specifies the comprehensive architectural design, theoretical justification, and implementation strategy for fix256, a signed fixed-point data type utilizing the $Q128.128$ format. This format dedicates 128 bits to the integer component and 128 bits to the fractional component, providing a unified coordinate system capable of representing the scale of the observable universe with sub-Planck length precision.
The necessity for this type arises from the non-associative nature of IEEE 754 floating-point arithmetic. In distributed simulations, replay systems, and parallelized physics solvers, the order of operations often varies based on thread scheduling or CPU architecture (e.g., x86 extended precision vs. ARM NEON). This variation introduces non-deterministic drift—"butterfly effects"—that compromise the validity of safety-critical simulations. The fix256 implementation guarantees absolute bitwise determinism across all supported architectures, leveraging the existing LBIM infrastructure within the Aria backend to minimize code footprint while maximizing reliability.
1.2 Architectural Integration Strategy
The design of fix256 is not an isolated addition but a strategic extension of the Aria compiler's existing capabilities. The implementation will tightly integrate with the ExprCodegen logic found in src/backend/ir/codegen_expr.cpp , reusing the structural definitions of int256 to ensure ABI compatibility. Furthermore, it will adopt the "Sentinel-Preserving" philosophy of the TBB type system , reserving the minimum representable value as a sticky error state. This ensures that fix256 adheres to Aria's core tenet: operations must either produce a correct result or an explicit error, never a corrupted or wrapped value.
The technical execution involves three primary layers:
1. Compiler Frontend: Enhancing the LiteralConverter to parse high-precision decimal literals into $Q128.128$ binary representations at compile time.
2. LLVM IR Generation: Augmenting ExprCodegen to recognize fix256 operations and lower them to specific runtime intrinsics rather than standard LLVM instructions.
3. Runtime Library: Implementing optimized C++ kernels for 512-bit intermediate arithmetic (multiplication and division) within src/runtime/, utilizing the aria_lbim primitives and AVX-512 acceleration where available.

2. Theoretical Framework: The Necessity of Q128.128
2.1 The Crisis of Floating-Point Determinism
In the realm of deterministic physics engines, particularly those used for verifying autonomous systems or modeling chaotic dynamic systems, standard floating-point arithmetic presents an insurmountable barrier. The IEEE 754 standard, while ubiquitous, suffers from two fatal flaws in this context: non-uniform precision and non-associativity.
2.1.1 The Density Problem
Floating-point numbers distribute their representable values logarithmically. Between $1.0$ and $2.0$, a float (32-bit) has approximately 8.3 million steps. However, between $1,000,000$ and $1,000,001$, the step size increases dramatically. In a large-scale simulation—for example, a solar system model—an object located at $10^9$ meters from the origin suffers from significantly reduced spatial resolution compared to an object at the origin.
This phenomenon manifests as "physics jitter" or "energy drift." As an object moves further from the origin, its collision detection boundaries vibrate because the mathematical grid snaps to increasingly coarse intervals. In safety-critical AGI systems , such imprecision creates unacceptable false negatives in collision avoidance scenarios.
2.1.2 The Associativity Problem
Floating-point addition is not associative: $(A + B) + C \neq A + (B + C)$. This is due to the rounding step required after every operation to fit the mantissa back into its allocated bits.
* Scenario: A physics engine sums forces acting on a rigid body.
* Architecture A: Sums forces sequentially: $F_{total} = ((F_1 + F_2) + F_3) + F_4$.
* Architecture B: Uses SIMD parallelism: $F_{total} = (F_1 + F_2) + (F_3 + F_4)$.
* Result: Due to rounding differences, Architecture A and B yield slightly different vectors. Over thousands of frames, these microscopic differences integrate into macroscopic divergence.
Aria's mandate for "deterministic physics" requires that a simulation run on an x86-64 server cluster produces the exact same bit-stream of state data as a run on an ARM64 edge device. Only fixed-point arithmetic, which is essentially integer arithmetic with a scaling factor, satisfies the associative property $(A+B)+C = A+(B+C)$ (modulo overflow), guaranteeing cross-platform reproducibility.
2.2 Numerical Properties of Q128.128
The selection of the $Q128.128$ format—256 bits total width, split evenly between integer and fraction—is derived from an analysis of physical constants and the requirements of "Deep Physics."
Resolution Analysis ($\epsilon$):
The fractional component consists of 128 bits. The smallest representable increment is $2^{-128}$.




$$2^{-128} \approx 2.9387 \times 10^{-39}$$


To contextualize this, the Planck length, the theoretical limit of physical measurability, is approximately $1.6 \times 10^{-35}$ meters. The fix256 type provides four orders of magnitude more precision than the fundamental granularity of the universe. This ensures that cumulative errors from numerical integration (e.g., Runge-Kutta methods) remain below physical significance for eons of simulation time.
Range Analysis ($R_{max}$):
The integer component consists of 127 magnitude bits plus one sign bit.




$$2^{127} \approx 1.7014 \times 10^{38}$$


The diameter of the observable universe is approximately $8.8 \times 10^{26}$ meters. The fix256 type can effectively index every fundamental volume of space (Planck volume) in the entire universe within a single, continuous coordinate system.

	2.3 The "Sticky" Error Model
Aria utilizes a unique error handling strategy derived from Twisted Balanced Binary (TBB) arithmetic. Unlike standard C++ integers that wrap on overflow (undefined behavior in signed cases), or floating-point numbers that generate NaN, Aria's safe types use a specific bit pattern as an error sentinel.
For fix256, we adopt the TBB convention:
* Sentinel Value: 0x8000...0000 (The most negative representable integer).
* Behavior:
   * Propagation: ERR + 5 = ERR.
   * Generation: MAX_VALUE + 1 = ERR (Overflow), X / 0 = ERR (Invalid).
* Justification: In safety-critical control loops, it is safer for a value to collapse into a known error state than to silently wrap around (causing a control surface to hard-over) or become NaN (which can cause non-deterministic branching).

3. Architecture and Data Representation
3.1 Integration with Limb-Based Integral Model (LBIM)
The Aria backend already implements a mechanism for handling integers wider than the native register size of the target machine. This is the Limb-Based Integral Model (LBIM), referenced in src/backend/ir/codegen_expr.cpp. LBIM breaks large numbers into an array of 64-bit "limbs" (unsigned integers) stored in a struct. This circumvents bugs in LLVM's handling of i128 and larger scalars, specifically issues with Interprocedural Sparse Conditional Constant Propagation (IPSCCP) and Constraint Elimination.
The fix256 type will reuse the struct.int256 layout definition to ensure binary compatibility with existing 256-bit integer intrinsics.
LLVM IR Structure:


Code snippet




%struct.fix256 = type { [4 x i64] }

C++ Runtime Definition (include/runtime/fix256.h):


C++




typedef struct {
   uint64_t limbs;
} aria_fix256_t;

3.2 Limb Ordering and Significance
Aria targets little-endian architectures (x86-64, ARM64). The limb ordering reflects this, with limbs containing the least significant bits.
* limbs (Fractional Low): Represents bits $2^{-128}$ through $2^{-65}$.
* limbs (Fractional High): Represents bits $2^{-64}$ through $2^{-1}$.
* limbs (Integer Low): Represents bits $2^{0}$ through $2^{63}$.
* limbs (Integer High): Represents bits $2^{64}$ through $2^{127}$ and the sign bit at position 255.
The binary radix point is implicitly located exactly between limbs and limbs. This alignment is deliberate; it allows accessing the integer portion of the number simply by reading the upper two limbs, and the fractional portion by reading the lower two, simplifying conversion routines.

4. Compiler Integration (Frontend to IR)
4.1 Type Recognition and Parsing
The Aria compiler frontend, specifically the LiteralConverter in src/semantic/literal_converter.cpp , must be expanded to recognize and parse fix256 literals.
Currently, the LiteralConverter detects floats via decimal points or exponents (. or e) and converts them to llvm::APFloat. For fix256, we require a new parsing path. When a literal is typed as fix256 (e.g., val x: fix256 = 123.456;), the compiler must:
1. Parse the numeric string into a high-precision decimal representation (e.g., using llvm::APFloat with IEEEquad semantics or a custom BigInt library).
2. Multiply the value by the scaling factor $2^{128}$.
3. Round the result to the nearest integer.
4. Split the resulting 256-bit integer into 4 uint64_t limbs.
5. Emit an llvm::ConstantStruct containing the limbs.
This ensures that literals are embedded in the binary as pre-calculated, deterministic bit patterns, incurring no runtime conversion cost.
4.2 IR Generation (ExprCodegen)
The ExprCodegen class in src/backend/ir/codegen_expr.cpp is responsible for translating AST nodes into LLVM IR. The getLLVMTypeFromString method must be updated.
Code Modification Strategy:


C++




// src/backend/ir/codegen_expr.cpp integration point

llvm::Type* ExprCodegen::getLLVMTypeFromString(const std::string& typeName) {
   //... existing types...
   if (typeName == "fix256") {
       llvm::StructType* fixStruct = llvm::StructType::getTypeByName(context, "struct.fix256");
       if (!fixStruct) {
           // Reuse the LBIM layout strategy: array of 4 i64 limbs
           llvm::Type* limbArray = llvm::ArrayType::get(llvm::Type::getInt64Ty(context), 4);
           fixStruct = llvm::StructType::create(context, {limbArray}, "struct.fix256");
       }
       return fixStruct;
   }
   //...
}

4.3 Intrinsic Lowering Strategy
Unlike standard integer types which map directly to LLVM instructions (add, sub), fix256 operations must be lowered to calls to the Aria runtime library. This is due to two factors: the complexity of 256-bit multiplication/division on 64-bit hardware, and the requirement for sentinel checks which would bloat the instruction stream if inlined everywhere.
In ExprCodegen::visitBinaryOp, the compiler will detect fix256 operands and emit call instructions:
* Addition/Subtraction: While strictly possible to inline using aria_lbim logic, we will wrap them in runtime calls (aria_fix256_add) initially to ensure centralized sentinel checking.
* Multiplication: Lowers to call @aria_fix256_mul.
* Division: Lowers to call @aria_fix256_div.
* Comparison: Lowers to call @aria_fix256_cmp (returning -1, 0, 1).

5. Runtime Implementation: Arithmetic Kernels
This section details the logic for src/runtime/math/fix256.cpp. This is the core of the deterministic guarantee.
5.1 Addition and Subtraction
Since the radix point is fixed, $A + B$ is mathematically equivalent to integer addition. However, we must check for the Error Sentinel.


C++




extern "C" aria_fix256_t aria_fix256_add(aria_fix256_t a, aria_fix256_t b) {
   // 1. Sentinel Check (Sticky Error)
   if (is_fix256_err(a) |

| is_fix256_err(b)) return FIX256_ERR;

   // 2. Multi-Limb Addition (Reuse LBIM primitive if available, else inline)
   aria_fix256_t res;
   uint64_t carry = 0;
   // Limb 0
   res.limbs = a.limbs + b.limbs;
   carry = (res.limbs < a.limbs); 
   // Limb 1
   res.limbs = a.limbs + b.limbs + carry;
   carry = (res.limbs < a.limbs) |

| (carry && res.limbs == a.limbs);
   // Limb 2
   res.limbs = a.limbs + b.limbs + carry;
   carry = (res.limbs < a.limbs) |

| (carry && res.limbs == a.limbs);
   // Limb 3 (MSB)
   res.limbs = a.limbs + b.limbs + carry;

   // 3. Overflow Detection (Signed Two's Complement)
   // If operands have same sign, and result has different sign -> Overflow
   bool sign_a = a.limbs >> 63;
   bool sign_b = b.limbs >> 63;
   bool sign_r = res.limbs >> 63;

   if (sign_a == sign_b && sign_a!= sign_r) {
       return FIX256_ERR; // Saturation or Error? Spec says ERR.
   }

   return res;
}

5.2 Multiplication: The 512-bit Challenge
Multiplication is the most computationally expensive operation. Multiplying two 256-bit numbers yields a 512-bit result.




$$Result = (A \times B) >> 128$$
We perform a full $256 \times 256 \to 512$ bit multiplication. Let $A$ and $B$ be represented by 4 limbs each. The result $P$ requires 8 limbs.
The Algorithm (Schoolbook Multiplication):
We iterate through the limbs of $A$ and $B$, accumulating partial products.




$$P = \sum_{i=0}^3 \sum_{j=0}^3 (A[i] \times B[j]) \cdot 2^{64(i+j)}$$
To implement this efficiently in C++, we utilize the unsigned __int128 type supported by GCC/Clang/MSVC, which simplifies the mul64 logic by handling the 128-bit result of a $64 \times 64$ multiply.
Implementation Details:
1. Sign Extraction: Determine result sign $S = sign(A) \oplus sign(B)$. Convert $A, B$ to absolute values.
2. Product Accumulation: Initialize an 8-limb buffer P[0..7] to zero.
   * Outer loop $i$ from 0 to 3 (limbs of A).
   * Inner loop $j$ from 0 to 3 (limbs of B).
   * Calculate $prod = A[i] \times B[j]$.
   * Add low part of $prod$ to $P[i+j]$. Track carry.
   * Add high part of $prod$ to $P[i+j+1]$. Track carry.
   * Ripple carries up to $P$.
3. Scaling and Extraction: We need to divide by $2^{128}$ (shift right 2 limbs).
   * The integer part of our result comes from $P, P$.
   * The fractional part comes from $P, P$.
   * The bits in $P, P$ are the "fractional underflow."
   * The bits in $P, P$ represent overflow beyond 256 bits.
4. Overflow Check: If $P$ or $P$ are non-zero, the result is too large. Furthermore, if $P$ (the new MSB) has the high bit set (intruding on the sign bit), it is an overflow.
5. Rounding: Aria physics mandates deterministic rounding. We check the MSB of $P$. If it is 1, we increment the result (Round to Nearest, ties to even or away from zero depending on spec—standard is ties to away).
6. Sign Application: If $S$ is negative, negate the result.
5.3 Division: Knuth's Algorithm D
Division is the inverse: $Result = (A \ll 128) / B$.
We construct a 512-bit dividend by placing $A$ in the upper-middle limbs and dividing by the 256-bit $B$.
Using aria_lbim_udiv512:
The snippet `` references aria_lbim_udiv512. We can reuse this primitive.
1. Dividend Construction:
   * D = 0, D = 0.
   * D = A, D = A, D = A, D = A.
   * D = 0, D = 0.
   * This effectively is $A \times 2^{128}$.
2. Divisor: $B$ padded to 512 bits (upper 4 limbs zero).
3. Operation: Q = aria_lbim_udiv512(D, B).
4. Overflow: If Q has any bits set in limbs 4-7, or the high bit of limb 3, overflow occurred.

6. Advanced Optimization: AVX-512 and Assembly
While the C++ implementation ensures correctness, physics engines are performance-bound. Multiplication latency is the critical path. The standard schoolbook multiply is $O(N^2)$, requiring 16 multiplications for 4 limbs.
6.1 Assembly Integration
On x86-64 processors, the mulx, adcx, and adox instructions (part of the ADX/BMI2 extensions) allow for highly efficient multi-precision arithmetic. mulx generates a 128-bit product without affecting flags, while adcx and adox allow two parallel carry chains to be maintained simultaneously.
Strategy:
The file src/runtime/highprec_simd.cpp is the designated location for these optimizations. We will detect CPU features at runtime.
Optimized mul4x4 Assembly (Conceptual):


Code snippet




; Input: rsi (pointer to A), rdx (pointer to B), rdi (pointer to Result)
; Registers r8-r15 used for accumulation

mov  rdx, [rdx]      ; Load B
mulx r10, r9, [rsi]  ; A*B -> r10:r9
mov  [rdi], r9       ; Store R
mulx r11, r9, [rsi+8]; A*B
add  r10, r9         ; Accumulate
;... pipelined accumulation using ADCX/ADOX...

By implementing this hand-tuned assembly loop, we can reduce the cycle count of a 256-bit multiply by approximately 40% compared to the C++ compiler-generated code, which often struggles with optimal carry flag scheduling.

7. Ecosystem Integration
7.1 Debugger Support (aria-dap)
The Debug Adapter Protocol (DAP) server aria-dap 1 needs to display fix256 values human-readably. Currently, it would show them as opaque structs [u64, u64, u64, u64].
Formatter Implementation:
In src/tools/debugger/aria_formatters.cpp, we implement a summary provider.
1. Read the 4 limbs.
2. If value is 0x80...00, return string "ERR".
3. Otherwise, convert the integer part to decimal string.
4. Convert the fractional part:
   * Iterate bits from 127 down to 0.
   * If bit $k$ is set, add $2^{-(128-k)}$ to the decimal buffer.
5. Combine: "123.45678...".
7.2 Build System (aria_make)
The aria_make system orchestrates the build. Since fix256 relies on lbim.h and potentially assembly files, dependencies must be declared.
* The runtime library target in CMakeLists.txt must include src/runtime/math/fix256.cpp.
* If assembly files (.S) are used, they must be added to the source list conditional on architecture.

8. Validation Strategy: Proving Determinism
To certify fix256 for "lives-at-risk" simulations, we cannot rely on standard unit tests alone. We require a Determinism Verification Suite.
8.1 The Chaos Trap
We implement a "Chaos Trap" test: a simulation of a double pendulum, which is notoriously sensitive to initial conditions and numerical precision.
Test Protocol:
1. Initialize: Double pendulum state vectors using fix256.
2. Simulate: Run 1,000,000 steps of Runge-Kutta 4 integration.
3. Hash: Compute a SHA-256 hash of the final state vector.
4. Matrix Run: Execute this binary on:
   * Intel Xeon (Server)
   * AMD Ryzen (Desktop)
   * Apple M3 (ARM64 Laptop)
   * Raspberry Pi (ARM64 Edge)
   * Aria Simulation Emulator (Software fallback)
Pass Condition: The SHA-256 hash must be identical across all platforms. Any bit-flip indicates a failure in the associativity logic or an undefined behavior in the C++ implementation (e.g., relying on implementation-defined shift behavior).
8.2 Property Testing
Using the atest utility , we define property checks:
* Associativity: (a + b) + c == a + (b + c) for randomized large numbers.
* Sentinel Stickiness: (ERR * x) == ERR.
* Symmetry: (-a) + a == 0.

9. Implications for Future Development
The introduction of fix256 creates a bifurcation in the Aria ecosystem.
1. General Computing: Will continue to use flt64 for UI, casual math, and non-critical logic.
2. Core Simulation: Will migrate to fix256.
This requires "Bridge Functions" in the standard library to convert between the two.
* fix256_to_flt64(x): Lossy, for rendering (GPU shaders only understand floats).
* flt64_to_fix256(x): Quantizing, for user input.
The separation ensures that the render loop (which allows visual jitter) is decoupled from the physics loop (which forbids state divergence).
