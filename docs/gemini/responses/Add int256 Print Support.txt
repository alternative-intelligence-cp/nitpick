Technical Research Report: Implementation of int256 Print Support in the Aria Runtime for LBIM Validation
1. Introduction and Strategic Imperative
The development of the Aria programming language has reached a critical juncture in its lifecycle, specifically regarding the verification of its Large Bounded Integer Math (LBIM) subsystem. As the foundational layer for a physics-based Artificial General Intelligence (AGI) designed to operate within a nine-dimensional torus memory architecture, the mathematical integrity of the Aria compiler is not merely a requirement for functional correctness but a prerequisite for operational safety. The specific use case—a companion system for neurodivergent children—imposes a "zero-failure" mandate. There is no tolerance for arithmetic errors, buffer overflows, or undefined behaviors that could manifest as unpredictable actions in the deployed model.
Current validation efforts for the int256 data type, a signed 256-bit integer crucial for the high-precision calculations of the AGI's wave processors, are stalled. While the compiler backend successfully generates code for int256 multiplication using the schoolbook algorithm, the runtime lacks the capability to visualize or verify these results via standard Input/Output (I/O) mechanisms. Specifically, the print() intrinsic function supports standard integer widths (up to i128) but fails to handle the int256 type, which is implemented via a custom Limb-Based Integral Model (LBIM) to circumvent upstream LLVM optimization defects.
This report outlines a comprehensive engineering strategy to implement print() support for int256. It details the architectural constraints of the LBIM design, the mathematical algorithms required for efficient and safe decimal conversion of 256-bit integers without external dependencies (such as GMP), and the integration of these capabilities into Aria's strict Six-Stream I/O topology. The proposed solution prioritizes correctness and memory safety above all else, ensuring that the resulting implementation meets the rigorous standards required for the Phase 4.2 milestone and the subsequent v0.1.0 feature freeze.
1.1 Project Status and Blocking Issues
The Aria project is currently in Phase 4.2, focusing on the stabilization of large integer arithmetic. The compiler's frontend successfully parses int256 literals, and the backend generates valid LLVM Intermediate Representation (IR) for arithmetic operations. However, the testing infrastructure relies on a "fuzzing" approach where millions of random multiplication operations are performed, and their results are compared against a trusted oracle (GNU Multiple Precision Arithmetic Library - GMP).
The inability to print int256 values creates a "black box" scenario. When a test fails or when verification is attempted, the test harness cannot output the actual value computed by the Aria binary. This lack of observability makes debugging the schoolbook multiplication algorithm impossible. Furthermore, the absence of a print function prevents the creation of reproducible failure cases in the bug tracking system.
The implementation of print(int256) is therefore the critical path dependency for unlocking the 1,000,000-iteration fuzz test suite. Without this capability, the mathematical core of the AGI cannot be certified as safe, and the project cannot proceed to the integration of the toroidal memory system.
1.2 Safety and Reliability Constraints
The intended application of the Aria language—powering an AGI for vulnerable populations—introduces specific non-functional requirements that dictate the technical implementation:
1. Memory Safety: The implementation must guarantee zero buffer overflows. Given that int256 values can exceed the capacity of standard 64-bit string buffers, rigorous bounds checking and static buffer sizing based on the mathematical maximum of the type ($2^{255}-1$) are mandatory.
2. Deterministic Behavior: The conversion algorithm must yield identical results across all target architectures. This necessitates a "pure" implementation that does not rely on platform-specific floating-point units or varying libc implementations of printf.
3. No External Runtime Dependencies: To minimize the attack surface and ensure the compiler's self-contained nature, the runtime cannot link against heavy external libraries like GMP or OpenSSL. The base conversion logic must be implemented in pure C++ within the aria_runtime library.
4. Integration with Six-Stream Topology: Aria utilizes a novel 6-stream I/O model (stdin, stdout, stderr, stddbg, stddati, stddato) to separate user interface text from binary data and telemetry. The print() function must strictly adhere to this topology, writing human-readable output to stdout (File Descriptor 1) while ensuring it does not pollute the binary data streams used for inter-process communication in the AGI's cognitive loops.

2. Architectural Context: The Limb-Based Integral Model (LBIM)
To design a correct printing mechanism, one must first understand the unique storage model employed by the Aria compiler for large integers. Unlike standard C compilers that might rely on __int128 or extended precision extensions, Aria fundamentally alters the representation of these types in the LLVM IR to guarantee correctness.
2.1 The Rationale for LBIM
The decision to implement the Limb-Based Integral Model (LBIM) stems from documented instability in the LLVM optimizer when handling integers wider than 128 bits. Specifically, the Interprocedural Sparse Conditional Constant Propagation (IPSCCP) and Constraint Elimination passes have been observed to miscompile logical operations and comparisons on i256 types, leading to silent data corruption.
To mitigate this, Aria specification ARIA-024 mandates that types larger than 64 bits (with the exception of i128 in some contexts) be treated not as scalar integers, but as aggregate structures (arrays) of machine-word "limbs." This prevents the LLVM optimizer from attempting aggressive integer transforms that trigger the known bugs, as the optimizer treats the value as a struct rather than a single arithmetic unit.
2.2 LBIM Internal Structure
Analysis of the compiler backend source code, specifically src/backend/ir/codegen_expr.cpp, reveals the exact internal structure of the int256 type. The type mapping logic is encapsulated in the getLLVMTypeFromString method:


C++




if (typeName == "int256" |

| typeName == "uint256") {
   llvm::StructType* int256Struct = llvm::StructType::getTypeByName(context, "struct.int256");
   if (!int256Struct) {
       llvm::Type* limbArray = llvm::ArrayType::get(llvm::Type::getInt64Ty(context), 4);
       int256Struct = llvm::StructType::create(context, {limbArray}, "struct.int256");
   }
   return int256Struct;
}

This code snippet confirms that an int256 is represented as a struct containing a single field: an array of four 64-bit integers ([4 x i64]). Assuming the standard little-endian memory layout used by x86-64 and ARM64 architectures, the limbs are ordered by significance:
* Limb 0 (limbs): Contains bits 0 to 63.
* Limb 1 (limbs): Contains bits 64 to 127.
* Limb 2 (limbs): Contains bits 128 to 191.
* Limb 3 (limbs): Contains bits 192 to 255.
For signed int256 types, the sign bit is the most significant bit (MSB) of Limb 3 (bit 63 of that limb). The representation uses two's complement, meaning negative numbers are stored as ~x + 1.
2.3 The "Acquire-Convert-Release" Pattern
This structural representation has profound implications for the implementation of the print() function. Standard C library functions like printf or std::cout do not know how to interpret a struct.int256. Passing this struct directly to a variadic function like printf would result in undefined behavior, likely printing only the first 64 bits or crashing due to ABI mismatches.
Therefore, the implementation must follow the Acquire-Convert-Release (ACR) pattern utilized elsewhere in the Aria standard library :
1. Acquire: The compiler's code generator must extract the int256 struct from the Aria variable. Since the struct is an aggregate, it is typically manipulated via pointer in the LLVM IR.
2. Convert: The compiler must emit a call to a specialized runtime function (aria_format_int256) that accepts the struct (or a pointer to it) and performs the mathematical conversion to a decimal string.
3. Release: The runtime returns a safe string object (likely an AriaString struct) which the compiler then passes to the low-level string printing routine.
This separates the concern of representation (handled by LBIM) from the concern of presentation (handled by the runtime formatter).

3.3 Buffer Management and Thread Safety
The AGI system relies on highly concurrent processing. The runtime implementation of print must be thread-safe. While glibc's printf is generally thread-safe regarding internal locking, the Aria runtime's formatter must ensure that the buffer allocation for the string conversion does not introduce race conditions.
Using a static buffer for the conversion is unsafe in a multi-threaded environment. Allocating memory on the heap (via malloc or Aria's GC) for the resulting string is the preferred approach, provided the memory is correctly managed or garbage collected. Given Aria's existing infrastructure , the formatter will likely return a pointer to a heap-allocated AriaString struct, which the caller (the code generated by the compiler) effectively owns or leaves to the GC.

4. Algorithmic Implementation Strategy
The core engineering challenge is converting a 256-bit binary integer (represented as four 64-bit limbs) into a decimal string without using external libraries.
4.1 Mathematical Analysis
The range of a signed 256-bit integer is:




$$-2^{255} \le x \le 2^{255} - 1$$
Approximating the maximum value:




$$2^{255} \approx 5.789 \times 10^{76}$$
This means the decimal representation requires up to 77 digits. Including a potential negative sign and a null terminator, the string buffer must be at least 79 bytes. To be safe and align with cache lines, a buffer size of 128 bytes is recommended.
4.2 Radix Conversion Optimization
The naive approach to binary-to-decimal conversion involves repeated division by 10 (modulo for the digit, division for the next iteration). However, performing a full 256-bit division by 10 for every single digit is computationally inefficient. It would require approximately 77 iterations of a multi-limb division algorithm.
A superior approach, widely used in high-performance libraries, is to divide by the largest power of 10 that fits within a single machine word (64 bits).




$$10^{19} = 10,000,000,000,000,000,000 < 2^{64}$$
By using $10^{19}$ as the divisor:
1. We extract 19 decimal digits in a single multi-precision division operation.
2. The number of expensive divisions is reduced from 77 to $\lceil 77/19 \rceil = 5$.
3. The remainder of each division is a 64-bit integer, which can be trivially converted to a string using standard sprintf or optimized integer-to-string logic.
4.3 The Division Algorithm (div_mod_1e19)
Since we cannot link GMP, we must implement a "schoolbook" short division algorithm specialized for our 4-limb structure. We treat the 256-bit number as a polynomial with base $2^{64}$:




$$N = L_3 \cdot (2^{64})^3 + L_2 \cdot (2^{64})^2 + L_1 \cdot (2^{64})^1 + L_0 \cdot (2^{64})^0$$
We divide $N$ by $D = 10^{19}$. The algorithm proceeds from the most significant limb ($L_3$) down to $L_0$.
Let $R$ be the remainder, initialized to 0. For $i = 3$ down to $0$:
1. Construct a double-width (128-bit) value: $Current = (R \ll 64) | L_i$.
2. Compute quotient limb: $Q_i = Current / D$.
3. Compute new remainder: $R = Current \% D$.
4. Update the limb $L_i$ with $Q_i$.
This operation effectively updates the 256-bit number in-place to be the quotient, and returns $R$ as the remainder (the extracted block of 19 decimal digits).
Hardware Requirement: This algorithm requires 128-bit division (uint128_t / uint64_t). Modern C++ compilers (GCC, Clang) support the unsigned __int128 type on 64-bit targets, which allows us to express this logic efficiently in C without writing raw assembly. This fits the "Pure Implementation" requirement.
4.4 Handling Negative Numbers
Aria uses two's complement. The conversion algorithm works on absolute magnitudes.
1. Check the sign bit (MSB of $L_3$).
2. If set, append '-' to the output buffer.
3. Negate the number: ~N + 1.
   * Edge Case: INT256_MIN ($ -2^{255} $). In two's complement, the absolute value of INT256_MIN cannot be represented as a positive signed int256. However, it can be represented as a uint256. Since our limb storage is effectively unsigned uint64_t arrays, the negation logic ~limbs + 1 correctly produces the bit pattern for $2^{255}$ in the unsigned domain. The printing algorithm simply treats the limbs as unsigned magnitude after the sign check, so this edge case is handled naturally.
4.5 Safety Verification
The safety of this implementation rests on the buffer sizing.
* We allocate 128 bytes.
* We perform at most 5 iterations of the div_mod_1e19 loop.
* Each iteration produces exactly 19 digits (zero-padded).
* $5 \times 19 = 95$ characters max? No, the first chunk (most significant) is not padded and may be small. The mathematical maximum is 78 characters.
* $78 < 128$. Buffer overflow is mathematically impossible provided the index logic is correct.

5. Code Implementation Blueprint
The implementation is split into two files: the runtime formatter and the compiler backend modification.
5.1 Runtime Implementation: src/runtime/fmt/formatters.cpp
This file handles the logic described in Section 4. We will define the aria_int256_t struct locally to match the LLVM IR layout and implement the formatting function.


C++




/**
* src/runtime/fmt/formatters.cpp
* 
* Implements safe formatting for int256 types using LBIM.
* Follows ARIA-024 specification for limb-based storage.
*/

#include "runtime/fmt/formatters.h"
#include <cstdio>
#include <cstring>
#include <cstdlib>
#include <vector>
#include <string>
#include <algorithm>

// Define the runtime layout of int256 to match LLVM struct.int256
// Layout: [4 x i64]
typedef struct {
   uint64_t limbs;
} aria_int256_t;

namespace {

// The Radix: 10^19
const uint64_t TEN_POW_19 = 10000000000000000000ULL;

// Helper: 128-bit division logic
// Modifies 'limbs' in-place to be the quotient.
// Returns the remainder (next 19 digits).
uint64_t div_mod_u64(std::vector<uint64_t>& limbs, uint64_t divisor) {
   uint64_t remainder = 0;
   // Iterate from Most Significant Limb (3) to Least Significant (0)
   for (int i = limbs.size() - 1; i >= 0; --i) {
       // Construct 128-bit value from remainder (high) and current limb (low)
       unsigned __int128 current = (unsigned __int128)remainder << 64 | limbs[i];
       
       // Division and Modulo
       limbs[i] = (uint64_t)(current / divisor);
       remainder = (uint64_t)(current % divisor);
   }
   return remainder;
}

// Helper: Check if the number is zero
bool is_zero(const std::vector<uint64_t>& limbs) {
   for (uint64_t limb : limbs) {
       if (limb!= 0) return false;
   }
   return true;
}

// Helper: Two's Complement Negation
void negate_int256(std::vector<uint64_t>& limbs) {
   uint64_t carry = 1;
   for (int i = 0; i < 4; ++i) {
       limbs[i] = ~limbs[i];
       unsigned __int128 sum = (unsigned __int128)limbs[i] + carry;
       limbs[i] = (uint64_t)sum;
       carry = (uint64_t)(sum >> 64);
   }
}

// Helper: Allocate AriaString (assumed existing helper in formatters.cpp)
// If not present, we use the standard layout.
AriaString* alloc_aria_string_safe(const char* cstr) {
   size_t len = strlen(cstr);
   // Assuming aria_gc_alloc is available for runtime strings
   // If pure stdlib, we might use malloc, but GC is safer for Aria.
   // For this implementation, we assume a wrapper exists.
   AriaString* str = (AriaString*)malloc(sizeof(AriaString));
   char* data = (char*)malloc(len + 1);
   memcpy(data, cstr, len);
   data[len] = '\0';
   str->data = data;
   str->length = len;
   return str;
}

} // namespace

extern "C" {

/**
* Main Entry Point: Format int256 to decimal string.
*/
AriaString* aria_format_int256(const aria_int256_t* val) {
   if (!val) return alloc_aria_string_safe("null");

   // Copy limbs to vector for manipulation
   std::vector<uint64_t> limbs = {
       val->limbs, val->limbs, val->limbs, val->limbs
   };

   // Edge Case: Zero
   if (is_zero(limbs)) {
       return alloc_aria_string_safe("0");
   }

   bool is_negative = false;
   // Check sign bit (Bit 63 of Limb 3)
   if ((limbs >> 63) & 1) {
       is_negative = true;
       negate_int256(limbs);
   }

   // Storage for 19-digit chunks
   std::vector<uint64_t> chunks;
   chunks.reserve(5); // Max 5 chunks for 256 bits

   // Extraction Loop
   while (!is_zero(limbs)) {
       chunks.push_back(div_mod_u64(limbs, TEN_POW_19));
   }

   // String Construction
   // Max size: 78 chars + safety. 
   std::string result;
   result.reserve(128);

   if (is_negative) {
       result += '-';
   }

   // Process chunks in reverse (Most Significant Chunk first)
   for (int i = chunks.size() - 1; i >= 0; --i) {
       char buf; // 19 digits + null is plenty
       if (i == chunks.size() - 1) {
           // Top chunk: No padding (e.g., "123" not "00...00123")
           snprintf(buf, sizeof(buf), "%llu", (unsigned long long)chunks[i]);
       } else {
           // Inner chunks: Must pad with zeros to 19 width
           snprintf(buf, sizeof(buf), "%019llu", (unsigned long long)chunks[i]);
       }
       result += buf;
   }

   return alloc_aria_string_safe(result.c_str());
}

} // extern "C"

5.2 Compiler Backend Implementation: src/backend/ir/codegen_expr.cpp
The compiler must be taught to recognize int256 arguments passed to print() and route them to our new runtime function. This modification happens in ExprCodegen::codegenCall.
The logic detects if the argument type is an LBIM struct (aggregate of 4 integers). If so, it cannot simply be passed by value to printf. Instead, we must spill it to the stack and pass the pointer.
File: src/backend/ir/codegen_expr.cpp


C++




// Within ExprCodegen::codegenCall...

// Detect Builtin: print()
if (callee->name == "print") {
   //... validation logic...
   
   llvm::Value* arg = codegenExpression(call->arguments.get());
   llvm::Type* argType = arg->getType();

   // Check specifically for int256 / uint256 LBIM structs
   // Logic matches getLLVMTypeFromString definition
   bool isInt256 = false;
   if (argType->isStructTy()) {
       llvm::StructType* st = llvm::cast<llvm::StructType>(argType);
       if (st->getName() == "struct.int256") {
           isInt256 = true;
       }
   }

   if (isInt256) {
       // Step 1: Declare the runtime function if not present
       // Signature: AriaString* aria_format_int256(int256*)
       llvm::Function* formatter = module->getFunction("aria_format_int256");
       if (!formatter) {
           llvm::Type* ptrTy = llvm::PointerType::get(context, 0); // Opaque ptr
           llvm::FunctionType* ft = llvm::FunctionType::get(
               ptrTy,      // Return: AriaString*
               { ptrTy },  // Arg: int256*
               false
           );
           formatter = llvm::Function::Create(ft, llvm::Function::ExternalLinkage, "aria_format_int256", module);
       }

       // Step 2: Spill the int256 value to the stack to get a pointer
       // We cannot pass a [4 x i64] struct by value easily to C ABI
       llvm::Value* stackSlot = builder.CreateAlloca(argType, nullptr, "int256_val_ptr");
       builder.CreateStore(arg, stackSlot);

       // Step 3: Call the formatter
       llvm::Value* ariaStringPtr = builder.CreateCall(formatter, { stackSlot }, "formatted_str");

       // Step 4: Extract the C-string pointer from AriaString
       // AriaString = { char* data, int64 len }
       // We need struct GEPS to get the data pointer at index 0
       llvm::StructType* stringStructTy = llvm::StructType::getTypeByName(context, "struct.AriaString");
       if (!stringStructTy) {
            // Fallback definition if not found 
            stringStructTy = llvm::StructType::create(context, {
                llvm::PointerType::get(context, 0), 
                llvm::Type::getInt64Ty(context)
            }, "struct.AriaString");
       }
       
       llvm::Value* dataPtrPtr = builder.CreateStructGEP(stringStructTy, ariaStringPtr, 0, "str_data_ptr");
       llvm::Value* charPtr = builder.CreateLoad(llvm::PointerType::get(context, 0), dataPtrPtr, "cstr");

       // Step 5: Print using system printf
       llvm::Function* printfFunc = module->getFunction("printf");
       if (!printfFunc) {
           llvm::FunctionType* printfType = llvm::FunctionType::get(
               llvm::Type::getInt32Ty(context),
               llvm::PointerType::get(context, 0),
               true // Varargs
           );
           printfFunc = llvm::Function::Create(printfType, llvm::Function::ExternalLinkage, "printf", module);
       }
       
       // Format string "%s" (no newline, print() usually adds none, println does)
       // User request shows `print(" \n");` separate, so print(x) should be raw.
       llvm::Value* fmtStr = builder.CreateGlobalStringPtr("%s", "fmt_s");
       return builder.CreateCall(printfFunc, { fmtStr, charPtr });
   }
}
