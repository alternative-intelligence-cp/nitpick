Architectural Specification and Implementation Strategy for the Aria Editor Ecosystem
1. Executive Summary and Strategic Context
The maturation of the Aria programming language ecosystem has reached a critical inflection point. With the foundational compiler infrastructure—specifically the Lexer, Parser, and Semantic Analysis phases—approaching stability, and the preliminary Language Server Protocol (LSP) implementation (aria-ls) successfully passing 1198 tests 1, the focus must now shift toward Developer Experience (DX). In the contemporary software engineering landscape, the viability of a systems programming language is determined not solely by its runtime performance or memory safety guarantees, but by the richness and responsiveness of its editing environment.
This comprehensive research report outlines the architectural roadmap for "Post-LSP Phase 7.3.7." The objective is to construct a unified, high-fidelity editor support system that transcends basic syntax highlighting. The proposed system integrates Aria's unique semantic features—such as Twisted Balanced Binary (TBB) arithmetic, the hybrid wild/gc memory model, and the pick/fall control flow constructs—directly into the developer's workspace.
The strategy necessitates a multi-platform approach, prioritizing Visual Studio Code as the primary target due to its market dominance and robust LSP client capabilities, while simultaneously delivering first-class support for Neovim (via Tree-sitter), Emacs, and other LSP-compliant editors. Furthermore, this report details the pioneering integration of the Model Context Protocol (MCP), positioning Aria as one of the first languages to offer native, standardized context bridging for AI-assisted development.2
2. Visual Studio Code Extension Architecture
The Visual Studio Code extension serves as the reference implementation for Aria's editor tooling. It acts as the bridge between the TypeScript-based editor environment and the C++-based aria-ls language server. The architecture must be robust, platform-agnostic, and capable of managing the lifecycle of the language server binary.
2.1 Extension Manifest (package.json) Configuration
The package.json file is the declarative core of the extension. It must be rigorously defined to ensure correct activation, file association, and configuration exposure. 2.1.2 Binary Bundling Strategy
Aria relies on a native C++ language server (aria-ls). To satisfy the "install and work" success criteria, the extension must bundle compiled binaries for all major platforms (Windows x64, Linux x64, macOS arm64/x64).4

Runtime Resolution Logic:
The extension's activate function (in TypeScript) must dynamically detect the host operating system and architecture to select the correct binary path. This logic should prioritize a user-defined aria.server.path if present, facilitating development and debugging of the server itself.5


TypeScript




import * as os from 'os';
import * as path from 'path';
import { workspace, ExtensionContext } from 'vscode';

export function getServerPath(context: ExtensionContext): string {
   const config = workspace.getConfiguration('aria');
   const customPath = config.get<string>('server.path');
   if (customPath) return customPath;

   const platform = os.platform();
   
   let executable = 'aria-ls';
   let folder = '';

   if (platform === 'win32') {
       folder = 'win32-x64';
       executable += '.exe';
   } else if (platform === 'linux') {
       folder = 'linux-x64';
   } else if (platform === 'darwin') {
       folder = 'darwin-universal';
   } else {
       throw new Error(`Unsupported platform: ${platform}`);
   }

   return context.asAbsolutePath(path.join('bin', folder, executable));
}

Grammar Logic Analysis:
The string_interpolation pattern uses #include: $self.7 This is critical for Aria because string interpolation allows arbitrary expressions, including nested strings or function calls. The TextMate engine must recursively apply the entire Aria grammar inside the &{... } block to ensure correct highlighting of code within strings.
4. Modern Parsing: Tree-sitter and Neovim Integration
While VS Code relies on TextMate, Neovim and increasingly Emacs leverage Tree-sitter for high-performance, incremental parsing. Tree-sitter generates a Concrete Syntax Tree (CST) that enables structural editing and precise code navigation.
4.1 Tree-sitter Grammar Definition (grammar.js)
The Tree-sitter grammar must define the operator precedence and structural rules of Aria. Based on the CodeGenContext and parser logic 1, the grammar hierarchy is established as follows:


JavaScript




module.exports = grammar({
 name: 'aria',

 extras: $ => [
   /\s/,
   $.comment,
 ],

 rules: {
   source_file: $ => repeat($._statement),

   _statement: $ => choice(
     $.variable_declaration,
     $.function_definition,
     $.expression_statement,
     $.return_statement,
     $.block,
     $.pick_statement, // Aria specific
     $.defer_statement // Aria specific
   ),

   // Variable: wild int8* ptr =...;
   variable_declaration: $ => seq(
     optional(choice('wild', 'gc', 'stack')), // Memory qualifiers
     field('type', $._type),
     field('name', $.identifier),
     optional(seq('=', field('value', $._expression))),
     ';'
   ),

   // Pick Statement: pick value {... }
   pick_statement: $ => seq(
     'pick',
     field('selector', $._expression),
     field('body', $.pick_body)
   ),

   pick_body: $ => seq(
     '{',
     repeat($.pick_case),
     optional($.pick_fall),
     '}'
   ),

   // Fall Statement: fall(label)
   pick_fall: $ => seq(
     'fall', '(', $.identifier, ')', ';'
   ),

   // Types including TBB and Generics
   _type: $ => choice(
     $.primitive_type,
     $.tbb_type,
     $.pointer_type,
     $.generic_type
   ),

   tbb_type: $ => /tbb(8|16|32|64)/,
   
   // Operator Precedence (inferred from common systems languages + Aria specs)
   _expression: $ => choice(
     $.identifier,
     $.literal,
     $.binary_expression,
     $.unary_expression
   ),

   binary_expression: $ => choice(
     prec.left(1, seq($._expression, '||', $._expression)),
     prec.left(2, seq($._expression, '&&', $._expression)),
     prec.left(3, seq($._expression, choice('==', '!='), $._expression)),
     prec.left(4, seq($._expression, choice('<', '>', '<=', '>='), $._expression)),
     prec.left(5, seq($._expression, choice('+', '-'), $._expression)),
     prec.left(6, seq($._expression, choice('*', '/', '%'), $._expression)),
     prec.left(7, seq($._expression, choice('|>', '?.', '!!'), $._expression)) // Pipe and Safe Nav
   )
 }
});

5.1 Emacs (aria-mode.el)
Emacs support requires a major mode that integrates with lsp-mode or eglot.

;; aria-mode.el
(require 'lsp-mode)

(defgroup aria nil "Support for Aria programming language" :group 'languages)

(defcustom aria-server-path "aria-ls"
 "Path to the Aria Language Server executable."
 :type 'string

(defconst aria-font-lock-keywords
 (list
  '("\\<\\(func\\|struct\\|enum\\|pick\\|fall\\|defer\\)\\>". font-lock-keyword-face)
  '("\\<\\(wild\\|gc\\|stack\\)\\>". font-lock-warning-face) ;; Highlight memory qualifiers
  '("\\<\\(tbb[0-9]+\\|int[0-9]+\\|flt[0-9]+\\)\\>". font-lock-type-face)
  '("\\<\\(ERR\\)\\>". font-lock-constant-face)))

(define-derived-mode aria-mode prog-mode "Aria"
 "Major mode for editing Aria source code."
 (setq-local comment-start "//")
 (setq-local comment-start-skip "\\(//+\\|/\\*+\\)\\s *")
 (setq-local font-lock-defaults '(aria-font-lock-keywords)))

;; LSP Integration
(add-to-list 'lsp-language-id-configuration '(aria-mode. "aria"))
(lsp-register-client
(make-lsp-client :new-connection (lsp-stdio-connection (lambda () aria-server-path))
                 :activation-fn (lsp-activate-on "aria")
                 :server-id 'aria-ls))

(add-hook 'aria-mode-hook #'lsp)
(provide 'aria-mode)

6. Model Context Protocol (MCP) Integration
The Model Context Protocol (MCP) represents a paradigm shift in how AI assistants interact with development environments. Unlike the LSP, which is designed for editor features (completion, diagnostics), MCP allows the Aria tooling to provide semantic context to LLMs (like Claude or Copilot).2
6.1 Role of MCP in Aria
Aria is a novel language with unique constraints (TBB arithmetic, memory qualifiers). General-purpose LLMs may hallucinate incorrect syntax or unsafe memory patterns. The aria-ls binary should function as a dual-protocol server, supporting both LSP and MCP, or a dedicated aria-mcp server should be created to bridge this gap.
6.2 Context Offerings
The Aria MCP server should expose the following resources and tools to the AI:
1. Resources (resources/list):
   * aria://docs/tbb: Returns the official specification for Twisted Balanced Binary arithmetic, explaining the ERR sentinel and sticky error propagation.1
   * aria://docs/memory: Returns documentation on wild vs gc allocation strategies and the defer mechanism.
   * aria://symbols/current_file: Returns the symbol table of the active file to help the AI understand available functions and structs.
2. Tools (tools/call):
   * explain_tbb_overflow(value, width): A tool that calculates whether a specific integer value would trigger an ERR state in a given TBB width (e.g., does 128 overflow tbb8?).
   * convert_c_to_aria(c_code): A specialized transpiler tool that takes a snippet of C code and suggests the idiomatic Aria equivalent, correctly mapping pointers to wild references.
6.3 Protocol Implementation
The MCP server communicates via JSON-RPC, similar to LSP but with different method signatures.10
* Request: tools/call
* Method: explain_tbb_overflow
* Params: { "value": 128, "width": "tbb8" }
* Response:
JSON
{
 "content":
}

7. Formatter and Linter Design (aria-fmt)
Consistency is paramount. The aria-fmt tool should be implemented as a subcommand of the compiler or a standalone binary that modifies the source code to adhere to the official style.
7.1 Implementation Strategy: Concrete Syntax Tree (CST)
Unlike the compiler which discards whitespace/comments during AST generation, aria-fmt must use a Concrete Syntax Tree or a "Lossless AST" to preserve comments.12
   * Algorithm:
   1. Parse source into CST (preserving trivia).
   2. Traverse CST.
   3. Emit code using a strict set of rules.
   4. Verify idempotence (output of formatting must not change if formatted again).
7.2 Configuration Rules (aria.toml)
Users should control formatting via a configuration file, though defaults should be opinionated.


Ini, TOML




[fmt]
tab_size = 4
use_spaces = true
brace_style = "1tbs" # One True Brace Style (K&R)
max_width = 100
import_sorting = true

7.3 Integration
The VS Code extension uses the DocumentFormattingEditProvider interface. It spawns aria-fmt --stdin and applies the text edits returned via stdout.

9. Conclusion and Release Roadmap
This specification defines a mature, industrial-grade tooling ecosystem for Aria. By leveraging the Language Server Protocol for semantics, TextMate/Tree-sitter for syntax, and the Model Context Protocol for AI integration, Aria can offer a developer experience that rivals established languages like Rust and Go.
Implementation Sequence:
   1. Phase A: Bundle aria-ls and release vscode-aria with TextMate grammar. (Immediate functionality).
   2. Phase B: Implement grammar.js for Tree-sitter and publish nvim-treesitter config. (Neovim support).
   3. Phase C: Develop aria-fmt and integrate into the LSP as a formatting provider. (Consistency).
   4. Phase D: Implement MCP endpoints within aria-ls to assist AI agents. (Future-proofing).
This roadmap ensures that as the language evolves, the tooling scales in parallel, providing a stable foundation for the Aria developer community.
