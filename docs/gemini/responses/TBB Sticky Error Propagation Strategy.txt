Architectural Implementation Report: Twisted Balanced Binary (TBB) Sticky Error Propagation in the Aria Compiler Backend
1. Introduction
The evolution of systems programming languages has long been characterized by a tension between the raw performance of hardware-native arithmetic and the rigorous safety requirements of modern software infrastructure. The Aria programming language, specifically in its v0.0.7 iteration, addresses this dichotomy through the introduction of the Twisted Balanced Binary (TBB) type system. Unlike traditional integer representations which prioritize hardware convenience over mathematical consistency, TBB types prioritize symmetry and deterministic error handling. This report details the architectural design and implementation strategy for "Sticky Error" propagation within the Aria compiler's LLVM backend, a feature critical to ensuring that the ERR sentinel state propagates transitively through all arithmetic computations.
1.1 The Systems Arithmetic Crisis
In the prevailing systems programming paradigm (C, C++, Rust), integer arithmetic is governed by the two's complement representation. While efficient for Arithmetic Logic Units (ALUs) designed in the mid-20th century, this model suffers from inherent defects when applied to high-reliability domains. The most egregious of these is the "silent overflow." When a computation exceeds the representable range of a standard integer, the hardware typically wraps the value modulo $2^N$. This behavior, while mathematically definable in modular arithmetic, is rarely the intent of the programmer in domains such as sensor fusion, financial transaction processing, or memory allocation sizing.
Furthermore, two's complement introduces a domain asymmetry. For an $N$-bit signed integer, the range is $[-2^{N-1}, 2^{N-1}-1]$. There exists a "singular" negative value (e.g., -128 in an 8-bit integer) that has no positive counterpart. Operations such as abs(INT_MIN) or -INT_MIN result in undefined behavior or identity mapping in standard C, leading to a class of bugs that are notoriously difficult to detect via static analysis.
1.2 The Aria Philosophy: TBB as a Correction
Aria's TBB types (tbb8, tbb16, tbb32, tbb64) resolve these fundamental issues by "twisting" the binary representation. By strictly enforcing a symmetric range—for example, [-127, +127] for an 8-bit type—Aria eliminates the asymmetry singularity. The bit pattern corresponding to the standard minimum value (e.g., 0x80 or -128) is excluded from the numeric domain and repurposed as a dedicated Error Sentinel (ERR).
The objective of this implementation effort is to reify the abstract semantics of TBB into concrete machine code. The compiler must ensure that this ERR sentinel behaves as an "absorbing element" in the algebra of integer operations. Conceptually, for any operator $\oplus$ and any value $x$, the identities $\text{ERR} \oplus x = \text{ERR}$ and $x \oplus \text{ERR} = \text{ERR}$ must hold. This "Sticky Error" property allows for Pipelined Validation, where a complex sequence of arithmetic operations can be executed without intermediate error checks, with the validity of the entire chain determined by a single check of the final result.
1.3 Scope of the Report
This document serves as a comprehensive engineering specification for modifying the Aria compiler backend (ariac) to support TBB sticky error propagation. It addresses the complete inventory of arithmetic operations, evaluates the trade-offs between branchy and branchless LLVM IR generation, defines the semantic behavior of ERR in comparisons, and provides a detailed integration plan for the ExprCodegen infrastructure. The analysis leverages research into the Aria ecosystem, including the atar archiver and aps process monitor, which rely on TBB safety guarantees for secure operation.1

2. Theoretical Framework: The Algebra of Sticky Errors
To implement a robust compiler backend, we must first establish a formal theoretical framework for TBB arithmetic. This framework defines the rules that the code generation logic must enforce.
2.1 Domain Definition and Symmetry
Let $T_w$ be a TBB type of width $w$ bits. In standard two's complement, the representable set $S_{std}$ is:




$$S_{std} = \{ i \in \mathbb{Z} \mid -2^{w-1} \le i \le 2^{w-1}-1 \}$$
The TBB domain $S_{tbb}$ is defined as the symmetric subset:




$$S_{tbb} = \{ i \in \mathbb{Z} \mid -(2^{w-1}-1) \le i \le 2^{w-1}-1 \}$$
The value $v_{min} = -2^{w-1}$ is excluded from $S_{tbb}$ and assigned the semantic label ERR.
For tbb8:
* Valid Domain: $[-127, +127]$
* ERR Sentinel: -128 (0x80)
This definition ensures that for all $x \in S_{tbb}$, $-x \in S_{tbb}$. The operation of negation is total and safe within the valid domain, removing the specific overflow hazard associated with INT_MIN.
2.2 The Propagation Invariant
The core requirement of the "Sticky Error" mechanism is the preservation of the invalid state. Formally, let $\mathbb{V} = S_{tbb} \cup \{\text{ERR}\}$ be the set of all possible values held by a TBB variable. We define an arithmetic operation $\odot: \mathbb{V} \times \mathbb{V} \rightarrow \mathbb{V}$ such that:
1. Input Propagation:

$$\forall y \in \mathbb{V}, \quad \text{ERR} \odot y = \text{ERR}$$
$$\forall x \in \mathbb{V}, \quad x \odot \text{ERR} = \text{ERR}$$
2. Closure on Overflow:
Let $R(x, y)$ be the mathematical result of $x \odot y$ over the infinite set of integers $\mathbb{Z}$.

$$x \odot y = \begin{cases} R(x, y) & \text{if } R(x, y) \in S_{tbb} \\ \text{ERR} & \text{if } R(x, y) \notin S_{tbb} \end{cases}$$
This second rule differentiates TBB from modular arithmetic (wrapping) and saturation arithmetic. While saturation clamps values to the boundary (destroying magnitude information), TBB returns ERR, explicitly signaling that the computation is no longer valid. This is critical for systems programming contexts like the atar utility, where a file size calculation overflow must prevent a memory allocation, not request a slightly smaller buffer.1
2.3 Comparison with IEEE 754 NaN
The TBB ERR sentinel shares similarities with the floating-point Not-a-Number (NaN) concept, but with critical divergences necessitated by integer semantics.
   * Similarities: Both represent invalid results (e.g., $0/0$) and propagate through arithmetic operations.
   * Differences:
   * Bit Pattern: NaN has many representations (payloads); ERR has exactly one (100...0).
   * Equality: IEEE 754 specifies NaN!= NaN. However, for integer sentinels in systems programming, the ability to check for the error state is paramount. As observed in the Aria shell integration (aria_shrc), scripts frequently check if (res == ERR).1 This necessitates an identity equality model (ERR == ERR is true).
   * Exceptions: Floating point units often support signaling NaNs that trap. TBB ERR is designed specifically to avoid traps, replacing hardware exceptions with data-flow propagation.

3. Comprehensive Arithmetic Operation Inventory
The research mandate requires an exhaustive catalog of operations. The compiler must intercept each of these and inject the necessary wrapper logic. Standard LLVM instructions (add, sub, icmp) are ignorant of the sentinel and the symmetric bounds; using them directly constitutes a compilation error in the context of Aria's specification.
3.1 Binary Arithmetic Operations
These operations form the core of the sticky error requirement.
Operator
	Symbol
	Semantics & Edge Cases
	Addition
	+, +=
	Standard addition. Overflow checks required. Edge Case: Mixed sign addition never overflows, but -64 + -64 = -128 (ERR) is a valid modular sum that must be caught as a sentinel collision.
	Subtraction
	-, -=
	Standard subtraction. Overflow checks required (127 - (-1)).
	Multiplication
	*, *=
	Standard multiplication. High risk of overflow. llvm.smul.with.overflow is essential.
	Division
	/, /=
	Div-by-Zero: Must return ERR, not trap. Overflow: INT_MIN / -1 is impossible in valid TBB inputs (since INT_MIN is ERR), but ERR / -1 must still be ERR.
	Modulo
	%, %=
	Mod-by-Zero: Must return ERR. Sign behavior follows C99 (truncated division).
	3.2 Unary Operations
Unary operations are simpler but must still respect the propagation invariant.
Operator
	Symbol
	Logic
	Negation
	-x
	If x is ERR, result is ERR. Since domain is symmetric, -127 -> 127 is valid. No overflow checking needed for valid inputs.
	Absolute
	abs(x)
	If x is ERR, result is ERR. abs(-127) -> 127. Safe.
	Bitwise Not
	~x
	Danger: ~ERR (~10000000) becomes 01111111 (127). This destroys the sentinel. Requires explicit handling: is_err(x)? ERR : ~x.
	Increment
	++
	x + 1. Checks for overflow at MAX. ERR++ -> ERR.
	Decrement
	--
	x - 1. Checks for underflow at -MAX. ERR-- -> ERR.
	3.3 Comparison Operations
Comparison logic determines how control flow interacts with errors.
Operator
	Semantics
	Implementation Logic
	Equality
	x == y
	true if bits match. ERR == ERR is true. Essential for error handling blocks.
	Inequality
	x!= y
	true if bits differ. ERR!= 5 is true.
	Less Than
	x < y
	false if x or y is ERR. ERR is not on the number line.
	Greater Than
	x > y
	false if x or y is ERR.
	LTE / GTE
	<=, >=
	false if x or y is ERR. Strict exclusion prevents logic bugs.
	Spaceship
	x <=> y
	Returns ERR (or equivalent ordering enum) if inputs are ERR.
	3.4 Bitwise Operations: The Semantic Conflict
Bitwise operations (&, |, ^, <<, >>) fundamentally operate on the representation, not the value. Applying them to TBB types is hazardous.
   * The Hazard: ERR & 0x7F results in 0. This operation strips the sign bit, effectively converting a fatal error state into the valid integer 0 without warning. This violates the "Sticky" principle and undermines the safety guarantees of the type.
   * Research Decision: While some languages might allow this, Aria's focus on safety 1 suggests that bitwise operations on TBB types should either be prohibited (requiring a cast to uint) or instrumented.
   * Instrumented Logic: If supported, x & y must be transformed to `(is_err(x) |
| is_err(y))? ERR : (x & y)`. This preserves the error state at the cost of performance. Given the performance goals, prohibiting them is the cleaner architectural choice, but for this implementation report, we will assume the instrumented path to satisfy the "Complete Inventory" requirement.

4. LLVM IR Implementation Strategy
Translating TBB semantics into LLVM Intermediate Representation (IR) requires a choice between control-flow based (branchy) and data-flow based (branchless) approaches.
4.1 The Branchy Approach (Naive)
A direct translation of the logic uses br instructions to jump to error handling blocks.


Code snippet




; Naive TBB Add
define i8 @tbb8_add(i8 %a, i8 %b) {
entry:
   ; Check Input A
   %a_err = icmp eq i8 %a, -128
   br i1 %a_err, label %return_err, label %check_b
check_b:
   ; Check Input B
   %b_err = icmp eq i8 %b, -128
   br i1 %b_err, label %return_err, label %perform_add
perform_add:
   ; Add with overflow intrinsic
   %res = call {i8, i1} @llvm.sadd.with.overflow.i8(i8 %a, i8 %b)
   %sum = extractvalue {i8, i1} %res, 0
   %ovf = extractvalue {i8, i1} %res, 1
   br i1 %ovf, label %return_err, label %check_sentinel
check_sentinel:
   ; Check for result collision (-128)
   %collision = icmp eq i8 %sum, -128
   br i1 %collision, label %return_err, label %return_sum
return_err:
   ret i8 -128
return_sum:
   ret i8 %sum
}

Critique: This structure introduces up to 4 basic blocks for a single addition. Modern CPU architectures rely heavily on branch prediction and speculative execution. While the "happy path" (no error) is predictable, the presence of multiple conditional branches consumes entries in the Branch Target Buffer (BTB) and inhibits the compiler's ability to reorder instructions. Furthermore, this structure destroys the potential for SIMD auto-vectorization, as LLVM's loop vectorizer struggles with control flow divergence inside arithmetic loops.2
4.2 The Branchless Approach (Optimized)
By utilizing the select instruction (ternary operator) and bitwise logic, we can flatten the control flow into a linear sequence of data dependencies. This is the recommended implementation strategy.


Code snippet




; Optimized TBB Add
define i8 @tbb8_add_fast(i8 %a, i8 %b) {
   ; Constants
   %ERR = add i8 0, -128 ; 0x80

   ; 1. Input Validation (Parallelizable)
   %a_is_err = icmp eq i8 %a, %ERR
   %b_is_err = icmp eq i8 %b, %ERR
   %input_invalid = or i1 %a_is_err, %b_is_err

   ; 2. Arithmetic with Overflow (Hardware Flag Usage)
   %res_struct = call {i8, i1} @llvm.sadd.with.overflow.i8(i8 %a, i8 %b)
   %sum = extractvalue {i8, i1} %res_struct, 0
   %ovf = extractvalue {i8, i1} %res_struct, 1

   ; 3. Sentinel Collision Check
   ; Did a valid calculation result in -128? e.g., -64 + -64
   %res_is_sentinel = icmp eq i8 %sum, %ERR

   ; 4. Aggregate Failure Flags
   %fail_1 = or i1 %input_invalid, %ovf
   %fail_final = or i1 %fail_1, %res_is_sentinel

   ; 5. Select Final Result (Conditional Move)
   %result = select i1 %fail_final, i8 %ERR, i8 %sum
   ret i8 %result
}

Advantages:
   1. Pipeline Efficiency: The instruction stream is linear. There are no branches to mispredict. The select instruction typically lowers to a cmov (Conditional Move) on x86 or a conditional selector on ARM (csel), which are executed in the ALU pipeline without stalling the fetch/decode stages.3
   2. Vectorization: The select instruction maps directly to vector masking instructions (e.g., vblendv on AVX). This allows the compiler to vectorize operations on arrays of TBB types, applying the error logic to 8, 16, or 32 values in parallel. This is crucial for high-performance workloads like the Nikola visualization mentioned in the research.1
   3. Instruction Level Parallelism (ILP): The input checks and the arithmetic operation are data-independent. A superscalar CPU can execute the input validation icmp instructions simultaneously with the sadd.with.overflow intrinsic.
4.3 Overflow Intrinsics Details
The use of llvm.sadd.with.overflow is non-negotiable for performance. Manually checking for overflow (e.g., if ((b > 0 && a > MAX - b)...) requires multiple comparisons and branches. The intrinsic gives direct access to the CPU's overflow flag (OF on x86, V flag on ARM).
   * x86 Lowering: add followed by seto (Set on Overflow) or jo (Jump on Overflow).
   * ARM Lowering: adds (Add setting flags) followed by condition code checks.
This approach reduces the overhead of overflow detection to nearly zero cycles in the best case, as the flag computation is part of the arithmetic instruction itself.4

5. Implementation Strategy: Modifying ExprCodegen
This section provides the concrete engineering blueprint for integrating the TBB logic into the Aria compiler source code. Based on standard compiler architectures, the ExprCodegen class (or CodeGenVisitor) is the locus of binary expression translation.
5.1 Architecture of ExprCodegen
The compiler utilizes a visitor pattern to traverse the Abstract Syntax Tree (AST). The codegenBinaryExpr method is responsible for generating IR for binary operators.
Integration Point: We must intercept the generation process before the standard integer logic is applied. This requires checking the type of the operands.
5.2 The generateTBBBinaryOp Helper
We define a comprehensive helper method that encapsulates the branchless logic. This keeps the main codegen loop clean.


C++




// Pseudocode for src/backend/codegen_tbb.cpp

llvm::Value* CodeGenContext::generateTBBBinaryOp(
   llvm::Value* lhs, 
   llvm::Value* rhs, 
   BinaryOp op, 
   int width // 8, 16, 32, 64
) {
   llvm::IRBuilder<>& builder = this->builder;
   llvm::Type* type = lhs->getType();
   
   // 1. Sentinel Definition (-2^(N-1))
   // We construct the APInt manually to ensure width-correctness.
   llvm::APInt minVal = llvm::APInt::getSignedMinValue(width);
   llvm::Value* errSentinel = llvm::ConstantInt::get(type, minVal);

   // 2. Input Validation (Sticky Error Input)
   // Check LHS and RHS against Sentinel
   llvm::Value* lhsIsErr = builder.CreateICmpEQ(lhs, errSentinel, "lhs_is_err");
   llvm::Value* rhsIsErr = builder.CreateICmpEQ(rhs, errSentinel, "rhs_is_err");
   llvm::Value* inputErr = builder.CreateOr(lhsIsErr, rhsIsErr, "input_err");

   // 3. Operation Dispatch
   llvm::Value* mathResult = nullptr;
   llvm::Value* opOverflow = nullptr;

   switch (op) {
       case BinOp::Add: {
           llvm::Function* intrinsic = llvm::Intrinsic::getDeclaration(
               this->module, llvm::Intrinsic::sadd_with_overflow, {type});
           llvm::Value* resStruct = builder.CreateCall(intrinsic, {lhs, rhs});
           mathResult = builder.CreateExtractValue(resStruct, 0);
           opOverflow = builder.CreateExtractValue(resStruct, 1);
           break;
       }
       case BinOp::Sub: {
           llvm::Function* intrinsic = llvm::Intrinsic::getDeclaration(
               this->module, llvm::Intrinsic::ssub_with_overflow, {type});
           llvm::Value* resStruct = builder.CreateCall(intrinsic, {lhs, rhs});
           mathResult = builder.CreateExtractValue(resStruct, 0);
           opOverflow = builder.CreateExtractValue(resStruct, 1);
           break;
       }
       case BinOp::Mul: {
           llvm::Function* intrinsic = llvm::Intrinsic::getDeclaration(
               this->module, llvm::Intrinsic::smul_with_overflow, {type});
           llvm::Value* resStruct = builder.CreateCall(intrinsic, {lhs, rhs});
           mathResult = builder.CreateExtractValue(resStruct, 0);
           opOverflow = builder.CreateExtractValue(resStruct, 1);
           break;
       }
       case BinOp::Div:
       case BinOp::Mod:
           return generateTBBDivMod(lhs, rhs, op, width, inputErr, errSentinel);
       default:
           throw CodeGenException("Unsupported TBB Binary Op");
   }

   // 4. Result Sentinel Collision Check
   // If a valid calculation results in the sentinel bit pattern (e.g. -64 + -64 = -128 in i8),
   // this is an underflow relative to the symmetric TBB range [-127, 127].
   llvm::Value* resIsSentinel = builder.CreateICmpEQ(mathResult, errSentinel, "res_collision");

   // 5. Aggregate Error Flags
   // Total Error = InputErr OR OpOverflow OR ResCollision
   llvm::Value* anyErr = builder.CreateOr(inputErr, opOverflow);
   anyErr = builder.CreateOr(anyErr, resIsSentinel, "tbb_err_flag");

   // 6. Select Final Result
   return builder.CreateSelect(anyErr, errSentinel, mathResult, "tbb_res");
}

5.3 Division and Modulo Complexity
Division logic is more intricate because there is no single "with overflow" intrinsic that covers all cases, and CPU division traps on zero. We must manually guard the instruction to prevent the compiled program from crashing.


C++




llvm::Value* generateTBBDivMod(...) {
   //... Input validation logic...

   // 1. Division by Zero Check
   llvm::Value* zero = llvm::ConstantInt::get(type, 0);
   llvm::Value* divByZero = builder.CreateICmpEQ(rhs, zero, "div_zero");

   // 2. Overflow Check (INT_MIN / -1)
   // While INT_MIN is ERR (and thus caught by input check), we include this
   // for robustness against future sentinel changes.
   llvm::Value* minusOne = llvm::ConstantInt::get(type, -1);
   llvm::Value* lhsIsMin = builder.CreateICmpEQ(lhs, errSentinel);
   llvm::Value* rhsIsNegOne = builder.CreateICmpEQ(rhs, minusOne);
   llvm::Value* overflow = builder.CreateAnd(lhsIsMin, rhsIsNegOne, "div_overflow");

   // 3. Aggregate Checks
   llvm::Value* opFault = builder.CreateOr(divByZero, overflow);
   llvm::Value* totalErr = builder.CreateOr(inputErr, opFault);

   // 4. Safe Divisor Selection
   // If the divisor is 0, replace it with 1 to allow the instruction to execute safely.
   // The result will be discarded by the final select, but the CPU won't trap.
   llvm::Value* safeRhs = builder.CreateSelect(divByZero, llvm::ConstantInt::get(type, 1), rhs);

   // 5. Perform Operation
   llvm::Value* mathResult;
   if (op == BinOp::Div)
       mathResult = builder.CreateSDiv(lhs, safeRhs);
   else
       mathResult = builder.CreateSRem(lhs, safeRhs);

   // 6. Result Collision (Rare but possible in Modulo)
   llvm::Value* resIsSentinel = builder.CreateICmpEQ(mathResult, errSentinel);
   totalErr = builder.CreateOr(totalErr, resIsSentinel);

   return builder.CreateSelect(totalErr, errSentinel, mathResult);
}

6. Comparison and Control Flow Semantics
The behavior of ERR in boolean contexts is a pivotal design decision. It determines how developers detect and handle errors.
6.1 The "Sentinel Identity" Model
Research into the usage patterns of Aria utilities like atar reveals constructs such as if (entry.size == ERR) {... }.1 This pattern necessitates that ERR is equal to itself. This contrasts with IEEE 754 floating point, where NaN!= NaN.
      * Decision: ERR == ERR must evaluate to True.
      * Implementation: Equality checks (==, !=) on TBB types are performed as raw integer comparisons. No special wrapper logic is needed for equality; the underlying bit pattern comparison suffices.
6.2 Ordered Comparisons
Ordering (<, >, <=, >=) against an error sentinel is semantically undefined. Is "Error" less than 5? Philosophically, no. Treating ERR as -128 would make ERR < 0 true, effectively treating a catastrophic failure as a valid negative number. This is dangerous for loop bounds and threshold checks.
      * Decision: Any ordered comparison involving ERR must evaluate to False. This mimics the "unordered" behavior of floats.
      * Implementation: We must wrap icmp instructions for ordered predicates.


Code snippet




; TBB Less Than (tbb8_lt)
define i1 @tbb8_lt(i8 %a, i8 %b) {
   %ERR = -128
   
   ; Check Inputs
   %a_err = icmp eq i8 %a, %ERR
   %b_err = icmp eq i8 %b, %ERR
   %any_err = or i1 %a_err, %b_err
   
   ; Raw Comparison
   %raw_res = icmp slt i8 %a, %b
   
   ; Force False if Error
   %res = select i1 %any_err, i1 false, i1 %raw_res
   ret i1 %res
}

This ensures that checks like if (val < limit) fail safely if val is corrupted, forcing the code into the else block (or requiring an explicit == ERR check).

7. Performance Impact and Optimization
Implementing safety has a cost. We quantify this cost and discuss mitigation.
7.1 Cycle Count Analysis
      * Standard Integer Add: ~1 cycle latency.
      * TBB Safe Add:
      * 2 icmp (input check) + 1 or.
      * 1 sadd.with.overflow.
      * 1 icmp (result check).
      * 2 or (flag aggregation).
      * 1 select.
      * Estimated Latency: 4-6 cycles.
While a 400-600% increase in latency seems high, the throughput impact is lower due to Instruction Level Parallelism (ILP). The input checks can execute in parallel with the addition.
7.2 Vectorization
The critical performance win of the branchless design is vectorization. If the compiler encounters a loop adding two arrays of tbb8, the LLVM autovectorizer can instantiate the TBB logic using vector instructions.
      * icmp -> vpcmpgtb (Vector Compare)
      * select -> vpblendvb (Vector Blend)
      * sadd.with.overflow -> No direct vector intrinsic on x86, but LLVM can synthesize it using vector adds and comparison checks.
This ensures that TBB arrays can be processed with high throughput, essential for the "Data Plane" applications (acurl, agrep) mentioned in the AriaX research.1
7.3 Inline vs. Runtime
We recommend a hybrid strategy:
         * Inline: Add, Sub, Neg. The instruction count (~10 ops) is small enough that the overhead of a function call (register saving, stack frame) would outweigh the code size savings. Inlining allows further optimizations (e.g., constant folding ERR + 5 to ERR at compile time).
         * Runtime Library: Mul, Div, Mod. These are larger (especially Div) and less frequent. Placing them in libaria_runtime reduces instruction cache pressure.

8. Testing and Validation
A rigorous test strategy is required to verify the implementation.
8.1 Test Categories
         1. Propagation Tests: Verify ERR flows through all ops.
         * assert((ERR + 1) == ERR)
         * assert((1 - ERR) == ERR)
         2. Boundary Tests: Verify symmetric limits.
         * assert((127 + 1) == ERR)
         * assert((-127 - 1) == ERR)
         3. Collision Tests: Verify the "hole" at -128.
         * assert((-64 + -64) == ERR) (Valid math, invalid result)
         4. Comparison Tests:
         * assert(ERR == ERR)
         * assert(!(ERR < 0))
         * assert(!(0 > ERR))
8.2 Fuzzing Strategy
A fuzzer should generate random sequences of operations on tbb8 values. Ideally, we implement a "Gold Standard" oracle using int16 in software. The oracle performs the math, checks the result against [-127, 127], and compares its output to the TBB implementation. Any divergence indicates a compiler bug.

9. Conclusion
The implementation of Sticky Error propagation transforms the Aria compiler from a standard tool into a safety-critical instrument. By shifting error handling from the developer's control flow (manual checks) to the language's data flow (sticky sentinels), we eliminate entire classes of vulnerability including silent overflow and uninitialized value usage.
The architectural decision to utilize branchless LLVM IR via select is the linchpin of this design. It aligns with modern superscalar processor design, enables SIMD vectorization, and minimizes the performance penalty of safety. While there is a computational cost (approx. 4-6x latency vs raw integers), the gain in system robustness—demonstrated by the reliability of the atar and aps utilities—is a decisive advantage for the Aria ecosystem.
