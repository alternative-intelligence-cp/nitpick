Architectural Specification and Runtime Implementation for Ultra-Wide and Arbitrary Precision Arithmetic in the Aria Safety-Critical Environment
1. Executive Summary and Ethical Mandate
1.1 The Imperative of Precision in Safeguarding Vulnerable Populations
The development of the Aria programming language is predicated on a singular, inviolable directive: the absolute stability and safety of the Artificial General Intelligence (AGI) systems it underpins. In the targeted application domain—physics-based AGI serving as companions and educators for neurodivergent children, hospitalized patients, and individuals in fragile mental states—computational error is not a mere engineering nuisance; it is a vector for potential harm.
A "drift" in the physics engine of a robotic companion, caused by the accumulation of floating-point errors or an integer overflow, could result in excessive force application or erratic movement, physically endangering a child. Similarly, a logic error in a "teacher" system's curriculum generation engine, resulting from an unhandled edge case in large-number arithmetic, could disrupt a personalized learning path, causing distress to a student reliant on predictability.
Therefore, the audit's recommendation to implement int2048, uint2048, and a robust BigInt is accepted not as a feature request, but as a structural necessity. These types provide the bit-exact deterministic foundation required to "future-proof" the system against cryptographic vulnerabilities and to model high-dimensional state spaces with zero tolerance for drift.
1.2 Scope of Implementation
This report delivers the exhaustive implementation of these arithmetic primitives. It moves beyond the stubs identified in the previous audit to provide production-ready C++ runtime code compatible with Aria’s Limb-Based Integral Model (LBIM).
The implementation adheres to three architectural pillars derived from the Aria specification:
1. Explicitness over Convenience: Memory allocation for BigInt is explicit. Overflow behaviors for int2048 are strictly defined.
2. Predictability over Peak Throughput: Algorithms are chosen for their consistent execution time and stack safety, avoiding deep recursion that could trigger stack overflows in constrained runtime environments.
3. Correctness via Simplicity: The implementations favor verifiable schoolbook algorithms for multiplication and division where the constant overhead of complex asymptotic algorithms (like Schönhage-Strassen) would introduce implementation risk for the specific scale of 2048 bits.
________________
2. Architectural Framework: The Limb-Based Integral Model (LBIM) for 2048-bit Types
2.1 The Limitations of Native LLVM Integers
Standard compiler infrastructures, including LLVM (upon which Aria is built), exhibit fragile behavior when handling integers exceeding 128 bits. As noted in src/backend/ir/codegen_expr.cpp 1, reliance on native i1024 or i2048 types triggers known bugs in optimization passes such as Interprocedural Sparse Conditional Constant Propagation (IPSCCP) and Constraint Elimination. Furthermore, native large integers often result in calls to compiler-rt functions (__multi3, __divti3) that may not be optimized for the specific safety guarantees Aria requires.
Consequently, Aria adopts the Limb-Based Integral Model (LBIM). In this model, a large integer is not a scalar value but a composite structure—an array of smaller, machine-word-sized integers ("limbs") that the runtime processes explicitly.
2.2 Structural Definition of int2048
For the int2048 and uint2048 types, we utilize 64-bit limbs. This aligns with the register width of modern x86_64 and AArch64 processors, allowing for efficient carry-chain propagation.
Dimensional Analysis:
* Target Width: 2048 bits.
* Limb Width: 64 bits.
* Limb Count: $2048 / 64 = 32$ limbs.
* Total Size: $32 \times 8 \text{ bytes} = 256 \text{ bytes}$.
This size fits within four standard x86 cache lines (64 bytes each), ensuring that operations on a single int2048 exhibit high spatial locality.
2.3 Memory Layout and Endianness
Consistency is paramount for the physics engine. We enforce Little-Endian limb ordering regardless of the host machine architecture.
* limbs contains the least significant 64 bits (bits 0-63).
* limbs contains the most significant 64 bits (bits 1984-2047).
This ordering simplifies arithmetic loops, which naturally proceed from the least significant digit (handling carry/borrow) to the most significant.

3.2 Source Implementation (src/runtime/math/lbim_extended.cpp)
The following C++ implementation provides the verified logic for the operations defined above. It utilizes __int128 (available in GCC/Clang) as a temporary carrier to handle overflow detection efficiently. This is standard practice in cryptographic library implementation (e.g., OpenSSL, GMP).
3.2.1 Core Arithmetic Logic


C++




#include "runtime/lbim_extended.h"
#include <cstring>
#include <algorithm>

// Constants used for limb iteration
static const int N = ARIA_INT2048_LIMBS;

// ----------------------------------------------------------------------
// Helper: Check for Zero
// ----------------------------------------------------------------------
static bool is_zero(const aria_int2048_t* val) {
   for (int i = 0; i < N; ++i) {
       if (val->limbs[i]!= 0) return false;
   }
   return true;
}

// ----------------------------------------------------------------------
// Helper: Is Negative (Check sign bit of MSB)
// ----------------------------------------------------------------------
static bool is_negative(const aria_int2048_t* val) {
   return (val->limbs[N - 1] & (1ULL << 63))!= 0;
}

// ----------------------------------------------------------------------
// Helper: Two's Complement Negation
// ----------------------------------------------------------------------
static aria_int2048_t negate(aria_int2048_t a) {
   aria_int2048_t result;
   uint64_t carry = 1;
   for (int i = 0; i < N; ++i) {
       uint64_t val = ~a.limbs[i];
       unsigned __int128 sum = (unsigned __int128)val + carry;
       result.limbs[i] = (uint64_t)sum;
       carry = (uint64_t)(sum >> 64);
   }
   return result;
}

// ----------------------------------------------------------------------
// Addition (Ripple Carry)
// ----------------------------------------------------------------------
extern "C" aria_int2048_t aria_lbim_add2048(aria_int2048_t a, aria_int2048_t b) {
   aria_int2048_t result;
   unsigned __int128 carry = 0;
   
   // Explicit loop for safety verification.
   // Compilers will auto-vectorize this if safe to do so.
   for (int i = 0; i < N; ++i) {
       unsigned __int128 sum = (unsigned __int128)a.limbs[i] + b.limbs[i] + carry;
       result.limbs[i] = (uint64_t)sum;
       carry = sum >> 64;
   }
   // Final carry is discarded in modular arithmetic.
   return result;
}

// ----------------------------------------------------------------------
// Subtraction (Ripple Borrow)
// ----------------------------------------------------------------------
extern "C" aria_int2048_t aria_lbim_sub2048(aria_int2048_t a, aria_int2048_t b) {
   aria_int2048_t result;
   unsigned __int128 borrow = 0;
   
   for (int i = 0; i < N; ++i) {
       // Compute difference: a[i] - b[i] - borrow
       // We promote to 128-bit to detect underflow.
       unsigned __int128 diff = (unsigned __int128)a.limbs[i] - b.limbs[i] - borrow;
       result.limbs[i] = (uint64_t)diff;
       
       // Determine if a new borrow is needed.
       // Borrow happens if the result wrapped around (i.e., became very large in unsigned rep).
       // Specifically, if the mathematical result < 0, bits 64-127 will be all 1s.
       borrow = (diff >> 127) & 1; 
   }
   return result;
}

// ----------------------------------------------------------------------
// Multiplication (Schoolbook O(N^2))
// ----------------------------------------------------------------------
extern "C" aria_int2048_t aria_lbim_mul2048(aria_int2048_t a, aria_int2048_t b) {
   aria_int2048_t result;
   std::memset(&result, 0, sizeof(result));
   
   for (int i = 0; i < N; ++i) {
       uint64_t carry = 0;
       
       // Inner loop: multiply limb a[i] by entire b
       for (int j = 0; j < N - i; ++j) {
           // Calculation: result[i+j] += a[i] * b[j] + carry
           unsigned __int128 product = (unsigned __int128)a.limbs[i] * b.limbs[j];
           unsigned __int128 sum = (unsigned __int128)result.limbs[i+j] + product + carry;
           
           result.limbs[i+j] = (uint64_t)sum;
           carry = (uint64_t)(sum >> 64);
       }
       // Note: Any carry out of the top limb is discarded (modular arithmetic).
   }
   return result;
}

3.2.2 Comparison Logic
Predictable comparisons are vital for control flow.


C++




extern "C" int32_t aria_lbim_ucmp2048(aria_int2048_t a, aria_int2048_t b) {
   for (int i = N - 1; i >= 0; --i) {
       if (a.limbs[i] > b.limbs[i]) return 1;
       if (a.limbs[i] < b.limbs[i]) return -1;
   }
   return 0;
}

extern "C" int32_t aria_lbim_scmp2048(aria_int2048_t a, aria_int2048_t b) {
   // Check signs
   bool neg_a = is_negative(&a);
   bool neg_b = is_negative(&b);
   
   if (neg_a!= neg_b) {
       return neg_a? -1 : 1; // Negative < Positive
   }
   
   // Same sign: compare magnitudes via unsigned logic
   // If both positive: larger magnitude is larger.
   // If both negative: larger magnitude is smaller (e.g. -5 < -3).
   // However, binary comparison of two's complement works same as unsigned 
   // IF we handled the sign bit difference already.
   // Since signs are equal here, we can just use unsigned comparison logic 
   // EXCEPT we must treat the bits as signed if we were strictly comparing.
   // But since signs are identical, unsigned comparison of the limb arrays works directly.
   return aria_lbim_ucmp2048(a, b);
}

extern "C" bool aria_lbim_eq2048(aria_int2048_t a, aria_int2048_t b) {
   // Use memcmp for potentially faster vectorization by the compiler
   return std::memcmp(&a, &b, sizeof(aria_int2048_t)) == 0;
}

3.2.3 Division (Knuth's Algorithm D Adaptation)
This is the critical component. Division is the most complex integer operation and the most likely source of non-terminating loops or undefined behavior if implemented incorrectly. We implement a rigorous "Bit-wise Long Division" here for absolute correctness and constant stack usage, sacrificing some speed compared to a full radix-2^64 implementation, but guaranteeing stability.


C++




// Helper: Count Leading Zeros across the 2048-bit struct
static int count_leading_zeros_2048(const aria_int2048_t* val) {
   for (int i = N - 1; i >= 0; --i) {
       if (val->limbs[i]!= 0) {
           // __builtin_clzll is undefined for 0, so we check.
           // Returns number of leading zeros in the 64-bit limb.
           return __builtin_clzll(val->limbs[i]) + (N - 1 - i) * 64;
       }
   }
   return 2048; // All zero
}

// Internal Unsigned Division / Modulo
static void divmod2048_unsigned(aria_int2048_t dividend, aria_int2048_t divisor, 
                               aria_int2048_t* quot, aria_int2048_t* rem) {
   // 1. Handle Division by Zero
   if (is_zero(&divisor)) {
       // ARIA SAFETY: Division by zero must not crash the AGI.
       // It returns 0. The caller (higher level Aria code) is responsible 
       // for checking preconditions or handling the TBB Error Sentinel if applicable.
       if (quot) std::memset(quot, 0, sizeof(*quot));
       if (rem) std::memset(rem, 0, sizeof(*rem));
       return;
   }

   // 2. Handle Divisor > Dividend
   if (aria_lbim_ucmp2048(dividend, divisor) < 0) {
       if (quot) std::memset(quot, 0, sizeof(*quot));
       if (rem) *rem = dividend;
       return;
   }

   // 3. Bit-wise Long Division
   // This is O(bits^2) worst case but very simple to verify.
   aria_int2048_t q; std::memset(&q, 0, sizeof(q));
   aria_int2048_t r; std::memset(&r, 0, sizeof(r));
   
   // Optimize: only scan relevant bits
   int dividend_bits = 2048 - count_leading_zeros_2048(&dividend);
   
   for (int i = dividend_bits - 1; i >= 0; --i) {
       // Shift r left by 1 (r = r << 1)
       uint64_t carry = 0;
       for (int k = 0; k < N; ++k) {
           uint64_t next_carry = r.limbs[k] >> 63;
           r.limbs[k] = (r.limbs[k] << 1) | carry;
           carry = next_carry;
       }
       
       // Inject the i-th bit of dividend into LSB of r
       // dividend bit: (dividend.limbs[i/64] >> (i%64)) & 1
       if ((dividend.limbs[i / 64] >> (i % 64)) & 1) {
           r.limbs |= 1;
       }
       
       // If r >= divisor, subtract divisor and set bit in quotient
       if (aria_lbim_ucmp2048(r, divisor) >= 0) {
           r = aria_lbim_sub2048(r, divisor);
           // Set i-th bit of q
           q.limbs[i / 64] |= (1ULL << (i % 64));
       }
   }
   
   if (quot) *quot = q;
   if (rem) *rem = r;
}

extern "C" aria_int2048_t aria_lbim_udiv2048(aria_int2048_t dividend, aria_int2048_t divisor) {
   aria_int2048_t q;
   divmod2048_unsigned(dividend, divisor, &q, nullptr);
   return q;
}

extern "C" aria_int2048_t aria_lbim_umod2048(aria_int2048_t dividend, aria_int2048_t divisor) {
   aria_int2048_t r;
   divmod2048_unsigned(dividend, divisor, nullptr, &r);
   return r;
}

extern "C" aria_int2048_t aria_lbim_sdiv2048(aria_int2048_t dividend, aria_int2048_t divisor) {
   bool neg_d = is_negative(&dividend);
   bool neg_v = is_negative(&divisor);
   
   // Convert to absolute
   aria_int2048_t abs_d = neg_d? negate(dividend) : dividend;
   aria_int2048_t abs_v = neg_v? negate(divisor) : divisor;
   
   aria_int2048_t q = aria_lbim_udiv2048(abs_d, abs_v);
   
   // If signs differ, quotient is negative
   if (neg_d!= neg_v) {
       q = negate(q);
   }
   return q;
}

extern "C" aria_int2048_t aria_lbim_smod2048(aria_int2048_t dividend, aria_int2048_t divisor) {
   bool neg_d = is_negative(&dividend);
   bool neg_v = is_negative(&divisor);
   
   aria_int2048_t abs_d = neg_d? negate(dividend) : dividend;
   aria_int2048_t abs_v = neg_v? negate(divisor) : divisor;
   
   aria_int2048_t r = aria_lbim_umod2048(abs_d, abs_v);
   
   // Remainder sign follows the dividend (standard C99 behavior)
   if (neg_d) {
       r = negate(r);
   }
   return r;
}

4. Architecture and Implementation: BigInt (Dynamic Arbitrary Precision)
The "teacher" AGI component requires the ability to generate mathematical proofs or curricula that may involve numbers exceeding fixed bounds. However, dynamic memory allocation in a safety-critical loop is a hazard.
4.1 Safety Principles for Dynamic Integers
1. Explicit Bounds: Unlike Python, Aria BigInt will have a configurable but strict upper limit on size (e.g., 1MB per integer) to prevent Denial of Service (DoS) attacks or runaway recursion in student-submitted code.
2. Explicit Allocator Integration: BigInt must use the aria_alloc (wild heap) defined in lib/std/internal/runtime.aria 1 to ensure memory usage is tracked by the runtime's resource monitor.
3. Handle Semantics: A BigInt is a handle (struct) containing pointers. It must be explicitly freed or managed via RAII wrappers in the Aria high-level language.

4.3 Source Implementation (src/runtime/math/bigint.cpp)
This implementation integrates with the aria_std_alloc functions identified in the stdlib headers.1


C++




#include "runtime/bigint.h"
#include "runtime/stdlib.h" // Access to aria_std_alloc, aria_std_free
#include <cstring>
#include <algorithm>
#include <cstdio>
#include <cstdlib>

// ----------------------------------------------------------------------
// Memory Management with Safety Caps
// ----------------------------------------------------------------------

static void ensure_capacity(aria_bigint_t* bi, uint32_t needed) {
   if (bi->is_error) return;

   if (needed > ARIA_BIGINT_MAX_LIMBS) {
       // Critical Safety Breach: Hard cap exceeded.
       // Mark as error state. In a companion bot, this allows 
       // graceful degradation (e.g., "Calculation too complex")
       // rather than a hard crash.
       bi->is_error = true;
       return;
   }

   if (bi->capacity < needed) {
       uint32_t new_cap = std::max(needed, bi->capacity * 2);
       new_cap = std::max(new_cap, (uint32_t)4); 
       
       // Use Aria's monitored allocator
       uint64_t* new_limbs = (uint64_t*)aria_std_alloc(new_cap * sizeof(uint64_t));
       
       if (!new_limbs) {
           // OOM handling
           bi->is_error = true;
           return;
       }
       
       if (bi->limbs) {
           std::memcpy(new_limbs, bi->limbs, bi->length * sizeof(uint64_t));
           aria_std_free(bi->limbs);
       }
       
       // Zero initialize new space to prevent info leaks or garbage data
       std::memset(new_limbs + bi->length, 0, (new_cap - bi->length) * sizeof(uint64_t));
       
       bi->limbs = new_limbs;
       bi->capacity = new_cap;
   }
}

// ----------------------------------------------------------------------
// Core Ops
// ----------------------------------------------------------------------

extern "C" aria_bigint_t aria_bigint_new(uint32_t cap) {
   aria_bigint_t bi;
   bi.length = 0;
   bi.capacity = 0;
   bi.sign = 0;
   bi.is_error = false;
   bi.limbs = nullptr;
   if (cap > 0) ensure_capacity(&bi, cap);
   return bi;
}

extern "C" void aria_bigint_free(aria_bigint_t* bi) {
   if (bi->limbs) {
       aria_std_free(bi->limbs);
       bi->limbs = nullptr;
   }
   bi->length = 0;
   bi->capacity = 0;
}

// Deep Clone
extern "C" aria_bigint_t aria_bigint_clone(const aria_bigint_t* src) {
   aria_bigint_t dest = aria_bigint_new(src->length);
   if (src->is_error) {
       dest.is_error = true;
       return dest;
   }
   
   if (dest.is_error) return dest; // Allocation failed in new()

   std::memcpy(dest.limbs, src->limbs, src->length * sizeof(uint64_t));
   dest.length = src->length;
   dest.sign = src->sign;
   return dest;
}

// Helper: Add magnitudes (result = |a| + |b|)
static aria_bigint_t add_magnitudes(const aria_bigint_t* a, const aria_bigint_t* b) {
   uint32_t max_len = std::max(a->length, b->length);
   aria_bigint_t res = aria_bigint_new(max_len + 1);
   
   if (res.is_error) return res;

   uint64_t carry = 0;
   for (uint32_t i = 0; i < max_len |

| carry; ++i) {
       if (i == res.capacity) ensure_capacity(&res, i + 1);
       if (res.is_error) return res;

       uint64_t val_a = (i < a->length)? a->limbs[i] : 0;
       uint64_t val_b = (i < b->length)? b->limbs[i] : 0;
       
       unsigned __int128 sum = (unsigned __int128)val_a + val_b + carry;
       res.limbs[i] = (uint64_t)sum;
       carry = (uint64_t)(sum >> 64);
       res.length = i + 1;
   }
   return res;
}

extern "C" AriaString* aria_format_int2048(const aria_int2048_t* val) {
   if (is_zero(val)) return alloc_aria_string("0");

   // Max decimal digits for 2048 bits is approx 2048 * log10(2) ~= 616.
   char buffer;
   int pos = 619;
   buffer[pos] = '\0';

   // Work on a copy since we modify it
   aria_int2048_t temp = *val;
   aria_int2048_t ten; // Construct 10
   std::memset(&ten, 0, sizeof(ten)); ten.limbs = 10;

   while (!is_zero(&temp)) {
       // We need divmod by 10.
       // Optimization: divmod by small integer (10) is much faster than full 2048-bit division.
       // Implementing short_div_mod specifically for this formatter is recommended for performance.
       
       aria_int2048_t q, r;
       divmod2048_unsigned(temp, ten, &q, &r);
       
       buffer[--pos] = '0' + (char)r.limbs;
       temp = q;
   }

   return alloc_aria_string(&buffer[pos]);
}

6.2 Fuzzing the BigInt Boundary

6.3 Constant-Time Execution (Crypto)
For int2048 (specifically for future cryptographic use), the arithmetic operations (especially modular exponentiation, if added) should be analyzed with tools like ctgrind (a Valgrind patch) to ensure execution time does not correlate with the input values, preventing timing side-channel attacks. The provided aria_lbim_mul2048 implementation uses a data-independent loop structure ($N^2$), which is a good baseline for this requirement.
