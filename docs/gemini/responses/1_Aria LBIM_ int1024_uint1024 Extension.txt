Architectural Extension of the Limb-Based Integral Model (LBIM) to 1024-bit Precision: Design, Implementation, and Safety Protocols
1. Executive Summary
In the domain of high-integrity computing, particularly for safety-critical AGI systems and cryptographic primitives, the precision and reliability of integral arithmetic are paramount. The Aria language ecosystem, designed as a "companion for vulnerable populations" 1, mandates a rigor in numerical processing that exceeds standard industrial requirements. As the ecosystem evolves to support post-quantum cryptographic standards—specifically RSA-2048/4096 and advanced Zero-Knowledge Proof (ZKP) systems—the current 512-bit integral limit (int512) imposed by the Limb-Based Integral Model (LBIM) has become a constraining factor.
This report details the comprehensive architectural specification and implementation strategy for extending the LBIM to support 1024-bit integers (int1024 and uint1024). This extension is not merely a widening of data paths; it represents a fundamental enhancement of the runtime's arithmetic core, adhering strictly to the "Twisted Balanced Binary" (TBB) philosophy. TBB ensures that arithmetic errors, such as overflows or divisions by zero, result in a "Sticky Error" state rather than undefined behavior or silent wraparound, preserving the integrity of data flow through complex computational graphs.1
The proposed solution involves a bipartite modification: upgrading the LLVM IR generation layer (src/backend/ir/codegen_expr.cpp) to recognize and allocate 1024-bit aggregate structures, and developing a highly optimized, constant-time C++ runtime library (src/runtime/math/lbim_1024.cpp). The runtime implementation leverages Comba multiplication for side-channel resistance and Knuth’s Algorithm D for robust division, utilizing the "wild allocator" for memory management to eliminate garbage collection pauses in critical paths.1
By moving from 512-bit to 1024-bit primitives, Aria achieves native parity with the requirements of modern cryptographic protocols without sacrificing the safety guarantees that distinguish the language. This report provides the complete theoretical justification, algorithmic derivation, and production-ready source code for this transition.

2. Theoretical Framework: The Limb-Based Integral Model (LBIM)
To understand the necessity of the implementation details provided later in this report, one must first appreciate the architectural constraints and design philosophy of the Aria compiler backend.
2.1 The Case Against Native LLVM Integers
Standard compiler engineering often leverages LLVM's support for arbitrary-width integers (e.g., i1024). While theoretically elegant, this approach is pragmatically flawed for production-grade, safety-critical languages. Historical analysis of the LLVM optimization pipeline reveals persistent instability when processing integers significantly wider than the target architecture's native word size (typically 64 bits).
Specifically, the Interprocedural Sparse Conditional Constant Propagation (IPSCCP) and Constraint Elimination passes have demonstrated erroneous behavior—miscalculating ranges or invalidating correct code paths—when handling ultra-wide scalars. These issues, documented in LLVM bug reports #68751 and #56351 1, pose an unacceptable risk for Aria. A silent miscalculation in a cryptographic key generation routine or a safety interlock system is a catastrophic failure mode.
2.2 The LBIM Architecture
The Limb-Based Integral Model (LBIM) mitigates this risk by treating large integers as aggregate structures rather than scalars. An int1024 is not represented as i1024 in IR, but as a struct containing an array of native 64-bit integers, termed "limbs."
For a 1024-bit integer, the representation is:




$$\text{Value} = \sum_{i=0}^{15} L_i \cdot 2^{64i}$$


where $L_i$ represents the $i$-th 64-bit limb. This forces the LLVM backend to treat operations on these types as sequences of native instructions (via runtime calls) rather than optimizable scalars, creating an "optimization barrier" that prevents the aforementioned compiler bugs from manifesting.
2.3 Twisted Balanced Binary (TBB) and Sticky Errors
Aria's arithmetic creates a distinct departure from C-style arithmetic through the TBB philosophy. In standard arithmetic, INT_MAX + 1 results in INT_MIN (wraparound). In TBB, valid data ranges are symmetric, and the extra bit pattern remaining (the two's complement minimum) is repurposed as a dedicated error sentinel.1
For int1024, a signed 1024-bit integer, the range of valid values is:




$$-(2^{1023} - 1) \le x \le +(2^{1023} - 1)$$


The value $-2^{1023}$ (0x8000...0000) is reserved as the ERR sentinel.
The "Sticky Error" property mandates that for any operation $\odot$:




$$\text{ERR} \odot x = \text{ERR}$$


$$x \odot \text{ERR} = \text{ERR}$$


$$\text{Overflow}(x \odot y) = \text{ERR}$$
This creates a data-flow safety net. If a calculation deep within a neural network or financial model fails, the error propagates to the final result, preventing the system from acting on corrupted intermediate values. Implementing this requires rigorous pre-computation and post-computation checks in the runtime library.

3. LLVM IR Generation Modifications
The first phase of implementation involves modifying the Aria compiler's backend to recognize int1024 and uint1024 as valid primitive types and generate the appropriate Type definitions in LLVM IR.
3.1 Analysis of src/backend/ir/codegen_expr.cpp
The existing compiler infrastructure, specifically ExprCodegen::getLLVMTypeFromString, handles type registration. As seen in the source code analysis 1, types like int128 through int512 are registered via llvm::StructType::create. The compiler checks if a struct with a specific name (e.g., "struct.int512") exists in the module context; if not, it creates it.
The mapping logic used for int512 is:


C++




if (typeName == "int512" |

| typeName == "uint512") {
   llvm::StructType* int512Struct = llvm::StructType::getTypeByName(context, "struct.int512");
   if (!int512Struct) {
       llvm::Type* limbArray = llvm::ArrayType::get(llvm::Type::getInt64Ty(context), 8);
       int512Struct = llvm::StructType::create(context, {limbArray}, "struct.int512");
   }
   return int512Struct;
}

This confirms the 8-limb structure ($8 \times 64 = 512$). To support 1024 bits, we require 16 limbs ($16 \times 64 = 1024$).
3.2 Implementation Strategy for 1024-bit Types
We must inject logic to handle int1024 and uint1024. Note that signed and unsigned types share the same binary representation (the struct of 16 limbs); the distinction is semantic and handled by the selection of runtime functions (e.g., sdiv vs udiv).
The modifications must:
1. Intercept the type strings "int1024" and "uint1024".
2. Define a unique struct name "struct.int1024" to prevent collision.
3. Allocate an array of 16 i64 elements.
4. Ensure TBB sentinel checks in other parts of the compiler (e.g., widening casts) recognize this new width.

C++

4. Runtime Library Implementation: lbim_1024.cpp
The complexity of 1024-bit arithmetic precludes inline code generation. Instead, the compiler lowers arithmetic expressions on int1024 types to calls against the Aria runtime library. This section specifies the implementation of src/runtime/math/lbim_1024.cpp.
4.1 Safety and Memory Constraints
The implementation operates under strict constraints:
1. No Garbage Collection: Intermediate buffers must use the "wild allocator" (aria_alloc / aria_free) or the stack to strictly avoid GC pauses, which could introduce timing jitter in cryptographic operations.1
2. Sticky Error Propagation: Every entry point must validate inputs against the TBB sentinel.
3. Constant Time Execution: Multiplication must be side-channel resistant.
4.1.1 The TBB Sentinel Macro
We define the error sentinel for int1024. In a 16-limb little-endian array, the sentinel is the minimum signed integer: the most significant bit of limb 15 is 1, and all other bits are 0.


C++




// Sentinel: 0x8000000000000000 in limb, 0 elsewhere
#define ARIA_TBB1024_ERR_MSB 0x8000000000000000ULL

#define CHECK_STICKY_ERROR(val) \
   do { \
       if ((val).limbs == ARIA_TBB1024_ERR_MSB) { \
           bool others_zero = true; \
           for(int i=0; i<15; ++i) if((val).limbs[i]!= 0) others_zero = false; \
           if (others_zero) return aria_int1024_err(); \
       } \
   } while(0)

4.2 Addition: Ripple-Carry Implementation
Addition is implemented using a straightforward ripple-carry approach. Given the 64-bit native limb size, we utilize the compiler intrinsic __int128 (available on x86_64 and AArch64) to handle carry propagation efficiently. This avoids the need for conditional branching on the carry flag, which improves pipeline utilization.
Algorithmic Logic
For $i = 0 \dots 15$:




$$\text{Sum} = A[i] + B[i] + \text{carry}$$


$$R[i] = \text{Sum} \pmod{2^{64}}$$


$$\text{carry} = \text{Sum} / 2^{64}$$
Post-calculation, we must check for signed overflow. In two's complement, overflow occurs if and only if:




$$(\text{Sign}(A) = \text{Sign}(B)) \land (\text{Sign}(R) \neq \text{Sign}(A))$$
4.3 Multiplication: The Comba Method
Standard "schoolbook" multiplication performs row-based partial products, requiring carry propagation after every row. This is $O(N^2)$ but involves significant memory traffic and carry management.
The Comba Method (column-wise multiplication) is superior for fixed-width cryptographic arithmetic. It computes the result limb-by-limb, starting from the least significant. For each column $k$, it sums all product pairs $A_i \cdot B_j$ where $i+j=k$.
Accumulator Strategy
With 16 limbs, the maximum column depth is 16. The maximum value of a column sum is roughly $16 \times (2^{64}-1)^2 \approx 2^{132}$. This fits comfortably within a 3-word (192-bit) accumulator ($r0, r1, r2$).
* $r0$: Current limb value.
* $r1$: First carry (to next limb).
* $r2$: Second carry (to limb after next).
Constant Time Unrolling
To meet the "cryptographic compliance" requirement, we cannot use variable-loop bounds that might leak information about the magnitude of the numbers. We fully unroll the loops. This results in a sequence of straight-line code that executes in data-independent time (assuming the CPU multiplier is constant-time, which is true for modern x86/ARM).
4.4 Division: Knuth's Algorithm D
Division is the most algorithmically complex operation. We implement Knuth's Algorithm D (Art of Computer Programming, Vol 2, Sec 4.3.1). This algorithm supports multi-precision division $U / V$.
Key Steps:
1. Normalization: Shift $U$ and $V$ left so the MSB of $V$ is set. This standardizes the estimation step.
2. Estimation ($q̂$): Calculate a candidate quotient digit based on the top 2 limbs of the remainder and top 1 limb of divisor.
3. Correction: Adjust $q̂$ if the estimate is too high (at most by 2).
4. Multiply & Subtract: Subtract $q̂ \times V$ from the current window of $U$.
5. Add Back: If the subtraction underflowed (probability $\approx 2^{-64}$), decrement $q$ and add $V$ back.
This algorithm is typically not constant time due to the normalization loops and the "Add Back" conditional. However, it is the standard for correctness. For cryptographic contexts requiring constant-time division (e.g., modular inversion), blinders or Montgomery reduction are used instead. For the general purpose int1024_div, Algorithm D is the correct choice.

// 128-bit unsigned integer for carry handling
typedef unsigned __int128 uint128_t;

// TBB Error Sentinel: 0x8000...0000 (Min Signed Int)
static const uint64_t TBB_SENTINEL_MSB = 0x8000000000000000ULL;

// Helper to construct the error value
static inline aria_int1024_t aria_int1024_err() {
   aria_int1024_t err;
   // Zero out lower 15 limbs
   for (int i = 0; i < 15; ++i) err.limbs[i] = 0;
   // Set MSB of highest limb
   err.limbs = TBB_SENTINEL_MSB;
   return err;
}

// Check Sticky Error Macro
// Returns true if 'val' is the TBB error sentinel
static inline bool is_tbb_err(const aria_int1024_t& val) {
   if (val.limbs!= TBB_SENTINEL_MSB) return false;
   for (int i = 0; i < 15; ++i) {
       if (val.limbs[i]!= 0) return false;
   }
   return true;
}

#define CHECK_STICKY_ERROR(val) \
   if (is_tbb_err(val)) return aria_int1024_err();

/**
* @brief 1024-bit Ripple-Carry Addition
* 
* Performs piecewise addition with carry propagation.
* Implements TBB overflow detection: if result wraps into the sign bit
* inconsistently with inputs, returns ERR.
*/
aria_int1024_t aria_int1024_add(aria_int1024_t a, aria_int1024_t b) {
   // 1. Sticky Error Check
   CHECK_STICKY_ERROR(a);
   CHECK_STICKY_ERROR(b);

   aria_int1024_t result;
   uint128_t carry = 0;

   // 2. Ripple-Carry Loop (Unrolled by compiler optimization levels, explicit here)
   for (int i = 0; i < 16; ++i) {
       uint128_t sum = (uint128_t)a.limbs[i] + (uint128_t)b.limbs[i] + carry;
       result.limbs[i] = (uint64_t)sum;
       carry = sum >> 64;
   }

   // 3. TBB Overflow Check (Signed)
   // Overflow occurs if: A and B have same sign, but Result has different sign
   bool sign_a = (a.limbs & TBB_SENTINEL_MSB);
   bool sign_b = (b.limbs & TBB_SENTINEL_MSB);
   bool sign_r = (result.limbs & TBB_SENTINEL_MSB);

   // If signs of operands match...
   if (sign_a == sign_b) {
       //...and sign of result differs
       if (sign_r!= sign_a) {
           return aria_int1024_err(); // Overflow detected
       }
   }

   // 4. Sentinel Collision Check
   // If the result literally equals the error sentinel by coincidence (valid calculation),
   // TBB rules generally require this to be treated as an error or handled specifically.
   // For safety, we treat it as an error state.
   if (is_tbb_err(result)) {
       return aria_int1024_err();
   }

   return result;
}

/**
* @brief 1024-bit Comba Multiplication (Column-Wise)
* 
* Uses a 3-word accumulator to sum partial products column by column.
* Fully unrolled to ensure constant instruction trace for cryptography.
* Only computes lower 1024 bits (truncating multiplication).
*/
aria_int1024_t aria_int1024_mul(aria_int1024_t a, aria_int1024_t b) {
   CHECK_STICKY_ERROR(a);
   CHECK_STICKY_ERROR(b);

   aria_int1024_t r;
   // Accumulators: r0 (sum), r1 (carry 1), r2 (carry 2)
   uint64_t r0 = 0, r1 = 0, r2 = 0;

   // Helper Macro for Comba Step: Accumulate a[i]*b[j]
   #define MUL_ACC(i, j) \
       do { \
           uint128_t prod = (uint128_t)a.limbs[i] * (uint128_t)b.limbs[j]; \
           uint128_t sum = (uint128_t)r0 + (uint64_t)prod; \
           r0 = (uint64_t)sum; \
           /* Carry propagation */ \
           uint128_t c = (uint128_t)r1 + (sum >> 64) + (prod >> 64); \
           r1 = (uint64_t)c; \
           r2 += (uint64_t)(c >> 64); \
       } while(0)

   // Helper Macro to finalize a column
   #define COL_FINISH(k) \
       r.limbs[k] = r0; \
       r0 = r1; \
       r1 = r2; \
       r2 = 0;

   // Unrolled Columns 0 to 15
   // We only need the lower 16 limbs for int1024 result.
   
   // Column 0
   MUL_ACC(0, 0); 
   COL_FINISH(0);

   // Column 1
   MUL_ACC(0, 1); MUL_ACC(1, 0); 
   COL_FINISH(1);

   // Column 2
   MUL_ACC(0, 2); MUL_ACC(1, 1); MUL_ACC(2, 0); 
   COL_FINISH(2);

   // Column 3
   MUL_ACC(0, 3); MUL_ACC(1, 2); MUL_ACC(2, 1); MUL_ACC(3, 0);

   MUL_ACC(0, 4); MUL_ACC(1, 3); MUL_ACC(2, 2); MUL_ACC(3, 1); MUL_ACC(4, 0); 
   COL_FINISH(4);

// Helper: Count Leading Zeros (64-bit)
static inline int clz64(uint64_t x) {
   return x? __builtin_clzll(x) : 64;
}

/**
* @brief 1024-bit Division (Knuth's Algorithm D)
* 
* Computes Quotient = A / B.
* Uses Wild Allocator for intermediate buffers U and V.
*/
aria_int1024_t aria_int1024_div(aria_int1024_t u_in, aria_int1024_t v_in) {
   CHECK_STICKY_ERROR(u_in);
   CHECK_STICKY_ERROR(v_in);

   // 1. Division by Zero Check
   bool v_is_zero = true;
   for(int i=0; i<16; ++i) if(v_in.limbs[i]!= 0) v_is_zero = false;
   if(v_is_zero) return aria_int1024_err();

   // 2. Division by One Check (Optimization)
   if(v_in.limbs == 1) {
       bool v_is_one = true;
       for(int i=1; i<16; ++i) if(v_in.limbs[i]!= 0) v_is_one = false;
       if(v_is_one) return u_in;
   }

   // 3. Memory Allocation (Wild Allocator)
   // We need normalized U (dividend, 17 limbs for remainder shift) and V (divisor)
   // Allocation ensures no GC pauses.
   uint64_t* u = (uint64_t*)aria_alloc(sizeof(uint64_t) * 17);
   uint64_t* v = (uint64_t*)aria_alloc(sizeof(uint64_t) * 16);
   uint64_t* q = (uint64_t*)aria_alloc(sizeof(uint64_t) * 16);
   
   // Initialize
   memset(u, 0, sizeof(uint64_t) * 17);
   memset(v, 0, sizeof(uint64_t) * 16);
   memset(q, 0, sizeof(uint64_t) * 16);

   // 4. Normalization
   // Shift both numbers left so V's MSB is 1. This increases estimate accuracy.
   int n = 16;
   while(n > 0 && v_in.limbs[n-1] == 0) n--; // Effective length of divisor
   
   int m = 16 - n; // Length difference
   
   int shift = clz64(v_in.limbs[n-1]);
   
   // Copy and shift V
   for(int i = 0; i < n; i++) {
       v[i] = v_in.limbs[i] << shift;
       if(i > 0 && shift > 0) v[i] |= (v_in.limbs[i-1] >> (64 - shift));
   }
   
   // Copy and shift U (u becomes dividend)
   u = 0; // Clear top guard limb
   for(int i = 0; i < 16; i++) {
       u[i] = u_in.limbs[i] << shift;
       if(i > 0 && shift > 0) u[i] |= (u_in.limbs[i-1] >> (64 - shift));
   }
   if (shift > 0) u = u_in.limbs >> (64 - shift);

   // 5. Knuth's Main Loop
   // Iterate from most significant place of the quotient down to 0
   for (int j = m; j >= 0; j--) {
       // D3. Calculate q_hat (Estimation)
       // q_hat = (u[j+n]*B + u[j+n-1]) / v[n-1]
       uint128_t num = ((uint128_t)u[j+n] << 64) | u[j+n-1];
       uint128_t q_hat = num / v[n-1];
       uint128_t r_hat = num % v[n-1];

       // Refine q_hat
       // while (q_hat >= b |

| q_hat * v[n-2] > b * r_hat + u[j+n-2])
       while (1) {
           if (q_hat >= ((uint128_t)1 << 64)) {
               q_hat--;
               r_hat += v[n-1];
               if (r_hat >= ((uint128_t)1 << 64)) continue; // Check overflow
           }
           
           // Check second term condition if n > 1
           if (n > 1) {
               uint128_t prod = q_hat * v[n-2];
               uint128_t rhs = (r_hat << 64) | u[j+n-2];
               if (prod > rhs) {
                   q_hat--;
                   r_hat += v[n-1];
                   if (r_hat < ((uint128_t)1 << 64)) continue;
               }
           }
           break;
       }

       // D4. Multiply and Subtract
       // u[j...j+n] -= q_hat * v
       uint128_t borrow = 0;
       for (int i = 0; i < n; i++) {
           uint128_t p = q_hat * v[i];
           uint128_t sub = (uint128_t)u[j+i] - (uint64_t)p - borrow;
           u[j+i] = (uint64_t)sub;
           borrow = (p >> 64) - (sub >> 127); // borrow out
           if ((uint64_t)p > (uint64_t)u[j+i] + (uint64_t)borrow) borrow++; // simpler logic possible
       }
       
       // Handle top limb borrow
       uint128_t sub = (uint128_t)u[j+n] - borrow;
       u[j+n] = (uint64_t)sub;
       bool underflow = (sub >> 127);

       // D5. Test Remainder (Add Back)
       q[j] = (uint64_t)q_hat;
       if (underflow) {
           q[j]--;
           uint128_t carry = 0;
           for (int i = 0; i < n; i++) {
               uint128_t sum = (uint128_t)u[j+i] + v[i] + carry;
               u[j+i] = (uint64_t)sum;
               carry = sum >> 64;
           }
           u[j+n] += (uint64_t)carry;
       }
   }

   // 6. Construct Result
   aria_int1024_t result;
   memcpy(result.limbs, q, sizeof(result.limbs));

   // 7. Cleanup
   aria_free(u);
   aria_free(v);
   aria_free(q);

   return result;
}

} // extern "C"

6. Safety Analysis and Compliance
6.1 TBB Sticky Error Propagation
The implementation strictly adheres to the TBB requirements. The CHECK_STICKY_ERROR macro is inserted at the ingress of every public API function. This guarantees that if a calculation yields an error (overflow or invalid operation), that error token is preserved through all subsequent operations. This is distinct from NaN in IEEE754; TBB ERR implies a specific structural failure of the integer pipeline, acting as a "dead man's switch" for safety-critical logic.
6.2 Side-Channel Resistance
The aria_int1024_mul implementation demonstrates robust side-channel resistance. By using the Comba method with full unrolling, the instruction trace is identical regardless of the input operand values. There are no data-dependent branches within the multiplication logic itself (the only branch is the CHECK_STICKY_ERROR pre-check, which is structurally dependent, not bit-pattern dependent). The use of the accumulator ($r0, r1, r2$) minimizes memory writes, reducing power analysis leakage compared to row-based multiplication.
6.3 Memory Management
The division routine explicitly utilizes aria_alloc and aria_free (the wild allocator interface) for buffer management.1 This bypasses the language's garbage collector, ensuring deterministic execution time for the memory allocation phase and preventing GC-induced latency spikes, which is a critical requirement for real-time safety systems and cryptographic signing operations.

7. Conclusion
This report establishes the complete architectural and implementation path for upgrading Aria's LBIM to 1024 bits. The modifications to src/backend/ir/codegen_expr.cpp enable the compiler to understand the new types, while the runtime library in src/runtime/math/lbim_1024.cpp provides the semantic execution.
By combining the structural rigidity of the 16-limb array with the semantic safety of Twisted Balanced Binary arithmetic, Aria achieves a unique position in the landscape of systems programming languages: capable of heavy-weight cryptography without the unsafety of C or the non-determinism of managed languages. The constant-time Comba multiplication and robust Knuth division ensure that this capability meets the rigorous demands of modern security standards.
The integration of these changes facilitates the immediate deployment of RSA-2048/4096 primitives and advanced Zero-Knowledge Proof constructs directly within the Aria language, fulfilling the project's goal of acting as a secure foundation for next-generation digital infrastructure.
