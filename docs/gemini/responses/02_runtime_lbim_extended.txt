1. Executive Summary: The Moral and Technical Imperative
This document constitutes a comprehensive, line-by-line safety audit of the Aria programming language runtime environment, specifically targeting the Limb-Based Integral Model (LBIM) extended precision operations (int1024, int2048, int4096). These mathematical primitives form the computational backbone of the physics-based Artificial General Intelligence (AGI) currently under development. The deployment context—robotic companions for neurodivergent children, patients in extended hospital care, and other vulnerable populations—mandates a zero-tolerance policy for software defects. In this domain, a runtime error is not merely an inconvenience; it is a vector for physical harm.
The directive for this audit is unambiguous: treat the code as a life-or-death scenario. The "Teacher" system capability, designed to customize educational interactions, relies on long-running state accumulation where numerical stability is paramount. A drift in variable state, caused by an unhandled integer overflow or a "healed" error sentinel, could lead to a robotic arm miscalculating force feedback limits, potentially resulting in injury. Consequently, this audit prioritizes explicitness, predictability, stability, security, and performance over any form of developer convenience.
Our analysis of the target files, particularly src/runtime/math/lbim_extended.cpp 1, has uncovered three (3) CRITICAL safety violations and one (1) MAJOR inconsistency that fundamentally undermine the safety guarantees Aria aims to provide. While the base 1024-bit implementation (lbim.cpp) serves as a robust reference standard, the extended 2048-bit and 4096-bit implementations exhibit dangerous regressions. Specifically, the extended integer operations fail to detect input error sentinels, allowing "poisoned" data to be processed as valid integers. Furthermore, division-by-zero operations in these extended types return a silent default value of zero rather than the mandatory Error Sentinel, creating a hazardous "false negative" state where a system failure masquerades as a stable physical state (e.g., zero velocity).
This report details these findings across five specific audit items, offering a rigorous deconstruction of the failure modes, the physics-based consequences of these bugs, and the necessary remediation to achieve Safety Integrity Level 4 (SIL-4) compliance.

2. Architectural Context and Threat Model
2.1 The Physics of Accumulating Error
In a physics-based AGI, the state of the world is maintained through continuous integration of forces, velocities, and positions. Aria utilizes fixed-point (fix256) and large-integer arithmetic (int1024 through int4096) to guarantee determinism—ensuring that the same inputs always produce the exact same outputs, bit-for-bit, regardless of the hardware platform.1 This determinism is essential for safety certification and for the "replayability" of failure states.
The Limb-Based Integral Model (LBIM) was introduced to bypass known bugs in the LLVM optimizer (specifically IPSCCP and ConstraintElimination passes) that affect native integers larger than 128 bits.1 LBIM splits massive integers into arrays of 64-bit "limbs" (e.g., 32 limbs for int2048, 64 limbs for int4096). Safety in this context relies on two non-negotiable invariants:
1. Value Integrity: The number stored in the limbs must accurately represent the physical quantity.
2. Error Propagation: If any part of a calculation fails (e.g., sensor disconnect, arithmetic overflow, division by zero), the result must be an explicit Error Sentinel (ERR). This sentinel must be "sticky"—if an ERR enters a calculation, the result must be ERR.
2.2 The Threat of "Sentinel Healing"
"Sentinel Healing" is a specific, catastrophic vulnerability in custom number systems. Aria's Twisted Balanced Binary (TBB) and extended integer specifications use the most significant bit (MSB) or specific bit patterns (like INT_MIN) to denote an error state.1
If a mathematical operation—such as a bit shift or a modular reduction—treats the sentinel as raw data rather than a status flag, it may alter the sentinel bit. For example, shifting a sentinel right might clear the MSB. The resulting bit pattern is no longer the Error Sentinel; it is now a valid, often massive, integer.
* Scenario: A velocity variable becomes ERR due to a sensor fault.
* Operation: The control loop calculates target_position = current_position + (velocity >> 4).
* Result: The ERR value in velocity is shifted. The error flag is lost. The resulting bits are interpreted as a valid integer (e.g., $2^{2044}$).
* Consequence: The robot arm attempts to move to a position light-years away instantly. The physical actuators saturate, causing a violent, unchecked movement.
2.3 SIL-4 Compliance Requirements
For this audit, we apply strict criteria derived from IEC 61508 (Functional Safety) and the specific mandates of the user query:
1. Deterministic Failure: Operations on invalid inputs must produce deterministic error outputs, never undefined behavior or silent defaults.
2. No Silent Defaults: A result of 0 is mathematically distinct from "Calculation Failed." Returning 0 on error is strictly prohibited.
3. Boundary Safety: All array accesses must be rigorously checked against the limb count (N).
4. Sentinel Persistency: Once a variable is tainted with ERR, no sequence of operations can untaint it unless explicit error-handling logic is invoked.

3. Detailed Audit: Item 1 - int1024_div() Function
Target Function: aria_lbim_udiv1024 / aria_lbim_sdiv1024
File: src/runtime/math/lbim.cpp 1
Status: ✅ COMPLIANT
3.1 Code Analysis
The int1024 implementation serves as the control group for this audit. It is located in the base lbim.cpp file and demonstrates a high degree of safety awareness. The implementation uses a template-based approach (div_mod), which ensures that the logic is consistent with the limb count.
The critical safety features observed in the source code are:


C++




extern "C" aria_int1024_t aria_lbim_udiv1024(aria_int1024_t dividend, aria_int1024_t divisor) {
   aria_int1024_t quotient, remainder;
   
   // TBB safety: check for ERR in inputs
   if (is_err_sentinel(dividend.limbs) |

| is_err_sentinel(divisor.limbs)) {
       set_err_sentinel(quotient.limbs);
       return quotient;
   }
   
   // Division by zero returns ERR
   if (is_zero(divisor.limbs)) {
       set_err_sentinel(quotient.limbs);
       return quotient;
   }
   
   div_mod(quotient.limbs, remainder.limbs, dividend.limbs, divisor.limbs);
   return quotient;
}

3.2 Safety Verification
Division by Zero Handling:
The code explicitly checks if (is_zero(divisor.limbs)). Upon detection, it does not crash, nor does it return zero. Instead, it calls set_err_sentinel(quotient.limbs). This aligns perfectly with the requirement that "Division by zero MUST return error sentinel." This behavior ensures that a mathematical singularity in the physics engine (e.g., dividing by a zero time delta) propagates as an error state rather than an infinite force or a zero result.
Sentinel Propagation:
The function includes a pre-computation check: `is_err_sentinel(dividend.limbs) |
| is_err_sentinel(divisor.limbs)`. This effectively implements the "sticky" error requirement. If either operand is already in an error state (perhaps from a previous overflow), the division operation is aborted, and the error state is propagated to the result. This prevents the "garbage in, garbage out" phenomenon where invalid data is processed as valid.
Precision and Algorithm:
The underlying div_mod template implements a binary long division algorithm. While this is computationally more expensive than hardware division, it is robust and predictable. For int1024, precision loss is not an issue in the floating-point sense; integer division is exact with a remainder. The implementation correctly separates the quotient and remainder, ensuring no data loss during the operation.
3.3 Conclusion for Item 1
The int1024 division functions are safe for deployment. They exhibit the defensive programming characteristics required for a SIL-4 system. The explicit checks for zero divisors and input sentinels act as a firewall against data corruption.

4. Detailed Audit: Item 2 - int2048_mod() Function
Target Function: aria_lbim_smod2048 / aria_lbim_umod2048
File: src/runtime/math/lbim_extended.cpp 1
Status: ❌ CRITICAL BUG FOUND
4.1 Code Analysis
The audit of the 2048-bit implementation reveals a significant and dangerous divergence from the 1024-bit standard. The code for int2048 operations is located in lbim_extended.cpp and utilizes a different internal helper function, divmod2048_unsigned.
Source Code Extraction:


C++




// Internal Unsigned Division / Modulo
static void divmod2048_unsigned(aria_int2048_t dividend, aria_int2048_t divisor, aria_int2048_t* quot, aria_int2048_t* rem) {
   // 1. Handle Division by Zero
   if (is_zero(&divisor)) {
       // ARIA SAFETY: Division by zero must not crash the AGI.
       // It returns 0. The caller (higher level Aria code) is responsible 
       // for checking preconditions or handling the TBB Error Sentinel if applicable.
       
       if (quot) std::memset(quot, 0, sizeof(*quot));
       if (rem) std::memset(rem, 0, sizeof(*rem));
       return;
   }
   //... actual division logic...
}

extern "C" aria_int2048_t aria_lbim_umod2048(const aria_int2048_t* dividend, const aria_int2048_t* divisor) {
   aria_int2048_t r;
   divmod2048_unsigned(*dividend, *divisor, nullptr, &r);
   return r;
}

4.2 Defect 1: The "Silent Default" Violation
The comment within divmod2048_unsigned is a damning admission of a safety violation: "It returns 0. The caller... is responsible for checking preconditions". This design choice fundamentally violates the "Fail Hard" requirement.
In a SIL-4 context, relying on the caller to check preconditions is a known anti-pattern that leads to latent defects. If the compiler generates a call to aria_lbim_umod2048 without emitting a pre-check instruction (which is likely, as the runtime is expected to be safe), a division by zero will result in a return value of 0.
Physics Impact Scenario:
Consider a robot arm calculating the required torque to lift an object based on a lever arm length. If the lever arm length approaches zero (a singularity), the calculation involves a division or modulo.
* Expected Behavior: Result is ERR. The motor controller sees ERR, engages emergency brakes, and alerts the Teacher system.
* Actual Behavior: Result is 0. The motor controller sees 0 torque required. It relaxes the motors. The object falls, potentially injuring a child.
4.3 Defect 2: Missing Sentinel Check
The wrapper function aria_lbim_umod2048 passes the operands directly to the helper function without checking if they are valid. Unlike int1024, there is no is_err_sentinel check at the entry point.
The Mechanism of Failure:
1. Assume dividend is the Error Sentinel (INT_MIN for 2048 bits). This is represented as the MSB set to 1, and all other bits 0.
2. The function proceeds to divmod2048_unsigned.
3. The sentinel value is treated as a valid, extremely large negative number (in signed context) or large positive number (in unsigned context).
4. The modulo operation is performed on this "error" value.
5. The result is a mathematical artifact of the error bit pattern, not a meaningful physical quantity.
This is a classic "Sentinel Healing" vulnerability. The error state is scrubbed from the system, and the "poisoned" data is laundered into a valid integer. This makes backtracking the source of the error impossible and allows the system to continue operating in an invalid state.
4.4 Conclusion for Item 2
The int2048_mod implementation is unsafe. It fails to propagate error sentinels and returns a silent default on zero division. This represents a critical risk to the stability of the AGI.

5. Detailed Audit: Item 3 - int4096_shl() and int4096_shr()
Target Functions: aria_lbim_shl4096, aria_lbim_lshr4096, aria_lbim_ashr4096
File: src/runtime/math/lbim_extended.cpp 1
Status: ❌ CRITICAL BUG FOUND
5.1 Code Analysis
Bitwise operations are fundamental for scaling fixed-point numbers and handling binary flags. The audit of the 4096-bit shift operations reveals a critical oversight in how these functions handle the specific bit patterns reserved for error sentinels.
Source Code Extraction:


C++




// Left Shift
extern "C" aria_int4096_t aria_lbim_shl4096(aria_int4096_t a, uint32_t shift) {
   aria_int4096_t result; memset(&result, 0, sizeof(result));
   
   if (shift >= 4096) return result; // Returns 0 (Valid)
   
   uint32_t limb_shift = shift / 64;
   uint32_t bit_shift = shift % 64;
   
   for (int i = N4096 - 1; i >= 0; --i) {
       //... shifting logic...
   }
   return result;
}

5.2 The "Sentinel Healing" Mechanism
The shl4096 function treats the input a purely as a sequence of 4096 bits. It does not check if a represents the Error Sentinel.
Scenario A: Left Shift (Healing via Erasure)
* Input: a is the Error Sentinel. In Aria's signed TBB/LBIM specification, this is INT_MIN ($1000...000_2$). The MSB is 1, all others are 0.
* Operation: aria_lbim_shl4096(a, 1) (Left shift by 1).
* Execution: The loop shifts bits left. The MSB (the 1) is shifted out of the 4096-bit window.
* Result: The result is all zeros ($0000...000_2$).
* Implication: An error state has been transformed into the scalar value 0. The system now believes it has a valid result.
Scenario B: Logical Right Shift (Healing via Displacement)
* Input: a is the Error Sentinel ($1000...000_2$).
* Operation: aria_lbim_lshr4096(a, 1) (Logical right shift by 1).
* Execution: The logical right shift inserts a 0 at the MSB. The 1 from the MSB is moved to the second most significant bit.
* Result: $0100...000_2$. This represents a massive positive integer ($2^{4094}$).
* Implication: An error state has been transformed into a massive valid value. If this value represents a target force, the robot will attempt to apply maximum force, believing it is a valid command.
5.3 Violation of Safety Requirements
This behavior explicitly violates the requirement: "Sentinel values must be inescapable (once error, always error)." By treating the sentinel bit pattern as manipulatable data, the shift operations provide a mechanism for errors to escape detection and manifest as valid, potentially dangerous control signals.
5.4 Conclusion for Item 3
The int4096 shift operations are critically unsafe. They must be patched to check for the sentinel pattern before performing any bitwise manipulation.

7. Detailed Audit: Item 5 - Limb Array Boundary Checks
Target: Array access patterns in lbim_extended.cpp
Status: ✅ COMPLIANT
7.1 Verification Methodology
We examined the loop structures and array indexing logic to ensure that no buffer overruns occur, even when processing 4096-bit (64-limb) integers.
Struct Definitions 1:


C++




#define ARIA_INT2048_LIMBS 32
#define ARIA_INT4096_LIMBS 64

typedef struct { uint64_t limbs; } aria_int2048_t;
typedef struct { uint64_t limbs; } aria_int4096_t;

7.2 Loop Analysis
The implementation generally uses constant bounds defined by the macros N or N4096.
Example: 4096-bit Shift:


C++




for (int i = N4096 - 1; i >= 0; --i) {
   int src_idx = i - limb_shift;
   //...
}

Here, i ranges from 63 down to 0. This is strictly within the bounds of the 64-element array.
Example: 2048-bit Multiplication:


C++




for (int i = 0; i < N; ++i) {
   for (int j = 0; j < N - i; ++j) {
       // Access result.limbs[i+j]
   }
}

The inner loop condition j < N - i guarantees that i + j < N. The maximum index accessed is N-1 (31). This proves that the schoolbook multiplication algorithm effectively bounds-checks itself through loop conditions.
7.3 Conclusion for Item 5
The array handling logic is compliant. We found no evidence of off-by-one errors or potential buffer overflows in the mathematical logic itself. The use of fixed-size structs and compile-time constants for loop bounds contributes to this stability.

8. Systemic Impact & Physics Scenarios
The identified bugs are not abstract software defects; they have concrete, physical implications for the robotic assistants controlled by this code.
8.1 Scenario: The "Zero Force" Fallacy
Trigger: A "division by zero" occurs in a load-balancing routine (e.g., calculating weight distribution across legs where one leg loses contact).
Bug: aria_lbim_udiv2048 returns 0 instead of ERR.
Physical Consequence: The system calculates that the load on a specific actuator is 0 Nm. The PID controller reduces power to that actuator to conserve energy. In reality, the actuator is under immense load. The leg collapses. If a child is near the robot, this uncontrolled fall presents a severe impact hazard.
8.2 Scenario: The "Hyperspeed" Shift
Trigger: A position sensor faults and returns ERR (INT_MIN).
Bug: A control loop right-shifts this value to smooth the data (val >> 2). aria_lbim_lshr4096 "heals" the sentinel.
Physical Consequence: The faulty sensor reading of ERR becomes a valid integer representing a coordinate far outside the safe working volume (e.g., $+2^{4090}$ mm). The robot's path planning algorithm sees a valid target and attempts to move the arm there at maximum velocity to compensate for the massive distance error. This results in a "hard-over" actuator command, causing the arm to lash out violently.

9. Remediation Strategy
To achieve SIL-4 compliance and ensure the safety of the target population, the following corrective actions must be taken immediately.
9.1 Immediate Code Patches
The lbim_extended.cpp file must be refactored to align with the safety patterns established in lbim.cpp.
1. Fix Sentinel Checks:
Insert the following check at the beginning of EVERY public function (add, sub, mul, div, mod, shl, shr) for both 2048 and 4096 types:


C++




if (is_err_sentinel_2048(a) |

| is_err_sentinel_2048(b)) {
   return make_err_sentinel_2048();
}

2. Fix Division by Zero:
Modify divmod2048_unsigned and divmod4096_unsigned to return the error sentinel, not zero.


C++




// CORRECTED LOGIC
if (is_zero(&divisor)) {
   // ARIA SAFETY: Division by zero is a critical fault.
   // Return ERR to propagate the failure state.
   if (quot) *quot = make_err_sentinel_2048();
   if (rem) *rem = make_err_sentinel_2048();
   return;
}

3. Fix Shift Operations:
Add a specific check for the sentinel in shift operations to prevent "healing."


C++




extern "C" aria_int4096_t aria_lbim_shl4096(aria_int4096_t a, uint32_t shift) {
   if (is_err_sentinel_4096(&a)) return a; // Sticky Error
   //... continue with shift...
}

9.2 Recommendation: Unified Template Architecture
The divergence between int1024 (safe) and extended types (unsafe) is a result of code duplication. We strongly recommend refactoring the runtime to use a single C++ template LBIM<NumLimbs> for all widths. This ensures that safety logic is written once and applied automatically to all integer sizes, eliminating the risk of human error in future extensions.
9.3 Conclusion
The current state of src/runtime/math/lbim_extended.cpp presents an unacceptable risk to life safety. The "Silent Default" and "Sentinel Healing" bugs fundamentally undermine the Aria language's safety guarantees. While the 1024-bit implementation is sound, the 2048-bit and 4096-bit types must be quarantined and strictly prohibited from use in physical control loops until the patches detailed above are applied and verified.
