Architectural Specification for the Aria Module System: Loader, Linker, and Symbol Resolution
1. Executive Summary
The Aria programming language has achieved a significant level of maturity in its syntactical frontend and LLVM-based backend, yet it faces a critical infrastructural disconnect: the absence of a functional module loading subsystem. While the parser correctly identifies use statements and the Abstract Syntax Tree (AST) retains this topological information, the semantic analysis and code generation phases currently treat translation units in isolation. This limitations restricts Aria programs to single-file scripts, rendering the "batteries-included" standard library inaccessible and hindering the development of complex, multi-file applications.
This report defines the comprehensive architectural specification for the Aria Module Loader and Symbol Importer. It addresses the "missing glue" required to transform the UseStmt AST node from a passive data container into an active directive that drives the compilation lifecycle. The proposed architecture introduces a deterministic algorithm for logical-to-physical resolution, mapping module identifiers to filesystem artifacts. It details a semantic graph construction mechanism to load, parse, and cache modules while rigorously detecting circularities. Furthermore, it establishes a scoping model for symbol import and visibility, ensuring strict encapsulation, and streamlines the Foreign Function Interface (FFI) integration to automate LLVM declarations for extern functions. By implementing this specification, the Aria compiler will transition from a single-file translator to a build-orchestrating compiler, capable of supporting the ecosystem scale expected of a modern systems language.
2. Theoretical Framework and Design Philosophy
To design a robust module system for Aria, it is imperative to analyze the theoretical underpinnings of modular programming and how they apply to Aria's specific constraints and goals. The design choices presented herein are informed by a comparative analysis of existing paradigms in systems programming, specifically those of Rust, Go, and Python, while adhering to Aria's core philosophy of explicit definitions and zero-cost abstractions.
2.1 The Role of the Module System in System Languages
In systems programming, a module system serves three primary functions: namespace management, compilation unit organization, and visibility control. Unlike interpreted languages where modules might be dynamic objects resolved at runtime, compiled languages like Aria must resolve all module dependencies at compile time to ensure type safety and efficient binary generation. The absence of a module loader 1 in the current Aria implementation creates a vacuum where the compiler cannot verify the existence or types of external symbols, leading to the Undefined identifier errors currently observed.
Aria's design rejects the C/C++ header inclusion model, which relies on textual copy-pasting via the preprocessor. While Aria possesses a preprocessor for macro expansion 1, the module system must be semantic, not textual. It operates on the AST level, merging symbol tables rather than source buffers. This distinction is crucial for compilation speed (parsing headers only once) and hygiene (preventing macro pollution).
2.2 Comparative Analysis of Reference Models
2.2.1 Rust: The Explicit Module Tree
Rust employs a strict, explicit module tree originally reliant on mod.rs files to define directory modules.2 The use keyword in Rust merely brings items into scope; it does not load files. File loading is driven by mod declarations in the crate root. This model provides extreme control over the compilation graph but introduces boilerplate that can be confusing for newcomers.3 Aria’s design simplifies this by merging loading and importing. Unlike Rust, where a file must be declared to be part of the compilation, Aria follows a discovery model where use implies dependency, reducing friction. However, Aria retains Rust’s strict pub visibility modifiers to enforce encapsulation, ensuring that internal implementation details of the standard library remain hidden.
2.2.2 Go: Package-Based Discovery
Go treats directories as packages, where an import path resolves to a directory and all files therein are part of the package.4 This "convention over configuration" approach simplifies project structure but ties the logical package structure strictly to the filesystem layout. Aria adopts Go's search path strategy (ARIA_PATH akin to GOPATH) for locating standard libraries 1, but rejects implicit visibility based on capitalization. Aria requires explicit pub keywords for exports, favoring explicitness over convention.
2.2.3 Python: Runtime Import Execution
Python executes modules at runtime, where import statements trigger the execution of top-level code.5 While flexible, this introduces runtime overhead and potential side effects during import. Aria, being a statically compiled language, must perform "loading" strictly at compile-time. Importing a module in Aria parses its declarations but does not execute top-level statements. Code execution happens only when functions are called at runtime, ensuring a separation between compilation logic and runtime behavior.
2.3 The Aria Hybrid Model
Aria's module system is designed to be declarative, static, and filesystem-centric.
* Atomic Unit: The atomic unit of compilation is the File (module).
* Identification: Modules are identified by logical paths (e.g., std.io) or direct file paths.
* Resolution: A ModuleResolver maps these logical paths to physical files using a defined search order.1
* Linkage: A statically linked runtime (libaria_runtime.a) ensures standard library functions are available without complex makefile configurations.1
3. Module Loading Algorithm
The module loading process acts as the bridge between the syntactic parsing of a UseStmt and the semantic availability of types and functions. It transforms a passive import declaration into a loaded, type-checked Module structure residing in the compiler's memory. This process must occur prior to the type-checking phase of the importing file to ensure that all dependencies are resolved and their types are available for validation.
3.1 Logical-to-Physical Resolution Strategy
The resolution algorithm is responsible for deterministically mapping a module path (e.g., std.io) to a concrete filesystem path (e.g., /usr/lib/aria/stdlib/io.aria).
Input:
* UseStmt* stmt: The AST node containing the import path vector (e.g., ["std", "io"]) or a file string.
* baseDir: The absolute directory path of the file currently being compiled.
Output:
* canonicalPath: The absolute, normalized filesystem path to the target .aria file.
Algorithm Steps:
1. Path Normalization:
The algorithm first converts the logical components into a relative filesystem path.
   * If the import is a logical path (e.g., std.io), components are joined by the operating system's directory separator: std/io.
   * If the import is a string literal (e.g., "./utils.aria"), it is treated as a relative path.
   2. Search Strategy:
The compiler queries a sequence of root directories. The order is significant to allow local overrides of system modules if necessary, although standard library shadowing is generally discouraged.
      * Attempt 1: Relative to Current File: Check baseDir/std/io.aria. This supports sibling modules and project-local dependencies.
      * Attempt 2: Project Root: If a project configuration exists (e.g., aria.toml), resolve relative to the project source root.
      * Attempt 3: Environment Paths (ARIA_PATH): Iterate through directories specified in the ARIA_PATH environment variable.1 This is crucial for locating third-party libraries.
      * Attempt 4: Standard Library Root: Check the compiler's intrinsic standard library location, typically ARIA_HOME/stdlib/std/io.aria.
      3. Directory-Based Module Fallback:
For every path P checked above, if P.aria does not exist, the resolver checks for P/mod.aria. This mirrors Rust’s mod.rs pattern, allowing a directory to function as a module by exposing an index file.2
      4. Failure Handling:
If the traversal of all search paths yields no valid file, the resolver must emit a Fatal Compilation Error detailing the logical path attempted and the list of directories searched. This provides immediate, actionable feedback to the developer.
3.2 The Recursive Loading Pipeline
Once a unique file path is resolved, the compiler must load the module. This process involves parsing the file and recursively loading its dependencies. A critical requirement is the detection of circular dependencies to prevent infinite recursion and ensure a deterministic initialization order.
Data Structures:
         * ModuleCache: A global hash map Map<std::string, Module*>, where keys are canonical file paths. This ensures each module is parsed exactly once.
         * LoadingStack: A stack of canonical paths currently being processed. This is the primary mechanism for cycle detection.
Algorithm Flow:
         1. Cache Lookup:
Query the ModuleCache with the canonicalPath.
            * If the module exists and its state is FullyLoaded, return the Module* immediately.
            * If the module exists but its state is Loading, a Circular Dependency is detected (e.g., A imports B, B imports A). The compiler must halt and report the cycle loop to the user.1
            2. Initialization:
If the module is not in the cache:
               * Create a new Module object.
               * Set its state to Loading.
               * Insert it into the ModuleCache and push the path onto the LoadingStack.
               3. Parsing and Discovery:
               * Read the file content from disk.
               * Instantiate a new Lexer and Parser for this file.
               * Generate the AST (ProgramNode).
               * Pre-Computation Scan: Traverse the AST specifically looking for UseStmt nodes. This can be done via a lightweight visitor or during the initial parse.
               * Recursive Load: For every UseStmt found, invoke this Loading Algorithm recursively. This depth-first approach ensures that leaf dependencies (like std.core) are fully loaded and type-checked before the modules that depend on them (like std.io).
               4. Finalization:
               * Once all dependencies are loaded, perform TypeChecking on the current module.
               * Populate the module's ExportTable with public symbols.
               * Set the module state to FullyLoaded.
               * Pop the path from the LoadingStack.
               * Return the Module*.

4. Symbol Import Strategy
Loading the module is only half the battle; the symbols defined within that module (functions, structs, constants) must be made accessible to the importing scope. This requires a strategy for visibility filtering, namespace management, and conflict detection.
4.1 Visibility and Export Filtering
Aria enforces a strict encapsulation model. By default, all symbols defined in a module are private. They are visible only within the file where they are defined. To make a symbol accessible to importers, it must be explicitly annotated with the pub keyword.
               * AST Annotation: The parser already handles the pub token, marking AST nodes (FuncDecl, StructDecl) with an isPublic flag.1
               * Export Table Generation: During the finalization phase of module loading, the compiler scans the module's top-level symbol table. It constructs a separate ExportTable containing only those symbols where isPublic is true. Private helpers, internal structs, and implementation details are excluded from this table, rendering them invisible to importers.
4.2 Namespace Creation and Symbol Injection
When a module is imported, its public symbols are injected into the importing file's symbol table. The manner of injection depends on the syntax of the UseStmt.
4.2.1 Namespace Import (Default)
Syntax: use std.io;
               * Mechanism: The compiler creates a single new symbol in the current scope named io (derived from the last component of the path).
               * Type: This symbol is of a special type ModuleSymbol. It does not map to an LLVM value directly but acts as a handle to the std.io module's ExportTable.
               * Resolution: When the compiler encounters a member access expression like io.print(...), it identifies io as a ModuleSymbol and performs a lookup for print within the associated ExportTable.
4.2.2 Aliased Import
Syntax: use std.io as my_io;
               * Mechanism: Identical to the default import, but the symbol created in the current scope is named my_io.
               * Use Case: This is essential for resolving name conflicts (e.g., importing json parsers from two different vendors) or shortening verbose module paths.
4.2.3 Wildcard Import
Syntax: use std.io.*;
               * Mechanism: The compiler iterates through every symbol in the std.io ExportTable.
               * Injection: Each public symbol (e.g., print, stdout, File) is inserted directly into the current scope.
               * Conflict Detection: If a symbol with the same name already exists in the current scope, the compiler must enforce a conflict resolution policy. The recommended policy for Aria is Explicit Shadowing Warning: allow the local definition to shadow the imported one but emit a compiler warning. If two wildcard imports provide the same symbol (ambiguity), an error is emitted only if that symbol is actually used.
4.3 Integration with the Symbol Table Class
The current CodeGenContext uses a scopeStack (std::vector<std::map<std::string, Symbol>>).1 To support modules, the Symbol struct must be extended or utilized to store module references.
               * ModuleSymbol: A Symbol where AllocStrategy is set to a new enum value MODULE and the val pointer (or a new field) points to the loaded Module instance.
               * Cross-Module Lookup: The TypeChecker's visit method for MemberAccessExpr must be updated. If the left-hand side resolves to a ModuleSymbol, the lookup for the right-hand side must be redirected from the local scopeStack to the ExportTable of the referenced module.

5. Extern Function Linkage
Aria's standard library is backed by a statically linked runtime, libaria_runtime.a.1 The compiler must bridge the gap between extern func declarations in Aria source files and the compiled machine code in the archive.
5.1 Declarations and ABI Mapping
When the parser encounters a declaration like:


Code snippet




extern func:aria_stdout_write = int64(string:str);

It creates a FuncDecl node with isExtern = true. During the code generation phase, the CodeGenVisitor handles this differently from standard functions.
               1. LLVM Declaration: Instead of generating a define block with a function body, the compiler emits an LLVM declare statement.
Code snippet
declare i64 @aria_stdout_write(%struct.AriaString*)

               2. Signature Mapping: The argument types and return types are mapped to their LLVM equivalents using the getLLVMType helper.1 Specifically, Aria strings (which are structs) are passed by pointer to match the C ABI expected by the runtime.
               3. Name Mangling: Critical to linking is the absence of name mangling for externs. While internal Aria functions might be mangled to support overloading or namespacing (e.g., _Aria_std_io_print), extern functions must retain their exact name (aria_stdout_write) to match the symbols in the C-based libaria_runtime.a.1
5.2 Linking Strategy
The actual resolution of these symbols occurs at the system linker level, not within the Aria compiler itself.
                  1. Compilation: ariac compiles the user's code and all used modules into a single object file (e.g., main.o) or a set of object files.
                  2. Linker Invocation: The final step of the build process involves ariac invoking the system linker (ld, clang, or lld).
                  * The compiler constructs a linker command that includes the user's object file and the path to the runtime library.
                  * Command: clang main.o -o main -L/usr/lib/aria -laria_runtime.
                  3. Resolution: The system linker matches the undefined symbol _aria_stdout_write in main.o with the defined symbol in libaria_runtime.a. If the symbol is missing, a link-time error ("Undefined reference") is generated.
5.3 Safety and Runtime Contracts
The validity of extern declarations relies on a "contract" between the stdlib modules and the runtime library.
                  * Verification: The compiler assumes the extern func signature matches the runtime implementation. Mismatches (e.g., passing int64 where int32 is expected) can lead to undefined behavior or segfaults at runtime.
                  * Mitigation: To minimize risk, end-users should rarely write extern declarations. Instead, they should use std.io, which contains the carefully vetted and type-checked extern definitions wrapping the runtime.

6. Implementation Roadmap
This roadmap breaks down the development of the module system into five distinct, sequential phases. Each phase builds upon the previous one, ensuring a stable progression from basic file resolution to full standard library support.
Phase 1: The Module Resolver
Objective: Implement the logic to locate files on the filesystem.
                  * Action: Create src/frontend/sema/module_resolver.cpp.
                  * Key Task: Implement readAriaPath() to parse environment variables.
                  * Key Task: Implement resolve(path) to check the sequence: Relative -> Project Root -> System Paths.
                  * Validation: Unit tests verifying that paths like std.io resolve correctly to /usr/lib/aria/stdlib/io.aria (mocked).
Phase 2: The Module Loader
Objective: Implement the graph construction and parsing pipeline.
                  * Action: Create src/frontend/sema/module_loader.cpp.
                  * Key Task: Define the Module class to hold the AST root and Symbol Table.
                  * Key Task: Implement the recursive load() function with the ModuleCache and LoadingStack for cycle detection.
                  * Integration: Modify the main compiler driver to initialize the ModuleLoader and trigger it for the main input file.
Phase 3: The Symbol Importer
Objective: Integrate loaded symbols into the semantic analysis scope.
                  * Action: Modify src/frontend/sema/type_checker.cpp and symbol_table.cpp.
                  * Key Task: Update UseStmt visitation to retrieve the loaded module and inject symbols into the current scope.
                  * Key Task: Implement ModuleSymbol type to support mod.func access.
                  * Key Task: Implement the logic for wildcard imports, iterating over the module's public exports.
Phase 4: Extern Linkage
Objective: Connect AST declarations to the binary runtime.
                  * Action: Modify src/backend/codegen.cpp.
                  * Key Task: Update visit(FuncDecl) to check for isExtern. If true, emit llvm::Function::Create without a body block.
                  * Key Task: Ensure name mangling is disabled for extern symbols.
                  * Key Task: Update the compiler driver to accept -L and -l flags and pass them to the linker invocation.
Phase 5: Standard Library Bootstrapping & Optimization
Objective: Enable "Hello World" with std.io.
                  * Action: Validate stdlib/io.aria and stdlib/core.aria.
                  * Key Task: Ensure stdlib files contain correct pub extern declarations matching the C runtime.
                  * Key Task: Implement AST caching (don't re-parse std.core if imported by multiple modules) to optimize build times.

8. Detailed Implementation Specifications
8.1 Class Architecture


C++




// include/frontend/sema/module_loader.h

struct ExportedSymbol {
   std::string name;
   Type* type;
   bool isPublic;
};

class Module {
public:
   std::string path; // Canonical path
   std::shared_ptr<ProgramNode> ast; // The parsed AST
   std::map<std::string, ExportedSymbol> exports; // Public interface
   bool isLoaded = false;
};

class ModuleLoader {
   std::map<std::string, std::shared_ptr<Module>> cache; // Path -> Module
   std::vector<std::string> loadingStack; // For cycle detection
   ModuleResolver resolver; 

public:
   // Main entry point
   std::shared_ptr<Module> load(const std::string& logicalPath);
   
   // Internal helper for recursive loading
   std::shared_ptr<Module> loadFile(const std::string& fsPath);
};

8.2 Dependency Cycle Detection Logic
The cycle detection logic is critical for compiler stability. It utilizes the loadingStack to track the active chain of imports.


C++




// src/frontend/sema/module_loader.cpp

std::shared_ptr<Module> ModuleLoader::loadFile(const std::string& fsPath) {
   // 1. Check Cycle
   if (std::find(loadingStack.begin(), loadingStack.end(), fsPath)!= loadingStack.end()) {
       std::cerr << "Error: Circular dependency detected involving " << fsPath << "\n";
       exit(1); // Fatal error
   }

   // 2. Check Cache
   if (cache.count(fsPath)) {
       return cache[fsPath];
   }

   // 3. Begin Loading
   loadingStack.push_back(fsPath);
   
   //... Parse file...
   //... Recursively load imports found in AST...

   // 4. Finish Loading
   auto mod = std::make_shared<Module>(...);
   cache[fsPath] = mod;
   loadingStack.pop_back();
   
   return mod;
}

8.3 UseStmt Code Generation (No-Op)
It is important to note that the UseStmt itself does not generate executable machine code. Its purpose is purely semantic.
                  * Codegen: The visit(UseStmt) method in the backend should technically be a no-op or simply return.
                  * Reason: The work of UseStmt is done during the Semantic Analysis phase, where it pulls symbols into the symbol table. By the time the compiler reaches Code Generation, all io.print calls have already been resolved to their extern func definitions. The use statement has served its purpose and "disappears" from the final binary.

9. Testing and Validation Strategy
To ensure the robustness of the module system, a tiered testing strategy is recommended.
9.1 Unit Testing (Compiler Internal)
                  * Resolver Tests: Verify that std.io correctly resolves to the mock standard library path and that relative imports (./utils) resolve correctly relative to the "current file."
                  * Cycle Tests: Create a mock filesystem with files A and B importing each other. Verify the loader detects the cycle and throws the correct error without crashing.
                  * Visibility Tests: Create a module with a private function. Verify that importing it does not add the private function to the symbol table.
9.2 Integration Testing (End-to-End)
                  * Hello World: Compile the failing test case provided in the problem statement.
                  * Success Criteria: The compiler finds std.io, resolves io.stdout.write, links with libaria_runtime.a, and the resulting binary prints "Hello, World!".
                  * Multi-Module Project: Create a project with main.aria, math/utils.aria, and data/types.aria. Ensure main can import math.utils which imports data.types.
9.3 Error Message Validation
                  * Missing Module: use std.nonexistent; -> Should print "Module not found" and list searched paths.
                  * Private Access: Accessing a non-pub function from an imported module -> Should print "Symbol is private".

10. Conclusion
The implementation of the Module Loader and Symbol Importer is the definitive step required to transform Aria from an experimental prototype into a capable systems programming language. By adopting the architecture outlined in this report—characterized by a deterministic resolution algorithm, a cached DAG-based loader, and strict visibility controls—Aria will achieve the "batteries-included" vision of its standard library. This design balances the ease of use found in Python with the rigorous compilation guarantees of Rust, providing a solid foundation for the growth of the Aria ecosystem. The engineering team is advised to proceed immediately with Phase 1 of the implementation roadmap.
