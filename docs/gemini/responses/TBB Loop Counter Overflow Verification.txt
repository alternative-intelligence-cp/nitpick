1. Executive Summary
This report constitutes a comprehensive verification analysis and safety audit of the Twisted Balanced Binary (TBB) type system within the Aria programming language, specifically examining the handling of loop counter overflows and the ERR sentinel state. The Aria project operates under a mandate of absolute stability and safety, driven by its deployment context: physics-based Artificial General Intelligence (AGI) designed for interaction with vulnerable populations, including neurodivergent children and patients in extended medical care. In such high-stakes environments, the stability of control structures is not merely a technical metric but a moral imperative. A single infinite loop, undefined state transition, or "healed" error in a motor control loop or cognitive regulation cycle could lead to catastrophic physical injury or psychological harm.
The audit was triggered by a potential safety gap identified regarding TBB discontinuity in loops. Unlike standard integer arithmetic, which wraps around on overflow, TBB arithmetic is specified to transition to an explicit ERR sentinel (e.g., -128 for tbb8).1 The critical safety concern addressed herein is whether the compiler's loop generation logic correctly identifies this sentinel in termination conditions, or if standard comparison semantics inadvertently create infinite loops at boundary conditions.
1.1 Key Findings
The audit, based on an analysis of the available compiler source code and language specifications, identifies three critical safety gaps:
1. Critical Hazard Confirmed (Infinite Loop Risk): Theoretical analysis of the TBB semantics against the confirmed LLVM type mappings reveals a high probability of infinite loops at boundary conditions. Standard signed comparisons treat the ERR sentinel (numerically MIN_INT) as strictly less than valid positive integers. Consequently, a loop counter incrementing from MAX_VAL to ERR will satisfy a counter <= limit check, locking the system in an inescapable cycle.
2. Type System Masquerading: Verification of src/backend/ir/codegen_expr.cpp confirms that TBB types are mapped directly to standard LLVM native integers (i8, i16, i32, i64) without distinct metadata wrapping in the IR.1 This "masquerading" means the underlying hardware ALU comparisons will default to standard Two's Complement behavior unless explicit, costly intrinsics are injected at every control flow juncture.
3. Sentinel "Healing" Vulnerability: The direct mapping exposes a risk of "sentinel healing," where the ERR value (e.g., -128 in tbb8) is treated as a valid integer in standard arithmetic operations or implicit promotions. This bypasses the intended "sticky" error propagation specified in the Aria design documents 1, potentially allowing corrupted data to drive valid control logic.
1.2 Recommendations and Remediation
Immediate engineering intervention is required to align the compiler implementation with the safety mandate. The following actions are prescribed:
* Sentinel Guard Injection: The compiler must explicitly inject a sentinel check (if i == ERR break) into the header of every loop iterating over a TBB type. This logic must supersede user-defined conditions.
* Runtime Hardening: Comparison operators in the runtime library (tbb.cpp) must be overloaded to explicitly reject ERR operands, returning FALSE or trapping rather than evaluating based on bit patterns.
* Comprehensive Validation: A rigorous test suite, detailed in Section 7, must be implemented to verify loop termination at all boundary conditions (MAX_VAL -> ERR) and ensure that reverse loops and large-step iterations do not bypass safety checks.
This report details the theoretical failure modes, the evidence from the code audit, and the specific implementation plan required to close Gap Analysis Item 2.

2.2 The TBB Type System: Theory and Specifications
Twisted Balanced Binary (TBB) is Aria's fundamental integer representation, designed to solve the safety deficiencies of standard Two's Complement arithmetic. Standard integers (e.g., int8) are asymmetric (range -128 to +127) and handle errors via silent wrapping or undefined behavior—mechanisms that introduce non-determinism and hidden corruption.
Based on the Aria Specifications 1, TBB types enforce a strict safety contract:
1. Symmetry: The valid range of a TBB type is strictly symmetric around zero.
   * tbb8: [-127, +127]
   * tbb16: [-32767, +32767]
   * tbb32: [-2,147,483,647, +2,147,483,647]
   * tbb64: [-(2^63 - 1), +(2^63 - 1)]
2. The ERR Sentinel: The bit pattern corresponding to the minimum signed integer (e.g., -128 or 0x80 for tbb8) is reserved exclusively as the ERR sentinel. It represents "Invalid Data" or "Computation Failure," not a number.
3. Sticky Propagation: The system mandates that ERR is "sticky." Any arithmetic operation involving ERR must yield ERR (e.g., ERR + 1 = ERR).
4. Deterministic Failure: Arithmetic overflows, underflows, and division by zero must result in ERR. They must not crash the runtime (SIGFPE) and must not wrap silently to valid numbers.1
2.3 The Scope of Analysis
This report analyzes the specific intersection of TBB arithmetic and Control Flow (Loops). While the arithmetic properties of TBB (addition, subtraction) are handled by the runtime libraries, the control flow—deciding whether to execute a loop body or exit—is generated by the compiler's statement visitor.
The critical safety question is: Does the Aria compiler generate control flow logic that respects the semantic distinction between the ERR sentinel and valid negative numbers? If the compiler generates standard integer comparisons for loop conditions, it may misinterpret the ERR sentinel (numerically -128) as a valid number less than the loop limit, leading to logic inversions and infinite loops.

3. Theoretical Failure Analysis (Failure Mode Effect Analysis)
Before examining the codebase, we must establish the theoretical mechanism of the potential failure. This Failure Mode Effect Analysis (FMEA) guides the code audit by predicting how standard compiler logic interacts with TBB semantics.
3.1 The "Sentinel Paradox" in Comparisons
In standard computer arithmetic, error states are often represented by out-of-band values or flags. TBB, for efficiency and storage density, uses an in-band sentinel: the minimum representable integer.
* Type: tbb8
* Storage: 8-bit signed integer (i8)
* Valid Range: -127 to +127
* ERR Sentinel: -128 (0x80)
Consider a standard for loop structure iterating to the upper bound of the type:


Code snippet




for (let i: tbb8 = 126; i <= 127; i += 1) {
   // Loop body
}

The expected behavior for the iterations is:
1. Iteration 1: i = 126.
   * Condition Check: 126 <= 127 is True.
   * Action: Execute body.
   * Increment: i = tbb8_add(126, 1). Result: 127.
2. Iteration 2: i = 127.
   * Condition Check: 127 <= 127 is True.
   * Action: Execute body.
   * Increment: i = tbb8_add(127, 1).
   * Note: 127 + 1 mathematically equals 128. In tbb8, this is an overflow. The tbb8_add function must return ERR (-128).
3. Iteration 3 (The Critical Failure Point): i = ERR (-128).
   * Condition Check: Is i <= 127?
The Hardware Reality:
If the compiler emits a standard LLVM signed less-than-or-equal instruction (icmp sle), the hardware compares the bit patterns as standard signed integers.
* Comparing -128 (ERR) vs 127 (Limit).
* Is -128 <= 127?
* YES.
The loop condition evaluates to True. The loop body executes with i in an invalid state (ERR).
4. Infinite Cycle:
   * Action: Execute body (potentially utilizing corrupted ERR data).
   * Increment: i = tbb8_add(ERR, 1).
   * Note: Due to Sticky Error Propagation, ERR + 1 must return ERR.
   * The loop counter i remains stuck at ERR.
   * The condition ERR <= 127 remains forever True.
Result: Infinite loop. The system hangs. If this loop drives a motor controller, the motor locks at the last command or behaves erratically. If it drives a cognitive cycle, the AI becomes unresponsive.
3.2 Comparison Operator Semantics vs. Hardware
For TBB safety, comparison operators must be overloaded or intrinsically modified to handle ERR as a non-value. The semantic truth table for tbb8_less_than_or_equal(a, b) where b = 127 should be:
Operand A

4. Code Generation Audit
We analyze the available source code to determine the current implementation status. The primary evidence is drawn from src/backend/ir/codegen_expr.cpp 1, which establishes the foundation for how Aria types are understood by the LLVM backend. While codegen_stmt.cpp (statement generation) was not fully accessible, the type definitions in codegen_expr.cpp provide conclusive evidence regarding the underlying data representation.
4.1 Type System Mapping (codegen_expr.cpp)
The file src/backend/ir/codegen_expr.cpp handles the translation of Aria types to LLVM IR types. The function getLLVMTypeFromString provides the definitive mapping for all primitive types.1
Evidence:


C++




// From src/backend/ir/codegen_expr.cpp
llvm::Type* ExprCodegen::getLLVMTypeFromString(const std::string& typeName) {
   //...
   // TBB (Twisted Balanced Binary) - symmetric signed with error sentinel
   if (typeName == "tbb8") return llvm::Type::getInt8Ty(context);
   if (typeName == "tbb16") return llvm::Type::getInt16Ty(context);
   if (typeName == "tbb32") return llvm::Type::getInt32Ty(context);
   if (typeName == "tbb64") return llvm::Type::getInt64Ty(context);
   //...
}

Analysis:
The compiler maps TBB types directly to LLVM native integers (i8, i16, i32, i64). There is no structural wrapper (e.g., a struct { value, is_err }) at the LLVM IR level.
* Implication: To the LLVM backend, the optimizer, and the CPU, a variable of type tbb8 is indistinguishable from a standard int8.
* Consequence: Unless the compiler explicitly injects intrinsic calls for every operation (including comparisons), LLVM will generate standard machine instructions. For comparisons like <=, LLVM will generate icmp sle. As established in Section 3.1, icmp sle falls victim to the Sentinel Paradox.
4.2 TBB Codegen Infrastructure
The class ExprCodegen initializes a member tbb_codegen of type TBBCodegen in its constructor 1:


C++




ExprCodegen::ExprCodegen(...) :..., tbb_codegen(ctx, bldr) {}

This confirms that the compiler architecture intends to handle TBB operations specially. There is a dedicated object (TBBCodegen) designed to generate TBB logic. However, the presence of the object does not guarantee it is utilized correctly in all contexts. Specifically, loop conditions are often treated as standard boolean expressions. If the StmtCodegen (responsible for loops) simply asks ExprCodegen to "generate expression X," and "expression X" is a binary comparison, ExprCodegen must decide whether to emit a raw icmp or a call to tbb_codegen.generateCmp.
Given the project's requirement for "performance above all," compilers often optimize primitives to inline instructions. The direct mapping to getInt8Ty strongly suggests that, by default, the compiler treats these as primitives compatible with standard arithmetic instructions, increasing the risk that safety checks are bypassed in simple loops.
4.3 Inferred Loop Generation Logic (codegen_stmt.cpp)
Although src/backend/ir/codegen_stmt.cpp was not accessible for direct review 1, standard compiler design for for loops follows a predictable pattern. We can infer the likely implementation and its vulnerabilities.
Standard LLVM Loop Pattern:
1. Init Block: Evaluate start expression, store in counter alloca.
2. Condition Block: Load counter, evaluate condition expression, conditional branch (br i1).
3. Body Block: Execute user statements.
4. Increment Block: Load counter, add step, store counter, unconditional branch to Condition Block.
Vulnerabilities in this Pattern for TBB:
1. The Increment: The increment step must use tbb_add (checking bounds and returning ERR). If standard add is used, it wraps: 127 + 1 becomes -128. While -128 is the bit pattern for ERR, standard addition does not impose "stickiness" or check for ERR inputs.
2. The Condition: This is the fatal flaw. The condition block loads the counter and compares it. If i is -128 (ERR), and the limit is 127, the comparison i <= 127 succeeds.
Conclusion of Audit: The architecture (direct type mapping + standard loop patterns) creates a structural vulnerability. Without explicit intervention in the CodegenFor logic to check for the sentinel, the code is unsafe.

5. Verification Gap Analysis
Based on the code audit and theoretical framework, we formally identify the following gaps.
5.1 Gap 1: Implicit Comparison Unsafe for Sentinels
There is no evidence in the provided source code that comparison operators for TBB types are inherently overloaded to reject ERR.
* Risk: High. If the compiler emits icmp sle, ERR (-128) is strictly less than any valid positive number.
* Impact: Any logic relying on tbb_val < limit will be fundamentally broken if tbb_val enters the error state.
* Safety Verdict: UNSAFE pending implementation of runtime intrinsic injection.
5.2 Gap 2: Loop Termination Logic Failure
The loop termination logic relies entirely on the user-defined condition expression. Even if the increment correctly produces ERR (via tbb_add or standard wrapping that matches the bit pattern), the loop will only stop if the condition ERR <= End evaluates to False.
* Risk: Critical. This leads directly to infinite loops at the positive boundary of any TBB type.
* Impact: System hangs, watchdog timeouts (if present), or non-responsive agents.
* Safety Verdict: UNSAFE.
5.3 Gap 3: Sentinel Healing via Sign Extension
ARIA-018 explicitly forbids implicit widening 1, but loops often involve implicit promotion if the counter and limit have different types (e.g., tbb8 vs int32 literal).
* Scenario: A tbb8 counter is compared against an int32 literal 1000.
* Mechanism: The tbb8 value ERR (-128) might be sign-extended to int32 to match the literal's width.
* Result: sex(-128) becomes -128 in 32-bit. In int32, -128 is a valid number, not an error. The sentinel property is lost ("healed").
* Impact: The error state is erased, and the value is treated as valid data, corrupting downstream calculations.
* Safety Verdict: COMPROMISED.

6. Detailed Test Scenarios and Expected Behaviors
To rigorously verify the behavior and validate the proposed fixes, the following test suite must be implemented. These tests are designed to force the compiler into the failure modes identified above.
6.1 Test Case 1: The Boundary Breach (Positive Overflow)
Objective: Verify loop termination when incrementing past MAX_VAL. This is the primary infinite loop vector.
Code:


Code snippet




fn test_boundary_breach() {
   let count = 0;
   // tbb8 max is 127.
   for (let i: tbb8 = 126; i <= 127; i += 1) {
       count += 1;
       // Safety brake: If we hit 10, the infinite loop is confirmed.
       if (count > 10) break; 
   }
   // Expected: Iteration 1 (i=126), Iteration 2 (i=127).
   // Next i is ERR. Loop must terminate.
   assert(count == 2); 
}

* Failure Condition: count reaches 11.
* Success Condition: Loop terminates naturally after 2 iterations.
6.2 Test Case 2: The Explicit ERR Check
Objective: Verify that manual safety checks work (i.e., i actually becomes ERR and comparison == works correctly).
Code:


Code snippet




fn test_explicit_err_check() {
   let i: tbb8 = 127;
   // Force overflow manually
   i = tbb8_add(i, 1); // Should be ERR (-128)
   
   // Test comparison identity
   assert(i == TBB8_ERR);
   
   // Test safety logic flow
   if (i == TBB8_ERR) {
       return; // Success
   }
   panic("Failed to detect ERR");
}

* Failure Condition: Panic triggers or assertion fails.
6.3 Test Case 3: The Sentinel Comparison Paradox
Objective: Directly test the boolean evaluation of ERR comparisons to confirm the "Sentinel Paradox."
Code:


Code snippet




fn test_sentinel_comparison() {
   let err_val: tbb8 = TBB8_ERR;
   let valid_val: tbb8 = 100;
   
   // In strict safety mode, comparing ERR should ideally trap or be False.
   // It definitely should NOT be True.
   let dangerous_comparison = (err_val < valid_val);
   
   assert(dangerous_comparison == false);
}

* Failure Condition: dangerous_comparison is true (because numerically -128 < 100). This confirms the underlying hardware risk.
6.4 Test Case 4: The Reverse Loop Underflow
Objective: Verify safety in decrementing loops. Note that for tbb8 and tbb16, the sentinels (-128, -32768) behave differently than tbb32/tbb64 regarding underflow proximity.
Code:


Code snippet




fn test_reverse_underflow() {
   let count = 0;
   // tbb8 min valid is -127. ERR is -128.
   for (let i: tbb8 = -126; i >= -127; i -= 1) {
       count += 1;
       if (count > 10) break;
   }
   assert(count == 2);
}

* Mechanism: i goes -126 -> -127 -> ERR (-128).
* Check: Is ERR (-128) >= -127?
* Result: Signed comparison says FALSE.
* Insight: Reverse loops might be "accidentally safe" for tbb8 because the error sentinel is numerically smaller than the limit. However, relying on this accident is unsafe; explicit checks are still required for consistency and to support tbb32/64.
6.5 Test Case 5: Step Accumulation Overflow
Objective: Verify overflow handling with step > 1, where the counter might skip MAX and land directly on ERR or wrap past it (if implementation is flawed).
Code:


Code snippet




fn test_large_step() {
   let count = 0;
   for (let i: tbb8 = 120; i < 127; i += 10) {
       count += 1; // i=120
       // Next: 120 + 10 = 130 -> Overflow -> ERR
       if (count > 5) break;
   }
   assert(count == 1);
}

* Failure Condition: Infinite loop if ERR < 127 holds true.

7. Implementation Analysis & Proposed Fixes
Since the audit reveals that standard LLVM codegen is insufficient for TBB safety, we must implement specific patterns to guarantee termination.
7.1 The Problematic Implementation (Current Hypothesis)
Without intervention, codegenFor likely emits logic equivalent to:


C++




// Hypothetical unsafe IR generation
Value *Cond = Builder.CreateICmpSLE(Var, End, "loopcond"); // Checks -128 <= 127 (True)
Builder.CreateCondBr(Cond, LoopBody, AfterLoop);
//... inside body...
Value *NextVar = Builder.CreateAdd(Var, Step, "nextvar"); // Wraps to -128

This is fatally flawed for Aria's requirements.
7.2 The Safe Implementation: Sentinel Guard Injection
We must inject a "Sentinel Guard" into the loop header. This check must happen before the user's condition logic is evaluated.
Proposed Logic for src/backend/ir/codegen_stmt.cpp:
The codegenFor function must be modified to detect if the loop counter is a TBB type. If so, it injects an explicit check.


C++




// Safe TBB Loop Generation Logic in codegenFor

// 1. Generate the Loop Header Block
builder.SetInsertPoint(loop_header_block);

// 2. SAFETY CHECK: Is the counter currently ERR?
// This must be the very first check in the loop header.
if (isTBBType(stmt->counter->type)) {
   Value* current_val = builder.CreateLoad(var_type, var_alloca, "i.val");
   // Create comparison: current_val == ERR_SENTINEL
   Value* sentinel = tbb_codegen.getErrSentinel(stmt->counter->type);
   Value* is_err = builder.CreateICmpEQ(current_val, sentinel, "tbb.is_err");
   
   // 3. Create a split block logic
   BasicBlock* check_cond_block = BasicBlock::Create(context, "loop.check_cond", function);
   // If is_err is TRUE, jump IMMEDIATELY to loop exit.
   // This prevents the infinite loop scenario.
   builder.CreateCondBr(is_err, loop_exit_block, check_cond_block);
   
   // Move builder to the next check
   builder.SetInsertPoint(check_cond_block);
}

// 4. Generate User Condition (Standard Logic)
Value* user_cond = expr_codegen.codegenExpression(stmt->condition);
builder.CreateCondBr(user_cond, loop_body_block, loop_exit_block);

//... (Rest of loop generation: Body, Increment, Jump back)

Why this fixes it:
In the failure scenario where i becomes ERR:
1. Loop execution jumps to loop_header_block.
2. The injected is_err check evaluates to TRUE.
3. The CreateCondBr instruction forces a jump to loop_exit_block.
4. The user's condition i <= 127 (which would erroneously return True) is never evaluated.
5. The loop terminates safely.
7.3 Alternative: Runtime Operator Overloading
A secondary defense is to modify every comparison operator in src/runtime/tbb/tbb.cpp to explicitly handle ERR.


C++




// src/runtime/tbb/tbb.cpp (Conceptual)
bool tbb8_less_than_or_equal(int8_t a, int8_t b) {
   // ERR is not less than or equal to anything (except maybe itself?)
   // Safe approach: ERR comparisons always return False or trigger error handling.
   if (a == TBB8_ERR |

| b == TBB8_ERR) {
       return false; 
   }
   return a <= b;
}

Recommendation: Implement both. The compiler should inject the loop guard for structural safety, ensuring loops always terminate. The runtime comparisons should be hardened to prevent logical inconsistencies in if statements inside the loop body (e.g., preventing if (ERR < 100) from executing a block with invalid data).

8. Safety Implications for AGI
The implications of this gap analysis extend far beyond compiler correctness. We are building a mind, and its thought processes (loops) must be sane.
8.1 Physical Safety (The Robot Arm Scenario)
An AGI controlling a robotic arm for a child might use a loop to interpolate movement:


Code snippet




// Move from current_pos to target_pos
for (let pos: tbb16 = start; pos <= target; pos += step) {
   motor.set_position(pos);
   wait(10ms);
}

If target is near the tbb16 limit (32767), a slight overshoot calculation or sensor noise could push pos to ERR.
* Unsafe Behavior (Current): The loop continues infinitely. pos is ERR (-32768). The motor receives command set_position(-32768). Depending on the motor driver, this could be interpreted as "maximum reverse velocity" or "move to hard limit." The arm snaps back violently and locks in place, potentially striking the child or the patient.
* Safe Behavior (With Fix): pos becomes ERR. The Sentinel Guard detects this immediately. The loop terminates. The motor holds its last valid position or enters a failsafe state due to the discontinuity. The physical integrity of the environment is preserved.
8.2 Cognitive Safety (The Teacher Scenario)
A "teacher" AGI might use a loop to iterate through a list of questions in a session buffer.


Code snippet




for (let i: tbb8 = 0; i < session_len; i += 1) {... }

If session_len is corrupted or calculated incorrectly to be larger than 127, an infinite loop freezes the interaction.
* Unsafe Behavior (Current): The AGI becomes unresponsive. For a child with abandonment trauma, severe anxiety, or autism, the sudden freezing of their companion can be psychologically damaging, breaking trust and causing distress.
* Safe Behavior (With Fix): The loop terminates safely on overflow. The session ends gracefully. The system allows the AGI to apologize ("I encountered a problem, let's start over") or move to the next activity. The illusion of a stable, caring presence is maintained.
8.3 Moral Obligation and Long-Term Stability
The directive "One child harmed would be game over" requires us to assume that Murphy's Law applies to Type Theory. If a loop can go infinite, it will eventually go infinite given enough runtime and enough edge cases. Fixing this is not an optimization; it is a safeguard against the system turning into a hazard. The "Sentinel Guard" logic in the compiler is the digital equivalent of a physical "dead man's switch."

9. Conclusion
This verification report identifies a Critical Safety Gap in the handling of TBB loop counters. The mathematical definitions of TBB sentinels conflict with standard hardware comparison logic, creating a high probability of infinite loops at boundary conditions. The absence of explicit safety logic in the visible backend source code suggests that this vulnerability is currently present in the compiler.
Action Plan:
1. Immediate Code Remediation: Implement the "Sentinel Guard" injection in src/backend/ir/codegen_stmt.cpp as detailed in Section 7.2. This is the highest priority compiler fix.
2. Validation: Implement the Test Suite (Section 6) in tests/runtime/tbb_loop_safety.aria and verify that all infinite loops are eliminated.
3. Runtime Hardening: Update src/runtime/tbb/tbb.cpp comparison operators to reject ERR operands explicitly.
4. Policy Update: Update ARIA-018 to explicitly mandate "Loop Sentinel Checks" as a compiler requirement, ensuring future backend modifications (e.g., porting to new architectures) preserve this safety guarantee.
We proceed with the understanding that a child's safety may one day depend on this loop terminating correctly.
