**AUDIT FOCUS: Runtime Safety - fix256 Deterministic Fixed-Point Arithmetic**

**Threat Model:**
fix256 (Q128.128 format) is used for all physics calculations where determinism is CRITICAL. Floating-point's non-determinism could cause: same inputs → different outputs on different CPUs → robot behavior diverges between training and deployment → trained "safe grip" becomes "crushing grip" on production hardware. Even 1 ULP difference can compound over time.

**Target Files:**
- `src/runtime/math/fix256.cpp`

**Specific Items to Audit (5 items max):**

1. **fix256_add() and fix256_sub()** - Basic arithmetic determinism
   - Verify fixed-point addition/subtraction is bit-exact
   - Check no rounding or truncation differences across platforms
   - Confirm overflow detection on whole-number part (high 128 bits)

2. **fix256_mul()** - Multiplication with scaling
   - Verify 256×256→512 bit multiply then shift-right by 128
   - Check no intermediate overflow or precision loss
   - Confirm result deterministic regardless of CPU architecture

3. **fix256_div()** - Division with scaling
   - Verify dividend shifted left 128 before division
   - Check division by zero returns error sentinel
   - Confirm no platform-specific rounding modes affect result

4. **fix256_sqrt()** - Square root for vector magnitude
   - Verify algorithm is bit-exact (no floating-point used)
   - Check negative input handled correctly (error or zero)
   - Confirm deterministic across ARM/x86/RISC-V

5. **Conversion functions: fix256_from_int64(), fix256_to_int64()**
   - Verify whole-number part correctly positioned in high 128 bits
   - Check fractional part correctly positioned in low 128 bits
   - Confirm no precision loss on round-trip conversions where possible
