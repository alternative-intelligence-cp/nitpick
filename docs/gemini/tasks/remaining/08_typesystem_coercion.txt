**AUDIT FOCUS: Type System - Type Coercion and Conversion Safety**

**Threat Model:**
Implicit type coercion is where most numeric bugs hide: i16→i32 loses error sentinel → downstream code uses invalid value → wrong force calculation → harm. Aria minimizes implicit conversions, but where they exist, they MUST be safe. Explicit conversions must also validate ranges.

**Target Files:**
- `src/semantic/type_checker.cpp` (coercion rules)
- `src/backend/ir/codegen_expr.cpp` (conversion code generation)

**Specific Items to Audit (5 items max):**

1. **Widening conversions (smaller → larger integer types)**
   - Verify sign extension used (not zero extension) for signed types
   - Check TBB sentinel values preserved during widening
   - Confirm no implicit widening where not explicitly allowed

2. **Narrowing conversions (larger → smaller integer types)**
   - Verify range check performed (value fits in target type)
   - Check out-of-range values cause error (not silent truncation)
   - Confirm sentinel propagation on narrowing failure

3. **Integer ↔ Fixed-point conversions**
   - Verify int→fix256 places value in whole part (high 128 bits)
   - Check fix256→int extracts whole part, detects fractional loss
   - Confirm no precision loss warnings where appropriate

4. **Implicit numeric promotions in binary operations**
   - Verify operands promoted to common type before operation
   - Check promotion rules match TBB/LBIM semantics
   - Confirm no unexpected promotions (e.g., i32 + i64 → ?)

5. **Explicit cast validation**
   - Verify casts only allowed where semantically valid
   - Check no casts that could break type safety (e.g., ptr→int→ptr)
   - Confirm runtime checks inserted for unsafe casts

**SIL-4 Compliance Requirements:**
- Narrowing MUST validate range (no silent truncation)
- Sentinel values MUST be preserved through conversions
- No implicit conversions that could lose precision without warning
- Explicit casts must validate at runtime where compile-time check impossible
- All conversions must be well-defined (no UB)

**Expected Deliverable:**
For each item, report:
- ✅ Compliant / ❌ Bug found / ⚠️ Unclear (need clarification)
- If bug: Exact line number, root cause, proposed fix
- If compliant: Briefly confirm what makes it safe
- If unclear: Specific question about spec or implementation

**Critical Note:**
Type coercion bugs are subtle. They often manifest as "works in test, fails in production" because test data happens to fit in smaller types. Check edge cases rigorously.
