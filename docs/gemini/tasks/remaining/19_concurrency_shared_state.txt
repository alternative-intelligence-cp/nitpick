**AUDIT FOCUS: Concurrency - Shared State Protection**

**Threat Model:**
When shared state is necessary, it must be protected. Bug: two threads modify motor position simultaneously → race condition → position value corrupted → robot thinks it's at (0,0) when actually at (1,1) → path planning wrong → collision. Synchronization must be correct.

**Target Files:**
- `src/runtime/atomic/atomic.cpp` (atomic operations)
- `src/runtime/threading/mutex.cpp` (mutex implementation)
- `src/backend/ir/codegen_sync.cpp` (if exists - synchronization code gen)

**Specific Items to Audit (5 items max):**

1. **Atomic operation correctness**
   - Verify atomic ops actually atomic (not split into load/modify/store)
   - Check memory ordering semantics correct (acquire/release/seq_cst)
   - Confirm atomic types match underlying hardware support

2. **Mutex lock/unlock pairing**
   - Verify every lock() has matching unlock()
   - Check unlock() only called by lock holder
   - Confirm no double-lock deadlocks

3. **Critical section protection**
   - Verify shared state accessed only within critical sections
   - Check critical sections minimal (no blocking operations)
   - Confirm no forgotten locks on shared state access

4. **Lock ordering to prevent deadlock**
   - Verify consistent lock acquisition order documented
   - Check no circular lock dependencies possible
   - Confirm deadlock prevention strategy enforced

5. **Read-Write lock correctness**
   - Verify multiple readers allowed simultaneously
   - Check writer excludes all readers and writers
   - Confirm no writer starvation from continuous readers

**SIL-4 Compliance Requirements:**
- Atomics MUST have correct memory ordering
- Mutexes MUST prevent simultaneous access
- All shared state MUST be protected
- Deadlocks MUST be impossible (or detected)
- Lock pairing MUST be enforced (RAII or similar)

**Expected Deliverable:**
For each item, report:
- ✅ Compliant / ❌ Bug found / ⚠️ Unclear (need clarification)
- If bug: Exact line number, root cause, proposed fix
- If compliant: Briefly confirm what makes it safe
- If unclear: Specific question about spec or implementation

**Critical Note:**
Concurrency bugs are notoriously hard to reproduce. Check synchronization primitives against hardware memory model and verify lock disciplines are enforced.
