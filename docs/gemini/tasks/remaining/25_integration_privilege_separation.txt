**AUDIT FOCUS: Integration - Privilege Separation and Sandboxing**

**Threat Model:**
Even with perfect code, environment may be hostile. Scenario: compromised dependency → arbitrary code execution → attacker gains full robot control. Privilege separation limits damage: motor control runs with elevated privileges, vision runs unprivileged → vision compromise can't access motors directly.

**Target Files:**
- `src/runtime/process/sandbox.cpp` (if exists - sandboxing implementation)
- `src/runtime/process/capabilities.cpp` (if exists - capability management)

**Specific Items to Audit (5 items max):**

1. **Process privilege levels**
   - Verify safety-critical processes run with minimal necessary privileges
   - Check privilege escalation points validated
   - Confirm no unnecessary elevated privileges

2. **Capability-based security**
   - Verify capabilities granted explicitly (not by default)
   - Check capability delegation controlled
   - Confirm revoked capabilities actually unusable

3. **Sandbox enforcement**
   - Verify sandboxed processes cannot escape
   - Check filesystem access restricted to needed paths
   - Confirm network access blocked for untrusted code

4. **Inter-process communication (IPC) validation**
   - Verify IPC messages validated at privilege boundary
   - Check no trusted process accepts arbitrary data from untrusted
   - Confirm IPC channels authenticated

5. **Privilege dropping after initialization**
   - Verify elevated privileges dropped when no longer needed
   - Check no way to regain dropped privileges
   - Confirm privileged operations completed before drop

**SIL-4 Compliance Requirements:**
- Privileges MUST be minimal (principle of least privilege)
- Privilege boundaries MUST be enforced by OS
- Sandbox escapes MUST be impossible
- IPC MUST validate across trust boundaries
- Capabilities MUST be unforgeable

- If bug: Exact line number, root cause, proposed fix
