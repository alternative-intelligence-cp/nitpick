**AUDIT FOCUS: Concurrency - Actor Message Passing**

**Threat Model:**
Actor model isolates state - actors communicate via messages. Bug: message queue overflow → new messages dropped → motor disable command lost → robot continues moving uncontrolled. Or: message corruption → actor receives wrong command → executes unintended action.

**Target Files:**
- `src/runtime/async/actor.cpp` (if exists - actor implementation)
- `src/backend/ir/codegen_async.cpp` (actor message send/receive)

**Specific Items to Audit (5 items max):**

1. **Message queue bounds checking**
   - Verify message queue has size limit
   - Check send() blocks or returns error when queue full
   - Confirm no silent message dropping

2. **Message serialization/deserialization**
   - Verify message type preserved across send/receive
   - Check no type confusion (message type A received as type B)
   - Confirm no corruption during message passing

3. **Actor state isolation**
   - Verify actors cannot access each other's state directly
   - Check only message passing allowed between actors
   - Confirm no shared mutable state between actors

4. **Actor lifecycle management**
   - Verify actor cleanup when stopped
   - Check pending messages handled on actor termination
   - Confirm no resource leaks from stopped actors

5. **Message ordering guarantees**
   - Verify messages from A to B delivered in send order
   - Check no message reordering within single channel
   - Confirm ordering guarantees documented and enforced

**SIL-4 Compliance Requirements:**
- Message queues MUST NOT overflow silently
- Message types MUST be preserved accurately
- Actor state MUST be isolated (no shared mutable state)
- Actor cleanup MUST be deterministic
- Message ordering MUST be defined and enforced

For each item, report:

- If bug: Exact line number, root cause, proposed fix

Message passing bugs are hard to detect - they manifest as missed commands or corrupted data. Check queue management and serialization carefully.
