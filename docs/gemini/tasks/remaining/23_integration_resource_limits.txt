**AUDIT FOCUS: Integration - Resource Limits and Quotas**

**Threat Model:**
Resource exhaustion can be as dangerous as bugs. Scenario: malicious input triggers exponential allocation → OOM → system kills robot control process → motors lose control → robot falls on child. Must enforce resource limits to prevent exhaustion attacks.

**Target Files:**
- `src/runtime/allocators/allocator.cpp` (memory allocation limits)
- `src/runtime/process/rlimit.cpp` (if exists - resource limit enforcement)

**Specific Items to Audit (5 items max):**

1. **Memory allocation limits**
   - Verify total allocation capped at reasonable limit
   - Check allocation failure handled gracefully (no crash)
   - Confirm no unbounded allocation in any code path

2. **Stack size limits**
   - Verify stack depth limited (ARIA-022/ARIA-026 guards)
   - Check stack overflow detected before corruption
   - Confirm alternative error handling for deep recursion

3. **File descriptor limits**
   - Verify open file descriptors counted and limited
   - Check FD exhaustion handled (closes old, returns error)
   - Confirm no FD leaks that could hit limit

4. **CPU time limits**
   - Verify long-running operations can be preempted
   - Check timeout mechanisms work correctly
   - Confirm infinite loops detectable and terminated

5. **I/O bandwidth limits**
   - Verify I/O rate limited to prevent saturation
   - Check backpressure applied when limit reached
   - Confirm no I/O starvation from rate limiting

**SIL-4 Compliance Requirements:**
- All resources MUST have defined limits
- Limit exceeded MUST trigger graceful degradation (not crash)
- No unbounded resource consumption possible
- Limits MUST be enforced (not just advisory)
- Resource exhaustion MUST NOT cause safety violation

Resource limits are defense-in-depth. Even if code has no bugs, malicious input shouldn't be able to exhaust resources and cause denial of service.
