**AUDIT FOCUS: Type System - Optional Types and Null Safety**

**Threat Model:**
Null pointer dereference is classic memory safety bug: optional value is None → code assumes Some(value) → dereferences null → segfault OR worse, reads garbage → uses garbage as sensor value → robot acts on invalid data → collision. Aria's optional types must enforce checking before use.

**Target Files:**
- `src/semantic/type_checker.cpp` (optional type validation)
- `src/backend/ir/codegen_expr.cpp` (optional value code generation)

**Specific Items to Audit (5 items max):**

1. **Optional type unwrapping safety**
   - Verify unwrap() without check causes compile error OR runtime panic
   - Check safe unwrapping (if-let, match) enforced by type checker
   - Confirm no implicit unwrapping anywhere

2. **Optional value representation**
   - Verify optional uses tagged union or similar safe representation
   - Check "None" state is distinguishable from any valid value
   - Confirm no confusion between None and default value (e.g., 0)

3. **Optional propagation (? operator or equivalent)**
   - Verify early return on None propagates correctly
   - Check type of propagated value matches expected optional type
   - Confirm no accidental unwrapping during propagation

4. **Default values for optionals (unwrap_or, etc.)**
   - Verify default value type matches optional's inner type
   - Check evaluation semantics (eager vs lazy)
   - Confirm no type confusion between T and Optional<T>

5. **Null pointer checks in FFI boundaries**
   - Verify pointers from C code wrapped in optional
   - Check null pointers converted to None, not dereferenced
   - Confirm all FFI pointer returns treated as potentially null

**SIL-4 Compliance Requirements:**
- Optional values MUST be checked before use (enforced by type system)
- "None" MUST be detectable without UB
- No implicit conversions Optional<T> → T
- FFI null pointers MUST be safely handled
- All code paths must handle None case or prove unreachable
