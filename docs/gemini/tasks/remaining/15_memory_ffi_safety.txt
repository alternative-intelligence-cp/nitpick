**AUDIT FOCUS: Memory Safety - FFI Boundary Safety**

**Threat Model:**
FFI (Foreign Function Interface) crosses trust boundary to C code. C has no safety guarantees. Bug: Aria pointer passed to C → C corrupts memory → corruption propagates back to Aria → robot state corrupted → unpredictable behavior. FFI must assume C code is hostile.

**Target Files:**
- `src/backend/ir/codegen.cpp` (FFI function declarations and calls)
- `src/semantic/ffi_validator.cpp` (if exists - FFI safety checking)

**Specific Items to Audit (5 items max):**

1. **FFI function signature validation**
   - Verify C function signatures match Aria declarations exactly
   - Check ABI compatibility (calling convention, alignment)
   - Confirm no type mismatches between Aria and C types

2. **Pointer argument safety**
   - Verify pointers to C validated non-null before passing
   - Check pointer lifetimes extend beyond C function call
   - Confirm C cannot modify Aria-owned memory unsafely

3. **Buffer ownership and bounds at FFI boundary**
   - Verify buffer size explicitly passed to C functions
   - Check C functions cannot write beyond buffer bounds
   - Confirm ownership transfer semantics clear (who frees?)

4. **Return value validation from C code**
   - Verify null pointers from C wrapped in Optional
   - Check error codes from C converted to Aria error types
   - Confirm no assumption that C returns valid data

5. **FFI callback safety (C calling back into Aria)**
   - Verify callback signatures type-safe
   - Check callbacks can't cause Aria undefined behavior
   - Confirm panic in callback handled safely (doesn't crash C code)

**SIL-4 Compliance Requirements:**
- All FFI calls MUST validate arguments before passing to C
- Return values from C MUST be validated before use
- Pointers MUST be bounds-checked at boundary
- Null pointers MUST be handled safely
- No undefined behavior from FFI interaction

**Expected Deliverable:**
For each item, report:
- ✅ Compliant / ❌ Bug found / ⚠️ Unclear (need clarification)
- If bug: Exact line number, root cause, proposed fix
- If compliant: Briefly confirm what makes it safe
- If unclear: Specific question about spec or implementation

**Critical Note:**
FFI is where safety guarantees are most vulnerable. Aria can be perfectly safe internally, but one FFI bug can undermine everything. Assume C code is adversarial.
