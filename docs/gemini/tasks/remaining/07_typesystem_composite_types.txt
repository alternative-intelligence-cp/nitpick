**AUDIT FOCUS: Type System - Composite Types (Structs, Arrays, Pointers)**

**Threat Model:**
Bugs in composite types can cause memory layout mismatches: compiler thinks field is at offset 8, runtime has it at offset 16 → reads wrong memory → loads garbage as force limit → robot applies 1000N instead of 10N. Alignment and padding are critical for safety-critical structs.

**Target Files:**
- `src/backend/ir/codegen_expr.cpp` (composite type handling in `getLLVMType()`)
- `src/semantic/type_checker.cpp` (struct layout validation)

**Specific Items to Audit (5 items max):**

1. **Struct type layout generation**
   - Verify fields mapped to correct LLVM struct members
   - Check field order matches source declaration order
   - Confirm padding inserted for alignment where needed

2. **Array type mapping**
   - Verify array element type correctly mapped recursively
   - Check array size properly encoded in LLVM type
   - Confirm multidimensional arrays handled correctly

3. **Pointer type mapping**
   - Verify pointer to T maps to LLVM pointer type
   - Check null pointer representation defined
   - Confirm pointer arithmetic type-checked correctly

4. **Nested composite types (struct containing array of structs, etc.)**
   - Verify recursive type mapping terminates correctly
   - Check no infinite recursion on self-referential types
   - Confirm consistent layout regardless of nesting depth

5. **Composite type alignment and size calculations**
   - Verify alignment matches runtime expectations
   - Check struct size includes all fields plus padding
   - Confirm no platform-dependent alignment assumptions

**SIL-4 Compliance Requirements:**
- Struct layouts MUST match ABI exactly (no silent reordering)
- Array bounds MUST be part of type (no decay to pointer)
- Alignment MUST be consistent between compiler and runtime
- Pointers MUST have well-defined null representation
- No undefined behavior from misaligned access
