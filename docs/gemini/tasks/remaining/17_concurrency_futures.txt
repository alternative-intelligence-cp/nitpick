**AUDIT FOCUS: Concurrency - Future State Management**

**Threat Model:**
Futures represent values that will be available later. Bug in future state: poll() returns Ready when value not actually ready → code uses uninitialized data → sensor reading is garbage → robot acts on invalid input → unpredictable behavior. Future state transitions must be correct.

**Target Files:**
- `src/runtime/async/future.cpp` (future implementation)
- `src/backend/ir/codegen_async.cpp` (future code generation)

**Specific Items to Audit (5 items max):**

1. **Future state transitions (Pending → Ready)**
   - Verify state only transitions when value actually available
   - Check no race conditions in state update
   - Confirm state visible to all threads consistently

2. **Future value storage**
   - Verify value storage allocated before storing
   - Check value type matches future type parameter
   - Confirm no corruption of stored value

3. **Poll operation correctness**
   - Verify poll() returns Pending when value not ready
   - Check poll() returns Ready(value) when complete
   - Confirm no spurious Ready returns

4. **Future cancellation safety**
   - Verify cancelled future never transitions to Ready
   - Check resources freed on cancellation
   - Confirm no use-after-free of cancelled future

5. **Multiple poll() calls handling**
   - Verify repeated poll() on Pending future is safe
   - Check poll() on Ready future returns same value
   - Confirm no state corruption from multiple pollers

**SIL-4 Compliance Requirements:**
- State transitions MUST be atomic and correct
- Poll MUST NOT return Ready unless value is valid
- Cancelled futures MUST NOT be usable
- All future storage MUST be freed when done
- No race conditions in future state

**Expected Deliverable:**
For each item, report:
- ✅ Compliant / ❌ Bug found / ⚠️ Unclear (need clarification)
- If bug: Exact line number, root cause, proposed fix
- If compliant: Briefly confirm what makes it safe
- If unclear: Specific question about spec or implementation

**Critical Note:**
Future state bugs often manifest as race conditions - hard to reproduce. Check synchronization primitives carefully, especially state transitions.
