**AUDIT FOCUS: Memory Safety - Defer/Panic Cleanup and Error Unwinding**

**Threat Model:**
Defer ensures cleanup happens even during error paths. Bug in defer: motor enabled → error occurs → defer doesn't run → motor stays enabled → robot continues moving with no control code running → collision. Panic unwinding must guarantee cleanup execution.

**Target Files:**
- `src/backend/ir/codegen_stmt.cpp` (defer statement code generation)
- `src/backend/ir/codegen.cpp` (panic/error unwinding)

**Specific Items to Audit (5 items max):**

1. **Defer statement code generation**
   - Verify defer blocks registered correctly for cleanup
   - Check defer execution order (LIFO - last registered, first executed)
   - Confirm defer runs on all exit paths (return, panic, normal)

2. **Defer block scope management**
   - Verify defers scoped to function (not global or block)
   - Check nested defer blocks execute in correct order
   - Confirm defer can access variables from declaration scope

3. **Panic unwinding correctness**
   - Verify panic triggers defer execution before stack unwind
   - Check all defers run even if earlier defer panics
   - Confirm panic message propagated after cleanup

4. **Resource cleanup guarantees**
   - Verify file handles/sockets closed even on panic
   - Check memory freed even on error paths
   - Confirm motor disable commands execute before panic propagates

5. **Defer with early returns**
   - Verify defer runs on early return from function
   - Check multiple return points all trigger defers
   - Confirm return value computed before defer execution

**SIL-4 Compliance Requirements:**
- Defer MUST execute on ALL exit paths (no exceptions)
- Cleanup order MUST be deterministic (LIFO)
- Panic MUST NOT prevent defer execution
- Resources MUST be freed even on error
- No cleanup code can be skipped
