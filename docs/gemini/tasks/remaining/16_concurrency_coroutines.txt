**AUDIT FOCUS: Concurrency - Coroutine Suspension Points**

**Threat Model:**
Coroutines enable async/await. Bug at suspension point: coroutine suspends mid-operation → state partially saved → resumes with corrupted state → motor command half-executed → robot arm position wrong → collision. Suspension points must be transactionally safe.

**Target Files:**
- `src/backend/ir/codegen_async.cpp` (if exists - async/await code generation)
- `src/runtime/async/coroutine.cpp` (coroutine runtime)

**Specific Items to Audit (5 items max):**

1. **Coroutine state capture at suspension points**
   - Verify all live variables saved before suspension
   - Check saved state size calculated correctly
   - Confirm no stack corruption from state save/restore

2. **Await expression safety**
   - Verify awaited value lifetime extends across suspension
   - Check temporaries not referenced after suspension
   - Confirm no dangling pointers in resumed context

3. **Suspension point atomicity**
   - Verify no partial operations visible at suspension boundary
   - Check critical sections not split by suspension
   - Confirm invariants maintained at all suspension points

4. **Coroutine frame allocation and deallocation**
   - Verify frame size includes all captured state
   - Check frame freed when coroutine completes
   - Confirm no memory leaks from abandoned coroutines

5. **Exception/panic interaction with suspension**
   - Verify panic during coroutine triggers cleanup
   - Check defer blocks run even if suspended
   - Confirm no resource leaks on coroutine panic

**SIL-4 Compliance Requirements:**
- State capture MUST be complete (no lost variables)
- Suspension points MUST be transactionally safe
- Resumed coroutine MUST see consistent state
- All coroutine frames MUST be freed (no leaks)
- Panic MUST NOT corrupt coroutine state

Coroutines are complex - state must be preserved across suspension. Any missed variable or type mismatch between save and restore can cause subtle corruption.
