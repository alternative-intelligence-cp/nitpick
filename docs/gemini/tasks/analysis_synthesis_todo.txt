1. Executive Summary
This report delivers an exhaustive technical audit and strategic implementation roadmap for the Aria programming language ecosystem, synthesizing findings from the provided architectural analyses, source code repositories, and feature audits. The primary objective of this analysis is to identify critical divergences between the Aria Master Specification (aria_specs.txt)—regarded as the immutable source of truth—and the current implementation state of the compiler (ariac), runtime environment, and auxiliary tooling (aria_make, aria_shell).
The analysis reveals a project at a pivotal juncture. While the foundational structures for the compiler backend are present—evidenced by the CodeGenContext class and initial LLVM integration—there exists a significant "semantic gap" between the language's safety guarantees (specifically regarding Twisted Balanced Binary arithmetic and the Borrow Checker) and their operational reality. Furthermore, the ecosystem faces a "bootstrapping paradox" where the build tools require standard library features (such as globbing and process orchestration) that are currently unimplemented or exist only as host-side C++ shims.
This report decomposes these challenges into a dependency-ordered Master Research Task List. This roadmap is designed to guide the engineering effort from the current v0.0.7 state to a feature-complete v0.1.0 release, prioritizing the resolution of semantic debts that compromise the language's core value proposition: safe systems programming with explicit memory management.
The document is structured to provide deep analytical insight into four core pillars: Core Semantics and Type Theory, Memory Model and Safety Enforcement, Runtime and Ecosystem Architecture, and Tooling Infrastructure. Each section rigorously compares the specification against the codebase, highlighting conflicts, overlaps, and critical omissions.
2. Core Semantic Gap Analysis: Type Theory and Arithmetic
The defining feature of Aria is its use of Twisted Balanced Binary (TBB) types and a hybrid memory model. The audit identifies a critical dissonance between the specification of these features and their backend realization.
2.1. Twisted Balanced Binary (TBB) Implementation Debt
The Aria Master Specification explicitly defines TBB types (tbb8, tbb16, tbb32, tbb64) as non-negotiable features with specific arithmetic behaviors that differ fundamentally from standard two's complement integers.1 The audit reveals that while the type definitions exist, the semantic enforcement of their behaviors is absent in the compiler backend.
2.1.1. Specification vs. Implementation Dissonance
Specification Requirement:
* Symmetric Range: tbb8 must represent the range $[-127, +127]$. This symmetry is crucial for eliminating "asymmetry bugs" found in standard two's complement arithmetic, where the absolute value of the minimum integer cannot be represented (e.g., abs(-128) overflows in int8).1
* ERR Sentinel: The minimum signed value (e.g., -128 for int8) is reserved exclusively as the ERR sentinel.1 This value represents an invalid state, overflow, or calculation error.
* Sticky Error Propagation: Any arithmetic operation involving ERR as an operand, or any operation that results in an overflow, must yield ERR. This allows for exception-free error handling that propagates through calculation chains, ensuring that errors are not silently ignored or wrapped into valid values.1

❌ STILL MISSING: Automatic Operator Lowering
Examination of aria_source_full.txt reveals that while the CodeGenContext correctly identifies TBB types, it maps them directly to standard LLVM integer types without injecting the necessary semantic logic to enforce TBB behaviors.
* In CodeGenContext::getLLVMType, the type tbb8 returns Type::getInt8Ty(llvmContext).1 This mapping instructs LLVM to treat the variable as a standard 8-bit integer.
* In CodeGenContext::getSentinel, the values are correctly identified (e.g., -128 for tbb8).1 However, identifying the value is insufficient without logic to enforce its meaning.
* Critical Gap: The TBBLowerer class, which is responsible for intercepting arithmetic operations and injecting the "sticky error" logic, is identified as missing or incomplete in the audit documents.1 Currently, the compiler generates standard LLVM add, sub, and mul instructions. These instructions strictly follow two's complement wrapping behavior on overflow.
Implication:
Until the TBBLowerer is fully implemented, Aria fails to meet its primary safety guarantee. A calculation like 127 + 1 in tbb8 currently results in -128. In standard integers, this is a valid (though overflowed) value. In Aria, -128 is ERR. However, because the stickiness logic is missing, a subsequent operation like ERR + 1 (i.e., -128 + 1) would result in -127. This "healing" of the error violates the specification, which dictates that ERR + x = ERR.1 This violation renders the type system unsound and potentially dangerous for systems relying on overflow detection, such as the blockchain primitives mentioned in the feature audit.1
2.1.2. The Mechanics of Required Lowering
To bridge this gap, the backend requires a sophisticated lowering pass. The standard LLVM add instruction is insufficient. The TBBLowerer must transform a simple Aria addition a + b into a complex sequence of LLVM IR instructions:
1. Check Operand A: Is a equal to the sentinel (e.g., -128)?
2. Check Operand B: Is b equal to the sentinel?
3. Perform Overflow-Checked Math: Use llvm.sadd.with.overflow intrinsic to perform the addition.
4. Select Result:
   * If a was sentinel OR b was sentinel OR overflow bit is set -> Result is Sentinel.
   * Else -> Result is the computed sum.
This logic creates a branching Control Flow Graph (CFG) or requires the heavy use of select instructions for every arithmetic operation. The current codebase lacks the infrastructure to generate this IR, representing a massive technical debt that blocks v0.1.0.
2.2. Exotic Type Arithmetic (Trits and Nits)
Aria supports "exotic" types based on non-binary logic systems: trit (balanced ternary: -1, 0, 1) and nit (balanced nonary).1 These types are not merely storage formats but require distinct algebraic operations.
Specification Requirement:
These types require specific ALUs or software emulation to perform arithmetic in their respective bases. For example, ternary logic does not follow binary boolean rules, and nonary arithmetic operates in base-9.1
Current Implementation Status:
The source code contains headers for codegen_ternary.h and codegen_nonary.h, suggesting the architecture is planned.1 The CodeGenContext maps trit to int8 and tryte (a collection of trits) to int16 for storage.1
However, the analysis indicates that the operational logic—the actual C++ runtime functions that the LLVM IR should call—is missing or not linked.1 The backend is currently aliasing these types to binary integers for storage, but it has no mechanism to perform arithmetic.
* Missing Runtime Support: There is no implementation of @aria_trit_add or @aria_nit_mul in the runtime library.
* Consequence: Users can declare trit variables, but attempting to add them would either fail to compile (due to missing intrinsics) or, worse, perform binary addition on the underlying int8 storage, yielding mathematically incorrect results for balanced ternary logic (e.g., 1 + 1 in balanced ternary should be 1 with a carry, or -1 depending on the specific ternary system used, whereas binary addition yields 2).
2.3. Floating Point Precision and Emulation
The specification mentions support for flt128 (quad-precision) and flt256 (octuple-precision) floating-point types.1
Current Implementation Status:
The source code acknowledges a hybrid implementation status:
* flt128 maps to Type::getFP128Ty(llvmContext), relying on hardware support (POWER architecture) or compiler-rt software emulation (x86/ARM).1
* flt256 is implemented as a software emulation struct StructType::get(llvmContext, {limbsType}), where limbsType is an array of 4 i64 integers.1
Analysis:
While the storage definition for flt256 exists, there is no evidence of the corresponding arithmetic library in the standard library audit.1 Without a "BigFloat" library linked to the runtime, flt256 is currently a passive data structure. The compiler needs to know how to lower operations like flt256 + flt256 into calls to runtime functions (e.g., __aria_addtf3), which appear to be absent.

	3. Memory Model and Safety Enforcement Analysis
Aria's hybrid memory model is its second defining pillar, differentiating between wild (manual), gc (managed), and stack allocation. The goal is to provide Rust-like safety with C-like flexibility.
3.1. The Missing Borrow Checker and Appendage Theory
The specification outlines an "Appendage Theory" for memory safety, enforcing strict rules on how references to different memory types can interact. The central tenet is the inequality Depth(Owner) <= Depth(Reference), ensuring that references do not outlive the data they point to.1
Current Implementation Status:
* Tracking Infrastructure: The CodeGenContext successfully tracks scope_depth and maintains a scopedVariables map.1 It differentiates between allocation strategies (AllocStrategy::WILD, AllocStrategy::GC, AllocStrategy::STACK).1
* Metadata Availability: The ScopedVariable struct includes a requires_drop flag, indicating specifically which variables need explicit cleanup (mostly WILD allocations).1
The Enforcement Gap:
The critical piece—the BorrowCheckVisitor—is missing.1 This is a static analysis pass that runs before code generation.
* Hypothetical Failure: Without this visitor, the compiler knows the depth and type of every variable but lacks the logic to reject invalid code. A user could write a function that returns a reference to a stack-allocated variable. The CodeGenContext would generate valid LLVM IR for this (returning a pointer to a stack slot), but the program would segfault at runtime when the stack frame is popped.
* Wild Memory Leaks: Similarly, the requires_drop flag is present, but there is no pass to verify that every wild allocation has a corresponding defer aria.free() or explicit free. The safety guarantees regarding memory leaks are currently unenforceable.
3.2. Pointer Semantics and Syntax Conflict
The specification is emphatic about the distinction between NIL (Aria "absence of value"), NULL (C-style 0x0 address), and void (FFI return type only).1 Furthermore, Aria specifies a distinct pointer syntax using the @ sigil (e.g., int64@ for a pointer to an int64) to avoid the ambiguity of the C-style * which overlaps with multiplication.1
Conflict Identified:
The audit of the source code reveals a "Syntax Debt." The C++ backend implementation occasionally uses standard C-style pointer notation in its internal logic or variable naming conventions, while the Spec mandates @.1
* Lexer Confusion: The aria_specs.txt examples show int64@:ptr = @value;.1 However, legacy parts of the parser or older documentation snippets in aria_analysis_todo.txt imply a struggle to enforce this shift.
* Resolution Requirement: The compiler frontend (Lexer/Parser) must enforce a strict ban on * for pointer declaration in native Aria code, reserving * exclusively for extern blocks where C compatibility is required. The CodeGenContext needs to handle the type resolution of Type@ distinct from Type* if they are to be treated differently (e.g., Type@ might imply a non-null constraint in future versions).
3.3. Garbage Collection Integration and Pinning
The source code reveals a "Shadow Stack" implementation (shadowStack, shadowStackTop) in CodeGenContext.1 This is a classic technique for precise or conservative garbage collection in environments like LLVM where stack maps are difficult to control.
Analysis of Pinning (#):
The specification introduces the # operator to "pin" a GC-managed object in memory, returning a fixed address (wild pointer) that is safe to pass to C functions or use in pointer arithmetic.1
* Implementation Evidence: The CodeGenContext contains a pinnedVariables map, tracking variables that have been pinned.1
* Missing Logic: While the map exists, the logic to prevent the GC from moving these objects is tied to the runtime GC implementation, which is only partially described. The backend must emit specific intrinsics (e.g., llvm.gcroot or custom calls to aria_gc_pin) when the # operator is encountered. The current code snippets do not show the emission of these specific pinning intrinsics, only the tracking structure.
* Safety Risk: If pinning is not correctly implemented in the backend, the GC might move an object while a wild pointer refers to its old location, leading to use-after-free bugs.
4. Runtime and Ecosystem Architecture
The "Batteries Included" promise of Aria relies on a robust runtime and standard library. The analysis exposes significant gaps that prevent the language from hosting its own tooling, creating a "Bootstrapping Paradox."
4.1. The Bootstrapping Paradox: Globbing

The build system (aria_make) is specified to use glob patterns (e.g., src/**/*.aria) to locate source files.1 This is a fundamental requirement for any build tool. However, the Aria standard library (std.fs) lacks a native readdir or glob implementation.1
Current Resolution & Conflict:
The project currently relies on a host-side C++ implementation of the globbing engine (aglob, glob_engine.hpp).1 This engine is sophisticated, providing deterministic, sorted path resolution to ensure reproducible builds.1
* The Paradox: aria_make is supposed to be an Aria program (or at least defined by Aria configs), but it depends on functionality that only exists in the C++ host compiler.
* Strategic Task: To break this dependency and allow Aria to self-host its build system, this C++ engine must be exposed to the Aria runtime via the Foreign Function Interface (FFI). The std.fs.glob module must wrap aria_glob_match.1 The report finds that while the C++ engine is "FFI-safe" (using integer error codes), the bridge code to actually call it from Aria is missing.
4.2. Hex-Stream Topology and Process Management
Aria specifies a unique "Hex-Stream" I/O topology for processes. Standard Unix processes use three File Descriptors (FD 0: stdin, 1: stdout, 2: stderr). Aria expands this to six:
* FD 3 (stddbg): Structured telemetry and debug logs.1
* FD 4 (stddati): Dedicated binary input stream.
* FD 5 (stddato): Dedicated binary output stream.
Implementation Status:
The process.cpp file 1 contains the beginnings of process management, but the audit flags a lack of "bootstrap code" to correctly set up these extra file descriptors during process spawning.1
* Linux Requirements: Spawning a child with 6 FDs requires specific usage of pipe2 (to create the pipes) and dup2 (to map them to the correct FDs in the child process after fork() but before exec()).1
* Windows Requirements: Windows CreateProcess does not natively support the concept of FDs 3, 4, 5 in the same way. It requires STARTUPINFOEX and handle inheritance lists.1
* Missing Logic: The current runtime lacks the complex cross-platform logic to negotiate this topology. Without it, tools like aria_shell and aria_make cannot separate build output (stdout) from build telemetry (stddbg) or object file transfer (stddato), crippling the envisioned tooling architecture.
4.3. Standard Library Bridges and "Acquire-Convert-Release"
The audit of stdlib_review_and_todo.txt highlights the need for a specific design pattern called "Acquire-Convert-Release" for bridging C++ runtime primitives to Aria.1
The Containment Field Pattern:
Since the C++ runtime returns "wild" pointers (standard malloc), and Aria prefers GC-managed memory, the standard library must act as a "containment field."
1. Acquire: Call C++ runtime (e.g., aria_read_file) -> receive wild byte*.
2. Convert: Allocate a GC string in Aria -> copy data from wild byte*.
3. Release: Immediately free the wild byte* via aria_free.
4. Return: Pass the safe GC string to the user.
Zero-Copy Implications:
The audit notes that for high-performance systems programming, this copying overhead is unacceptable. Therefore, the standard library must offer dual interfaces:
* std.fs.read_file(path) -> Returns string (Safe, Copied).
* std.fs.read_file_wild(path) -> Returns wild byte@ (Unsafe, Zero-Copy).1
This dual-pathing is currently unimplemented, leading to a library that is either too slow (always copies) or too unsafe (exposes raw pointers by default).
4.4. Collections and Hashing
The std.collections module faces a challenge in bridging Aria's monomorphized generics with the runtime's likely type-erased hash map implementation.1
   * The Shim Problem: The runtime map.cpp likely operates on void* keys and values. Aria's HashMap<K, V> is a generic struct.
   * Requirement: The compiler must generate "shim" functions for every HashMap<K,V> instantiation that calculate the size of K and V and pass them to the runtime. Furthermore, hashing logic must be consistent. If Aria strings are fat pointers (ptr + len), the runtime hash function must know how to hash that structure, or the standard library must pass the raw data pointer.
5. Tooling Infrastructure: Make and Shell
The tooling layer sits atop the runtime and is currently in a conceptual phase with significant gaps in implementation.
5.1. AriaBuild (aria_make)
The build system requires a configuration parser for the "ABC" (Aria Build Configuration) format.
   * Parser Status: The audit suggests a lightweight, recursive-descent parser is needed, supporting {} blocks for maps, `` for lists, and interpolation syntax &{VAR}.1 This parser does not currently exist.
   * Dependency Engine: aria_make requires a Directed Acyclic Graph (DAG) engine to schedule builds. This engine must implement Kahn’s Algorithm for topological sorting to correctly order dependencies and identify parallelizable compilation tasks.1 Currently, there is no logic to detect cycles (e.g., A -> B -> A) or manage the build graph.
5.2. AriaShell (aria_shell)
The shell is intended to be a REPL and process orchestrator.
   * Host-Side TBB Evaluation: A critical gap is "Host-Side TBB Evaluation".1 The shell needs to evaluate Aria expressions (like 1 + 2) interactively. Since the JIT compiler might be too heavy for simple expressions, or bootstrapping might be incomplete, the shell needs a C++ implementation of TBB arithmetic (a HostTBB8 class) to replicate the language's semantics immediately.
   * Job Control: A JobManager subsystem is required to handle process groups, backgrounding (&), and signal propagation (SIGINT, SIGTSTP). This is a complex state machine that interacts directly with the OS kernel, entirely separate from the language runtime.1

7. Conclusion
The audit confirms that while Aria's specification is ambitious and robust, the current v0.0.7 implementation is structurally sound but semantically hollow. The reliance on standard LLVM integers for TBB types and the absence of the Borrow Checker constitute the highest-severity technical debts. Addressing the Master Research Task List in the prescribed order—securing semantics first, then memory safety, then runtime capabilities—is the only viable path to a functional v0.1.0 release. The "Bootstrapping Paradox" regarding globbing and build tools is a critical dependency that must be resolved via FFI before the ecosystem can become self-sustaining. The implementation of the Hex-Stream I/O topology will define Aria's unique value in the systems programming landscape, but it requires immediate attention to the low-level process bootstrapping logic.
