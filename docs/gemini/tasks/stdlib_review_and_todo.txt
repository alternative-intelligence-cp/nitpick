Architectural Gap Analysis and Implementation Strategy for the Aria Standard Library v0.1.0
1. Executive Overview and Ecosystem State Assessment
The Aria programming language project stands at a pivotal juncture in its development lifecycle. A rigorous audit of the provided source code repositories—specifically the compiler infrastructure (src/backend, src/frontend), the runtime environment (src/runtime), and the existing standard library definitions (lib/std, stdlib)—reveals a fundamental dichotomy in the system's maturity. The architecture exhibits a condition best described as "Runtime-Rich, Interface-Poor." The underlying C++ runtime infrastructure (libaria_runtime.a) has achieved a high degree of functional completeness, boasting robust implementations for cross-platform file I/O, hash map data structures, mathematical operations, and the language's signature six-stream I/O topology. Conversely, the Aria-level standard library—the user-facing layer intended to expose these capabilities to developers—remains nascent, characterized by rudimentary wrappers and significant functional voids.
This report serves as a comprehensive architectural audit and a strategic implementation roadmap. Its primary objective is to define the specific engineering tasks required to bridge the gap between the C++ runtime's capabilities and the Aria language's public interface. By synthesizing the research directives found in the documentation with the reality of the codebase, we identify that the critical path for the v0.1.0 release is not the invention of new features, but the systematic construction of the Aria Standard Library Bridge. This involves creating high-level, type-safe Aria modules (e.g., std.fs, std.collections, std.math, std.process) that wrap existing C++ runtime primitives using Aria's advanced generic system, Twisted Balanced Binary (TBB) arithmetic, and hybrid memory model.
The analysis is structured to provide deep technical insight into the distinct architectural pillars of Aria: the integration of Wild and Garbage Collected memory, the propagation of TBB sticky errors, and the mechanics of the Hex-Stream I/O topology. It culminates in a prioritized todo list and specific, actionable prompts designed to guide an AI assistant in generating the missing production-grade code.

2. Architectural Analysis of Core System Primitives
To accurately prescribe the missing standard library components, one must first deconstruct the capabilities and constraints of the underlying compiler and runtime architecture. The source code reveals a sophisticated system that demands a standard library far more complex than simple C bindings.
2.1 The Hybrid Memory Model: Bridging "Wild" and "GC" Universes
The most distinct feature of the Aria architecture, as evidenced by the codegen_context.h and core.aria source files, is its bifurcated memory model. The compiler explicitly tracks allocation strategies for every variable, distinguishing between GC (Garbage Collected), WILD (Manual/Arena), and STACK memory. This design choice has profound implications for the standard library, which serves as the primary interface for resource acquisition.
The GC universe is managed by a mark-and-sweep collector, referenced in the compiler's backend via shadow stack implementations for root tracking. This memory is safe, automatic, and intended for high-level business logic. In contrast, the WILD universe, backed by the mimalloc allocator, offers raw performance and manual control but demands explicit deallocation. The standard library acts as the "containment field" for this wild memory. For instance, the runtime function aria_read_file returns a raw buffer (void* or AriaString*) allocated in wild memory. A naïve wrapper would simply return this pointer to the user, effectively leaking memory if the user fails to free it. A robust standard library implementation must encapsulate this wild resource within a managed type, copying the data into a GC-allocated string or wrapping the wild pointer in a struct with a destructor-like defer pattern (though Aria lacks automatic destructors, relying on defer blocks).
Furthermore, the standard library must support "Zero-Copy" operations for high-performance systems programming. This requires dual interfaces for many operations: a safe, copying interface (e.g., readFile) that returns a managed string, and an unsafe, zero-copy interface (e.g., readFileWild) that returns a wild pointer. The absence of these dual paths in the current lib/std is a significant gap. The core.aria file demonstrates the capability to allocate wild memory (wild *T@:ptr = std.mem.alloc(...)), but without standard patterns for ownership transfer, the ecosystem invites memory leaks. The implementation of std.mem must therefore provide not just allocators, but also safety primitives—smart pointer-like wrappers that leverage Aria's unique reference capabilities ($) to lend wild memory safely to GC contexts.
2.2 Generic Monomorphization and the Type Erasure Gap
The compiler's handling of generics, as verified by generics_main.aria, utilizes a monomorphization strategy similar to C++ templates or Rust generics. When a function func<T> is called with int32, the compiler generates a specialized function symbol (e.g., identity_int32). This is a powerful feature for performance, but it creates a complex bridging problem for the standard library, particularly for collections.
The runtime implementation of the Hash Map (src/runtime/collections/map.cpp) is type-erased. It operates on void* pointers and requires explicit size_t arguments for key and value sizes. It has no concept of Aria's type system. The standard library's std.collections module must bridge this gap. Because Aria generics are monomorphized, the wrapper code has access to the concrete types at compile time. However, the current standard library lacks the "shim" logic to translate these concrete Aria types into the void* interface expected by the runtime.
For example, a Map<string, int32> needs to pass string keys by value (deep copy or reference count) and integers by value. The C++ runtime expects pointers to both. The Aria wrapper must handle this impedance mismatch, potentially by creating temporary allocations for primitive types or by passing addresses directly. The absence of this generic bridging layer means that despite having a robust hash map in the runtime, Aria developers currently have no access to associative arrays. This is a critical deficiency for any non-trivial application development.
2.3 Twisted Balanced Binary (TBB) Semantics and Error Propagation
The codegen_context.h file reveals explicit mapping for tbb8 through tbb64 types. Unlike standard two's complement integers, TBB types reserve their minimum value (e.g., -128 for tbb8) as an ERR sentinel. The compiler enforces "sticky error" propagation, where any operation involving an ERR value results in ERR.
This unique semantic imposes strict requirements on the standard library, particularly for I/O and mathematical operations. A standard read_byte function in C might return an int to allow for -1 (EOF). In Aria, returning a tbb8 is dangerous if the byte 0x80 (unsigned 128) is valid data, as it interprets to -128 (ERR). The standard library must be meticulously designed to avoid this ambiguity. I/O functions interacting with binary data should likely return standard uint8 or int16 types, while math functions utilizing TBB must explicitly handle the sentinel.
The current stdlib/math/ternary.aria provides trit operations but ignores the broader TBB ecosystem. The missing std.math module must provide wrappers that respect this error propagation, perhaps by returning result<tbb> types or by providing explicit is_err() checks before passing values to the raw C++ runtime functions, which may not be TBB-aware. The disconnect between the TBB-aware compiler and the potentially TBB-ignorant C++ runtime functions (like standard <cmath> wrappers) represents a significant potential for subtle bugs if not addressed in the Aria wrapper layer.

3. Deep Dive Gap Analysis: The Missing Modules
This section provides a granular analysis of the specific modules missing from the standard library, contrasting the existing runtime support with the void in the Aria-level interface. This analysis directly informs the "Todo List" and prompts generated in the final section of this report.
3.1 The File System Void (std.fs)
Runtime Reality:
The C++ runtime (src/runtime/io/io.cpp) is surprisingly complete. It implements aria_read_file, aria_write_file, aria_file_exists, aria_delete_file, and streaming operations like aria_stream_read_line. These functions handle the platform-specific complexity of POSIX and Windows file APIs. They typically return an AriaResult* struct, which encapsulates either a successful value or an error message.
Library Gap:
Despite this rich runtime support, lib/std/io.aria is skeletal. It defines wrappers for stdout and stderr but provides zero capability to open, read, or manipulate files on disk. A developer cannot write a simple script to read a config file or save a log.
Required Implementation:
We must implement std.fs. This module must define the extern signatures for the C++ functions. Crucially, it must handle the AriaResult* lifecycle. The C++ runtime allocates this result on the heap. The Aria wrapper must:
1. Call the extern function.
2. Check the err field.
3. If error: Extract the message, free the C++ result, and return an Aria result::err.
4. If success: Extract the data, potentially copy it to a managed Aria string, free the C++ result, and return result::ok.
Additionally, the File struct needs to be defined as an RAII wrapper around the AriaStream* handle. It should expose methods for read, write, seek, and close. The absence of a std.path module for joining and manipulating file paths is also a blocker for cross-platform file system work.
3.2 The Collections Crisis (std.collections)
Runtime Reality:
The src/runtime/collections/map.cpp file contains a production-ready hash map implementation using FNV-1a hashing and open addressing (linear probing). It supports dynamic resizing and basic CRUD operations. It is designed as a generic C container, using void* for data storage.
Library Gap:
stdlib/core.aria defines a dynamic array<T>, but there is no Map or Set. This is the single largest blocker for application logic. The challenge is not implementing the map algorithm (which is done) but bridging the type system.
Required Implementation:
We need std.collections. The Map<K,V> struct must be a generic type that holds a wild void@ handle to the underlying C++ map. The methods (put, get, remove) must perform the necessary unsafe casts to interact with the runtime.
A specific challenge identified is hashing. The C++ runtime likely hashes the raw bytes of the key. For primitive keys (int, tbb), this works fine. For complex keys (string, struct), the std.collections module must ensure that the pointer to the data is passed correctly, or that the C++ runtime is aware of Aria string layouts. Given the "batteries-included" requirement, the wrapper should likely enforce that keys are either primitives or strings, or require a Hashable constraint (if Aria supports traits/interfaces fully) to provide a custom hash.
3.3 The Math & Logic Deficit (std.math)
Runtime Reality:
src/runtime/math/math.cpp provides wrappers for the standard C math library: sin, cos, exp, log, etc. These operate on standard double types.
Library Gap:
There is no Aria module exposing these. stdlib/math/ternary.aria exists but is niche. General floating-point math is inaccessible.
Required Implementation:
A std.math module is required to simply wrap these calls. However, it should also introduce the constants PI, E, etc., potentially as const globals. Furthermore, standard integer math functions like abs, min, max, and clamp should be implemented as generics in Aria directly, leveraging the monomorphizer to generate optimal code for int32, flt64, etc., without crossing the FFI boundary for simple comparisons.
3.4 The Concurrency Disconnect (std.async & std.process)
Runtime Reality:
The research files (ASYNC_MODEL.txt) describe a sophisticated M:N scheduler and a Future<T> based async system. The codegen.cpp handles coroutine intrinsics. io.cpp (and references to aria_spawn) suggests support for spawning processes with the unique 6-stream topology.
Library Gap:
There are no Aria types for Future, Task, or Channel. There is no API to spawn a process or communicate with it via the 6 streams. The kernel-level support for FDs 3, 4, and 5 (stddbg, stddati, stddato) is useless without a spawn function that correctly maps them.
Required Implementation:
* std.async: Must define the Future<T> struct layout expected by the compiler's coroutine lowering. It must provide the waker mechanism and the await intrinsic wrappers (if not handled entirely by syntax).
* std.process: Must wrap aria_spawn. This function needs a complex configuration struct to specify how the 6 streams are handled (inherit, pipe, or close) for the child process. This is the implementation of the "Hex-Stream Topology" described in the project goals.

4. The Memory Model Integration Strategy
The successful implementation of the standard library hinges on a consistent strategy for managing the boundary between Aria's hybrid memory model and the C++ runtime.
4.1 The "Containment Field" Pattern
The standard library must act as a containment field for wild memory. The C++ runtime functions are inherently "wild"—they allocate memory using malloc (or mimalloc) that is not tracked by the Aria GC. If a user calls aria_read_file directly, they receive a pointer they must manually free. This violates the safety guarantees expected of a high-level language.
The proposed strategy for all standard library modules is the Acquire-Convert-Release pattern:
1. Acquire (Wild): The wrapper calls the C++ runtime function. It receives a wild pointer (e.g., AriaString* or AriaResult*).
2. Convert (Managed): The wrapper allocates a new GC-managed object (e.g., string or result<T>) and copies the data from the wild pointer into this safe container. For strings, this implies a memory copy. For structs, a field-wise copy.
3. Release (Wild): The wrapper immediately calls aria_free or the specific destructor (aria_result_free) on the wild pointer.
4. Return (Safe): The wrapper returns the GC-managed object to the user.
This pattern ensures that memory leaks are impossible in user code, at the cost of a copy operation. For performance-critical sections, a secondary API (suffixed with _wild, e.g., read_file_wild) should be exposed that skips steps 2 and 3, returning the raw pointer to the user with the expectation of manual management.
4.2 Handling TBB "Sticky Errors" in FFI
When passing data between Aria (TBB-aware) and C++ (TBB-ignorant), the standard library must explicitly handle the ERR sentinel. The C++ math functions, for example, will happily process -128 as a valid integer.
* Input Sanitization: Aria wrappers for math functions taking tbb types must check for ERR inputs. if (val == ERR) return ERR; This simulates the sticky error propagation in the library layer that the compiler enforces in the language layer.
* Output Validation: If a C++ function returns a value that coincides with an Aria ERR sentinel (unlikely for 64-bit, but possible for 8-bit types), the wrapper must decide whether to map this to a generic error result or pass it through.

6. Conclusion and Strategic Recommendation
The Aria ecosystem is currently in a state of latent potential. The hard engineering problems—the compiler backend, the hybrid memory allocator, the hex-stream kernel integration—are largely solved and implemented in C++. However, these capabilities remain locked away from the Aria developer due to the emptiness of the lib/std directory.
The "Standard Library Gap" is the single defining obstacle to the v0.1.0 release. It is not a research problem; it is an integration problem. By executing the five tasks outlined above, we will effectively "plug in" the runtime to the language.
Recommendation: Proceed immediately with Task 1 (File System) and Task 2 (Collections). These are the foundational dependencies for all higher-level application logic, including the build system (abuild) and package manager (apk). Without them, self-hosting is impossible. The accompanying prompts are engineered to generate code that respects Aria's unique memory and type safety constraints, ensuring a robust and idiomatic standard library.

2. Execute Prompt 2 to generate lib/std/collections.aria.
3. Execute Prompt 3 to generate lib/std/path.aria.
4. Execute Prompt 4 to generate lib/std/process.aria.
5. Execute Prompt 5 to generate lib/std/math.aria.
6. Compile & verify the new modules against libaria_runtime.a.
This roadmap transforms Aria from a compiler prototype into a usable systems language.
