## EXISTING BASELINE

**IMPORTANT CLARIFICATION:**

Our SegFit allocator (`/educational/allocators/segfit_advanced/`) is a **segregated-fit allocator**, NOT a pool or slab allocator:

- **SegFit characteristics:**
  - 15 size classes (8B - 65KB)
  - General-purpose allocation (any size within class)
  - Doubly-linked free lists per class
  - Performance: 56ns allocation, 245ns deallocation
  - Use case: General-purpose memory management

- **Pool allocator characteristics (different):**
  - Single size class (fixed-size objects only)
  - Optimized for specific object types (e.g., struct Node)
  - Simpler data structures (bitmap or stack)
  - Performance: <20ns allocation (faster than SegFit)
  - Use case: High-frequency same-type allocations

- **Slab allocator characteristics (different):**
  - Object caching (reuse initialized objects)
  - Constructor/destructor optimization
  - Cache coloring (reduce false sharing)
  - Performance: <30ns with object reuse
  - Use case: Kernel object management (inodes, dentries)

**Research should focus on:** Pool/slab-specific optimizations NOT present in SegFit.

   - What problem does slab solve (object initialization overhead)?
   - What is object caching?

2. **Terminology:**

   - What is a cache (collection of slabs for one object type)?

   - What are full/partial/empty slabs?

3. **Architecture:**
   - What data structures track slabs?
   - How are slabs organized (linked lists: full, partial, empty)?
   - What is slab coloring (cache line offset to reduce conflicts)?
   - How does slab interact with kernel page allocator?

4. **Object Caching:**
   - How does object caching work (reuse initialized objects)?
   - How are constructors/destructors optimized?
   - Example: Allocating struct inode (Linux kernel)
     - First allocation: Call constructor, initialize fields
     - Deallocation: Keep object initialized, return to cache
     - Next allocation: Reuse cached object (skip constructor)
   - Performance benefit: Avoid repeated initialization

5. **Allocation Algorithm:**
   - What is the allocation algorithm (pseudocode)?
   - How is a slab selected (partial → empty → allocate new)?
   - How is a free object found within a slab?
   - What is the deallocation algorithm?

6. **Slab Coloring:**
   - What is slab coloring?
   - How does coloring reduce cache conflicts?
   - Example: Offset objects by cache line size
   - Performance benefit: Reduce false sharing

7. **Performance:**
   - What performance benchmarks exist for slab allocators?
   - How does slab compare to malloc (ns per allocation)?
   - Target: <30ns allocation with object reuse
   - What is the memory overhead (slab headers, alignment)?

8. **Linux Kernel Implementation:**
   - What is SLUB (Simple Linux Unqueued Allocator)?
   - How does SLUB differ from original SLAB?
   - What optimizations does SLUB provide?
   - Code analysis: `mm/slub.c` in Linux kernel

5. **Aria Allocator Interface Recommendation:**
   - What interface should Aria use?
   - Trait-like or protocol-based?
   - Example proposed API:
     ```aria
     interface Allocator {
         fn alloc(size: uint64, align: uint64) -> wild void@;
         fn free(ptr: wild void@) -> void;
         fn realloc(ptr: wild void@, new_size: uint64) -> wild void@;
     }
     ```
   - How should type-safe wrappers work?
     ```aria
     fn alloc<T>(allocator: Allocator) -> wild T@ {
         ptr := allocator.alloc(@sizeof(T), @alignof(T));
         return @cast<wild T@>(ptr);
     }
     ```
