# RESEARCH TASK: Arena Allocator Block Chaining Patterns

**Task ID:** alloc_001_arena_allocator_patterns
**Priority:** HIGH
**Category:** Memory Management - Allocators
**Status:** PENDING
**Created:** 2025-12-21
**Estimated Complexity:** Medium

---

## PROBLEM STATEMENT

Research arena/bump allocator patterns from Rust bumpalo, Zig ArenaAllocator, and modern C++ implementations to design Aria's arena allocator with multi-block chaining, memory reuse capabilities, and type-safe API integration. This is for Aria Phase 4.2 Stage 1.2 (Arena Allocators).

**Critical Gap:** We have bump allocator baseline (Stabilized: 1.6-45ns) but need arena-specific multi-block chaining patterns and reset-with-reuse mechanisms not present in our existing C implementations.

---

## SCOPE

### Arena Fundamentals
- Core concepts: arena/region/bump allocator definition
- Use cases: compiler phases, web request processing, temporary computations
- Performance characteristics: allocation speed, memory overhead, cache locality
- Trade-offs: bulk deallocation vs individual free, simplicity vs flexibility

### Block Chaining
- Multi-block arena growth strategies (fixed-size blocks, exponential growth, hybrid)
- Linked list vs array-based block tracking
- Block allocation triggers (overflow handling)
- Large allocation handling (allocations > block_size)

### Memory Reuse
- Arena reset() patterns that preserve allocated blocks for reuse
- Difference between reset() and destroy()
- Zero-copy memory reuse strategies
- Performance implications of reset vs destroy+recreate

### Type Safety
- Type-safe arena API design leveraging language features
- Integration with generics/templates
- Alignment handling for mixed-type allocations
- Lifetime/scope safety (Rust lifetimes, Aria defer)

## EXISTING BASELINE

We have implemented the following allocators in `/educational/allocators/`:

1. **Stabilized Bump Allocator**:
   - Performance: 1.6-45ns allocation
   - Architecture: Pure bump pointer (no chaining)
   - Limitation: Single pool, no memory reuse
   - File: `educational/allocators/stabilized/sm_consciousness_stabilized.c`

2. **SegFit Segregated Allocator**:
   - Performance: 56ns allocation
   - Architecture: 15 size classes, doubly-linked free lists
   - Block chaining: Demonstrated for size class chunks
   - File: `educational/allocators/segfit_advanced/segfit_consciousness_allocator.h`

3. **Wild Allocator (TBB)**:
   - Performance: ~100ns allocation
   - Architecture: Individual malloc/free with TBB sticky errors
   - File: `src/runtime/allocators/wild_alloc.cpp`

**What we need:** Arena-specific multi-block chaining (missing from Stabilized) + reset-with-reuse (missing from all).

## KEY QUESTIONS

### Rust bumpalo Crate Analysis

1. **Multi-Block Chaining:**
   - How does bumpalo implement chunk overflow (when allocation doesn't fit in current chunk)?
   - What data structure tracks multiple chunks (linked list, Vec)?
   - What is the initial chunk size and growth policy?
   - How does bumpalo decide when to allocate a new chunk vs reuse existing?

2. **Allocation Algorithm:**
   - What is the exact allocation algorithm (pseudocode or Rust code)?
   - How does bumpalo handle alignment (align_to, ptr.align_offset)?
   - What is the overhead per allocation (just alignment padding)?
   - How does bumpalo achieve zero-cost abstractions?

3. **Memory Reuse:**
   - What is bumpalo's reset() behavior (does it exist)?
   - How does reset() differ from drop (destructor)?
   - Can bumpalo reuse allocated chunks after reset?
   - What is the performance of reset vs destroy+recreate?

4. **Type Safety & Lifetimes:**
   - How does bumpalo integrate with Rust's lifetime system?
   - How does bumpalo ensure allocated references don't outlive arena?
   - What is bumpalo's API for type-safe allocation (Bump::alloc<T>)?
   - How does bumpalo handle Copy vs non-Copy types?

5. **Large Allocations:**
   - How does bumpalo handle allocations larger than chunk size?
   - Does bumpalo allocate oversized chunks or reject?
   - What is the performance impact of large allocations?

6. **Performance:**
   - What benchmark results exist for bumpalo (ns per allocation)?
   - How does bumpalo compare to malloc in real-world scenarios?
   - What is the memory overhead (wasted space in chunks)?

### Aria Integration Questions

1. **Type-Safe API:**
   - How should `Arena.alloc<T>()` provide type safety in Aria?
   - What should the function signature be?
   - How should generics interact with arena allocation?
   - Example: `fn Arena.alloc<T>(arena: wild Arena@) -> wild T@`

2. **Defer Integration:**
   - How can Aria's `defer` integrate with arena cleanup?
   - Example: `defer Arena.destroy(arena);`
   - Should `Arena` implement a destructor-like mechanism?

3. **Reset API:**
   - What should `Arena.reset()` provide?
   - Should reset preserve allocated blocks for reuse?
   - What is the expected behavior: `Arena.reset() -> void`
   - Performance target: O(1) reset time

4. **Alignment Handling:**
   - How should Arena handle `@alignof(T)` for mixed types?
   - What is the alignment algorithm (align_forward bitwise AND)?
   - Example: Allocating uint8, uint32, uint64, vec3 in sequence
   - Expected behavior: Proper alignment for each type

5. **Statistics API:**
   - What statistics should Arena provide?
   - Candidates: `Arena.bytes_allocated()`, `Arena.bytes_reserved()`, `Arena.block_count()`
   - Should statistics be optional (compile-time flag)?

6. **Large Allocation Handling:**
   - How should Arena handle allocations > block_size?
   - Options: (a) Error, (b) Allocate oversized block, (c) Fallback to malloc
   - Recommended approach for Aria?

7. **Validation:**
   - What validation mechanisms should Arena provide?
   - Magic numbers for corruption detection (following SegFit pattern)?
   - Example: `#define ARIA_ARENA_MAGIC 0x4152454E  // "AREN"`

8. **Wild Pointer Integration:**
   - How should Arena integrate with wild pointer semantics?
   - Arena allocations are wild (not GC-tracked), confirm behavior
   - Can wild pointers from arena be converted to GC pointers?
   - Lifetime rules: pointers invalid after arena destroyed
