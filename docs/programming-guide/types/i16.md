# i16 (16-bit Signed Integer)

The 16-bit signed integer type for medium-range signed values.

## Overview

`i16` is a 16-bit signed integer type that can represent values from -32,768 to +32,767. It's a medium-sized signed integer using two's complement representation with wrapping overflow behavior.

**Use `i16` when:**
- You need signed values (can be negative)
- Range of -32,768 to +32,767 is sufficient
- Values exceed i8 range (-128 to +127)
- Audio processing (16-bit PCM standard)
- Geographic coordinates, sensor data, temperature sensors
- C interoperability with `int16_t` or `short`

**Don't use `i16` when:**
- Values are always positive (use `u16` instead)
- Range might exceed -32,768 to +32,767 (use `i32`)
- Values fit in i8 range (use smaller `i8` instead)
- Overflow detection is critical (use TBB types)
- You need symmetric range for `abs()` (use `tbb16`)

**Key characteristics:**
- Range: -32,768 to +32,767 (65,536 total values)
- Size: 2 bytes (16 bits)
- Overflow: Wraps around (32,767 + 1 = -32,768)
- Underflow: Wraps around (-32,768 - 1 = 32,767)
- Sign: Two's complement (MSB is sign bit)
- Performance: Native machine instructions, zero overhead

## i16 vs i8

Choose between `i16` and `i8` based on the range you need:

```aria
// i8: Small range (-128 to +127)
offset: i8 = -50;  // ✅ Fits in i8

// i16: Medium range (-32,768 to +32,767)
altitude: i16 = -1000;  // ❌ Doesn't fit in i8, need i16
```

**Key differences:**

| Feature | i8 | i16 |
|---------|----|----|
| **Range** | -128 to +127 | -32,768 to +32,767 |
| **Size** | 1 byte | 2 bytes (2× larger) |
| **Values** | 256 | 65,536 (256× more) |
| **Use for** | Tiny signed values | Medium signed values |

**When to use i16:**
```aria
// Altitude in meters (-1000 to +10,000)
altitude: i16 = -500;  // ✅ i16 needed

// Audio sample (16-bit PCM: -32,768 to +32,767)
audio_sample: i16 = -15000;  // ✅ Standard audio format

// Temperature sensor (extended precision)
temp_celsius: i16 = -273;  // ✅ Absolute zero in Celsius

// Screen coordinate (1920×1080 HD)
screen_x: i16 = 1920;  // ✅ Fits in i16
screen_y: i16 = 1080;
```

**When to use i8:**
```aria
// Small temperature offset
temp_delta: i8 = -15;  // ✅ i8 is enough

// Small coordinate offset
dx: i8 = -20;  // ✅ Within -128 to +127
dy: i8 = 50;
```

## i16 vs u16

Choose between signed (`i16`) and unsigned (`u16`) based on whether values can be negative:

```aria
// i16: Can be negative (signed)
altitude: i16 = -1000;  // Below sea level

// u16: Always positive (unsigned)
port: u16 = 8080;  // Network ports can't be negative
```

**Key differences:**

| Feature | i16 | u16 |
|---------|----|----|
| **Range** | -32,768 to +32,767 | 0 to 65,535 |
| **Can be negative** | ✅ Yes | ❌ No |
| **Maximum positive** | 32,767 | 65,535 (2× larger) |
| **Use for** | Signed data, coordinates | Ports, counts, dimensions |
| **Overflow** | 32,767 + 1 = -32,768 | 65,535 + 1 = 0 |
| **Underflow** | -32,768 - 1 = 32,767 | 0 - 1 = 65,535 |

**When to use i16:**
```aria
// Temperature (can be negative)
temp_celsius: i16 = -40;  // ✅ Makes sense

// Altitude (can be negative - below sea level)
altitude: i16 = -400;  // Dead Sea (-430m)

// Coordinate offset (can be negative)
dx: i16 = -500;  // Move 500 units left
dy: i16 = 300;   // Move 300 units up

// Audio sample (signed PCM)
audio: i16 = -15000;  // ✅ 16-bit audio is signed
```

**When to use u16:**
```aria
// Network port (never negative)
port: u16 = 8080;  // ✅ u16 is correct (0-65535)

// Image dimensions (never negative)
width: u16 = 1920;   // ✅ u16 for dimensions
height: u16 = 1080;

// Count (never negative)
count: u16 = 5000;  // ✅ u16 for counts
```

**Conversion gotcha:**
```aria
signed: i16 = -1;
unsigned: u16 = signed as u16;  // 65,535! (bit pattern reinterpreted)

// -1 = 0xFFFF (i16)
//    = 65,535 (u16, same bit pattern)
```

## i16 vs i32

Choose between `i16` and `i32` based on range and memory needs:

```aria
// i16: Medium range (-32,768 to +32,767)
screen_y: i16 = 1080;  // ✅ i16 is enough

// i32: Large range (±2.1 billion)
population: i32 = 8000000;  // ❌ Beyond i16 range
```

**Key differences:**

| Feature | i16 | i32 |
|---------|----|----|
| **Range** | -32,768 to +32,767 | -2.1B to +2.1B |
| **Size** | 2 bytes | 4 bytes (2× larger) |
| **Default** | No | ✅ Yes (Aria's default int) |
| **Use for** | Medium signed values | General-purpose integers |

**When to use i16:**
```aria
// Array of coordinates (save memory)
coordinates: [i16; 10000];  // 20 KB (vs 40 KB with i32)

for i in 0..10000 {
    coordinates[i] = calculate_position();  // Guaranteed ±32K
}

// Audio samples (standard 16-bit PCM)
audio_buffer: [i16; 44100];  // 1 second at 44.1kHz (88.2 KB)

// Geographic coordinates (scaled)
// Latitude/longitude * 100 for 2 decimal places
lat_scaled: i16 = 12345;  // Actually 123.45°
lon_scaled: i16 = -5678;  // Actually -56.78°
```

**When to use i32:**
```aria
// General arithmetic (unknown range)
sum: i32 = 0;
for value in values {
    sum += value;  // ✅ i32 safe for accumulation
}

// Large values
pixels: i32 = 1920 * 1080;  // 2,073,600 (beyond i16!)

// Loop counter (large range)
for i: i32 in 0..100000 {  // ✅ Beyond i16 range
    process(i);
}
```

**Prefer i32 as default**, use i16 only when you know values fit and need memory efficiency:
```aria
// DEFAULT: Use i32 (Aria's default integer type)
count: i32 = 10000;  // ✅ Safe choice

// SPECIAL CASE: Use i16 when range known and memory matters
audio_samples: [i16; 1000000];  // ✅ Save 2 MB vs i32
```

## Range and Representation

### Range

`i16` can represent 65,536 distinct values:

- **Minimum:** -32,768 (-2¹⁵)
- **Maximum:** +32,767 (2¹⁵ - 1)
- **Total values:** 65,536 (2¹⁶)
- **Asymmetric:** One more negative value than positive

**Critical asymmetry:**
```aria
// 32,768 negative values: -32,768 to -1
// 32,768 positive values: 0 to +32,767

// This asymmetry matters for abs()!
min: i16 = -32768;
// abs(-32768) = 32,768, but 32,768 doesn't fit in i16!
// Calling abs(min) PANICS in Aria
```

### Two's Complement Representation

`i16` uses two's complement binary representation:

```aria
// Positive values (MSB = 0)
+32767 = 0b0111111111111111 = 0x7FFF
+1     = 0b0000000000000001 = 0x0001
 0     = 0b0000000000000000 = 0x0000

// Negative values (MSB = 1)
-1     = 0b1111111111111111 = 0xFFFF
-32768 = 0b1000000000000000 = 0x8000
```

**Sign bit (MSB):**
- 0 = positive or zero
- 1 = negative

**Example breakdown:**
```aria
value: i16 = -1000;

// Binary: 0b1111110000011000
// Hex:    0xFC18
// Sign:   1 (negative, MSB = 1)
// Value:  -1000 (two's complement)
```

### Computing Negative Values

To negate a value in two's complement:
1. Flip all bits (bitwise NOT)
2. Add 1

```aria
positive: i16 = 1000;  // 0b0000001111101000

// Manual negation:
inverted: i16 = ~positive;  // 0b1111110000010111 (bitwise NOT)
negative: i16 = inverted + 1;  // 0b1111110000011000 = -1000

// Or just:
negative: i16 = -positive;  // -1000 (compiler does the above)
```

**Verification:**
```aria
value: i16 = -1000;  // 0b1111110000011000

// Negate again to get original:
original: i16 = -value;  // 1000 (0b0000001111101000)
```

### Boundary Values

```aria
// Minimum value (most negative)
min: i16 = -32768;  // 0b1000000000000000 = 0x8000

// Maximum value (most positive)
max: i16 = 32767;   // 0b0111111111111111 = 0x7FFF

// Zero
zero: i16 = 0;    // 0b0000000000000000 = 0x0000

// -1 (all bits set)
neg_one: i16 = -1;  // 0b1111111111111111 = 0xFFFF
```

## Overflow and Underflow

### Wrapping Behavior (Defined in Aria)

Unlike C/C++ where signed overflow is undefined behavior, Aria **defines** overflow as two's complement wrapping:

```aria
// Overflow: wraps to minimum
max: i16 = 32767;
max += 1;  // max = -32768 (wraps around!)

// Underflow: wraps to maximum
min: i16 = -32768;
min -= 1;  // min = 32767 (wraps around!)
```

**This is defined behavior**, not undefined! Aria guarantees two's complement wrapping.

### Addition Overflow

```aria
a: i16 = 30000;
b: i16 = 5000;
sum: i16 = a + b;  // 35,000... but wraps to -30,536!

// Why?
// 35,000 = 0b1000100010111000
// MSB = 1, so interpreted as negative
// 0b1000100010111000 = -30,536 in two's complement
```

**Step-by-step wrapping:**
```aria
value: i16 = 32765;
value += 1;  // 32766
value += 1;  // 32767
value += 1;  // -32768 (wrapped!)
value += 1;  // -32767
value += 1;  // -32766
```

### Subtraction Underflow

```aria
a: i16 = -30000;
b: i16 = 5000;
diff: i16 = a - b;  // -35,000... but wraps to 30,536!

// Why?
// -35,000 would be 0b0111011101001000 when wrapped
// 0b0111011101001000 = 30,536 in two's complement
```

**Step-by-step wrapping:**
```aria
value: i16 = -32766;
value -= 1;  // -32767
value -= 1;  // -32768
value -= 1;  // 32767 (wrapped!)
value -= 1;  // 32766
value -= 1;  // 32765
```

### Multiplication Overflow

```aria
a: i16 = 200;
b: i16 = 200;
product: i16 = a * b;  // 40,000... but wraps to -25,536!

// Why?
// 40,000 = 0b1001110001000000
// MSB = 1, so interpreted as negative
// 0b1001110001000000 = -25,536
```

**Audio processing example:**
```aria
// Common gotcha with audio gain
audio_sample: i16 = 20000;
gain: i16 = 2;
amplified: i16 = audio_sample * gain;  // 40,000 wraps to -25,536!

// CORRECT: Widen before multiplication
wide: i32 = (audio_sample as i32) * (gain as i32);  // 40,000
// Then clamp to i16 range
if wide > 32767 {
    amplified: i16 = 32767;  // Clip
} else if wide < -32768 {
    amplified: i16 = -32768;
} else {
    amplified: i16 = wide as i16;
}
```

### Division (No Overflow)

Division doesn't overflow (quotient always fits), but has special cases:

```aria
a: i16 = 10000;
b: i16 = 3;
quotient: i16 = a / b;  // 3,333 (truncates toward zero)

a: i16 = -10000;
b: i16 = 3;
quotient: i16 = a / b;  // -3,333 (truncates toward zero)
```

**Division by zero:**
```aria
a: i16 = 10000;
b: i16 = 0;
quotient: i16 = a / b;  // PANICS! Runtime error
```

### Negation Overflow (Critical!)

```aria
min: i16 = -32768;
negated: i16 = -min;  // Should be 32,768, but 32,768 doesn't fit!
// Result: -32768 (wraps!)

// Why?
// -(-32768) = 32,768, but i16 max is 32,767
// 32,768 wraps to -32,768
```

**This is why `abs(-32768)` is dangerous:**
```aria
min: i16 = -32768;
absolute: i16 = abs(min);  // PANICS! (No representation for 32,768)

// abs() is negation for negative values:
// abs(-32768) = -(-32768) = 32,768... doesn't fit!
```

## Operations

### Arithmetic Operations

```aria
a: i16 = 1000;
b: i16 = 2000;

sum: i16 = a + b;          // 3000
difference: i16 = a - b;   // -1000
product: i16 = a * b;      // 2,000,000 wraps!
quotient: i16 = a / b;     // 0 (truncates toward zero)
remainder: i16 = a % b;    // 1000

// Compound assignment
a += 500;   // 1500
a -= 300;   // 1200
a *= 2;     // 2400
a /= 4;     // 600
a %= 100;   // 0
```

### Comparisons

```aria
a: i16 = -1000;
b: i16 = 2000;

is_equal: bool = a == b;        // false
is_not_equal: bool = a != b;    // true
is_less: bool = a < b;          // true (-1000 < 2000)
is_less_eq: bool = a <= b;      // true
is_greater: bool = a > b;       // false
is_greater_eq: bool = a >= b;   // false
```

**Signed comparison (respects sign):**
```aria
negative: i16 = -1;
positive: i16 = 32767;

result: bool = negative < positive;  // true (signed comparison)

// -1 = 0xFFFF
// 32767 = 0x7FFF
// -1 < 32767 in signed comparison
```

### Bitwise Operations

```aria
a: i16 = 0b0000001111110000;  // 1008
b: i16 = 0b0000000011111111;  // 255

and_result: i16 = a & b;   // 0b0000000011110000 = 240
or_result: i16 = a | b;    // 0b0000001111111111 = 1023
xor_result: i16 = a ^ b;   // 0b0000001100001111 = 783
not_result: i16 = ~a;      // 0b1111110000001111 = -1009

// Bit shifts
left: i16 = 1 << 10;   // 0b0000010000000000 = 1024
right: i16 = 1024 >> 2; // 0b0000000100000000 = 256
```

**Arithmetic right shift (sign extension):**
```aria
negative: i16 = -1024;  // 0b1111110000000000

shifted: i16 = negative >> 1;  // 0b1111111000000000 = -512

// MSB propagates (sign extends)
// 0b1111110000000000 >> 1 = 0b1111111000000000 (not 0b0111111000000000)
```

### Negation and Absolute Value

```aria
positive: i16 = 1000;
negative: i16 = -positive;  // -1000

value: i16 = -500;
abs_value: i16 = abs(value);  // 500
```

**Critical: abs() panics on -32768:**
```aria
min: i16 = -32768;
abs_min: i16 = abs(min);  // PANICS! (32,768 doesn't fit in i16)

// Solution 1: Check before calling abs()
if value != -32768 {
    abs_value: i16 = abs(value);  // Safe
}

// Solution 2: Use wider type
wide: i32 = value as i32;
abs_wide: i32 = abs(wide);  // abs(-32768) = 32,768 (fits in i32)

// Solution 3: Use TBB type (symmetric range)
safe: tbb16 = value as tbb16;
abs_safe: tbb16 = abs(safe);  // tbb16 min is -32767, abs always works
```

### Saturating Arithmetic

Saturating arithmetic clamps results to min/max instead of wrapping:

```aria
fn saturating_add(a: i16, b: i16) -> i16 {
    // Widen to avoid wrapping
    wide: i32 = (a as i32) + (b as i32);
    
    if wide > 32767 {
        return 32767;  // Saturate at max
    } else if wide < -32768 {
        return -32768;  // Saturate at min
    } else {
        return wide as i16;
    }
}

a: i16 = 30000;
b: i16 = 5000;
result: i16 = saturating_add(a, b);  // 32,767 (saturated, not -30,536)
```

**Audio clipping example:**
```aria
fn clip_audio(sample: i16, gain: i16) -> i16 {
    wide: i32 = (sample as i32) * (gain as i32);
    
    if wide > 32767 {
        return 32767;  // Clip to max
    } else if wide < -32768 {
        return -32768;  // Clip to min
    } else {
        return wide as i16;
    }
}

audio: i16 = 20000;
amplified: i16 = clip_audio(audio, 2);  // 32,767 (clipped)
```

### Checked Arithmetic

Checked arithmetic detects overflow and returns an error:

```aria
fn checked_add(a: i16, b: i16) -> Result<i16, string> {
    // Overflow if signs match and result sign differs
    sum: i16 = a + b;
    
    if (a > 0 && b > 0 && sum < 0) ||
       (a < 0 && b < 0 && sum > 0) {
        return Err("Overflow");
    }
    
    return Ok(sum);
}

a: i16 = 30000;
b: i16 = 5000;

match checked_add(a, b) {
    Ok(sum) => println("Sum: {}", sum),
    Err(msg) => println("Error: {}", msg),  // "Error: Overflow"
}
```

**Or use TBB types for automatic checking:**
```aria
a: tbb16 = 30000;
b: tbb16 = 5000;
sum: tbb16 = a + b;  // ERR (overflow detected automatically)

if sum == ERR {
    println("Overflow detected!");
}
```

## Type Conversions

### Widening (Always Safe)

Converting `i16` to a larger signed type is always safe:

```aria
medium: i16 = -10000;
large: i32 = medium as i32;   // -10,000 (exact)
huge: i64 = medium as i64;    // -10,000 (exact)

// Sign extends properly
// -10000 (i16) = 0xD8F0
// -10000 (i32) = 0xFFFFD8F0 (sign-extended)
```

### Narrowing (Potentially Unsafe)

Converting larger types to `i16` truncates if value doesn't fit:

```aria
large: i32 = 100000;
medium: i16 = large as i16;  // -31,072 (truncated!)

// 100,000 = 0x000186A0
// Truncate to 16 bits: 0x86A0 = -31,072 in two's complement
```

**Safe narrowing with range check:**
```aria
fn safe_i32_to_i16(value: i32) -> ?i16 {
    if value < -32768 || value > 32767 {
        return NIL;  // Out of range
    }
    return value as i16;  // Safe cast
}

large: i32 = 100000;
match safe_i32_to_i16(large) {
    ?medium => println("Converted: {}", medium),
    NIL => println("Value too large for i16"),
}
```

### From i8 (Always Safe)

Converting `i8` to `i16` is always safe (widening):

```aria
small: i8 = -100;
medium: i16 = small as i16;  // -100 (exact, sign-extended)

// -100 (i8) = 0x9C
// -100 (i16) = 0xFF9C (sign-extended)
```

### To i8 (Potentially Unsafe)

Converting `i16` to `i8` truncates if value doesn't fit:

```aria
medium: i16 = 1000;
small: i8 = medium as i8;  // -24 (truncated!)

// 1000 = 0x03E8
// Truncate to 8 bits: 0xE8 = -24 in two's complement
```

### Signed to Unsigned

Converting `i16` to `u16` reinterprets the bit pattern:

```aria
signed: i16 = -1;  // 0xFFFF
unsigned: u16 = signed as u16;  // 65,535 (same bit pattern)

signed: i16 = -32768;  // 0x8000
unsigned: u16 = signed as u16;  // 32,768 (same bit pattern)

signed: i16 = 32767;  // 0x7FFF
unsigned: u16 = signed as u16;  // 32,767 (same bit pattern)
```

**Safe conversion (check sign first):**
```aria
fn safe_i16_to_u16(value: i16) -> ?u16 {
    if value < 0 {
        return NIL;  // Negative values don't fit in u16
    }
    return value as u16;  // Safe (0 to 32,767)
}

signed: i16 = -1000;
match safe_i16_to_u16(signed) {
    ?unsigned => println("Converted: {}", unsigned),
    NIL => println("Negative value can't convert to u16"),
}
```

### Unsigned to Signed

Converting `u16` to `i16` reinterprets if value > 32,767:

```aria
unsigned: u16 = 40000;  // 0x9C40
signed: i16 = unsigned as i16;  // -25,536 (reinterpreted!)

// 40,000 > 32,767, so MSB = 1 in i16
// 0x9C40 = -25,536 in two's complement
```

**Safe conversion:**
```aria
fn safe_u16_to_i16(value: u16) -> ?i16 {
    if value > 32767 {
        return NIL;  // Doesn't fit in i16 positive range
    }
    return value as i16;  // Safe (0 to 32,767)
}

unsigned: u16 = 40000;
match safe_u16_to_i16(unsigned) {
    ?signed => println("Converted: {}", signed),
    NIL => println("Value too large for i16"),
}
```

### To Floating-Point

Converting `i16` to floating-point is exact:

```aria
integer: i16 = -10000;
float32: f32 = integer as f32;  // -10,000.0 (exact)
float64: f64 = integer as f64;  // -10,000.0 (exact)

// All i16 values fit exactly in f32 and f64
// (i16 has 65,536 values, f32 has 23-bit mantissa)
```

### From Floating-Point

Converting floating-point to `i16` truncates toward zero:

```aria
float: f64 = 1234.7;
integer: i16 = float as i16;  // 1234 (truncates decimal)

float: f64 = -1234.9;
integer: i16 = float as i16;  // -1234 (truncates toward zero)
```

**Out of range values wrap or saturate:**
```aria
large: f64 = 100000.0;
small: i16 = large as i16;  // Undefined (implementation-dependent)

// Better: Check range first
fn safe_f64_to_i16(value: f64) -> ?i16 {
    if value < -32768.0 || value > 32767.0 {
        return NIL;
    }
    return value as i16;
}
```

## Common Patterns

### Audio Processing (16-bit PCM)

```aria
// Standard 16-bit PCM audio
const SAMPLE_RATE: i32 = 44100;  // 44.1 kHz

// Audio buffer (1 second)
audio_buffer: [i16; SAMPLE_RATE];

// Read audio samples
fn read_audio_samples(buffer: &mut [i16]) {
    for i in 0..buffer.len() {
        buffer[i] = read_pcm16_sample();
    }
}

// Apply gain with clipping
fn apply_gain(buffer: &mut [i16], gain: f32) {
    for i in 0..buffer.len() {
        // Widen to avoid overflow
        sample: i32 = (buffer[i] as f32 * gain) as i32;
        
        // Clip to i16 range
        if sample > 32767 {
            buffer[i] = 32767;
        } else if sample < -32768 {
            buffer[i] = -32768;
        } else {
            buffer[i] = sample as i16;
        }
    }
}
```

### Geographic Coordinates (Scaled)

```aria
// Latitude/longitude scaled by 100 for 2 decimal places
struct GeoCoordinate {
    lat: i16,  // -9000 to +9000 (actually -90.00° to +90.00°)
    lon: i16,  // -18000 to +18000 (actually -180.00° to +180.00°)
}

fn create_coordinate(lat_degrees: f32, lon_degrees: f32) -> GeoCoordinate {
    return GeoCoordinate {
        lat: (lat_degrees * 100.0) as i16,
        lon: (lon_degrees * 100.0) as i16,
    };
}

fn to_degrees(coord: GeoCoordinate) -> (f32, f32) {
    lat: f32 = coord.lat as f32 / 100.0;
    lon: f32 = coord.lon as f32 / 100.0;
    return (lat, lon);
}

// Usage
sf: GeoCoordinate = create_coordinate(37.77, -122.42);  // San Francisco
// lat: 3777 (-90.00° to +90.00°)
// lon: -12242 (-180.00° to +180.00°)
```

### Temperature Sensors (Extended Range)

```aria
// Temperature sensor with 0.01°C precision
struct TemperatureSensor {
    temp_scaled: i16,  // Temperature × 100 (±327.67°C range)
}

fn read_temperature() -> i16 {
    // Read from sensor (scaled by 100)
    // Range: -27315 to +32767 (actually -273.15°C to +327.67°C)
    raw: i16 = read_sensor_i16();
    return raw;
}

fn celsius_to_scaled(celsius: f32) -> i16 {
    return (celsius * 100.0) as i16;
}

fn scaled_to_celsius(scaled: i16) -> f32 {
```

### Delta Encoding for Compression

```aria
// Compress array by storing deltas instead of absolute values
fn delta_encode(values: [i32]) -> [i16] {
    deltas: [i16; values.len()];
    deltas[0] = 0;  // First value has no delta
    
    for i in 1..values.len() {
        delta: i32 = values[i] - values[i - 1];
        
        // Clamp to i16 range
        if delta > 32767 {
            deltas[i] = 32767;
        } else if delta < -32768 {
            deltas[i] = -32768;
        } else {
            deltas[i] = delta as i16;
        }
    }
    
    return deltas;
}

// Reconstruct from deltas
fn delta_decode(base: i32, deltas: [i16]) -> [i32] {
    values: [i32; deltas.len()];
    values[0] = base;
    
    for i in 1..deltas.len() {
        values[i] = values[i - 1] + (deltas[i] as i32);
    }
    
    return values;
}
```

## Anti-Patterns

### ❌ Using i16 When Values Can't Be Negative

```aria
// WRONG: Port number is never negative, use u16!
port: i16 = 8080;  // ❌ Wastes half the range

// RIGHT: Use u16 for values that can't be negative
port: u16 = 8080;  // ✅ 0-65535 range, semantically correct
```

### ❌ Audio Multiplication Without Widening

```aria
// WRONG: Overflow!
audio: i16 = 20000;
gain: i16 = 2;
amplified: i16 = audio * gain;  // ❌ 40,000 wraps to -25,536!

// RIGHT: Widen before multiplication, then clip
wide: i32 = (audio as i32) * (gain as i32);
amplified: i16 = clamp_to_i16(wide);  // ✅ 32,767 (clipped)
```

### ❌ Ignoring Overflow in Accumulation

```aria
// WRONG: Overflows silently!
sum: i16 = 0;
for value in audio_samples {
    sum += value;  // ❌ Can overflow!
}

// RIGHT: Use larger type or checked arithmetic
sum: i32 = 0;  // ✅ Much larger range
for value in audio_samples {
    sum += value as i32;  // Safe accumulation
}
```

### ❌ Calling abs() on Unchecked Value

```aria
// WRONG: Panics if value is -32768!
value: i16 = read_sensor();
absolute: i16 = abs(value);  // ❌ Panics on -32768

// RIGHT: Check before abs()
value: i16 = read_sensor();
if value == -32768 {
    absolute: i16 = 32767;  // Or handle specially
} else {
    absolute: i16 = abs(value);  // ✅ Safe
}

// OR: Use wider type
wide: i32 = value as i32;
absolute: i32 = abs(wide);  // ✅ abs(-32768) = 32,768 (fits in i32)
```

### ❌ Unsafe Narrowing Without Range Check

```aria
// WRONG: Truncates silently!
large: i32 = 100000;
medium: i16 = large as i16;  // ❌ -31,072 (wrapped!)

// RIGHT: Check range before casting
large: i32 = 100000;
if large >= -32768 && large <= 32767 {
    medium: i16 = large as i16;  // ✅ Safe
} else {
    println("Value doesn't fit in i16");
}
```

### ❌ Mixing i16 and u16 Without Care

```aria
// WRONG: Comparison doesn't work as expected!
signed: i16 = -1000;
unsigned: u16 = 1000;

// Implicit conversion can cause issues
if signed < unsigned as i16 {  // ❌ Confusing
    // -1000 < 1000, but bit patterns might confuse
}

// RIGHT: Explicit widening to common type
if signed as i32 < unsigned as i32 {  // ✅ Clear
    // Explicit conversion, clear semantics
}
```

### ❌ Using i16 for Large Loop Counters

```aria

    process(i);
}

// RIGHT: Use appropriate type
for i: i32 in 0..100000 {  // ✅ i32 fits 100,000
    process(i);
}
```

### ❌ Coordinate Multiplication Without Widening

```aria
// WRONG: Distance calculation overflows!

dy: i16 = p2.y - p1.y;  // e.g., 1000

// RIGHT: Widen before squaring

dy: i32 = (p2.y as i32) - (p1.y as i32);
```

## Performance Characteristics

### Memory Efficiency

`i16` is half the size of `i32`:

```aria

// Array of 1 million i16

// Compare to i32:
array32: [i32; 1000000];  // 4 MB (2× larger!)
```

**Cache efficiency:** More values fit in cache lines, improving performance for large arrays.

### Native Instructions

`i16` operations compile to efficient native instructions:

```aria

b: i16 = 2000;

```

**x86-64 assembly:**
```asm

; Result in AX (one instruction!)
```

**Zero overhead:** Same as hand-written assembly.

### SIMD Vectorization

`i16` has excellent SIMD support:


### Audio Performance

`i16` is the industry standard for audio because:
- **CD quality:** 16-bit PCM at 44.1 kHz
- **Efficient:** Half the size of 32-bit audio
- **Hardware support:** DSPs and audio codecs use 16-bit
- **SIMD:** Process 8-32 samples in parallel

- Industry standard for audio

## Common Gotchas

### Gotcha 1: Asymmetric Range (abs Trap)

```aria
// i16 has one more negative value than positive
min: i16 = -32768;
max: i16 = 32767;

// This means abs() doesn't always work!
absolute: i16 = abs(-32768);  // PANICS! (32,768 doesn't fit)

// ✅ Solution: Check before abs()
if value != -32768 {
    absolute: i16 = abs(value);
}

// OR: Use wider type
wide: i32 = value as i32;
absolute: i32 = abs(wide);  // abs(-32768) = 32,768 (fits)
```

### Gotcha 2: Audio Multiplication Overflow

```aria
// Common audio processing mistake
audio: i16 = 20000;
gain: i16 = 2;
amplified: i16 = audio * gain;  // 40,000 wraps to -25,536!

// ✅ Solution: Widen, multiply, clip
wide: i32 = (audio as i32) * (gain as i32);
amplified: i16 = clamp_to_i16(wide);  // 32,767 (clipped)
```

### Gotcha 3: Coordinate Distance Overflow

```aria
// Distance calculation overflows!
dx: i16 = 1000;
dy: i16 = 1000;
dist_sq: i16 = dx * dx + dy * dy;  // 2,000,000 wraps!

// ✅ Solution: Use i32 for intermediate
dx32: i32 = dx as i32;
dy32: i32 = dy as i32;
dist_sq: i32 = dx32 * dx32 + dy32 * dy32;  // 2,000,000 (fits)
```

### Gotcha 4: Silent Overflow Wrapping

```aria
// Overflow wraps silently (no error!)
value: i16 = 30000;
value += 5000;  // 35,000... wraps to -30,536!

// ✅ Solution: Use checked arithmetic or TBB
tbb_value: tbb16 = 30000;
tbb_value += 5000;  // ERR (automatic overflow detection)
```

### Gotcha 5: Signed vs Unsigned Confusion

```aria
// i16 and u16 have different ranges!
signed: i16 = -1000;
unsigned: u16 = signed as u16;  // 64,536 (same bits!)

// ✅ Solution: Check sign before converting
if signed >= 0 {
    unsigned: u16 = signed as u16;  // Safe
}
```

### Gotcha 6: Narrowing Truncation

```aria
// Narrowing from i32 truncates!
large: i32 = 100000;
medium: i16 = large as i16;  // -31,072 (truncated!)

// ✅ Solution: Check range
if large >= -32768 && large <= 32767 {
    medium: i16 = large as i16;  // Safe
}
```

### Gotcha 7: Loop Counter Overflow

```aria
// Large loop overflows i16!
for i: i16 in 0..100000 {  // ❌ i16 max is 32,767
    process(i);
}

// ✅ Solution: Use i32 for large loops
for i: i32 in 0..100000 {  // Safe
    process(i);
}
```

### Gotcha 8: Division by Zero

```aria
// Division by zero panics!
a: i16 = 10000;
b: i16 = 0;
quotient: i16 = a / b;  // PANICS!

// ✅ Solution: Check before dividing
if b != 0 {
    quotient: i16 = a / b;  // Safe
}

// OR: Use TBB types
tbb_a: tbb16 = 10000;
tbb_b: tbb16 = 0;
tbb_quotient: tbb16 = tbb_a / tbb_b;  // ERR (no panic)
```

## Summary

`i16` is a 16-bit signed integer type for medium-range values that can be negative:

**Key Properties:**
- **Range:** -32,768 to +32,767 (65,536 values)
- **Size:** 2 bytes (16 bits)
- **Representation:** Two's complement
- **Overflow:** Wraps around (defined behavior)
- **Performance:** Native instructions, excellent SIMD support

**When to use i16:**
- ✅ Audio processing (16-bit PCM standard)
- ✅ Screen coordinates (supports up to 32K×32K)
- ✅ Temperature sensors (extended range, high precision)
- ✅ Geographic coordinates (scaled for precision)
- ✅ Accelerometer data (±32g range)
- ✅ Values beyond i8 but within ±32K
- ✅ C interoperability (`int16_t`, `short`)

**When NOT to use i16:**
- ❌ Values always positive (use `u16` instead)
- ❌ Range might exceed ±32K (use `i32`)
- ❌ Values fit in i8 (use smaller `i8` instead)
- ❌ Overflow detection critical (use TBB types)
- ❌ Need symmetric range for `abs()` (use `tbb16`)

**Critical gotchas:**
- ⚠️ `abs(-32768)` panics (32,768 doesn't fit)
- ⚠️ Overflow wraps silently (32,767 + 1 = -32,768)
- ⚠️ Audio multiplication overflows (20,000 × 2 wraps!)
- ⚠️ Coordinate distance overflows (1000² + 1000² wraps!)
- ⚠️ Asymmetric range (one more negative value)
- ⚠️ Narrowing truncates (100,000 as i16 = -31,072)

**Best practices:**
- Widen before multiplication (especially audio/coordinates)
- Clip audio samples instead of wrapping
- Check range when converting from larger types
- Avoid `abs()` on unchecked values (check for -32,768)
- Prefer `u16` when values never negative
- Use `i32` for accumulation and large loops
- Use TBB types (`tbb16`) for safety-critical code

**vs Other Types:**

| Comparison | Key Difference | When to Prefer i16 |
|------------|----------------|-------------------|
| **i16 vs i8** | 256× larger range | Values beyond ±128 |
| **i16 vs u16** | Signed vs unsigned | Values can be negative |
| **i16 vs i32** | 2 bytes vs 4 bytes | Values within ±32K, memory critical |
| **i16 vs tbb16** | Wrapping vs error detection | Performance > safety |

**Audio Standard:**
- 16-bit PCM is industry standard for CD-quality audio
- Range -32,768 to +32,767 represents full dynamic range
- Always widen before multiplication (gain/mixing)
- Clip instead of wrap for audio processing

**Remember:** `i16` is the audio industry standard and perfect for medium-range signed values. Critical for audio: always widen before multiplication, then clip to range!
