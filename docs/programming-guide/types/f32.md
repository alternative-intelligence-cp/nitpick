# f32: 32-Bit Floating-Point Type

## Overview

**Purpose**: The `f32` type represents 32-bit single-precision floating-point numbers following the IEEE 754 standard. It's designed for performance-critical applications where approximate arithmetic is acceptable and memory/bandwidth efficiency matters.

**Characteristics**:
- Size: 32 bits (4 bytes)
- Range: ±1.18×10⁻³⁸ to ±3.4×10³⁸ (approximately)
- Precision: ~7 decimal digits
- Format: IEEE 754 single precision (1 sign + 8 exponent + 23 mantissa bits)
- Hardware: Native CPU/GPU support, SIMD-friendly
- Special values: +0.0, -0.0, +Infinity, -Infinity, NaN

**When to Use f32**:
- Graphics rendering (vertices, colors, transformations)
- Real-time audio processing (samples, filters, effects)
- Physics simulations (games, approximate models)
- Machine learning (neural network weights, activations)
- Signal processing (DSP, image filters)
- Performance-critical float operations
- Memory/bandwidth constrained (vs f64)
- SIMD vectorization (process 4/8/16 values in parallel)

**When NOT to Use f32**:
- Financial calculations (use `fix256` for exact decimal arithmetic)
- Safety-critical systems (use `fix256` for deterministic behavior)
- Need more than 7 digits of precision (use `f64` for ~15 digits)
- Cross-platform determinism required (use deterministic float types)
- Exact decimal values (0.1, 0.2, 0.3) needed (use `frac` or `fix256`)
- Large integer values beyond ±16,777,216 (use `i32` or `i64`)
- Counting or indexing (use integer types)

## f32 vs f64: Precision vs Performance

The choice between `f32` (single precision) and `f64` (double precision) involves trade-offs between precision, memory, and performance.

### Key Differences

| Aspect | f32 (Single) | f64 (Double) | Notes |
|--------|--------------|--------------|-------|
| **Size** | 4 bytes | 8 bytes | f32 uses half the memory |
| **Precision** | ~7 decimal digits | ~15-16 decimal digits | f64 has 2× the precision |
| **Range** | ±3.4×10³⁸ | ±1.8×10³⁰⁸ | Both cover most use cases |
| **SIMD Width** | 4/8/16 values | 2/4/8 values | f32 processes 2× data per instruction |
| **Bandwidth** | 1× | 2× | f32 transfers data 2× faster |
| **Cache Efficiency** | 2× entries | 1× entries | f32 fits twice as many values |
| **GPU Performance** | Fastest | Slower | GPUs optimized for f32 |
| **Epsilon** | ~1.2×10⁻⁷ | ~2.2×10⁻¹⁶ | f64 much finer comparisons |

### When to Choose f32 Over f64

**Graphics/Rendering**:
```aria
// Vertex transformation (millions per frame)
// f32 is 2× faster in SIMD, uses half the memory
struct Vertex {
    f32:x, f32:y, f32:z;     // Position
    f32:nx, f32:ny, f32:nz;  // Normal
    f32:u, f32:v;            // Texture coords
}

// Matrix multiply (4×4 matrix = 16 floats = 64 bytes vs 128 for f64)
f32[16]:transform_matrix = get_model_view_projection();
```

**Audio Processing**:
```aria
// 48 kHz audio = 48,000 samples/second
// Need real-time processing, f32 precision is plenty
f32:apply_lowpass_filter(f32:sample, f32:cutoff, f32:resonance) -> f32 {
    // 7 digits precision far exceeds human hearing
    return sample * compute_filter_coefficient(cutoff, resonance);
}
```

**Machine Learning**:
```aria
// Neural network inference (millions of operations)
// f32 is standard (GPUs optimized for it)
f32[]:forward_pass(f32[]:input, f32[]:weights) -> f32[] {
    // Precision less important than speed at scale
    return matrix_multiply(input, weights);
}
```

### When to Choose f64 Over f32

**Scientific Computing**:
```aria
// Astronomy calculations (need high precision)
f64:orbital_period = compute_kepler_period(semi_major_axis, mass);

// Accumulated error over many iterations
f64:sum = 0.0;
loop (1_000_000:i) {
    sum += small_value;  // f64 preserves precision better
}
```

**High-Precision Requirements**:
```aria
// Need more than 7 digits
f64:precise_constant = 3.14159265358979323846;  // f32 would round after 7 digits

// Large values with fine detail
f64:distance_meters = 149_597_870_700.0;  // Earth to Sun (needs precision)
f64:delta = distance_meters + 0.001;      // f32 would lose the .001
```

### Decision Matrix

```
Choose f32 when:
✓ Precision of ~7 digits sufficient
✓ Performance critical (graphics, games, audio)
✓ Memory/bandwidth constrained
✓ SIMD vectorization important
✓ GPU acceleration needed
✓ Processing large arrays of values

Choose f64 when:
✓ Need ~15 digits of precision
✓ Scientific calculations
✓ Long accumulations (many additions)
✓ Large values requiring fine adjustment
✓ Precision more important than speed
```

## f32 vs i32: Floating vs Integer

`f32` (floating-point) and `i32` (integer) serve different purposes despite both being 32 bits.

### Fundamental Differences

| Aspect | f32 | i32 | Notes |
|--------|-----|-----|-------|
| **Range** | ±3.4×10³⁸ | -2,147,483,648 to 2,147,483,647 | f32 vastly larger range |
| **Precision** | ~7 digits | Exact | i32 always exact within range |
| **Decimals** | Yes (approximate) | No | 3.14, 0.1 possible in f32 only |
| **Overflow** | → Infinity | Wraps around | Different failure modes |
| **Equality** | Unreliable | Reliable | i32 can use `==` safely |
| **Performance** | Variable | Consistent | i32 often faster for small ops |
| **Special Values** | NaN, ±Infinity, -0.0 | None | f32 has special cases |

### When f32 Loses Integer Precision

**Critical**: f32 can only represent consecutive integers up to **±16,777,216** (2²⁴). Beyond this, gaps appear:

```aria
// Safe: Small integers exact in f32
f32:small = 1000.0;
i32:back = small.to_i32();  // 1000 (exact round-trip)

// DANGER: Large integers lose precision
i32:big_int = 16777217;       // 2^24 + 1 (exact in i32)
f32:as_float = big_int.to_f32();  // 16777216.0 (rounded down!)
i32:round_trip = as_float.to_i32();  // 16777216 (lost 1!)

big_int != round_trip;  // true (precision lost)

// f32 can't represent all values
f32:x = 16777216.0;  // 2^24 (last exact integer)
f32:y = x + 1.0;     // 16777216.0 (can't represent 16777217.0!)
x == y;              // true! (gap in representation)
```

### Use Cases Comparison

**Array Indexing** (use i32):
```aria
// NEVER use f32 for indexing
i32:index = 42;
value = array[index];  // Correct

// BAD: Floating-point for indices
f32:bad_index = 42.0;  // Why would you do this?
value = array[bad_index.to_i32()];  // Wasteful, error-prone
```

**Counting** (use i32):
```aria
// CORRECT: Integer counter
i32:count = 0;
loop (items.length:i) {
    if (items[i].is_valid()) {
        count += 1;  // Exact
    }
}

// BAD: Float counter
f32:bad_count = 0.0;  // Rounding errors possible
loop (items.length:i) {
    if (items[i].is_valid()) {
        bad_count += 1.0;  // Why risk it?
    }
}
```

**Measurements** (use f32):
```aria
// Temperature, distance, voltage (continuous values)
f32:temperature_celsius = 23.8;
f32:distance_meters = 15.432;
f32:voltage = 3.3;

// These aren't integers, f32 appropriate
```

**Ratios/Percentages** (use f32):
```aria
// Division produces fractions
i32:hits = 75;
i32:total = 100;
f32:accuracy = hits.to_f32() / total.to_f32();  // 0.75

// WRONG: Integer division truncates
i32:bad_accuracy = hits / total;  // 0 (truncated!)
```

### Decision Matrix

```
Use f32 when:
✓ Fractional values needed (3.14, 0.5, -2.718)
✓ Very large or very small magnitudes
✓ Continuous measurements
✓ Ratios, percentages, probabilities
✓ Trigonometry, exponentials, logarithms

Use i32 when:
✓ Counting discrete items
✓ Array indices
✓ Exact integer arithmetic needed
✓ Values guaranteed within ±2.1 billion
✓ Bit manipulation required
✓ No fractional component
```

## IEEE 754 Single Precision Format

### Bit Layout

The `f32` type uses IEEE 754 single precision format:

```
Bit Position:   31      30-23           22-0
Component:     Sign    Exponent      Mantissa
Bits:           1        8 bits       23 bits
```

**Sign Bit** (bit 31):
- `0` = positive number
- `1` = negative number

**Exponent** (bits 30-23, 8 bits):
- Biased representation (bias = 127)
- Stored value 0: Zero or subnormal numbers
- Stored value 1-254: Normal numbers
- Stored value 255: Infinity or NaN
- Actual exponent = stored_value - 127

**Mantissa/Significand** (bits 22-0, 23 bits):
- Fractional part after the implicit leading 1
- Implicit leading 1 for normal numbers
- 23 bits provide ~7 decimal digits precision

### Value Calculation

**Normal Numbers** (exponent 1-254):
```
value = (-1)^sign × (1 + mantissa/2^23) × 2^(exponent - 127)
```

**Example**: Representing 12.5 as f32
```
12.5 = 1.5625 × 2³

Sign: 0 (positive)
Exponent: 3 + 127 = 130 = 0b10000010
Mantissa: 0.5625 = 0.1001 binary = 0b10010000000000000000000

Bit Pattern: 0 10000010 10010000000000000000000
Hexadecimal: 0x41480000

Verification:
(-1)^0 × (1 + 0.5625) × 2^(130-127)
= 1 × 1.5625 × 2^3
= 1.5625 × 8
= 12.5 ✓
```

**Example**: Representing -0.25 as f32
```
-0.25 = -1.0 × 2⁻²

Sign: 1 (negative)
Exponent: -2 + 127 = 125 = 0b01111101
Mantissa: 0.0 (since 1.0 × 2^(-2) = 0.25)

Bit Pattern: 1 01111101 00000000000000000000000
Hexadecimal: 0xBE800000
```

### Precision Characteristics

**Approximately 7 Decimal Digits**:
```aria
// Maximum precision demo
f32:seven_digits = 1234567.0;    // Exact
f32:eight_digits = 12345678.0;   // Rounded (8th digit lost)

// Relative precision (not absolute)
f32:large = 1000000.0;
f32:large_delta = large + 0.1;   // Rounds to 1000000.0 (0.1 too small)

f32:small = 0.001;
f32:small_delta = small + 0.0001;  // = 0.0011 (retained)
```

**Consecutive Integer Precision**:
```aria
// Can exactly represent integers from -16,777,216 to +16,777,216
f32:max_exact_int = 16777216.0;  // 2^24
f32:next = max_exact_int + 1.0;  // 16777216.0 (can't represent!)
max_exact_int == next;  // true (precision limit reached)

// Beyond this, gaps appear
f32:beyond = 16777218.0;  // Actually stored as 16777218.0 (gap of 2)
```

## Special Values

### Zero (+0.0 and -0.0)

IEEE 754 has **two zeros** with different signs:

```aria
f32:positive_zero = 0.0;   // Sign=0, Exponent=0, Mantissa=0
f32:negative_zero = -0.0;  // Sign=1, Exponent=0, Mantissa=0

// Zeros compare equal
positive_zero == negative_zero;  // true

// But behave differently in some operations
f32:pos_result = 1.0 / positive_zero;  // +Infinity
f32:neg_result = 1.0 / negative_zero;  // -Infinity

// Sign preservation in multiplication
f32:a = -2.0 * 0.0;   // -0.0
f32:b = -2.0 * -0.0;  // +0.0

// Checking for zero (either sign)
bool:is_zero = (value == 0.0);  // true for both ±0.0
```

**When Signed Zero Matters**:
```aria
// Branch cuts in complex functions
f32:angle1 = atan2(+0.0, -1.0);  // +π
f32:angle2 = atan2(-0.0, -1.0);  // -π

// Limit directions
f32:lim1 = 1.0 / positive_zero;  // Approaching from above: +∞
f32:lim2 = 1.0 / negative_zero;  // Approaching from below: -∞
```

### Infinity (+∞ and -∞)

Represents values too large for f32 range:

```aria
f32:pos_inf = f32.infinity;       // Exponent=255, Mantissa=0, Sign=0
f32:neg_inf = -f32.infinity;      // Exponent=255, Mantissa=0, Sign=1

// Generation
f32:overflow = 1e38 * 1e38;       // +Infinity (overflow)
f32:division = 1.0 / 0.0;         // +Infinity
f32:neg_division = -1.0 / 0.0;    // -Infinity

// Arithmetic with infinity
pos_inf + 100.0;           // +Infinity
pos_inf * 2.0;             // +Infinity
pos_inf * -1.0;            // -Infinity
1.0 / pos_inf;             // +0.0
pos_inf - pos_inf;         // NaN (undefined)
pos_inf / pos_inf;         // NaN (undefined)

// Comparisons
pos_inf > 1e38;            // true (greater than all finite values)
neg_inf < -1e38;           // true (less than all finite values)
pos_inf == pos_inf;        // true (infinity equals itself)

// Checking for infinity
bool:is_inf = value.is_infinity();
bool:is_pos_inf = (value == f32.infinity);
bool:is_neg_inf = (value == -f32.infinity);
```

### NaN (Not a Number)

Represents undefined or invalid results:

```aria
f32:nan = f32.nan;  // Exponent=255, Mantissa≠0

// NaN generation
f32:zero_div = 0.0 / 0.0;           // NaN
f32:inf_sub = f32.infinity - f32.infinity;  // NaN
f32:neg_sqrt = (-1.0).sqrt();       // NaN (domain error)
f32:invalid_log = (-5.0).log();     // NaN (domain error)

// NaN arithmetic (NaN infects everything)
nan + 5.0;         // NaN
nan * 0.0;         // NaN (even × 0!)
nan / nan;         // NaN

// NaN comparisons (ALWAYS false, even with itself!)
nan == nan;        // false (!!)
nan != nan;        // true (only reliable NaN test)
nan < 5.0;         // false
nan >= 5.0;        // false
nan == 0.0;        // false

// CORRECT way to check for NaN
bool:is_nan = value.is_nan();  // Use this!
bool:wrong = (value != value);  // Works but obscure

// NaN payload (diagnostic information)
f32:custom_nan = f32.nan_with_payload(0x123456);
u32:payload = custom_nan.nan_payload();  // 0x123456
```

**Critical NaN Behavior**:
```aria
// NaN in conditionals (always false)
f32:result = compute_value();
if (result > threshold) {
    // NEVER executes if result is NaN!
}

// NaN in sorting (unpredictable)
f32[]:values = [1.0, NaN, 3.0, 2.0];
values.sort();  // Undefined behavior with NaN!

// ALWAYS validate inputs
void:process(f32:value) {
    if (value.is_nan() || value.is_infinity()) {
        error("Invalid input: NaN or Infinity");
        return;
    }
    // Safe to process
}
```

## Arithmetic Operations

### Basic Arithmetic

```aria
f32:a = 10.5;
f32:b = 3.2;

f32:sum = a + b;       // 13.7 (approximate)
f32:diff = a - b;      // 7.3
f32:product = a * b;   // 33.6
f32:quotient = a / b;  // 3.28125
f32:negated = -a;      // -10.5

// Compound assignment
a += b;   // a = a + b
a -= b;   // a = a - b
a *= b;   // a = a * b
a /= b;   // a = a / b
```

### Division by Zero

```aria
f32:positive = 1.0;
f32:negative = -1.0;
f32:zero = 0.0;

f32:pos_div_zero = positive / zero;   // +Infinity
f32:neg_div_zero = negative / zero;   // -Infinity
f32:zero_div_zero = zero / zero;      // NaN (undefined)

// Checking before division
f32:safe_divide(f32:numerator, f32:denominator) -> f32 {
    if (denominator == 0.0) {
        error("Division by zero");
        return 0.0;  // Or handle appropriately
    }
    return numerator / denominator;
}
```

### Rounding and Precision

```aria
// Rounding modes
f32:value = 3.7;
f32:rounded = value.round();    // 4.0 (nearest)
f32:floored = value.floor();    // 3.0 (toward -∞)
f32:ceiled = value.ceil();      // 4.0 (toward +∞)
f32:truncated = value.trunc();  // 3.0 (toward zero)

// Negative values
f32:neg = -3.7;
neg.round();   // -4.0
neg.floor();   // -4.0 (toward -∞)
neg.ceil();    // -3.0 (toward +∞)
neg.trunc();   // -3.0 (toward zero)

// Fractional part
f32:fract = value.fract();  // 0.7 (value - floor(value))
```

### Precision Loss Examples

```aria
// Small + Large (small value lost)
f32:large = 1e7;
f32:small = 0.1;
f32:result = large + small;  // 10000000.0 (0.1 lost in rounding)

// Catastrophic cancellation
f32:nearly_equal_a = 1.0000001;
f32:nearly_equal_b = 1.0000000;
f32:tiny_diff = nearly_equal_a - nearly_equal_b;
// Lost most significant digits, result dominated by rounding error

// Accumulation drift
f32:sum = 0.0;
loop (10000:i) {
    sum += 0.1;  // Each addition rounds
}
// Expected: 1000.0
// Actual: ~999.9 or ~1000.1 (accumulated rounding error)
```

## Comparisons

### Equality (Unreliable!)

**CRITICAL**: Never use `==` for f32 comparisons due to rounding errors:

```aria
// DON'T DO THIS
f32:a = 0.1 + 0.2;
if (a == 0.3) {  // NEVER TRUE!
    // This code never executes
}
// Why? 0.1, 0.2, 0.3 not exactly representable in binary

// CORRECT: Use epsilon comparison
f32:epsilon = 1e-6;  // Tolerance (adjust for use case)
if (abs(a - 0.3) < epsilon) {
    // Approximately equal
}

// Or use built-in
if (a.approx_equal(0.3, epsilon)) {
    // Same as above
}
```

### Epsilon Selection

```aria
// Absolute epsilon (for values near zero)
f32:abs_epsilon = 1e-6;
bool:nearly_zero = abs(value) < abs_epsilon;

// Relative epsilon (for arbitrary magnitudes)
f32:relative_epsilon(f32:a, f32:b) -> bool {
    f32:epsilon = 1e-6;
    f32:abs_diff = abs(a - b);
    f32:magnitude = max(abs(a), abs(b));
    return abs_diff <= epsilon * magnitude;
}

// Machine epsilon (smallest representable difference)
f32:machine_eps = f32.epsilon;  // ~1.19×10^-7
```

### Ordering Comparisons

```aria
f32:a = 5.5;
f32:b = 3.2;

bool:greater = a > b;         // true
bool:greater_equal = a >= b;  // true
bool:less = a < b;            // false
bool:less_equal = a <= b;     // false

// NaN in comparisons (ALWAYS false!)
f32:nan = f32.nan;
nan > 5.0;   // false
nan < 5.0;   // false
nan >= 5.0;  // false
nan == nan;  // false

// Safe comparison with NaN handling
bool:safe_less_than(f32:a, f32:b) -> bool {
    if (a.is_nan() || b.is_nan()) {
        return false;  // Or handle NaN appropriately
    }
    return a < b;
}
```

### Min/Max with NaN Handling

```aria
f32:a = 5.5;
f32:b = 3.2;

f32:minimum = min(a, b);  // 3.2
f32:maximum = max(a, b);  // 5.5

// NaN handling (IEEE 754 semantics)
f32:nan = f32.nan;
min(nan, 5.0);  // NaN (NaN propagates)
max(nan, 5.0);  // NaN

// NaN-safe min/max
f32:safe_min(f32:a, f32:b) -> f32 {
    if (a.is_nan()) return b;
    if (b.is_nan()) return a;
    return min(a, b);
}
```

### Clamping and Interpolation

```aria
// Clamp value to range
f32:clamp(f32:value, f32:min_val, f32:max_val) -> f32 {
    return max(min_val, min(max_val, value));
}

f32:clamped = clamp(150.0, 0.0, 100.0);  // 100.0

// Linear interpolation
f32:lerp(f32:start, f32:end, f32:t) -> f32 {
    return start + t * (end - start);
}

f32:interpolated = lerp(0.0, 100.0, 0.5);  // 50.0

// Inverse lerp (find t given value)
f32:inverse_lerp(f32:start, f32:end, f32:value) -> f32 {
    return (value - start) / (end - start);
}

f32:t = inverse_lerp(0.0, 100.0, 25.0);  // 0.25
```

## Type Conversions

### From Integers

```aria
// i32 → f32
i32:int_val = 42;
f32:float_val = int_val.to_f32();  // 42.0 (exact for small values)

// Large integers may lose precision
i32:large = 16777217;  // 2^24 + 1
f32:rounded = large.to_f32();  // 16777216.0 (rounded!)

// Negative integers
i32:neg = -100;
f32:neg_float = neg.to_f32();  // -100.0

// i64 → f32 (precision loss more common)
i64:big = 9007199254740993;  // Larger than f32 can represent exactly
f32:imprecise = big.to_f32();  // Rounded to nearest f32 value
```

### To Integers

```aria
// f32 → i32 (truncates toward zero)
f32:positive = 3.9;
i32:truncated = positive.to_i32();  // 3

f32:negative = -3.9;
i32:neg_truncated = negative.to_i32();  // -3 (toward zero)

// Rounding before conversion
f32:value = 3.6;
i32:rounded_int = value.round().to_i32();  // 4
i32:floor_int = value.floor().to_i32();    // 3
i32:ceil_int = value.ceil().to_i32();      // 4

// Overflow handling
f32:too_big = 1e20;
i32:overflowed = too_big.to_i32();  // Undefined! Check ranges first

// Safe conversion
i32:safe_to_i32(f32:value) -> Result<i32> {
    if (value.is_nan() || value.is_infinity()) {
        return Error("Cannot convert NaN/Infinity to integer");
    }
    if (value > i32.max.to_f32() || value < i32.min.to_f32()) {
        return Error("Value out of i32 range");
    }
    return Ok(value.to_i32());
}
```

### f32 ↔ f64

```aria
// f32 → f64 (always safe, no precision loss)
f32:single = 3.14159;
f64:double = single.to_f64();  // 3.141590118408203125 (exact representation of f32 value)

// f64 → f32 (may round)
f64:precise = 3.141592653589793;
f32:rounded = precise.to_f32();  // 3.1415927 (rounded to f32 precision)

// Large f64 values may overflow
f64:huge = 1e100;
f32:overflow = huge.to_f32();  // +Infinity (out of f32 range)
```

### String Conversions

```aria
// f32 → string
f32:value = 3.14159;
string:str = value.to_string();  // "3.14159"
string:formatted = value.format(2);  // "3.14" (2 decimal places)
string:scientific = value.to_scientific();  // "3.14159e0"

// string → f32
string:input = "42.5";
Result<f32>:parsed = f32.parse(input);
match (parsed) {
    Ok(num) => { /* Use num */ }
    Error(msg) => { error("Parse failed: " + msg); }
}

// Handles special values

f32.parse("NaN");   // Ok(f32.nan)
f32.parse("xyz");   // Error
```

## Common Patterns

### Averaging Values

```aria
// Simple average
f32:average(f32[]:values) -> f32 {
    if (values.length == 0) return 0.0;
    
    f32:sum = 0.0;
    loop (values.length:i) {
        sum += values[i];
    }
    return sum / values.length.to_f32();
}

// Numerically stable averaging (reduces rounding error)
f32:stable_average(f32[]:values) -> f32 {
    if (values.length == 0) return 0.0;
    
    f32:mean = 0.0;
    loop (values.length:i) {
        // Welford's online algorithm
        mean += (values[i] - mean) / (i + 1).to_f32();
    }
    return mean;
}
```

### Distance Calculations

```aria
// Euclidean distance (2D)
f32:distance_2d(f32:x1, f32:y1, f32:x2, f32:y2) -> f32 {
    f32:dx = x2 - x1;
    f32:dy = y2 - y1;
    return (dx * dx + dy * dy).sqrt();
}

// Euclidean distance (3D)
f32:distance_3d(f32:x1, f32:y1, f32:z1, f32:x2, f32:y2, f32:z2) -> f32 {
    f32:dx = x2 - x1;
    f32:dy = y2 - y1;
    f32:dz = z2 - z1;
    return (dx*dx + dy*dy + dz*dz).sqrt();
}

// Manhattan distance (grid-based)
f32:manhattan_distance(f32:x1, f32:y1, f32:x2, f32:y2) -> f32 {
    return abs(x2 - x1) + abs(y2 - y1);
}
```

### Smoothing and Filtering

```aria
// Linear interpolation (smoothing)
f32:smooth_step(f32:edge0, f32:edge1, f32:x) -> f32 {
    f32:t = clamp((x - edge0) / (edge1 - edge0), 0.0, 1.0);
    return t * t * (3.0 - 2.0 * t);  // Hermite interpolation
}

// Exponential moving average
struct ExponentialAverage {
    f32:value;
    f32:alpha;  // Smoothing factor (0-1)
}

void:update(ExponentialAverage&:avg, f32:new_value) {
    avg.value = avg.alpha * new_value + (1.0 - avg.alpha) * avg.value;
}
```

### Normalization

```aria
// Normalize to [0, 1] range
f32:normalize(f32:value, f32:min_val, f32:max_val) -> f32 {
    return (value - min_val) / (max_val - min_val);
}

// Vector normalization (unit vector)
struct Vec3 { f32:x, f32:y, f32:z; }

Vec3:normalize_vector(Vec3:v) -> Vec3 {
    f32:length = (v.x*v.x + v.y*v.y + v.z*v.z).sqrt();
    if (length == 0.0) {
        return Vec3{0.0, 0.0, 0.0};  // Avoid division by zero
    }
    return Vec3{
        v.x / length,
        v.y / length,
        v.z / length
    };
}
```

### Physics Calculations

```aria
// Velocity from acceleration
f32:update_velocity(f32:current_velocity, f32:acceleration, f32:delta_time) -> f32 {
    return current_velocity + acceleration * delta_time;
}

// Position from velocity
f32:update_position(f32:current_position, f32:velocity, f32:delta_time) -> f32 {
    return current_position + velocity * delta_time;
}

// Projectile motion
struct Projectile {
    f32:x, f32:y;          // Position
    f32:vx, f32:vy;        // Velocity
    f32:gravity = -9.81;   // Acceleration
}

void:update(Projectile&:p, f32:dt) {
    p.vy += p.gravity * dt;  // Update velocity
    p.x += p.vx * dt;        // Update position
    p.y += p.vy * dt;
}
```

## Anti-Patterns

### ❌ Using f32 for Money

```aria
// WRONG: f32 for currency
f32:price = 0.1;
f32:quantity = 3.0;
f32:total = price * quantity;  // 0.30000001 (rounding error!)

// Legal/accounting issue if this propagates

// CORRECT: Use fix256 for exact decimal arithmetic
fix256:exact_price = fix256.from_decimal("0.10");
fix256:exact_quantity = fix256.from_int(3);
fix256:exact_total = exact_price * exact_quantity;  // Exactly 0.30
```

### ❌ Equality Comparison

```aria
// WRONG: Direct equality
f32:x = 0.1 + 0.2;
if (x == 0.3) {  // NEVER true!
    unreachable_code();
}

// CORRECT: Epsilon comparison
f32:epsilon = 1e-6;
if (abs(x - 0.3) < epsilon) {
    correct_code();
}
```

### ❌ Using f32 for Loop Counters

```aria
// WRONG: Float for counting
f32:counter = 0.0;
while (counter < 1000.0) {
    process_item();
    counter += 1.0;  // Accumulates rounding error
}

// CORRECT: Integer for counting
i32:counter = 0;
while (counter < 1000) {
    process_item();
    counter += 1;  // Exact
}
```

### ❌ Ignoring NaN Checks

```aria
// WRONG: Assume valid input
void:process(f32:user_input) {
    f32:result = 100.0 / user_input;  // NaN if user_input is NaN!
    display_result(result);  // Displays "NaN"
}

// CORRECT: Validate input
void:process(f32:user_input) {
    if (user_input.is_nan() || user_input.is_infinity()) {
        error("Invalid input");
        return;
    }
    f32:result = 100.0 / user_input;
    display_result(result);
}
```

### ❌ Adding Small to Large Repeatedly

```aria
// WRONG: Loses precision
f32:large_sum = 1e7;
loop (10000:i) {
    large_sum += 0.1;  // 0.1 too small to affect large_sum
}
// Expected: 10001000.0
// Actual: 10000000.0 (increments lost)

// CORRECT: Accumulate small values first
f32:small_sum = 0.0;
loop (10000:i) {
    small_sum += 0.1;
}
f32:total = 1e7 + small_sum;  // Correct result
```

### ❌ Using f32 for Large Integer IDs

```aria
// WRONG: f32 for 64-bit IDs
i64:user_id = 9007199254740993;  // Larger than f32 precision
f32:bad_id = user_id.to_f32();   // Rounds, loses precision
i64:recovered = bad_id.to_i64(); // Different value!

user_id != recovered;  // true (data corruption!)

// CORRECT: Use i64 for integer IDs
i64:correct_id = user_id;  // Exact
```

## Best Practices

### ✓ Use f32 for Visual/Audio Precision

```aria
// Graphics (humans can't perceive f32 rounding)
struct Color {
    f32:r, f32:g, f32:b, f32:a;  // RGBA [0.0, 1.0]
}

// Audio (f32 precision exceeds human hearing)
f32:sample_rate = 48000.0;
f32:amplitude = 0.8;
```

### ✓ Always Check for NaN/Infinity

```aria
void:safe_compute(f32:input) {
    if (!input.is_finite()) {  // Checks both NaN and Infinity
        error("Invalid input: must be finite");
        return;
    }
    // Safe to proceed
}
```

### ✓ Use Epsilon for Comparisons

```aria
// Context-appropriate epsilon
f32:epsilon_for_physics = 1e-3;   // 1mm tolerance
f32:epsilon_for_rendering = 1e-5; // Finer visual tolerance

bool:approximately_equal(f32:a, f32:b, f32:epsilon) -> bool {
    return abs(a - b) < epsilon;
}
```

### ✓ Prefer f64 for Accumulations

```aria
// Many operations? Use f64 to reduce error
f64:sum = 0.0;
loop (million_values.length:i) {
    sum += million_values[i].to_f64();  // Less accumulated error
}
f32:final_avg = (sum / million_values.length.to_f64()).to_f32();
```

### ✓ Validate Ranges Before Conversion

```aria
f32:safe_to_u8(f32:value) -> Result<u8> {
    if (value < 0.0 || value > 255.0) {
        return Error("Value out of u8 range");
    }
    return Ok(value.to_u8());
}
```

### ✓ Document Precision Requirements

```aria
// GOOD: Explicit precision contract
// @precision: ±0.01 meters (1cm tolerance)
f32:measure_distance() -> f32;

// BAD: Unclear precision expectations
f32:get_value() -> f32;
```

## Performance Characteristics

### Cycle Counts (Approximate, Modern CPUs)

| Operation | Cycles | Notes |
|-----------|--------|-------|
| Addition/Subtraction | 3-5 | Fully pipelined |
| Multiplication | 5 | Fully pipelined |
| Division | 10-20 | Slower, not fully pipelined |
| Square root | 10-15 | Hardware accelerated |
| Trigonometric | 30-100 | Uses approximations |
| Logarithm/Exponential | 30-60 | Polynomial approximations |

### SIMD Throughput

**SSE (128-bit)**:
- Process 4× f32 values per instruction
- Example: `add_ps` adds 4 floats in ~3 cycles

**AVX (256-bit)**:
- Process 8× f32 values per instruction
- ~2× throughput of SSE

**AVX-512 (512-bit)**:
- Process 16× f32 values per instruction
- ~4× throughput of SSE

### Memory Efficiency

```aria
// f32 array: 4 bytes per element
f32[1000]:small_array;  // 4 KB

// f64 array: 8 bytes per element
f64[1000]:large_array;  // 8 KB (2× memory)

// Cache impact
// L1 cache: ~32 KB
// Can fit 8000 f32 values vs 4000 f64 values
```

### GPU Performance

**CUDA/GPU Compute**:
- f32: Maximum throughput (native)
- f64: Often 2-32× slower than f32 (depending on GPU)

**Graphics Shaders**:
- f32: Standard precision (fast)
- f16: Half precision (2× faster, limited precision)
- f64: Rarely supported in shaders

## Common Gotchas

### 1. 0.1 + 0.2 ≠ 0.3

```aria
f32:a = 0.1;
f32:b = 0.2;
f32:sum = a + b;

sum == 0.3;  // false!
// Reason: 0.1, 0.2, 0.3 cannot be represented exactly in binary

// sum is approximately 0.30000001192092896
```

### 2. NaN != NaN

```aria
f32:nan = f32.nan;
nan == nan;  // false (only value not equal to itself!)

// MUST use is_nan()
bool:is_nan = nan.is_nan();  // true
```

### 3. Associativity Lost

```aria
f32:a = 1e20;
f32:b = -1e20;
f32:c = 1.0;

(a + b) + c;  // 1.0 (correct)
a + (b + c);  // 0.0 (c lost in rounding with b!)

// Order of operations matters!
```

### 4. Negative Zero

```aria
f32:neg_zero = -0.0;
neg_zero == 0.0;  // true (compares equal)

// But behaves differently
1.0 / neg_zero;   // -Infinity
1.0 / 0.0;        // +Infinity
```

### 5. Large + Small Vanishes

```aria
f32:large = 1e7;
f32:small = 0.1;
f32:result = large + small;  // 10000000.0 (0.1 lost!)

// 0.1 is insignificant at this magnitude
```

### 6. Integer Precision Limit

```aria
f32:x = 16777216.0;  // 2^24
f32:y = x + 1.0;

x == y;  // true! (can't represent 16777217.0)
```

### 7. Subnormal Performance

```aria
// Very small numbers (near zero) use subnormal representation
f32:tiny = 1e-40;  // Subnormal (slower arithmetic)

// Can be 10-100× slower than normal numbers on some CPUs
// Consider flushing to zero for performance-critical code
```

## Summary

**f32** is the **performance-oriented floating-point type** for approximate arithmetic:

**Use f32 when**:
- Graphics, audio, physics, machine learning
- Performance critical (SIMD, GPU)
- ~7 digits of precision sufficient
- Memory/bandwidth matters

**Avoid f32 when**:
- Exact decimal values needed (use `fix256`)
- More than 7 digits precision (use `f64`)
- Counting or indexing (use `i32`)
- Financial calculations (use `fix256`)

**Key Principles**:
1. **Never use `==`** for comparisons (use epsilon)
2. **Always check NaN/Infinity** in user input
3. **Accumulate carefully** (small + large loses small)
4. **7 digits precision** is the limit
5. **Integer precision** only up to ±16,777,216

**Strengths**:
- ✓ Extremely fast (native hardware)
- ✓ SIMD vectorization (4/8/16 values in parallel)
- ✓ GPU acceleration
- ✓ Half the memory of f64
- ✓ Wide range (±3.4×10³⁸)

**Limitations**:
- ✗ Approximate, not exact
- ✗ Rounding errors accumulate
- ✗ NaN/Infinity edge cases
- ✗ Limited integer precision
- ✗ Order of operations matters

The `f32` type is the **workhorse of real-time computing**—fast, efficient, and good enough for most visual and audio applications. Just respect its limitations and validate your inputs!
