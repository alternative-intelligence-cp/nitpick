# u64: 64-Bit Unsigned Integer Type

## Overview

**Purpose**: The `u64` type represents 64-bit unsigned integers for large non-negative whole numbers. It's the modern standard for file sizes, memory addresses, high-resolution timestamps, and large-scale counting.

**Characteristics**:
- Size: 64 bits (8 bytes)
- Range: 0 to 18,446,744,073,709,551,615 (0 to ~18.4 quintillion)
- Always non-negative (unsigned)
- Wrapping overflow and underflow (modular arithmetic)
- Zero-overhead on 64-bit CPUs
- Binary representation: All 64 bits for magnitude (no sign bit)

**When to Use u64**:
- File sizes >4GB (modern standard)
- Memory addresses (64-bit pointers)
- High-resolution timestamps (nanoseconds, milliseconds)
- Large-scale counting (>4.3 billion)
- Network byte counters (petabytes of traffic)
- Database record IDs (billions of entries)
- Performance counters (CPU cycles)
- Large array sizes (>4.3B elements)
- Filesystem operations (inodes, block counts)
- C interop (`uint64_t`, `size_t` on 64-bit, `uintptr_t`)

**When NOT to Use u64**:
- Values that can be negative (use `i64` instead)
- Small values that fit in u32 (<4.3B, save memory)
- Subtraction that might go below zero (underflow wraps to MAX!)
- Mixed arithmetic with signed types (conversion issues)
- Security-critical calculations (wrapping can create vulnerabilities)
- When overflow detection needed (use bounds-checked types)
- Loop counters that decrement to zero (underflow trap)

## u64 vs u32: Size Selection

The choice between `u64` (8 bytes) and `u32` (4 bytes) depends on maximum value needed and memory considerations.

### Key Differences

| Aspect | u32 (32-bit) | u64 (64-bit) | Notes |
|--------|--------------|--------------|-------|
| **Range** | 0 to 4.3 billion | 0 to 18.4 quintillion | u64 is 4 billion× larger |
| **Size** | 4 bytes | 8 bytes | u64 uses 2× memory |
| **Max File Size** | 4GB | 18 exabytes | u64 for modern files |
| **Timestamp Range** | Year 2106 (seconds) | Year 2554 (nanoseconds) | u64 for high-res time |
| **Performance** | Same on 64-bit | Same on 64-bit | No speed difference |
| **SIMD Width** | 4/8/16 values | 2/4/8 values | u32 better vectorization |
| **Cache** | 2× more per line | 1× | u32 better cache density |

### When to Choose u64 Over u32

**Modern file sizes**:
```aria
// Files >4GB (standard for video, databases, backups)
u64:video_size = 549_755_813_888;  // 512GB 4K video
u64:database_size = 1_099_511_627_776;  // 1TB database
u64:backup_size = 5_497_558_138_880;  // 5TB backup

// Filesystem operations
u64:total_space = get_disk_space();  // Might be >4GB
u64:free_space = get_free_space();
u64:file_offset = seek_to_position(file, 10_000_000_000);  // 10GB offset
```

**High-resolution timestamps**:
```aria
// Nanosecond precision (u32 would overflow in ~4.3 seconds!)
u64:start_ns = get_timestamp_nanoseconds();
u64:end_ns = get_timestamp_nanoseconds();
u64:elapsed_ns = end_ns - start_ns;

// Milliseconds since epoch (good until year 2554)
u64:timestamp_ms = get_timestamp_milliseconds();
// u32 timestamp_ms would overflow in year 2106
```

**Large-scale counting**:
```aria
// Traffic beyond 4.3 billion
u64:total_requests = 50_000_000_000;  // 50 billion requests
u64:bytes_transferred = 5_000_000_000_000_000;  // 5 petabytes
u64:records_processed = 100_000_000_000;  // 100 billion records
```

**Memory addresses (64-bit systems)**:
```aria
// Pointer values (64-bit virtual address space)
u64:address = get_memory_address(ptr);
u64:heap_start = 0x0000_7FFF_0000_0000;  // Typical user space
```

### When to Choose u32 Over u64

**Small values (<4.3 billion)**:
```aria
// Array lengths that won't exceed 4.3B
u32:length = 10_000_000;  // 10 million (fits easily)
u32:pixel_count = width * height;  // < 4.3B pixels

// Counters unlikely to exceed billions
u32:connection_count = active_connections.length();
```

**Memory-constrained environments**:
```aria
// Embedded system: Save memory
struct SensorData {
    u32:timestamp;     // 4 bytes (vs 8 for u64)
    u32:sample_count;  // 4 bytes
    u32:checksum;      // 4 bytes
}  // 12 bytes total (vs 24 for u64 fields)

// Array of 10 million readings
// u32: 120MB, u64: 240MB (2× memory!)
```

**Legacy 32-bit compatibility**:
```aria
// Interop with 32-bit systems or file formats
u32:legacy_file_size;  // Old format limited to 4GB
u32:ipv4_address;  // Exactly 32 bits
u32:crc32_checksum;  // Standard is 32-bit
```

### Decision Matrix

```
Choose u64 when:
✓ Files >4GB (modern standard)
✓ Memory addresses (64-bit pointers)
✓ Nanosecond/microsecond timestamps
✓ Counts >4.3 billion
✓ Network traffic totals
✓ Future-proofing (avoid u32 limits)
✓ Database record IDs (billions of rows)

Choose u32 when:
✓ Values guaranteed <4.3 billion
✓ Memory matters (embedded, mobile, arrays)
✓ Legacy 32-bit compatibility
✓ Array indices for reasonable-sized arrays
✓ Hash values (32-bit standard)
✓ IPv4 addresses (exactly 32 bits)

Modern default: u64 for file sizes, u32 for counts
```

## u64 vs i64: Signed vs Unsigned

The choice between `u64` (unsigned) and `i64` (signed) determines whether negative values are possible.

### Key Differences

| Aspect | u64 (Unsigned) | i64 (Signed) | Notes |
|--------|----------------|--------------|-------|
| **Range** | 0 to 18.4 quintillion | -9.2 to +9.2 quintillion | u64 has 2× positive range |
| **Negative Values** | ❌ Never | ✅ Yes | u64 wraps on underflow |
| **Overflow** | Wraps to 0 | Wraps to MIN | Both defined behavior |
| **Underflow** | Wraps to MAX | Wraps to MAX | **Critical danger for u64!** |
| **Sign Bit** | No (all 64 bits for value) | Yes (bit 63) | u64 uses extra bit |
| **Timestamps** | Year 2554 (ns) | Year 2262 (ns) | u64 extends range |

### The Underflow Trap (Magnified!)

**CRITICAL**: u64 underflow creates astronomically large values:

```aria
// DANGER: Underflow wrapping
u64:count = 5;
count -= 10;  // Wraps to 18,446,744,073,709,551,611! (not -5)

// Even more dangerous at scale
u64:bytes_sent = 1_000;
u64:bytes_received = 5_000;
u64:net = bytes_sent - bytes_received;  
// Wraps to 18,446,744,073,709,547,616 (looks like petabytes!)

// CORRECT: Use i64 for values that can be negative
i64:net_signed = bytes_sent.to_i64() - bytes_received.to_i64();
// -4,000 (correct)
```

### When to Choose u64 Over i64

**File sizes and byte counts** (never negative):
```aria
// File operations
u64:file_size = get_file_size(path);
u64:bytes_written = write_to_file(data);
u64:total_disk_usage = calculate_usage();

// Network traffic
u64:bytes_sent = 0;
u64:bytes_received = 0;
loop {
    bytes_sent += send_packet(data);
    bytes_received += receive_packet(buffer);
}
```

**Memory addresses** (always positive):
```aria
// Pointer arithmetic
u64:address = get_address(ptr);
u64:offset = address - base_address;  // Safe if address > base
u64:next_page = address + 4096;
```

**Timestamps** (Unix epoch is positive only):
```aria
// Milliseconds since epoch
u64:timestamp_ms = get_current_time_ms();

// Nanoseconds (need u64 for range)
u64:timestamp_ns = get_current_time_ns();
```

### When to Choose i64 Over u64

**Values that might be negative**:
```aria
// Time differences (can be negative if reversed)
i64:time_diff = end_time.to_i64() - start_time.to_i64();
if (time_diff < 0) {
    error("End time before start time");
}

// File offsets (seek backwards from end)
i64:offset = -1024;  // Seek 1KB from end
file.seek(offset, SEEK_END);

// Temperature, altitude, balance (can be negative)
i64:temperature = -40;  // -40°C
i64:altitude = -400;  // 400m below sea level
i64:balance = -1000;  // Overdrawn
```

**Subtraction results**:
```aria
// Differences between values
i64:delta = new_value.to_i64() - old_value.to_i64();
if (delta < 0) {
    log("Decreased by ${-delta}");
} else {
    log("Increased by ${delta}");
}
```

### Decision Matrix

```
Choose u64 when:
✓ Value CANNOT be negative (semantic guarantee)
✓ Need full 18.4 quintillion positive range
✓ File sizes, byte counts
✓ Memory addresses
✓ Unix timestamps (seconds/ms/ns since epoch)
✓ Counts that can't be negative

Choose i64 when:
✓ Value MIGHT be negative
✓ Time differences, offsets
✓ General-purpose large integer
✓ Subtraction involved (risk of underflow)
✓ Range fits in ±9.2 quintillion
✓ Mixed arithmetic with signed types

Default: i64 for general use, u64 for sizes/addresses
```

## Range and Representation

### Numeric Range

**Minimum**: 0  
**Maximum**: 18,446,744,073,709,551,615 (2⁶⁴ - 1)  
**Total values**: 18,446,744,073,709,551,616 (2⁶⁴)

**Approximately**: 18.4 quintillion (18.4 × 10¹⁸)

**In different bases**:
```aria
u64:max_decimal = 18_446_744_073_709_551_615;
u64:max_hex = 0xFFFF_FFFF_FFFF_FFFF;
u64:max_binary = 0b11111111_11111111_11111111_11111111_11111111_11111111_11111111_11111111;

// Common boundaries
u64:zero = 0;
u64:one_trillion = 1_000_000_000_000;
u64:one_quadrillion = 1_000_000_000_000_000;
u64:max = 18_446_744_073_709_551_615;
```

### Scale Comparison

```aria
// 1 thousand
u64:thousand = 1_000;

// 1 million
u64:million = 1_000_000;

// 1 billion (u32 max is 4.3 billion)
u64:billion = 1_000_000_000;

// 1 trillion (needs u64)
u64:trillion = 1_000_000_000_000;

// 1 quadrillion (needs u64)
u64:quadrillion = 1_000_000_000_000_000;

// 1 quintillion (near u64 max)
u64:quintillion = 1_000_000_000_000_000_000;

// u64 max: ~18.4 quintillion
```

### Binary Representation

All 64 bits represent magnitude (no sign bit):

```
Bit positions:  63 62 61 ... 2  1  0
Value:          0  0  0  ... 0  0  0  = 0
Value:          1  1  1  ... 1  1  1  = 18,446,744,073,709,551,615

Example: 1,000 in u64
Binary: 00000000 00000000 00000000 00000000 00000000 00000000 00000011 11101000
Hex:    0x0000000000000 3E8
```

**Comparison with i64** (same 64 bits, different interpretation):
```aria
// Same bit pattern, different meaning
u64:unsigned_max = 0xFFFF_FFFF_FFFF_FFFF;  // 18,446,744,073,709,551,615
i64:signed = -1;                            // Same bits: 0xFFFFFFFFFFFFFFFF

// Conversion (bit reinterpretation)
i64:negative = -1;
u64:as_unsigned = negative.to_u64();  // 18,446,744,073,709,551,615
```

## Overflow and Underflow

### Overflow (Addition Beyond MAX)

u64 wraps modulo 2⁶⁴:

```aria
// Overflow wrapping
u64:max = 18_446_744_073_709_551_615;
u64:overflow = max + 1;  // Wraps to 0

u64:near_max = 18_446_744_073_709_551_610;
near_max += 10;  // Wraps to 4

// Large multiplication overflow
u64:large = 10_000_000_000;  // 10 billion
u64:result = large * large;  // Overflows! (10²⁰ > 10¹⁹)

// Checking for overflow before operation
u64:safe_add(u64:a, u64:b) -> Result<u64> {
    if (a > u64.max - b) {
        return Error("Overflow would occur");
    }
    return Ok(a + b);
}
```

### Underflow (Subtraction Below Zero)

**CRITICAL DANGER**: u64 underflow creates massive values:

```aria
// Classic underflow bug
u64:count = 5;
count -= 10;  // Wraps to 18,446,744,073,709,551,611 (!!)

// Real-world scenario
u64:available_memory = 8_589_934_592;  // 8GB
u64:requested_memory = 17_179_869_184;  // 16GB
u64:remaining = available_memory - requested_memory;  
// Wraps to huge number instead of going negative!

if (remaining < 1_000_000_000) {
    // Never executes because remaining is enormous
    error("Low memory");
}

// CORRECT: Check before subtraction
if (requested_memory > available_memory) {
    error("Insufficient memory");
} else {
    u64:remaining = available_memory - requested_memory;
}
```

### Loop Underflow Trap

```aria
// WRONG: Decrementing u64 loop (infinite!)
u64:i = 10;
while (i >= 0) {  // ALWAYS TRUE (u64 always >= 0)
    process(i);
    i -= 1;  // At i=0, wraps to 18,446,744,073,709,551,615
}

// CORRECT: Use i64 or count up
i64:i = 10;
while (i >= 0) {
    process(i.to_u64());
    i -= 1;  // Goes to -1, exits loop
}

// ALTERNATIVE: Count up
u64:count = 10;
loop (count:i) {
    u64:reverse = count - i - 1;
    process(reverse);
}
```

## Operations

### Arithmetic

```aria
u64:a = 1_000_000;
u64:b = 500_000;

u64:sum = a + b;          // 1,500,000
u64:difference = a - b;   // 500,000 (safe: a > b)
u64:product = a * b;      // 500,000,000,000
u64:quotient = a / b;     // 2
u64:remainder = a % b;    // 0

// Compound assignment
a += b;   // 1,500,000
a -= b;   // 1,000,000
a *= 2;   // 2,000,000
a /= 10;  // 200,000
a %= 7;   // 4
```

### Large Number Arithmetic

```aria
// Adding terabytes
u64:file1_size = 1_099_511_627_776;  // 1TB
u64:file2_size = 2_199_023_255_552;  // 2TB
u64:total_size = file1_size + file2_size;  // 3TB

// Multiplying for total bytes
u64:sectors = 1_000_000_000;  // 1 billion sectors
u64:sector_size = 512;
u64:total_bytes = sectors * sector_size;  // 512GB
```

### Bitwise Operations

```aria
u64:a = 0xFFFF_0000_0000_0000;
u64:b = 0x0000_FFFF_0000_0000;

u64:and = a & b;      // 0x0000_0000_0000_0000
u64:or = a | b;       // 0xFFFF_FFFF_0000_0000
u64:xor = a ^ b;      // 0xFFFF_FFFF_0000_0000
u64:not = ~a;         // 0x0000_FFFF_FFFF_FFFF

// Bit shifts (large shifts possible)
u64:value = 1;
u64:bit32 = value << 32;  // 0x0000_0001_0000_0000
u64:bit63 = value << 63;  // 0x8000_0000_0000_0000

// Memory address alignment
u64:address = 0x7FFF_0000_1234;

```

### Comparisons

```aria
u64:a = 1_000_000_000_000;
u64:b = 500_000_000_000;

bool:equal = (a == b);        // false
bool:not_equal = (a != b);    // true
bool:greater = (a > b);       // true
bool:less = (a < b);          // false

// Always false (u64 never negative)
bool:is_negative = (a < 0);  // Always false!

// Checking for zero
bool:is_zero = (a == 0);
bool:is_nonzero = (a != 0);
```

## Type Conversions

### From Smaller Unsigned Types

Safe (no data loss):

```aria
// u8 → u64 (always safe)
u8:byte = 255;
u64:extended = byte.to_u64();  // 255

// u32 → u64 (always safe)
u32:value = 4_000_000_000;
u64:big = value.to_u64();  // 4,000,000,000 (exact)
```

### To Smaller Unsigned Types

Truncates (data loss if too large):

```aria
// u64 → u32 (truncates if >4.3B)
u64:large = 5_000_000_000;  // 5 billion
u32:truncated = large.to_u32();  // Wraps! (loses high bits)

// u64 → u8 (truncates to lowest 8 bits)
u64:value = 300;
u8:byte = value.to_u8();  // 44 (300 mod 256)

// Safe conversion with bounds check
u64:safe_to_u32(u64:value) -> Result<u32> {
    if (value > u32.max.to_u64()) {
        return Error("Value too large for u32");
    }
    return Ok(value.to_u32());
}
```

### From Signed Types

Bit reinterpretation (negative becomes large positive):

```aria
// i64 → u64 (bit pattern unchanged)
i64:negative = -1;
u64:as_unsigned = negative.to_u64();  // 18,446,744,073,709,551,615

i64:small_neg = -1000;
u64:wrapped = small_neg.to_u64();  // 18,446,744,073,709,550,616

// Safe conversion (reject negatives)
i64:safe_to_u64(i64:value) -> Result<u64> {
    if (value < 0) {
        return Error("Cannot convert negative to unsigned");
    }
    return Ok(value.to_u64());
}
```

### To Signed Types

Wraps if > i64.max:

```aria
// u64 → i64 (values >9.2 quintillion wrap to negative)
u64:small = 1_000_000;
i64:safe = small.to_i64();  // 1,000,000

u64:large = 10_000_000_000_000_000_000;  // 10 quintillion
i64:wrapped = large.to_i64();  // Negative! (wraps)

// Safe conversion
u64:safe_to_i64(u64:value) -> Result<i64> {
    if (value > i64.max.to_u64()) {
        return Error("Value too large for i64");
    }
    return Ok(value.to_i64());
}
```

## Common Patterns

### File Operations

```aria
// Reading large files
struct FileMetadata {
    string:path;
    u64:size_bytes;
    u64:modification_time;
    u64:inode;
}

FileMetadata:get_file_info(string:path) -> Result<FileMetadata> {
    // Filesystem API calls
    u64:size = get_file_size(path);
    u64:mtime = get_modification_time(path);
    u64:inode = get_inode_number(path);
    
    return Ok(FileMetadata {
        path: path,
        size_bytes: size,
        modification_time: mtime,
        inode: inode
    });
}

// Processing large files in chunks
void:process_large_file(string:path, u64:chunk_size) {
    u64:file_size = get_file_size(path);
    u64:offset = 0;
    
    while (offset < file_size) {
        u64:bytes_to_read = min(chunk_size, file_size - offset);
        u8[]:chunk = read_file_chunk(path, offset, bytes_to_read);
        
        process_chunk(chunk);
        
        offset += bytes_to_read;
    }
}
```

### High-Resolution Timestamps

```aria
// Nanosecond precision timing
struct Timer {
    u64:start_ns;
}

Timer:start() -> Timer {
    return Timer { start_ns: get_timestamp_nanoseconds() };
}

u64:elapsed_ns(Timer:timer) -> u64 {
    u64:end_ns = get_timestamp_nanoseconds();
    return end_ns - timer.start_ns;
}

// Usage
Timer:timer = start();
perform_operation();
u64:duration_ns = elapsed_ns(timer);

f64:duration_ms = duration_ns.to_f64() / 1_000_000.0;
print("Operation took ${duration_ms} ms\n");
```

### Network Traffic Statistics

```aria
// Tracking petabytes of data
struct NetworkStats {
    u64:bytes_sent;
    u64:bytes_received;
    u64:packets_sent;
    u64:packets_received;
    u64:errors;
}

void:update_stats(NetworkStats&:stats, u64:bytes, bool:is_send) {
    if (is_send) {
        stats.bytes_sent += bytes;
        stats.packets_sent += 1;
    } else {
        stats.bytes_received += bytes;
        stats.packets_received += 1;
    }
    
    // Check for overflow (unlikely but possible with u64)
    if (stats.bytes_sent < bytes) {
        error("Byte counter overflow!");
    }
}

// Display in human-readable format
string:format_bytes(u64:bytes) -> string {
    if (bytes >= 1_000_000_000_000_000) {  // Petabytes
        f64:pb = bytes.to_f64() / 1_000_000_000_000_000.0;
        return "${pb:.2f} PB";
    } else if (bytes >= 1_000_000_000_000) {  // Terabytes
        f64:tb = bytes.to_f64() / 1_000_000_000_000.0;
        return "${tb:.2f} TB";
    } else if (bytes >= 1_000_000_000) {  // Gigabytes
        f64:gb = bytes.to_f64() / 1_000_000_000.0;
        return "${gb:.2f} GB";
    } else {
        return "${bytes} bytes";
    }
}
```

### Memory Address Calculations

```aria
// Virtual memory management
struct MemoryRegion {
    u64:start_address;
    u64:size_bytes;
}

bool:contains_address(MemoryRegion:region, u64:address) {
    u64:end_address = region.start_address + region.size_bytes;
    return address >= region.start_address && address < end_address;
}

u64:align_up(u64:value, u64:alignment) -> u64 {
    u64:mask = alignment - 1;
    return (value + mask) & ~mask;
}

u64:align_down(u64:value, u64:alignment) -> u64 {
    u64:mask = alignment - 1;
    return value & ~mask;
}

// Example: Page-align addresses
u64:address = 0x7FFF_0000_1234;
u64:page_aligned = align_down(address, 4096);  // 0x7FFF_0000_1000
```

## Anti-Patterns

### ❌ Underflow in Loops

```aria
// WRONG: Decrementing u64 loop (infinite!)
u64:i = 100;
while (i >= 0) {  // ALWAYS TRUE!
    process(i);
    i -= 1;  // At i=0, wraps to 18,446,744,073,709,551,615
}

// CORRECT: Use i64 or count up
i64:i = 100;
while (i >= 0) {
    process(i.to_u64());
    i -= 1;
}
```

### ❌ Using u64 for Values That Can Be Negative

```aria
// WRONG: Time difference in u64
u64:start = 1000;
u64:end = 500;
u64:duration = end - start;  // Wraps to huge number!

// CORRECT: Use i64 for differences
i64:duration = end.to_i64() - start.to_i64();
if (duration < 0) {
    error("End before start");
}
```

### ❌ Ignoring Overflow in Accumulation

```aria
// WRONG: No overflow checking
u64:total = 0;
loop (data.length():i) {
    total += data[i];  // Might overflow at extreme scale
}

// CORRECT: Check for overflow (or use i128 for accumulation)
u64:total = 0;
loop (data.length():i) {
    if (total > u64.max - data[i]) {
        error("Accumulation overflow");
        break;
    }
    total += data[i];
}
```

### ❌ Using u64 for Small Values

```aria
// WRONG: Wasting memory with u64 for small values
struct Point {
    u64:x;  // Coordinates rarely exceed 4.3B
    u64:y;
    u64:z;
}  // 24 bytes

// CORRECT: Use u32 for reasonable-sized coordinates
struct Point {
    u32:x;
    u32:y;
    u32:z;
}  // 12 bytes (half the memory)
```

### ❌ Mixing Signed and Unsigned Without Validation

```aria
// WRONG: Implicit conversion
i64:signed = -100;
u64:unsigned = 1000;

if (signed < unsigned) {  // DANGER!
    // Implicit conversion: -100 → huge positive
}

// CORRECT: Explicit handling
if (signed < 0 || signed.to_u64() < unsigned) {
    // Properly handles negative
}
```

## Best Practices

### ✓ Use u64 for Modern File Sizes

```aria
// Always u64 for file operations (>4GB standard now)
u64:file_size = get_file_size(path);
u64:bytes_written = write_file(path, data);
```

### ✓ Check Bounds Before Subtraction

```aria
// Validate subtraction won't underflow
if (b > a) {
    error("Underflow would occur");
} else {
```

## Performance Characteristics

### CPU Operations

**Native 64-bit instructions** on 64-bit CPUs (same as u32):

| Operation | Cycles | Notes |
|-----------|--------|-------|
| Addition/Subtraction | 1 | Fully pipelined |
| Multiplication | 3-5 | Fast on modern CPUs |
| Division | 10-30 | Slower, avoid in hot loops |
| Bitwise ops | 1 | AND, OR, XOR, NOT |
| Bit shifts | 1 | Left, right shifts |
| Comparisons | 1 | Equal, less, greater |

**32-bit CPUs**: u64 operations require multiple instructions (slower).

### SIMD Performance

**Vectorization** (process multiple u64 values in parallel):

- **SSE (128-bit)**: Process 2× u64 values per instruction
- **AVX2 (256-bit)**: Process 4× u64 values per instruction
- **AVX-512 (512-bit)**: Process 8× u64 values per instruction

```aria
// SIMD-friendly: Compilers can auto-vectorize
void:add_arrays(u64[]:a, u64[]:b, u64[]:result) {
    loop (a.length():i) {
        result[i] = a[i] + b[i];  // Vectorizes to 2/4/8 at a time
    }
}
```

### Memory Efficiency

**Size**: 8 bytes per value (2× u32)

```aria
// Array memory usage
u64[1_000_000]:data;  // 8MB

// vs u32
u32[1_000_000]:data;  // 4MB (half size)

// Trade-off: Range vs memory
```

**Cache efficiency**: u64 arrays fit half as many values per cache line as u32.

## Common Gotchas

### 1. Underflow Wrapping (Most Common!)

```aria
u64:count = 100;
count -= 200;  // Wraps to 18,446,744,073,709,551,516 (not -100!)

// Always check before subtraction
if (decrement > count) {
    error("Underflow");
}
```

### 2. Always >= 0 Checks

```aria
u64:value = get_value();
if (value < 0) {  // ALWAYS FALSE!
    // Never executes
}
```

### 3. Overflow in Multiplication

```aria
u64:large = 10_000_000_000;  // 10 billion
u64:result = large * large;  // Overflows! (100 quintillion > 18.4)

// Check or use wider type for intermediate
```

### 4. Mixing with Signed Types

```aria
i64:signed = -1;
u64:unsigned = 10;

bool:comparison = (signed < unsigned);  // FALSE! (-1 → huge positive)
```

### 5. Division Truncates

```aria
u64:a = 7;
u64:b = 3;
u64:result = a / b;  // 2 (not 2.333...)

// Use f64 for fractional results
f64:quotient = a.to_f64() / b.to_f64();
```

### 6. Two's Complement Confusion

```aria
// Bitwise NOT doesn't make negative
u64:value = 5;
u64:inverted = ~value;  // 18,446,744,073,709,551,610 (not -6!)

// Use i64 if you need negation
```

### 7. Pointer Size Assumptions

```aria
// u64 is NOT always pointer size
// 32-bit systems: pointers are 32-bit
// 64-bit systems: pointers are 64-bit

// Use usize for portable pointer-sized values
```

## Summary

**u64** is the **large unsigned integer** for modern computing:

**Use u64 when**:
- File sizes >4GB
- Memory addresses (64-bit)
- Nanosecond timestamps
- Large counts >4.3B
- Network traffic totals
- Future-proofing

**Avoid u64 when**:
- Values can be negative (use `i64`)
- Small values <4.3B (use `u32` to save memory)
- Subtraction might underflow

**Key Principles**:
1. **Never negative** (always >= 0)
2. **Underflow wraps to MAX** (18.4 quintillion!)
3. **Check before subtracting**
4. **Modern default for files** (>4GB standard)
5. **Use i64 for differences**

**Strengths**:
- ✓ 18.4 quintillion range
- ✓ Modern file size standard
- ✓ High-resolution timestamps
- ✓ Native 64-bit performance
- ✓ Memory address capable

**Limitations**:
- ✗ No negative values
- ✗ Underflow wraps silently
- ✗ 2× memory vs u32
- ✗ Mixing with signed is error-prone

The `u64` type is the **modern standard** for large non-negative values. Perfect for files, timestamps, and addresses in 64-bit computing!
