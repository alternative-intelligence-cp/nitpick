# i64: 64-bit Signed Integer

## Overview

**Purpose**: Large signed integers for values beyond ±2.1 billion, modern timestamps, and big data.

**Characteristics**:
- 64-bit two's complement signed integer
- Range: -9,223,372,036,854,775,808 to +9,223,372,036,854,775,807 (approximately ±9.2 quintillion)
- Wrapping overflow (defined behavior, not UB)
- Zero overhead on 64-bit CPUs
- Asymmetric range (one more negative value than positive)
- Full name: `int64`, common alias: `i64`

**When to Use**:
- **Modern timestamps** (nanoseconds since epoch - good until year 2262!)
- **Large file sizes** (files > 2GB, up to 8 exabytes)
- **Memory addresses** (64-bit pointers, offsets)
- **Large datasets** (genomic data, astronomical counts)
- **High-value financial** (transactions > $21 million in cents)
- **Accumulation** (prevent overflow when summing many i32 values)
- **Large array sizes** (billions of elements)

**When NOT to Use**:
- Small values where i32 sufficient (wastes 4 bytes per value)
- Memory-critical applications with billions of small integers
- Performance-critical tight loops on 32-bit platforms (2× slower)
- When overflow detection needed (use `tbb64` for error propagation)
- Always-positive large values (consider `u64` for unsigned semantics)

## i64 vs i32: When to Use Which

### Size and Range Comparison

```aria
// i32: ±2.1 billion
i32:small_max = 2147483647;           // ~2.1 × 10^9
i32:small_min = -2147483648;

// i64: ±9.2 quintillion
i64:large_max = 9223372036854775807;  // ~9.2 × 10^18
i64:large_min = -9223372036854775808;
```

**i64 range is ~4.3 billion times larger than i32!**

### Decision Matrix

| Use Case | Type | Reason |
|----------|------|--------|
| Loop counter (< 2B iterations) | `i32` | Sufficient range, saves memory |
| Loop counter (> 2B iterations) | `i64` | Exceeds i32 range |
| Array index (< 2B elements) | `i32` | Standard, sufficient |
| Array index (> 2B elements) | `i64` | Exceeds i32 range |
| **Unix timestamp (seconds)** | `i64` | **i32 causes Year 2038 problem!** |
| **Nanosecond timestamp** | `i64` | **Required (good until 2262)** |
| File size (< 2GB) | `i32` | Sufficient |
| File size (> 2GB) | `i64` | Required for large files |
| Database ID (< 2B records) | `i32` | Standard |
| Database ID (> 2B records) | `i64` | Exceeds i32 range |
| Memory offset/pointer | `i64` | 64-bit addressing |
| Financial (< $21M) | `i32` | Sufficient (in cents) |
| Financial (> $21M) | `i64` | Large transactions |
| Accumulation of many values | `i64` | Prevent overflow |
| Milliseconds since epoch | `i64` | Good for 290 million years |

### Memory vs Range Trade-off

```aria
// i32 array: 1 million elements
i32:data_32[1000000];
// Memory: 4 MB (4 bytes × 1 million)

// i64 array: 1 million elements
i64:data_64[1000000];
// Memory: 8 MB (8 bytes × 1 million) - 2× larger!

// For small values, i32 saves memory
// For large values, i64 prevents overflow
```

### When i32 Overflows, Use i64

```aria
// WRONG: i32 overflow for large multiplication
i32:a = 100000;
i32:b = 50000;
i32:product = a * b;  // 5,000,000,000 - OVERFLOW! Wraps to negative

// RIGHT: Use i64 for large products
i64:a = 100000;
i64:b = 50000;
i64:product = a * b;  // 5,000,000,000 (fits!)

// Or widen before multiply
i64:product2 = i64(a) * i64(b);  // Safe
```

## i64 vs u64: Signed vs Unsigned

### The Critical Distinction

**i64**: Signed integer (can be negative)
- Purpose: Large values that can be negative
- Range: -9,223,372,036,854,775,808 to +9,223,372,036,854,775,807
- Intent: "This can be large AND negative" (deltas, offsets, signed computations)

**u64**: Unsigned integer (always non-negative)
- Purpose: Large quantities that cannot be negative
- Range: 0 to 18,446,744,073,709,551,615 (0 to ~18.4 quintillion)
- Intent: "This is always positive" (sizes, counts, addresses)

### Examples of the Difference

```aria
// Signed: Time delta (can be negative)
i64:time_delta_ns = end_time - start_time;  // Can be negative if end < start

// Unsigned: File size (cannot be negative)
u64:file_size = get_file_size("movie.mp4");  // Always positive

// Signed: Memory offset (can go backward)
i64:offset = current_position - base_address;  // Can be negative

// Unsigned: Memory address (always forward from 0)
u64:address = 0x7FFF_FFFF_0000;  // Memory address (always positive)
```

### Conversion Dangers

```aria
// DANGER: Negative i64 → u64 wraps to huge positive!
i64:negative = -1;
u64:unsigned = u64(negative);  // 18,446,744,073,709,551,615 (wrapped!)

// Safe: Check before converting
i64:value = get_delta();
if (value < 0) {
    log.error("Cannot convert negative to unsigned\n");
} else {
    u64:safe = u64(value);  // Only convert if positive
}
```

## Nanosecond Timestamps: The Year 2262 Solution

### Why i64 Nanoseconds?

**i32 seconds problem**:
- Range: ±2.1 billion seconds
- From 1970: Overflows in **2038** (68 years total)
- Silent wraparound causes disasters

**i64 nanoseconds solution**:
- Range: ±9.2 quintillion nanoseconds
- From 1970: Good until **2262** (292 years total)
- High precision (billionth of a second)
- Never wraps in any realistic scenario

### Nanosecond Timestamp Pattern

```aria
// Get current time as i64 nanoseconds since Unix epoch
func get_current_time_ns(): i64 {
    // Returns nanoseconds since January 1, 1970, 00:00:00 UTC
    return system_time_nanoseconds();
}

// Usage
i64:now = get_current_time_ns();
// Example: 1739318400000000000 (Feb 12, 2025)

// High-precision event timing
i64:start = get_current_time_ns();
perform_operation();
i64:end = get_current_time_ns();
i64:elapsed_ns = end - start;

log.write("Operation took ");
log.write_i64(elapsed_ns);
log.write(" nanoseconds\n");
```

### Timestamp Conversions

```aria
// Nanoseconds ↔ Seconds
i64:timestamp_ns = 1739318400000000000;
i64:timestamp_s = timestamp_ns / 1000000000;  // Seconds

i64:seconds = 1739318400;
i64:nanoseconds = seconds * 1000000000;  // Nanoseconds

// Nanoseconds ↔ Milliseconds
i64:timestamp_ms = timestamp_ns / 1000000;  // Milliseconds
i64:ns_from_ms = timestamp_ms * 1000000;    // Back to nanoseconds

// Nanoseconds ↔ Microseconds
i64:timestamp_us = timestamp_ns / 1000;  // Microseconds
i64:ns_from_us = timestamp_us * 1000;    // Back to nanoseconds
```

### High-Resolution Event Logging

```aria
struct Event {
    i64:timestamp_ns;
    string:event_type;
    string:data;
}

Event:events[1000];
i32:event_count = 0;

func log_event(event_type: string, data: string) {
    events[event_count] = Event{
        timestamp_ns: get_current_time_ns(),
        event_type: event_type,
        data: data
    };
    event_count += 1;
}

// Later: Compute precise time between events
i64:duration_ns = events[1].timestamp_ns - events[0].timestamp_ns;
f64:duration_ms = f64(duration_ns) / 1000000.0;  // Convert to milliseconds
```

## Large File Sizes

### Why i64 for File Sizes?

**i32 limit**: 2,147,483,647 bytes = ~2.14 GB (too small!)

**i64 limit**: 9,223,372,036,854,775,807 bytes = ~8 exabytes (8,192 petabytes)

Modern files frequently exceed 2GB:
- 4K movies: 50-100 GB
- Blu-ray images: 25-50 GB
- Game installations: 100+ GB
- Database files: Terabytes
- Scientific datasets: Petabytes

### File Size Pattern

```aria
// Get file size (returns i64 for large files)
func get_file_size(path: string): Result<i64> {
    // Open file, seek to end, return position
    return file_system_get_size(path);
}

// Usage
Result<i64>:size_result = get_file_size("large_video.mp4");

if (size_result.is_ok()) {
    i64:size_bytes = size_result.unwrap();
    
    // Convert to human-readable
    if (size_bytes < 1024) {
        log.write("${size_bytes} bytes\n");
    } else if (size_bytes < 1024 * 1024) {
        f64:kb = f64(size_bytes) / 1024.0;
        log.write("${kb} KB\n");
    } else if (size_bytes < 1024 * 1024 * 1024) {
        f64:mb = f64(size_bytes) / (1024.0 * 1024.0);
        log.write("${mb} MB\n");
    } else {
        f64:gb = f64(size_bytes) / (1024.0 * 1024.0 * 1024.0);
        log.write("${gb} GB\n");
    }
}
```

### Accumulating File Sizes

```aria
// Calculate total size of directory
i64:total_bytes = 0;

for (file in files) {
    Result<i64>:file_size = get_file_size(file);
    if (file_size.is_ok()) {
        total_bytes += file_size.unwrap();
    }
}

log.write("Total directory size: ");
log.write_i64(total_bytes);
log.write(" bytes\n");
```

## Memory Addresses and Offsets

### 64-bit Addressing

Modern systems use 64-bit memory addressing:
- Virtual address space: 0 to 2^64 - 1
- i64 (signed) range: Adequate for pointer arithmetic
- u64 (unsigned) range: Better semantic fit for addresses

```aria
// Memory offset calculation
i64:base_address = 0x7FFF_0000_0000;
i64:offset = 0x1234;
i64:target_address = base_address + offset;

// Pointer arithmetic (signed for bidirectional)
i64:current = get_buffer_position();
i64:start = get_buffer_start();
i64:bytes_processed = current - start;  // Can be negative if current < start
```

### Large Array Indexing

```aria
// Array with billions of elements
i64:array_size = 5000000000;  // 5 billion elements
f64:data[array_size];

for (i: i64 = 0; i < array_size; i += 1) {
    data[i] = f64(i) * 3.14159;
}

// i32 would overflow at 2.1 billion!
```

## Arithmetic Operations

### Basic Arithmetic

```aria
i64:a = 5000000000;
i64:b = 3000000000;

i64:sum = a + b;         // 8,000,000,000
i64:difference = a - b;  // 2,000,000,000
i64:product = a * b;     // 15,000,000,000,000,000,000 (15 quintillion)
i64:quotient = a / b;    // 1 (integer division)
i64:remainder = a % b;   // 2,000,000,000
```

### Overflow Still Possible!

```aria
// i64 can still overflow (just much larger range)
i64:max = 9223372036854775807;
i64:result = max + 1;  // -9223372036854775808 (wraps to MIN)

// Example: Extremely large multiplication
i64:huge = 1000000000000;  // 1 trillion
i64:product = huge * huge;  // Overflow! (1 septillion doesn't fit)

// For safety-critical: Use tbb64
tbb64:safe = 1000000000000;
tbb64:safe_product = safe * safe;
if (safe_product == ERR) {
    log.error("Multiplication overflow!\n");
}
```

### Mixed i32/i64 Arithmetic

```aria
i32:small = 1000000;
i64:large = 5000000000;

// Implicit widening (i32 → i64)
i64:result = large + small;  // OK: small widened to i64

// Multiplication
i64:product = large * small;  // OK

// DANGER: Overflow if i32 computed first!
i64:wrong = 2 * 2000000000;  // Overflow! 2 and 2000000000 both i32
i64:correct = 2i64 * 2000000000i64;  // OK: Both i64
```

## Type Conversions

### Widening (Always Safe)

```aria
// i8 → i64
i8:tiny = -100;
i64:large = i64(tiny);  // -100 (sign-extended)

// i32 → i64 (most common)
i32:medium = -2000000000;
i64:large2 = i64(medium);  // -2000000000 (sign-extended)

// Always safe, no data loss
```

### Narrowing (Check Range!)

```aria
// i64 → i32 (UNSAFE - can overflow!)
i64:large = 5000000000;  // 5 billion
// i32:truncated = i32(large);  // Truncates! Undefined result

// Safe narrowing with check
i64:value = 5000000000;
if (value >= -2147483648 && value <= 2147483647) {
    i32:safe = i32(value);
    // Use safe
} else {
    log.error("Value out of i32 range\n");
}

// Or use Result
func narrow_to_i32(value: i64): Result<i32> {
    if (value < -2147483648 || value > 2147483647) {
        return Err("Out of i32 range");
    }
    return Ok(i32(value));
}
```

### Float Conversions

```aria
// i64 → f64 (may lose precision for very large values)
i64:large = 9223372036854775807;  // i64:MAX
f64:float_val = f64(large);  // May lose precision (f64 has ~15-16 significant digits)

// f64 → i64 (truncates decimal)
f64:pi = 3.14159;
i64:truncated = i64(pi);  // 3 (decimal part discarded)

// Large float → i64
f64:large_float = 1.5e18;  // 1.5 quintillion
i64:converted = i64(large_float);  // 1,500,000,000,000,000,000
```

### Unsigned Conversions

```aria
// i64 → u64 (reinterpret bits)
i64:negative = -1;
u64:unsigned = u64(negative);  // 18,446,744,073,709,551,615 (wrapped)

// u64 → i64 (reinterpret bits)
u64:large_unsigned = 10000000000000000000;  // > i64:MAX
i64:signed = i64(large_unsigned);  // Negative! (bit pattern reinterpreted)
```

## Anti-Patterns

### ❌ Using i64 Everywhere (Memory Waste)

```aria
// WRONG: Using i64 for small values
i64:loop_counter = 0;  // Wastes 4 bytes if only counting to 100

for (i: i64 = 0; i < 100; i += 1) {  // i32 sufficient!
    process(i);
}

// RIGHT: Use i32 for small ranges
i32:loop_counter = 0;

for (i: i32 = 0; i < 100; i += 1) {
    process(i);
}
```

### ❌ Not Using i64 for Timestamps

```aria
// WRONG: i32 timestamp (Year 2038!)
i32:timestamp = get_current_time_seconds();  // Breaks in 2038!

// RIGHT: i64 nanoseconds
i64:timestamp_ns = get_current_time_nanoseconds();  // Good till 2262
```

### ❌ Narrowing Without Check

```aria
// WRONG: Blind narrowing from i64 to i32
i64:large = get_user_input_large();
i32:small = i32(large);  // Silently truncates if > i32:MAX!

// RIGHT: Validate range
i64:value = get_user_input_large();
if (value >= i32:MIN && value <= i32:MAX) {
    i32:safe = i32(value);
} else {
    log.error("Value exceeds i32 range\n");
}
```

### ❌ Ignoring i64 Overflow

```aria
// WRONG: Assuming i64 never overflows
i64:huge = 1000000000000;  // 1 trillion
i64:product = huge * huge;  // Overflow! (1 septillion doesn't fit)

// RIGHT: Check or use wider type for extreme values
// Or use tbb64 for error detection
tbb64:safe = 1000000000000;
tbb64:safe_product = safe * safe;
if (safe_product == ERR) {
    log.error("Overflow detected\n");
}
```

## Performance Characteristics

### 64-bit vs 32-bit Platforms

**64-bit CPU** (modern systems):
- i64 arithmetic: Single instruction (same as i32)
- Registers: 64-bit native
- Memory bandwidth: Can load 8 bytes at once
- **Performance**: i64 and i32 identical

**32-bit CPU** (legacy systems):
- i64 arithmetic: Two instructions (2× slower than i32)
- Registers: Must use register pairs
- Memory bandwidth: Needs two loads
- **Performance**: i64 noticeably slower

```aria
// On 64-bit CPU: Same performance
i32:a_32 = 1000;
i32:b_32 = 2000;
i32:sum_32 = a_32 + b_32;  // 1 instruction

i64:a_64 = 1000;
i64:b_64 = 2000;
i64:sum_64 = a_64 + b_64;  // 1 instruction (same!)
```

### Memory Bandwidth

```aria
// i32 array: 100 million elements
i32:data_32[100000000];
// Memory: 400 MB
// Cache lines: ~6.25 million (64 bytes each, 16 i32s per line)

// i64 array: 100 million elements
i64:data_64[100000000];
// Memory: 800 MB (2× larger!)
// Cache lines: ~12.5 million (64 bytes each, 8 i64s per line)

// Trade-off: i64 uses 2× memory but prevents overflow
```

### Register Pressure

```aria
// More i32 values fit in available registers
// x86-64 has 16 general-purpose 64-bit registers
// Can hold: 16 i64 values OR 16 i32 values (same)

// But for operations, i32 may allow more values in registers
// due to smaller memory footprint (better cache utilization)
```

## Common Gotchas

### 1. Large Literals Need Suffix

```aria
// ERROR: Literal exceeds i32 range
// i64:value = 3000000000;  // Compile error!

// CORRECT: Use i64 suffix
i64:value = 3000000000i64;  // OK
```

### 2. Mixed Arithmetic Can Overflow

```aria
// DANGER: Multiplication computed as i32, THEN widened!
i64:wrong = 2000000000 * 3;  // Overflow! (both i32)

// CORRECT: Explicit i64 literals
i64:correct = 2000000000i64 * 3i64;  // OK

// Or widen first
i32:a = 2000000000;
i64:correct2 = i64(a) * 3i64;  // OK
```

### 3. Division by Zero Still Panics

```aria
i64:a = 1000000000000;
i64:b = 0;
// i64:result = a / b;  // PANIC! Runtime error

// Always check divisor
if (b != 0) {
    i64:result = a / b;
} else {
    log.error("Division by zero\n");
}
```

### 4. abs(INT64_MIN) Panics

```aria
i64:min_value = -9223372036854775808;
// i64:absolute = abs(min_value);  // PANIC! (|min| = 2^63, doesn't fit)

// Check first or use tbb64
if (min_value == i64:MIN) {
    log.error("Cannot take abs of i64:MIN\n");
} else {
    i64:absolute = abs(min_value);
}
```

### 5. i64 Can Still Overflow

```aria
// i64 is large but not infinite!
i64:max = 9223372036854775807;
i64:result = max + 1;  // -9223372036854775808 (wrapped!)

// For extreme values, use tbb64 or arbitrary precision
```


### ✅ DO: Use i64 for Large File Sizes

```aria
// Files > 2GB require i64
i64:file_size = get_file_size("large_movie.mp4");

if (file_size > 2147483647) {
    log.write("File exceeds 2GB\n");
}
```

### ✅ DO: Use i64 for Accumulation

```aria
// Prevent overflow when summing many values
i64:total = 0;
for (value in values) {
    total += i64(value);
}
```

### ✅ DO: Check Range When Narrowing

```aria
// Always validate before narrowing
i64:large = get_database_id();

if (large >= i32:MIN && large <= i32:MAX) {
    i32:id = i32(large);
} else {
    log.error("ID out of i32 range\n");
}
```

### ❌ DON'T: Use i64 for Small Values

```aria
// WRONG: Wastes memory
i64:tiny_counter = 0;  // Only counts to 100

// RIGHT: Use i32
i32:counter = 0;
```

### ❌ DON'T: Assume i64 Never Overflows

```aria
// WRONG: i64 can overflow too!
i64:huge = 1000000000000;
i64:product = huge * huge;  // Overflow!

// RIGHT: Check or use tbb64
tbb64:safe_huge = 1000000000000;
tbb64:safe_product = safe_huge * safe_huge;
if (safe_product == ERR) {
    log.error("Overflow\n");
}
```

### ❌ DON'T: Forget i64 Suffix for Large Literals

```aria
// WRONG: Compiler error
// i64:value = 5000000000;  // Exceeds i32

// RIGHT:
i64:value = 5000000000i64;
```
