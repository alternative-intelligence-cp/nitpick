LAYER 1: Result<T> SYSTEM - HARD ERRORS THAT MUST BE HANDLED

Purpose: Make errors impossible to ignore
Status: IMPLEMENTED (February 2026)

Structure:
struct Result<T> {
    T:val;           // Success value
    void*:err;       // Error pointer (NULL if no error)
    bool:is_error;   // Error flag
};

Creating Results:
    pass(value);     // Success: { val: value, err: NULL, is_error: false }
    fail(error);     // Failure: { val: undefined, err: error, is_error: true }

Unwrapping operators:
    ?   - Safe unwrap with default value
          Config:cfg = load_config() ? default_config;
          
    ?!  - Emphatic unwrap (triggers failsafe on error)
          Config:cfg = load_critical_config()?!;  // MUST succeed
          
    ??  - Null coalescing (for NIL optional values)
          string:name = user?.name ?? "Unknown";

Sticky error propagation:
    ERR + anything = ERR     // Errors propagate automatically
    10 + ERR_SENTINEL = ERR  // TBB types preserve errors
    
Philosophy:
    "Errors are data, not exceptions" - errors flow through type system
    "Errors cannot be ignored" - compiler forces handling
    "Errors are cheap" - zero overhead on success path (~1 cycle branch)

Use cases:
    ✅ File I/O failures
    ✅ Network errors
    ✅ Parse failures
    ✅ Contract violations (preconditions, postconditions)
    ✅ Resource allocation failures
    ✅ Any recoverable error condition

LAYER 2: unknown SYSTEM - SOFT ERRORS FOR UNDEFINED STATES

Purpose: Map crash-inducing operations to defined "unknown" state
Status: IMPLEMENTED (February 2026)

Philosophy:
    "undefined → unknown" - Give developers a chance to recover
    "Soft error propagation" - Errors flow but don't force immediate handling
    "Explicit acknowledgment required" - Use ok() to propagate unknown values

The unknown value:
    - Special sentinel value (like ERR, but for undefined states)
    - Propagates through operations (sticky, like NaN but controlled)
    - Can be checked with ok(value) function
    - Must be explicitly acknowledged with ok(value) to pass to Result<T>

Operations that return unknown (instead of crashing):
    ✅ Division by zero: 10 / 0 → unknown
    ✅ Array out of bounds: arr[999] → unknown (if array length < 999)
    ✅ Integer overflow: INT_MAX + 1 → unknown (in checked mode)
    ✅ Square root of negative: sqrt(-1) → unknown (for real types)
    ✅ Invalid type conversions: string_to_int("not a number") → unknown
    ✅ Null pointer dereference: <-null_ptr → unknown
    ✅ Any operation on unknown: unknown + 5 → unknown (propagates)

Checking for unknown:
    ok(value) - Returns:
        - 1 if value is valid (not unknown)
        - 0 if value is unknown
        
    int32:result = divide(10, 0);  // result = unknown
    if (ok(result) == 0) {
        // Handle unknown state
        stderr.write("Division resulted in unknown\n");
        result = 0;  // Recover with default value
    }

Propagating unknown explicitly:
    func:risky_calc = Result<int32>() {
        int32:val = divide(10, maybe_zero) ? 0;
        if (ok(val) == 0) {
            // Explicitly acknowledge unknown and propagate
            pass(ok(val));  // ok() strips unknown taint
        }
        pass(val);
    };

Why two systems (Result vs unknown)?
    - Result<T>: For EXPECTED errors (file not found, parse error)
    - unknown: For UNEXPECTED undefined states (division by zero, OOB)
    - Result forces handling, unknown allows recovery
    - Together: Complete coverage of error space

LAYER 3: failsafe() - MANDATORY FINAL SAFETY NET

Purpose: Catch everything that Result<T> and unknown missed
Status: IMPLEMENTED (February 2026)

MANDATORY FUNCTION:
Every Aria program MUST define failsafe() or compilation fails.
Cannot be empty - must at least accept error code and take action.

Signature:
func:failsafe = void(int32:err_code) {
    // Emergency cleanup and controlled termination
    // MUST be defined by every program
};

When failsafe() is called:
    ✅ Emphatic unwrap (?!) on error Result
    ✅ Explicit call: failsafe(ERR_CODE) or !!! ERR_CODE
    ✅ Unhandled errors that propagate to top level (future)
    ✅ Critical contract violations (future)
    ✅ Out-of-memory conditions (future)
    ✅ Stack overflow (future)

Error codes:
    Negative: System errors (-1: OOM, -2: stack overflow, -3: div by zero)
    Positive: Application errors (1: sensor offline, 2: config invalid, etc.)
    Zero: Reserved (should not be used)

Calling failsafe:
    failsafe(ERR_HARDWARE_FAILURE);  // Direct call
    !!! ERR_HARDWARE_FAILURE;        // Direct call (operator form)
    
    Config:cfg = load()?!;  // Calls failsafe if load() returns error

Philosophy:
    "Compiler-enforced accountability" - Cannot ignore error handling
    "Last line of defense" - Controlled shutdown vs undefined behavior
    "Fail to safe state" - Release resources, log errors, notify operators

Example failsafe (robotics):
func:failsafe = void(int32:err_code) {
    // 1. Emergency stop (cut power to motors)
    emergency_stop_all_motors();
    
    // 2. Release grippers (safety)
    release_all_grippers();
    
    // 3. Sound alarm
    sound_error_alarm(err_code);
    
    // 4. Log to persistent storage
    write_error_to_eeprom(err_code);
    
    // 5. Wait for manual reset
    while (true) {
        flash_error_led(err_code);
        delay_ms(500);
    }
};

HANDLE<T> TYPE - GENERATIONAL INDEX HANDLES:

Handle<T> is a safe alternative to raw pointers for arena-allocated memory:
  - NOT a pointer (no direct memory address)
  - Index + generation counter (detects stale references)
  - Works with arena allocators (region-based memory)
  - Prevents use-after-free when arenas reallocate

Type signature:
Handle<T>, !!! struct<T>:Handle { uint64:index; uint32:generation; }

LLVM IR representation:
%Handle = type { i64, i32 }  ; {index, generation}

Memory layout:
┌──────────────────┬────────────────┐
│  index (uint64)  │ generation(u32)│
│  8 bytes         │ 4 bytes        │
└──────────────────┴────────────────┘
= 12 bytes total (+ 4 bytes padding = 16 bytes aligned)

---

WHY GENERATIONAL HANDLES EXIST:

THE PROBLEM: Raw pointers become invalid during arena reallocation

Example scenario (using raw pointers - UNSAFE):
Neuron*:ptr = arena.alloc();  // Pointer to slot 0 at address 0x1000
arena.grow();                 // Reallocates to larger memory block at 0x2000
ptr->activate();              // 💥 USE-AFTER-FREE! ptr still points to 0x1000 (now invalid)

Result:
  - Memory corruption (reading/writing freed memory)
  - Segmentation fault (accessing unmapped memory)
  - Arbitrary code execution vulnerability (CWE-416, CVSS 9.8)
  - Mars Climate Orbiter-class failure ($125M loss from pointer bugs)

THE SOLUTION: Generational handles detect stale references

Example (using Handle<T> - SAFE):
Handle<Neuron>:h = arena.alloc();  // h = {index: 0, generation: 1}
arena.grow();                       // Reallocates, increments generation to 2
Neuron:n = arena.get(h) ? default;  // ✅ Returns ERR! Generation mismatch (1 != 2)

Result:
  - Safe failure (ERR propagation, no corruption)
  - Deterministic behavior (always detects stale handles)
  - No segfault (generation check prevents access)
  - Type-safe (ERR sticky propagation halts computation)

HOW GENERATIONAL HANDLES WORK:

Arena Structure (C runtime):
typedef struct {
    void** slots;              // Array of pointers to elements
    uint32_t* generations;     // Generation counter for each slot
    bool* occupied;            // Is slot allocated?
    size_t capacity;           // Total slots
    size_t count;              // Used slots
} aria_gen_arena;

Slot Structure:
typedef struct {
    void* data;                // Actual element (T)
    uint32_t generation;       // Version counter (1, 2, 3, ...)
    bool occupied;             // Is this slot in use?
} aria_gen_slot;

Allocation Flow:
1. User calls: Handle<T>:h = arena.alloc(data);
2. Arena finds free slot (index i)
3. Arena stores data in slots[i]
4. Arena sets occupied[i] = true
5. Arena returns Handle{index: i, generation: generations[i]}

Access Flow:
1. User calls: T:val = arena.get(h) ? default;
2. Arena checks: generations[h.index] == h.generation?
3. If match: Return slots[h.index] (SUCCESS)
4. If mismatch: Return ERR (STALE HANDLE DETECTED)

Reallocation Flow:
1. Arena grows (reallocates larger memory block)
2. Arena copies all slots to new memory
3. Arena increments ALL generation counters: generations[i]++
4. Old handles now have mismatched generations (stale handle detection)

GENERATIONAL SAFETY MECHANISM:

Generation Increment Triggers:
  - Arena reallocation (memory moves to new address)
  - Slot deallocation (prevent use-after-free of freed slot)
  - Explicit invalidation (arena.invalidate_all())

Generation Matching:
  Valid handle: h.generation == arena.generations[h.index]
  Stale handle: h.generation != arena.generations[h.index]

PERFORMANCE CHARACTERISTICS:

Allocation: O(1) amortized
  - Find free slot (bitmap scan or free list)
  - Store data, set generation, return Handle
  - Cost: ~10-20 cycles

Access (valid handle): O(1)
  - Generate check: 1 comparison (~1 cycle)
  - Index lookup: 1 array access (~3-5 cycles)
  - Total: ~4-6 cycles (branch prediction makes this essentially free)

Access (stale handle): O(1)
  - Generation mismatch detected immediately
  - Return ERR (no access to invalid memory)
  - Cost: ~2-3 cycles (branch misprediction, but rare)

Reallocation: O(n)
  - Copy all slots to new memory
  - Increment all generation counters
  - Cost: Same as regular realloc, plus generation increment
  - Rare event (exponential growth strategy minimizes reallocations)

Zero-overhead when valid:
  - Modern CPUs predict generation match (hot path)
  - Branch predictor makes check essentially free
  - Same performance as pointer dereference for valid handles

/* MEMORY MODEL OVERVIEW

Aria defines a memory model that prioritizes SAFETY while enabling PERFORMANCE

/* MEMORY ORDERING LEVELS (SafetyFirst™ Design)

DEFAULT: Sequential Consistency (SeqCst)
- All atomic operations appear to execute in a single total order
- Matches programmer intuition - no surprises  
- Performance cost: Memory barriers on most platforms
- Use case: Default for all atomic<T> operations unless explicitly weakened
- Guarantees: No reordering across atomic operations
- Example:
    atomic<int64>:x = 0;
    atomic<int64>:y = 0;
    // Thread 1: x.store(1); y.store(2);  
    // Thread 2: if (y.load() == 2) { assert(x.load() == 1); } // GUARANTEED

Acquire/Release (AcqRel) — EXPERT ONLY
- Acquire (load): Prevent later operations from moving before this load
- Release (store): Prevent earlier operations from moving after this store
- Use case: Implementing efficient mutexes, seqlocks, lock-free queues
- Performance: Cheaper than SeqCst (no full barrier, only one-way)
- The function name carries the intent: _acquire/_release means you own the ordering
- Example (Seqlock pattern for Nikola's metric tensor):
    uint64:seq1 = seq_counter.load_acquire();
    metric_data = tensor.read();  // Cannot move before seq1 load
    uint64:seq2 = seq_counter.load_acquire();
    if (seq1 == seq2 && seq1 % 2 == 0) { /* consistent read */ }

Relaxed — EXPERT ONLY
- No ordering guarantees except atomicity of individual operations
- Use case: Counters, statistics where order doesn't matter
- Performance: Fastest (no memory barriers at all)
- The function name carries the intent: _relaxed means you own the lack of ordering
- Example (metabolic energy counter):
    energy_total.fetch_add_relaxed(delta);  // Order doesn't matter for total

MEMORY FENCES (explicit synchronization):
- atomic_fence_acquire() - Prevent loads/stores from moving before this point
- atomic_fence_release() - Prevent loads/stores from moving after this point
- atomic_fence_seqcst() - Full bidirectional barrier
- Used with Relaxed operations to build custom synchronization

*/

/* ATOMIC OPERATIONS - atomic<T> PRIMITIVES

/* DATA RACE DETECTION AND PREVENTION

DEFINITION OF DATA RACE:
Two threads access same memory location concurrently, at least one is write, 
and accesses are not ordered by synchronization → UNDEFINED BEHAVIOR

COMPILER DETECTION (where possible):
- Static analysis detects obvious races (shared mutable state without atomic)
- Warning: "Potential data race on variable 'x' - use atomic<T> for shared state"
- Error: Violates Aria's safety requirements compile with --allow-undefined-behavior to override

PREVENTION STRATEGIES:
1. Use atomic<T> for all shared mutable state
2. Use Handle<T> + arena allocators (never raw pointers for shared data)
3. Message passing (send data between threads via channels - ownership transfer)
4. Immutable data sharing (no mutation = no race)
5. Thread-local storage (each thread has own copy)

HAPPENS-BEFORE RELATIONSHIP:
- Operation A "happens-before" operation B if:
  1. A and B in same thread, A before B in program order, OR
  2. A is Release store, B is Acquire load of same atomic, OR
  3. A happens-before C, C happens-before B (transitivity)
- Compiler/hardware CANNOT reorder across happens-before edges
- Guarantees memory visibility: If A writes X and A happens-before B, B sees X

*/

RESULT<T> STRUCTURE (3-field struct):

Result<T> is implemented as a struct with three fields:

/* GENERICS/TEMPLATES - ZERO-COST ABSTRACTIONS (February 2026)

!! CRITICAL DESIGN PHILOSOPHY !!
Aria uses MONOMORPHIZATION (like C++ templates, Rust generics) for generic code.
This means: Write code once, compiler generates specialized versions for each type used.
Result: ZERO runtime overhead, but increased binary size.

PHILOSOPHY: "Zero-cost abstractions" - generics as fast as hand-written specialized code

---

WHAT ARE GENERICS?

Generics let you write functions, structs, and algorithms that work with ANY type:
  - Write once: func<T>:identity = *T(*T:value) { pass(value); }
  - Use anywhere: identity<int32>(42), identity<string>("hello"), identity<fix256>(3.14)
  - Compiler generates specialized versions

Benefits:
  ✅ Code reuse (write once, use with any type)
  ✅ Type safety (compiler checks each specialization)
  ✅ Zero runtime cost (monomorphization = no vtables, no boxing)
  ✅ Full optimization (each version optimized independently)

Trade-off:
  ⚠️ Binary size increases (each type used creates new function copy)

---

GENERIC FUNCTION SYNTAX:

Declaration:
func<T, U>:name = returnType(params) { body };

Where:
  - <T, U> declares type parameters (can be any identifier)
  - returnType uses *T, *U for generic types
  - params use *T, *U for generic parameter types
  - body uses *T, *U for generic types

*T PREFIX RULE:
The * prefix marks generic type usage in function bodies:
  - *T in return type → "returns value of type T"
  - *T:param → "parameter of type T"
  - *T:local → "local variable of type T"

Example:
func<T>:identity = *T(*T:value) {
    pass(value);  // Returns Result<T>
};

---

GENERIC FUNCTION EXAMPLES:

// Single type parameter
func<T>:identity = *T(*T:value) {
    pass(value);
};

// Multiple type parameters

    pass((first, second));  // Returns tuple
};

// Generic with conditional logic

    if (a > b) {
        pass(a);
    }
    pass(b);
};

// Generic array operations

    if (len > 0) {
        pass(arr[0]);
    }
    fail(-1);  // Error if empty
};

        pass(default_val);
    }

};

MONOMORPHIZATION - HOW IT WORKS:

Monomorphization = "Generating specialized versions at compile-time"

Source code:

    pass(a + b);

};

Each specialized version:
  ✅ Optimized independently (int32 uses CPU register, flt64 uses FPU)
  ✅ No runtime type checking (type known at compile-time)
  ✅ No indirection/vtables (direct function call)

/* CLOSURES AND LAMBDAS - ANONYMOUS FUNCTIONS (February 2026)

!! IMPORTANT: => IS THE CAST OPERATOR IN ARIA, NOT A LAMBDA OPERATOR !!
Aria does not use => for lambdas. Lambda syntax is simply: returnType(params) { body }

=> is used exclusively for type casting (coercion):
   int8:b = a => int8;       // cast a to int8
   float64:d = val => float64;  // widen val to float64
   int8:c = b + a => int8;   // casts a, then adds to b (cast binds tighter than +)

PHILOSOPHY: "Functions are first-class values" - pass them around like any other data

---

WHAT ARE CLOSURES?

Closures are anonymous functions that:
  - Can be created inline (no need to name them)
  - Can capture variables from enclosing scope
  - Can be passed as function parameters
  - Can be returned from functions
  - Can be stored in variables

Use cases:
  ✅ Callbacks (event handlers, async operations)
  ✅ Functional programming (map, filter, reduce)
  ✅ Deferred execution (lazy evaluation)
  ✅ Encapsulation (private state in closures)

---

LAMBDA SYNTAX:

Basic lambda (no captures):
returnType(paramType:param) { body }

Example:
int32(int32:x) { pass(x * 2); }  // Lambda that doubles input

Assigned to variable:
(int32)(int32):doubler = int32(int32:x) { pass(x * 2); };
int32:result = doubler(5);  // result = 10

Multiple parameters:
(int32, int32)(int32:a, int32:b) { pass(a + b); }

No parameters:
int32() { pass(42); }

---

FUNCTION POINTERS AND LAMBDA TYPES:

Function pointer type syntax:
(returnType)(paramTypes)

Examples:
(int32)(int32):unary_op;           // Takes int32, returns int32
(int32)(int32, int32):binary_op;   // Takes two int32, returns int32
(NIL)():callback;                  // Takes nothing, returns NIL
(string)(obj):formatter;           // Takes obj, returns string

Assigning lambdas to function pointers:
(int32)(int32):square = int32(int32:x) { pass(x * x); };
int32:result = square(5);  // result = 25

Closures can capture variables from enclosing scope:

};

};

CLOSURE CAPTURE SEMANTICS:

Value capture (safe, copies data):

  ⚠️ Captured pointers must outlive closure

  ✅ Use reference capture only when necessary (performance)

CLOSURE PERFORMANCE:

Stack-allocated closures (no heap allocation):

  - No function call overhead
  - Compiler can optimize aggressively

/* DEFER STATEMENT - GUARANTEED CLEANUP (February 2026)

!! CRITICAL FOR RESOURCE MANAGEMENT !!
The defer statement guarantees cleanup code runs when scope exits.
Essential for preventing resource leaks in safety-critical systems.

PHILOSOPHY: "Cleanup is automatic, leaks are impossible"

---

WHAT IS DEFER?

defer schedules code to execute when current scope exits:
  - Runs at end of function (before return)
  - Runs on all exit paths (normal return, early return, error)
  - Multiple defers execute in reverse order (LIFO - Last In, First Out)
  - Similar to: C++ destructors, Go defer, Swift defer

Use cases:
  ✅ Resource cleanup (free memory, close files, unlock mutexes)
  ✅ Error handling (ensure cleanup happens even on error paths)
  ✅ Transaction rollback (database operations)
  ✅ Logging (function entry/exit tracing)
  ✅ Invariant restoration (restore state after temporary modification)

---

DEFER SYNTAX:

defer statement;

Examples:

// Defer single statement
defer aria.free(ptr);

// Defer block
defer {
    aria.free(buffer);
    close_file(handle);
};

// Multiple defers (execute in reverse order)
defer cleanup1();
defer cleanup2();
defer cleanup3();
// Execution order: cleanup3(), cleanup2(), cleanup1()

---

DEFER EXECUTION ORDER:

Defers execute in LIFO (Last In, First Out) order:

func:process = NIL() {
    defer println("First defer");   // Executes 3rd
    defer println("Second defer");  // Executes 2nd
    defer println("Third defer");   // Executes 1st
    
    println("Function body");
    pass(NIL);
};

// Output:
// Function body
// Third defer
// Second defer
// First defer

Rationale for LIFO:
  - Mirrors construction/destruction order
  - Innermost resource allocated last, cleaned up first
  - Natural pairing with allocation (alloc → defer free)

---

DEFER FOR MEMORY MANAGEMENT:

Automatic cleanup prevents memory leaks:

func:process_file = Result<NIL>(string:path) {
    // Allocate buffer
    int8->:buffer = aria.alloc<int8>(1024);
    defer aria.free(buffer);  // ✅ ALWAYS freed, even on error
    
    // Read file
    Result<int64>:bytes_read = read_file(path, buffer, 1024);
    if (bytes_read.err != NULL) {
        fail(-1);  // defer ensures buffer freed before return!
    }
    
    // Process buffer...
    process_buffer(buffer);
    
    pass(NIL);  // defer ensures buffer freed on normal return too
};

Without defer (manual cleanup - ERROR PRONE):
func:process_file_manual = Result<NIL>(string:path) {
    int8->:buffer = aria.alloc<int8>(1024);
    
    Result<int64>:bytes_read = read_file(path, buffer, 1024);
    if (bytes_read.err != NULL) {
        aria.free(buffer);  // ❌ MUST REMEMBER to free!
        fail(-1);
    }
    
    process_buffer(buffer);
    
    aria.free(buffer);  // ❌ MUST REMEMBER to free!
    pass(NIL);
};
// Multiple exit paths → multiple free() calls → easy to forget one → LEAK

!!! OPERATOR RATIONALE:

Why three exclamation marks?

1. **Avoid clashes:**
   - ! = logical NOT (standard)
   - !! = potential double negation or typo
   - !!! = clearly intentional, no accidental formation

2. **Semantic clarity:**
   - !!!(code) reads as "VERY NOT RIGHT HERE"
   - Maintains continuity with ! (negation/error family)
   - Visual impact matches severity (emergency termination)
   - Already recognized in informal writing as "holy shit" or serious warning

3. **Visual prominence:**
   - Stands out among letters and numbers
   - Easy to spot when reading/debugging code
   - Makes code review easier (can't miss a failsafe call)
   - Like a 🚨 red alert sign in ASCII form

4. **User design:**
   - Originally ?! was !! (double exclamation)
   - Changed to ?! to fit question-mark operator family
   - Freed up !!! for direct failsafe calls
   - Clear distinction: ?!(conditional) vs !!!(unconditional)

};

!!! OPERATOR SUMMARY:

The !!! operator provides direct failsafe invocation:
  ✅ Syntax: !!!(error_code);
  ✅ Unconditional program termination
  ✅ Calls failsafe(error_code) immediately
  ✅ Used for assertions, invariants, impossible states
  ✅ Different from ?! (conditional vs unconditional)
  ✅ Three marks avoid clashes, provide visual impact
  ✅ "VERY NOT RIGHT HERE" - emergency shutdown

Operator family:
  - ! = logical NOT
  - ?! = emphatic unwrap (conditional failsafe)
  - !!! = direct failsafe (unconditional termination)

"When you reach impossible territory, !!! is your emergency brake."

FAILSAFE SUMMARY:

failsafe() is the mandatory last-resort error handler:
  ✅ MUST be defined in every Aria program
  ✅ Signature: func:failsafe = void(int32:err_code) { ... };
  ✅ Never returns (terminates program or halts system)
  ✅ Triggered by ?! operator on error Results
  ✅ Provides controlled shutdown instead of undefined behavior
  ✅ Essential for safety-critical systems (Nikola, robotics, medical)
  ✅ Compiler enforces definition (compile error if missing)

Error handling spectrum:
  - Expected errors → Result<T> + unwrap (?)
  - Unrecoverable errors → failsafe + emphatic unwrap (?!)

"When all your error handling fails, failsafe() is your last hope for a graceful death."

WHAT ARE MODULES?

Modules provide:

/* MEMORY ALLOCATION MODES - EXPLICIT ALLOCATION CONTROL (February 2026)

!! CRITICAL FOR PERFORMANCE AND SAFETY !!
Aria provides multiple allocation modes for different use cases.
Understanding when to use each mode is essential for efficiency and safety.

PHILOSOPHY: "Explicit allocation = predictable behavior" - no hidden allocations

---

MEMORY ALLOCATION MODES:

Aria supports four allocation modes:
  1. **stack** - Stack allocation (default for locals, fastest)
  2. **gc** - Garbage collected heap (automatic, safe but unpredictable timing)
  3. **wild** - Manual heap allocation (unsafe, full control)
  4. **wildx** - Executable memory (JIT compilation, unsafe)

Syntax for explicit modes:
stack TypeName:var;   // Allocate on stack (usually default)
gc TypeName:var;      // Allocate with GC
wild TypeName:var;    // Manual allocation (must free)
wildx TypeName:var;   // Executable memory (must free)

---

STACK ALLOCATION (DEFAULT):

Stack-allocated variables:
  - Allocated on function stack frame
  - Automatically freed when scope exits
  - Fastest allocation (pointer bump)
  - Limited size (stack size limits)
  - LIFO (Last In, First Out) lifetime

Syntax (stack is usually implicit):
int32:x = 42;              // Stack allocated (default)
stack int32:y = 100;       // Explicit stack annotation

int32[1024]:buffer;        // Stack array (4KB)
stack Point:p = {x: 10, y: 20};  // Stack struct

When to use:
  ✅ Local variables (most common case)
  ✅ Small to medium structs
  ✅ Temporary buffers
  ✅ Performance-critical code
  ✅ Deterministic lifetime (scope-based)

When NOT to use:
  ❌ Large allocations (>1MB, risk stack overflow)
  ❌ Variable-sized data (use heap/gc)
  ❌ Long-lived data outliving function
  ❌ Shared across threads

Performance:
  - Allocation: ~1 cycle (pointer bump)
  - Deallocation: 0 cycles (automatic at scope exit)
  - Cache-friendly (stack is hot in L1 cache)

Example:
func:compute = int32() {
    stack int32[100]:temp_buffer;  // Stack array
    
    for (int32:i = 0; i < 100; i++) {
        temp_buffer[i] = i * i;
    }
    
    pass(temp_buffer[50]);
};  // temp_buffer automatically freed here

---

GC ALLOCATION (GARBAGE COLLECTED):

GC-allocated variables:
  - Allocated on garbage-collected heap
  - Automatically freed when no references remain
  - Safe (no manual free needed)
  - Unpredictable timing (GC pause)
  - Shared ownership possible

Syntax:
gc TypeName:var = value;

gc int32->:ptr = aria.gc_alloc<int32>();
<-ptr = 42;

gc string:text = "Hello, GC!";
gc Vec<int32>:numbers = Vec.new();

When to use:
  ✅ Shared ownership (multiple references)
  ✅ Complex lifetimes (unclear when to free)
  ✅ Long-lived data
  ✅ Dynamic data structures (graphs, trees)
  ✅ When determinism not required

When NOT to use:
  ❌ Real-time systems (GC pause breaks timing)
  ❌ Safety-critical (unpredictable pauses)
  ❌ Performance-critical inner loops
  ❌ Embedded systems (memory constrained)

Performance:
  - Allocation: ~10-50 cycles (find free space)
  - Deallocation: Amortized (GC runs periodically)
  - GC pause: 1-100ms depending on heap size
  - Throughput: Good (bulk collection efficient)
  - Latency: Poor (unpredictable pauses)

Example warning from Nikola context:
// ❌ BAD: GC pause breaks physics timestep
func:propagate_wave = wavefunction(wavefunction:psi, fix256:dt) {
    gc Buffer:temp = aria.gc_alloc<Buffer>();  // GC pause → missed timestep!
    // ... wave propagation
};

// ✅ GOOD: Use stack or arena allocation
func:propagate_wave = wavefunction(wavefunction:psi, fix256:dt) {
    stack Buffer:temp;  // Deterministic, zero overhead
    // ... wave propagation
};

---

WILD ALLOCATION (MANUAL HEAP):

Wild-allocated variables:
  - Manual heap allocation (like C malloc/free)
  - UNSAFE: Must manually free (risk of leaks)
  - No GC overhead
  - Full control over lifetime
  - Deterministic performance

Syntax:
wild TypeName:var;
wild TypeName->:ptr = aria.alloc<TypeName>(size);

Allocation and deallocation:
wild int32->:ptr = aria.alloc<int32>(100);  // Allocate 100 int32s
defer aria.free(ptr);  // MUST free manually!

wild string->:text = aria.alloc_string(1024);
defer aria.free(text);

When to use:
  ✅ Real-time systems (deterministic allocation)
  ✅ Performance-critical code (no GC overhead)
  ✅ Embedded systems (predictable memory)
  ✅ Large buffers (arenas for bulk data)
  ✅ FFI with C libraries (C expects malloc/free)

When NOT to use:
  ❌ General application code (GC safer)
  ❌ Complex ownership (easy to leak)
  ❌ Unless defer used (leak risk)

Safety with defer:
wild int32->:buffer = aria.alloc<int32>(1024);
defer aria.free(buffer);  // ✅ ALWAYS freed

// Use buffer...
for (int32:i = 0; i < 1024; i++) {
    buffer[i] = i;
}

Performance:
  - Allocation: ~20-100 cycles (malloc overhead)
  - Deallocation: ~10-50 cycles (free overhead)
  - No GC pause (deterministic)
  - Cache-unfriendly (heap scattered)

Example (Nikola physics buffer):
wild fix256->:wave_buffer = aria.alloc<fix256>(GRID_SIZE);
defer aria.free(wave_buffer);

// Propagate waves deterministically
for (int64:i = 0; i < GRID_SIZE; i++) {
    wave_buffer[i] = compute_wave(i);
}

---

WILDX ALLOCATION (EXECUTABLE MEMORY):

Wildx-allocated memory:
  - Executable heap allocation (for JIT compilers)
  - UNSAFE: Must manually free
  - Memory is executable (can hold machine code)
  - Used for runtime code generation

Syntax:
wildx uint8->:code_buffer = aria.alloc_executable(size);

When to use:
  ✅ JIT compilers (generating machine code at runtime)
  ✅ Dynamic code loading
  ✅ Trampoline functions
  ✅ Self-modifying code (very rare)

When NOT to use:
  ❌ Regular data (use wild instead)
  ❌ Unless implementing JIT/dynamic code
  ❌ Security-sensitive code (executable heap risk)

Example (JIT compiler):
wildx uint8->:jit_code = aria.alloc_executable(4096);  // 4KB code page
defer aria.free_executable(jit_code);

// Generate x86-64 machine code
jit_code[0] = 0x48;  // REX.W prefix
jit_code[1] = 0x89;  // MOV instruction
jit_code[2] = 0xC8;  // MOV RAX, RCX
jit_code[3] = 0xC3;  // RET

// Cast to function pointer and execute
(int64)(int64):jit_func = (int64)(int64)jit_code;
int64:result = jit_func(42);  // Executes JIT-compiled code

---

ALLOCATION MODE COMPARISON:

| Mode | Speed | Safety | Lifetime | Use Case |
|------|-------|--------|----------|----------|
| **stack** | Fastest | Safe | Scope | Local variables |
| **gc** | Medium | Safest | Automatic | Complex ownership |
| **wild** | Fast | Unsafe | Manual | Real-time, embedded |
| **wildx** | Fast | Unsafe | Manual | JIT compilation |

Memory overhead:
  - stack: 0 bytes (part of stack frame)
  - gc: ~8-16 bytes per object (GC metadata)
  - wild: ~8-16 bytes per allocation (malloc metadata)
  - wildx: ~8-16 bytes + page alignment (OS overhead)

---

ALLOCATION MODE BEST PRACTICES:

1. **Default to stack** (fastest and safest for locals)
   int32:x = 42;  // Stack (default)

2. **Use gc for shared ownership** (complex lifetimes)
   gc TreeNode->:root = build_tree();  // Multiple references

3. **Use wild for real-time** (deterministic, no GC pause)
   wild tfp64->:physics_buffer = aria.alloc<tfp64>(GRID_SIZE);
   defer aria.free(physics_buffer);

4. **ALWAYS defer wild/wildx allocations** (prevent leaks)
   wild int32->:ptr = aria.alloc<int32>(100);
   defer aria.free(ptr);  // ✅ Mandatory

5. **Avoid gc in real-time loops** (unpredictable pause)
   Bad: gc Buffer:temp in 1ms physics timestep
   Good: stack or arena allocation

6. **Use wildx ONLY for JIT** (security risk otherwise)
   Only needed for runtime code generation

7. **Consider arena allocators** (bulk allocation + bulk free)
   arena<WaveNode>:temp_arena for neurogenesis

---

ALLOCATION MODE SUMMARY:

Aria provides explicit allocation control:
  ✅ **stack**: Default, fastest, scope-based (safe)
  ✅ **gc**: Garbage collected, safest, unpredictable timing
  ✅ **wild**: Manual heap, deterministic, unsafe (requires free)
  ✅ **wildx**: Executable memory, JIT only, unsafe

Choose based on requirements:
  - Safety priority → gc (automatic, safe)
  - Performance priority → stack (fastest) or wild (deterministic)
  - Real-time priority → wild or arena (no GC pause)
  - Code generation → wildx (executable memory)

"Explicit allocation modes = predictable performance = safety-critical readiness"

*/

/* STRING INTERPOLATION - TEMPLATE LITERALS (February 2026)

!! CRITICAL FOR DEBUGGING AND STRING FORMATTING !!
String interpolation enables embedding expressions directly in strings.
Essential for readable debugging, logging, and dynamic string generation.

PHILOSOPHY: "Strings should speak the language of data" - direct expression embedding

---

WHAT IS STRING INTERPOLATION?

String interpolation allows embedding expressions in strings:
  - Backtick strings (`) support interpolation
  - & prefix marks interpolated expressions
  - Syntax: `&{expression}` within backtick strings
  - Expressions evaluated at runtime
  - Results automatically converted to strings

Regular strings vs template literals:
"Hello, world"      // Regular string (double quotes), no interpolation
'Hello, world'      // Regular string (single quotes), no interpolation
`Hello, world`      // Template literal (backticks), can use interpolation

---

BASIC INTERPOLATION:

Embedding variables:
int32:value = 42;
string:msg = `The answer is &{value}`;  // "The answer is 42"

string:name = "Alice";
int32:age = 30;
string:intro = `My name is &{name} and I am &{age} years old`;
// "My name is Alice and I am 30 years old"

Multiple interpolations:
int32:x = 10, y = 20, z = 30;
print(`Point: (&{x}, &{y}, &{z})`);  // "Point: (10, 20, 30)"

---

EXPRESSION INTERPOLATION:

Arithmetic expressions:
int32:a = 5, b = 3;
print(`&{a} + &{b} = &{a + b}`);  // "5 + 3 = 8"
print(`&{a} * &{b} = &{a * b}`);  // "5 * 3 = 15"

Function calls:
func:square = int32(int32:x) { pass(x * x); };
int32:num = 7;
print(`&{num} squared is &{square(num)}`);  // "7 squared is 49"

1. **Prefer interpolation over concatenation** (more readable)
   Good: `Result: &{value}`
   Bad: "Result: " + value

2. **Keep expressions simple** (complex logic in functions)
   Good: `Status: &{format_status(code)}`

// LAMBDA/FUNCTION SYNTAX RULES
// Complete anonymous declaration syntax:
//   returnType(type:arg, type:arg) { /* function body */ }(/* optional immediate execution args */);
//
// Named function (assigned to func-type variable):
//   func:name = returnType(type:arg, type:arg) { body };
//
// Anonymous with immediate execution (assigned to result-type variable):
//   result<T>:r = returnType(type:arg, type:arg) { body }(arg, arg);
//   // Immediate execution () is REQUIRED when assigning to non-func variable
//   // Otherwise you'd assign the function itself (type mismatch)
//
// CRITICAL: ALL functions return result type {value: T, err: void*, is_error: bool}
//   pass(value) → syntactic sugar for return result with value and is_error=false
//   fail(code)  → syntactic sugar for return result with err and is_error=true
//
// Examples:
//   func:add = int8(int8:a, int8:b) { pass(a + b); };          // Named function
//   int8:sum = int8(int8:a, int8:b) { pass(a + b); }(3, 4) ? 0; // Anonymous + immediate + unwrap = 7
//   func:f = int8(int8:a, int8:b) { pass(a + b); };            // Store function
//   int8:res = f(3, 4) ? 0;                                     // Call stored function, unwrap = 7
//
// LAMBDAS AS ARGUMENTS (JS-style callbacks):
//   // Function that takes a func parameter
//   func:processValues = int8(func:operation, int8:a, int8:b) {
//       int8:r = operation(a, b) ? -1i8;
//       pass(r);  // Re-pass
//   };
//   // Call with inline anonymous lambda
//   Result<int8>:r = processValues(
//       int8(int8:x, int8:y) { pass(x * y); },  // Lambda passed as argument
//       4, 5
//   );
//   int8:res = r ? 0;  // Unwrap result = 20
//
// LAMBDA + IMMEDIATE EXECUTION + UNWRAP (composable features):
//   // Lambda returns result, executed immediately, unwrapped with ?
//   func:test = int8(int8:a, int8:b) { pass(a+b); };
//   Result<int8>:r = test(
//       int8(int8:x, int8:y) { pass(x*y); }(4, 5) ? 0,  // Execute & unwrap = 20
//       12
//   );  // Passes unwrapped value 20 and 12 to test()
//   int8:final = r ? 0;  // final = 32
//
// IMPORTANT: Arrow operator (->) is ONLY for pointer member dereference (C-style):
//   ptr->member  // Access member through pointer (dereference + member access)
//   obj.member   // Access member of object (dot notation)
//
// NO SPECIAL OPERATORS FOR FUNCTIONS:
//   Functions use: func:name = returnType(params) { body };
//   NO -> for return types
//   NO => for lambdas (that's just inline function syntax)
//   ALL functions return result type implicitly with {err, val} fields

/* ============================================================================
   SAFETY SYSTEM: LAYERED ERROR HANDLING ARCHITECTURE (Feb 2026)
   ============================================================================
   
DESIGN PHILOSOPHY: "Safety baked in from the very bottom"
   - Multiple degradation points for error handling
   - Never reaching undefined state and crashing uncontrollably
   - Mandatory error accountability via failsafe() function
   - Explicit handling required at each safety layer

SAFETY LAYERS (from safest to most permissive):
   
   Layer 1: ? operator - Safe unwrap with default value
      - Purpose: Handle potential errors gracefully  
      - Syntax: result ? default_value
      - Behavior: Returns value if success, default if error
      - Example: int32:value = divide(10, 0) ? 0;  // Returns 0 on error
      
   Layer 2: ?! operator - Emphatic unwrap (failsafe-triggering)
      - Purpose: "This better work, but call failsafe if it doesn't"
      - Syntax: result ?! default_value
      - Behavior: Same as ? but triggers failsafe() with error code on failure
      - Example: int32:value = critical_read() ?! 0;  // Calls failsafe(-1) on error
      - Note: Created from agent miscommunication, but kept both solutions!
      
   Layer 3: unknown type - Propagates through operations
      - Purpose: Track uncertain/error states through computation chains
      - Behavior: Operations on unknown return unknown (sticky propagation)
      - Detection: Use ok(value) to check if value is unknown
      - Example:
          int32:a = divide(10, 0);  // a is unknown (divide-by-zero)
          int32:b = a + 5;  // b is also unknown (propagates)
          if (ok(b) == 0) { /* handle unknown state */ }
      
   Layer 4: ok() function - Explicit unknown value handling
      - Purpose: Check if value has entered unknown/error state
      - Syntax: ok(value)
      - Returns: 1 if value is valid, 0 if value is unknown
      - Example:
          int32:result = calculate();
          if (ok(result) == 0) {
              stderr.write("Calculation entered unknown state");
              result = ok(result);  // Strip unknown taint, acknowledge it
          }
      - Note: ok() can also strip "unknown taint" when used as wrapper
      
   Layer 5: failsafe(int32:err_code) - Mandatory error accountability
      - Purpose: REQUIRED function for all programs, catches unhandled errors
      - Signature: func:failsafe = void(int32:err_code) { /* handle error */ }
      - Behavior: Called automatically when:
         * ?! operator encounters error
         * Unwrap operation fails in certain contexts
         * Unhandled errors propagate to top level
      - Example:
          func:failsafe = void(int32:err_code) {
              stderr.write("CRITICAL ERROR: ");
              stderr.write(err_code);
              // Log to file, cleanup, emergency shutdown, etc.
          };
      - CRITICAL: Every program MUST define failsafe() or compilation fails
      - Philosophy: "Failsafe tax" - forces explicit error handling strategy

OPERATOR DETAILS:

? (Unwrap Operator):
   - Syntax: optional_value ? default
            result_value ? default
   - Returns: value if present/success, default otherwise
   - Use case: Safe fallback when error is acceptable
   - Example: string:name = getUserName() ? "Guest";

?! (Emphatic Unwrap Operator):
   - Syntax: result_value?!
   - Returns: value on success; calls failsafe(err_code) on failure
   - Side effect: Calls failsafe(err_code) on failure
   - Use case: Expected success — failure is unrecoverable, must call failsafe
   - Example: Config:cfg = loadConfig()?!;  // failsafe fires if config load fails
   - Note: Origin story - agent misunderstood "unwrap with failsafe" as needing
           either failsafe() function OR ?! operator. User said "it turned out
           great. We did both." - happy accident of redundant safety!

<=> (Spaceship Operator):
   - Syntax: a <=> b
   - Returns: -1 if a < b, 0 if a == b, 1 if a > b
   - Use case: Three-way comparison for sorting
   - Example: array.sort((a,b) => a.priority <=> b.priority);
   - Status: FULLY IMPLEMENTED (Feb 2026)

<- (Left-Arrow Dereference):
   - Syntax: value <- pointer
   - Returns: Value pointed to by pointer
   - Use case: "Pull value FROM pointer" (directional clarity)
   - Example: 
       int64:value = 42;
       int64->:ptr = @value;
       int64:retrieved = <-ptr;  // retrieved = 42
   - Status: FULLY IMPLEMENTED (Feb 2026)
   - Philosophy: Blueprint-style - arrow shows data flow direction

|> (Pipeline Forward):
   - Syntax: value |> func1 |> func2
   - Behavior: value |> f(args) becomes f(value, args)
   - Use case: Functional composition, data transformation chains
   - Example: result = data |> filter(isValid) |> transform(normalize);
   - Status: PARTIALLY IMPLEMENTED - type-checking works, needs Result auto-unwrap in codegen
   - Known Issue: Functions return Result<T>, pipeline needs auto-unwrapping between calls

<| (Pipeline Backward):
   - Syntax: func <| value
   - Behavior: f <| value becomes f(value)
   - Use case: Right-to-left data flow (reverse of |>)
   - Example: result <| transform <| filter <| data;
   - Status: PARTIALLY IMPLEMENTED - same as pipeline forward

?. (Safe Navigation):
   - Syntax: object?.member
   - Returns: member if object is not NIL, NIL otherwise
   - Use case: Navigate through potentially NIL optional chains
   - Example: string:name = user?.profile?.name ?? "Unknown";
   - Status: NOT YET IMPLEMENTED - lexer recognizes token, codegen pending
   - Note: Critical for pipeline operators - prevents NIL dereferencing

RESULT vs OPTIONAL vs NIL vs NULL:

Result<T>:   {value: T, err: void*, is_error: bool} - Function return type with error code
   - Purpose: Represent success or failure from operations
   - Fields: is_error (true on failure), err (error info on failure, NULL on success)
             value accessed via raw() on success
   - Example: result<int32>:r = divide(10, 2);
              if (!r.is_error) { use raw(r); }

Optional<T>:  T? - May or may not have a value
   - Purpose: Represent presence/absence of value (not an error)
   - Syntax: int64?, string?, obj?
   - Check: value == NIL or value != NIL
   - Example: int64?:age = user.age;
              if (age != NIL) { use age; }

NIL:  Unit type - "Absence of value" for optionals
   - Purpose: Explicit "no value" state (not an error)
   - Type: Compatible with any optional type (T?)
   - Example: int64?:none = NIL;
              string?:empty = NIL;
   - CRITICAL: Can ONLY be used with optional types (T?)
   - COMPILE ERROR: int64:x = NIL;  // ERROR! NIL requires optional type
   - CORRECT: int64?:x = NIL;  // ✓ NIL assigned to optional

NULL:  Null pointer - Address 0x0 for pointers
   - Purpose: Invalid/no pointer sentinel
   - Type: Compatible with any pointer type (T-> or wild T*)
   - Example: int64->:ptr = NULL;
              if (ptr == NULL) { /* handle null */ }
   - CRITICAL: Can ONLY be used with pointer types (T-> or wild T*)
   - COMPILE ERROR: int64?:x = NULL;  // ERROR! NULL requires pointer type
   - CORRECT: int64->:ptr = NULL;  // ✓ NULL assigned to pointer

NIL vs NULL Type Safety (ENFORCED Feb 2026):
   - NIL → Optional types only (int64?, string?, obj?)
   - NULL → Pointer types only (int64->, wild int8*, void*)
   - Compile-time validation prevents misuse
   - Examples:
       int64?:good1 = NIL;   // ✓ OK - NIL to optional
       int64->:good2 = NULL; // ✓ OK - NULL to pointer
       int64?:bad1 = NULL;   // ✗ ERROR - NULL to optional
       int64->:bad2 = NIL;   // ✗ ERROR - NIL to pointer

ERROR HANDLING PHILOSOPHIES:

1. Fail Fast (use ?! or explicit checks):
   - For critical operations where failure is exceptional
   - Example: readConfig() ?! defaultConfig();  // This MUST work

2. Graceful Degradation (use ? with defaults):
   - For operations where failure is expected/acceptable
   - Example: getCache() ? computeFresh();  // Cache miss is normal

3. Explicit Handling (use Result pattern):
   - For operations where error details matter
   - Example:
       result<obj>:r = parseJSON(data);
       if (r.is_error) {
           stderr.write(`Parse error: &{r.err}`);
           return r.err;
       }

4. Error Propagation (use unknown + ok()):
   - For computational chains where errors should propagate
   - Example:
       int32:result = step1() + step2() + step3();
       if (ok(result) == 0) { /* handle propagated error */ }

MANDATORY FAILSAFE PATTERN:

Every Aria program MUST define a failsafe function:

func:failsafe = void(int32:err_code) {
    // Handle unrecoverable errors
    // Examples: log to file, cleanup resources, emergency shutdown
    stderr.write("CRITICAL ERROR: ");
    stderr.write(err_code);
    
    // Application-specific error handling
    if (err_code == -1) {
        stderr.write("Configuration load failed");
    } else if (err_code == -2) {
        stderr.write("Resource initialization failed");
    }
    
    // Cleanup, logging, etc.
};

WHY MANDATORY?
- Forces developers to think about error handling strategy upfront
- Prevents "TODO: handle this error" that never gets done
- Provides single point for error logging/recovery
- "Failsafe tax" - small upfront cost for safety-critical reliability

!! SPECIAL BEHAVIOR: main FUNCTION - RESULT AUTO-UNWRAP !!
The main function is a special case for C ABI compatibility.

DECLARATION: func:main = int32(type:arg...) { ... };
- Returns raw int32 (NOT result<int32>) for OS compatibility
- Developer still uses pass(value) and fail(err) statements
- Compiler automatically unwraps Result internally

IMPLEMENTATION:
- pass(42i32) → Compiler extracts int32 value → returns 42 to OS
- fail(err) → Compiler converts TBB error to appropriate exit code
- NO manual unwrapping needed - compiler handles it transparently

WHY: All Aria functions return result<T> for safety-critical error handling,
but main must return raw int32 for OS process exit codes. The compiler bridges
this gap so developers maintain consistent pass/fail semantics everywhere.

EXAMPLE:
func:main = int32(type:arg...) {
    // Use pass/fail like any Aria function
    if (init_failed) {
        fail(ERR_INIT);  // Compiler converts to exit code
    }
    pass(0i32);  // Compiler extracts 0 and returns to OS
};

LAYERED SAFETY SYSTEM (Lines 7-317)

Layer 1: Result<T> - Hard errors (MUST handle)
